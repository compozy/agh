# ADR-003: gocron v2 as In-Process Scheduling Runtime

## Status

Accepted

## Date

2026-04-10

## Context

AGH needs an in-process cron scheduling library for time-based job execution. The library must support: POSIX cron expressions, context.Context propagation to jobs, graceful shutdown, singleton mode per job, lifecycle hooks, UUID-based job identity, testability with fake clocks, and active maintenance.

## Decision

Use **go-co-op/gocron v2** as the in-process scheduling runtime. gocron wraps robfig/cron's battle-tested expression parser while adding context propagation, lifecycle hooks, singleton mode, UUID identity, and a monitor interface that robfig/cron lacks.

## Alternatives Considered

### Alternative 1: robfig/cron v3

- **Description**: The most popular Go cron library (14.1k stars)
- **Pros**: Battle-tested, ubiquitous, zero dependencies, clean `Schedule` interface
- **Cons**: Effectively unmaintained (last release 2019, 111 open issues). No `context.Context` propagation to jobs (`Job.Run()` is `func()`). No lifecycle hooks, no UUID identity, no monitoring interface. Critical bugs unfixed (double-fire, panic on malformed TZ).
- **Why rejected**: Cannot propagate context for graceful cancellation on daemon shutdown — a hard requirement for AGH's subprocess-managed agent sessions.

### Alternative 2: adhocore/gronx

- **Description**: Zero-dependency cron parser with basic tasker (501 stars)
- **Pros**: Strong parser (supports L, W, # modifiers), zero dependencies
- **Cons**: No context propagation, no UUID identity, no lifecycle hooks, no monitoring. Parser-first, scheduler-second.
- **Why rejected**: Insufficient scheduler capabilities. Only the parser is competitive.

### Alternative 3: Custom scheduler on time.AfterFunc

- **Description**: Build scheduling logic from scratch using Go standard library
- **Pros**: Zero dependencies, full control
- **Cons**: Significant development effort for: cron parsing, next-tick calculation, timezone handling, singleton mode, concurrent execution limits, graceful shutdown coordination
- **Why rejected**: gocron already solves all of these correctly. Reimplementing is not justified.

## Consequences

### Positive

- Context propagation enables graceful cancellation on daemon shutdown
- UUID-based job identity maps naturally to SQLite-stored definitions
- Lifecycle hooks (`BeforeJobRuns`, `AfterJobRuns`, `AfterJobRunsWithError`) decouple scheduler from storage
- `SchedulerMonitor` interface plugs into AGH's observability layer
- `clockwork.FakeClock` via `WithClock()` enables deterministic testing
- `WithSingletonMode` prevents double-spawning expensive agent sessions
- Actively maintained (monthly releases, 7k stars, MIT license)

### Negative

- Adds dependencies: `robfig/cron/v3` (parsing), `google/uuid` (already in AGH), `jonboulle/clockwork`
- In-memory scheduler state — job definitions must be loaded from SQLite on daemon boot

### Risks

- gocron maintenance could decline. Mitigation: the library is a thin wrapper; if abandoned, fork or replace with minimal effort since AGH wraps it behind its own interface.

## Implementation Notes

- Dependency: `go get github.com/go-co-op/gocron/v2`
- Wrapper in `internal/automation/schedule.go` — isolates gocron behind AGH interfaces
- Scheduler created with: `WithLocation`, `WithLimitConcurrentJobs`, `WithLogger`, `WithStopTimeout`
- Jobs registered with: `CronJob`, `DurationJob`, `OneTimeJob`, `WithSingletonMode`, `WithEventListeners`
- Graceful shutdown: `scheduler.Shutdown()` cancels all job contexts and waits
