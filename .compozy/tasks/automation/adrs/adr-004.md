# ADR-004: Configurable Per-Job Retry with Fire Limits

## Status

Accepted

## Date

2026-04-10

## Context

When a scheduled or triggered job fails (agent session crashes, times out, or errors), the system must decide whether to retry. Agent sessions are expensive (LLM API calls, token costs, time). Blindly retrying can amplify costs and create thundering herd effects. Different jobs have different reliability requirements.

## Decision

Implement **configurable per-job retry strategy** with three options: `none` (default), `backoff` (exponential with configurable max retries and base delay), and global **fire limits** (max fires per rolling window) to prevent runaway execution regardless of retry strategy.

Default: `none` — failure is recorded, next scheduled run proceeds normally. This is the safest default for LLM-powered agent sessions.

## Alternatives Considered

### Alternative 1: No Retry, Record + Reschedule (OpenFang Pattern)

- **Description**: Never retry. Record failure, let next scheduled run proceed. Users manually trigger re-runs.
- **Pros**: Simplest, most predictable, no cost amplification
- **Cons**: Too rigid for jobs with transient failures (network blips, rate limits)
- **Why rejected**: Good as a default but insufficient as the only option. Some jobs benefit from retry.

### Alternative 2: Global Exponential Backoff (GoClaw Pattern)

- **Description**: All jobs retry with exponential backoff (3 attempts, 1s/2s/4s)
- **Pros**: Handles transient failures automatically
- **Cons**: One-size-fits-all; critical jobs and best-effort jobs get same treatment; 3 failed sessions per failure
- **Why rejected**: Agent sessions are too expensive for blanket retry. Per-job control is essential.

## Consequences

### Positive

- Users choose appropriate reliability level per job
- Default `none` prevents unexpected cost amplification
- Fire limits provide safety net regardless of retry strategy
- Extensible — new strategies (e.g., `fixed_delay`) can be added later

### Negative

- More configuration surface than simpler alternatives
- Retry attempt state still exists in memory during active execution, but fire-limit evaluation must survive daemon restart

### Risks

- Users may over-configure retries, leading to high costs. Mitigation: fire limits cap total executions per window; documentation warns about LLM cost implications.

## Implementation Notes

```toml
[[automation.jobs]]
name = "critical-report"
retry = { strategy = "backoff", max_retries = 3, base_delay = "2s" }
fire_limit = { max = 12, window = "1h" }

[[automation.jobs]]
name = "best-effort"
retry = { strategy = "none" }
```

- Default retry: `{ strategy = "none" }`
- Default fire_limit: `{ max = 12, window = "1h" }`
- Backoff formula: `base_delay * 2^attempt` (capped at max_retries)
- Fire limit tracking: evaluate against persisted recent runs in SQLite for the active rolling window; in-memory caching may optimize reads but is not the source of truth

## References

- OpenFang fire limits: rolling hourly window, configurable per schedule
- GoClaw retry: exponential backoff (3 retries, 1s/2s/4s base)
- Hermes: no retry, failure recorded, next run proceeds
