# ADR-001: Built-In Daemon Component with Extension Integration Points

## Status

Accepted

## Date

2026-04-10

## Context

AGH needs an automation system (scheduled and event-triggered agent execution). The extension architecture TechSpec (P1) already reserves a "future seam" for `scheduler.driver` as an extension capability. However, the extension system is only partially implemented (manifest parsing, registry, subprocess lifecycle exist; full Host API, CLI commands, and TypeScript SDK are pending).

The automation system must be agent-manageable (CLI + API) and human-configurable (TOML), consistent with AGH's "highly extensible, highly agent-managed" design philosophy.

## Decision

Implement the automation system as a **built-in daemon component** in `internal/automation/`, wired into the daemon boot sequence as a `Server`. The package exposes interfaces that the extension system can consume once P1 is complete — extensions will be able to observe, manage, and extend the automation engine via Host API methods.

## Alternatives Considered

### Alternative 1: Extension-Only

- **Description**: Implement scheduler as a subprocess extension providing `scheduler.driver` capability
- **Pros**: Clean isolation, hot-reloadable, language-agnostic
- **Cons**: Depends on finishing P1 extension architecture first; IPC overhead (~100-500us per Host API call); more complex deployment; cannot ship until extension system is complete
- **Why rejected**: Blocks delivery on unfinished infrastructure. Agent sessions are expensive enough that IPC latency is negligible, but the dependency chain is not.

### Alternative 2: Hybrid with Driver Interface

- **Description**: Built-in scheduler with a `Driver` interface, default gocron implementation, future extension-provided drivers
- **Pros**: Maximum flexibility
- **Cons**: Over-engineering for v1; no evidence that alternative scheduling backends are needed
- **Why rejected**: YAGNI. The extension integration points (Host API for CRUD, hook events for observation) provide sufficient extensibility without a driver abstraction.

## Consequences

### Positive

- Ships without waiting for extension system completion
- Zero IPC overhead for scheduling operations
- Direct access to session manager, observer, and store without Host API serialization
- Extension integration points are additive — exposed when P1 completes

### Negative

- Scheduling logic compiled into daemon binary (not hot-swappable)
- Extensions observe/manage but cannot replace the core engine

### Risks

- Extension integration points may need revision when P1 finalizes. Mitigation: define narrow interfaces now, widen later.

## Implementation Notes

- Package: `internal/automation/`
- Boot phase: after hooks, before servers
- Implements `daemon.Server` interface (`Start`, `Shutdown`)
- Functional options constructor: `New(opts ...Option)`
- Extension integration via: new Host API methods (`automation/jobs`, `automation/triggers`), automation lifecycle hook events (`automation.job.pre_fire`, `automation.job.post_fire`, `automation.trigger.pre_fire`, `automation.trigger.post_fire`, `automation.run.completed`, `automation.run.failed`)
