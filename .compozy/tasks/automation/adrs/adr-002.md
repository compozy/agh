# ADR-002: Unified Automation Model — Schedules and Triggers

## Status

Accepted

## Date

2026-04-10

## Context

Competing agent OS projects handle time-based and event-based execution differently. OpenFang explicitly separates CronScheduler (time) from TriggerEngine (events). GoClaw merges everything into a CronService with three modes. Hermes uses a single polling loop.

AGH needs both time-based scheduling (run agent every day at 9am) and event-driven triggers (run agent when a session completes, when a webhook fires, when memory consolidates). Both result in the same action: dispatching a session with an agent and prompt.

## Decision

Implement a **unified automation system** in `internal/automation/` that handles both schedules and triggers through a shared dispatch mechanism. Schedules are time-based activations (cron, interval, one-shot). Triggers are event-based activations (session events, webhook, memory events, hook events). Both use the same `Job` definition and `Dispatcher` for session creation.

## Alternatives Considered

### Alternative 1: Schedules Only (Triggers Later)

- **Description**: Ship cron/interval/one-shot first, add trigger engine as separate TechSpec
- **Pros**: Faster delivery, smaller scope, lower risk
- **Cons**: Two separate systems that do the same thing (dispatch sessions); trigger architecture may not align with scheduler if designed independently
- **Why rejected**: The shared dispatch model is architecturally cleaner. Both systems need the same storage, API surface, CLI commands, and UI. Building them together avoids duplication.

### Alternative 2: Everything Is a Trigger

- **Description**: Cron schedules are just `TriggerSource` implementations (time-based trigger)
- **Pros**: Maximum conceptual uniformity
- **Cons**: Obscures the fundamental difference between predictable time-based scheduling and reactive event handling; harder to reason about "when does this run next?"
- **Why rejected**: OpenFang's experience shows that operators think about "what fires when?" (schedules) and "what reacts to what?" (triggers) differently. Separate concepts with shared dispatch respects this mental model.

## Consequences

### Positive

- Single package owns all automation logic
- Shared dispatch mechanism prevents duplication
- Consistent API surface (`/api/automation/jobs`, `/api/automation/triggers`)
- Cascading automations natural (schedule fires → session completes → trigger fires)

### Negative

- Larger initial scope than schedules-only
- Trigger event subscription adds coupling to session, memory, and hooks packages

### Risks

- Event subscription fan-out may impact performance if many triggers are defined. Mitigation: fire limits per trigger (rolling window), efficient event matching.

## Implementation Notes

- `internal/automation/schedule.go` — gocron v2 wrapper for time-based jobs
- `internal/automation/trigger.go` — event subscriber for reactive triggers
- `internal/automation/dispatch.go` — shared session creation dispatch
- `internal/automation/job.go` — unified job definition (source: schedule or trigger)
- Trigger sources: `session.created`, `session.stopped`, `memory.consolidated`, `hook.*.completed`, `webhook`
