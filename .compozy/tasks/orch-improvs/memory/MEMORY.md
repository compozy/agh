## Shared Decisions

- `state.yaml` and future `progress.checklist[]` entries are the durable execution source for this loop.
- Prior `.codex/ledger/2026-05-04-MEMORY-orch-improvs-techspec.md` remains historical spec-authoring context only; implementation progress belongs in this workflow memory and state file.
- Task orchestration config now lives under root `Config.Task` with `[task.orchestration]`, `[task.orchestration.profile]`, and `[task.orchestration.review]` TOML overlays.
- New task orchestration config keys are part of the existing agent-mutable config surface so agents can manage them through current config tooling.
- Queryable orchestration/profile state now has a typed global DB schema foundation in migration v17 instead of living in `metadata_json`.
- Task orchestration profile schema statements are now shared between fresh DB creation and migration v17 through `taskOrchestrationProfileSchemaStatements`; fresh schema includes `idx_tasks_current_run` exactly once.
- Review-gate persistence now has a numbered global DB migration v18 and shared review-gate DDL helpers consumed by fresh DB boot and the migration.
- `tasks.current_run_id` is a store-owned projection on `task.Task` and `task.Summary`; it is not agent-writable directly.
- Empty global DB bootstrap still records every numbered migration and preserves the upgrade path, but fresh databases apply the registry in a single transaction to keep test and daemon boot cost bounded.
- Extension lifecycle shutdown/recovery must snapshot mutable `managedExtension` fields under `Manager.mu`; goroutines must not read `process`, runtime, or info fields from the shared struct without synchronization.
- `internal/notifications` is a narrow delivery-progress boundary, not an event bus, task queue, hook dispatcher, review bus, or task authority surface.
- Notification cursor identity is `(consumer_id, stream_name, subject_id)`, with empty `subject_id` representing an unscoped stream.
- Notification cursor advancement is monotonic; same-sequence replay is accepted only when the delivery id matches, and reset is the only lowering path.
- Bridge task subscriptions own target state only. Their cursor identity is fixed to `consumer_id = bridge_task_subscription:<subscription_id>`, `stream_name = task_events`, `subject_id = <task_id>`.
- Bridge terminal notification cursor advancement is ack-gated; failed delivery records bounded cursor diagnostics without moving `last_sequence`.
- Bridge notification subscription diagnostics are an additive cursor read model on the existing `cursor` object. Missing cursor rows represent no delivery progress yet and render as zero-sequence identity; persisted rows add sequence, delivery id, delivered timestamp, error, and updated timestamp.
- Bridge task subscription delete removes only the active target row. Stale cursor diagnostics remain inspectable by cursor key and same-subscription recreation resumes from the preserved cursor, while active subscription reads/deletes return deterministic not-found errors after cleanup.
- Terminal notifier state is now explicit: accepted finals deliver and advance; non-terminal task state defers without cursor mutation or mismatch error; accepted-final state disagreement fails closed with `ErrTerminalTaskNotificationMismatch`, records bounded `last_error`, and does not advance.
- Task execution profile state must stay typed in `task_execution_profiles` plus selector side tables; it must not be reconstructed from `metadata_json`.
- Task execution profiles are exposed in Go as `task.ExecutionProfile` to avoid Go package stutter while still mapping to the persisted `task_execution_profiles` schema and selector tables.
- Task-service profile mutation is rejected while `tasks.current_run_id` is non-empty; profile edits are control-plane changes and must not rewrite an active run under execution.
- Review request authority starts in `task.Service`: request creation validates task/run ownership, terminal run status, review policy, review round/attempt, actor identity, and then delegates idempotent persistence to GlobalDB.
- GlobalDB review request insertion is idempotent on `(run_id, review_round, attempt)` and links `task_runs.review_request_id` in the same transaction as the request row creation.
- Reviewer session binding is persisted through `task.Service.BindRunReviewSession`; it validates reviewer agent/peer/channel metadata before moving the request to `in_review`.
- Review verdicts are persisted only through `task.Service.RecordRunReview`; rejected verdicts enqueue at most one continuation run linked by `task_runs.review_id` and replayed by delivery id.
- Task-run review lineage is grouped under `task.Run.Review *RunReviewLineage` instead of direct fields on `Run`, preserving typed persistence while keeping hot task-run values small enough for lint and call-site copying.
- Worker and participant profile selectors are store-level claim eligibility filters. They narrow `ClaimNextRun` and do not create new authority outside task/run status, scope, lease, and capability rules.
- Task session starts now carry the effective `task.ExecutionProfile` through `task.StartTaskSession`, and only the daemon session bridge maps worker runtime fields into `session.CreateOpts`.
- Runtime provider/model overrides must still flow through `config.ResolveSessionAgentWithRuntime`; profile model override is not allowed to bypass provider config validation.
- Task sandbox profile overrides are session-start options only: `ref` still resolves through workspace config sandbox validation, `none` explicitly skips sandbox startup, and `inherit` leaves the existing workspace/default sandbox path untouched.
- Bundled orchestration skills are instructional-only assets. They may guide coordinator, worker, and reviewer sessions, but task state, claims, review requests, verdicts, and continuation runs remain owned by task-service/native-tool authority.
- The native `submit_run_review` tool is model-facing as `submit_run_review` but registered internally as `agh__task_run_review_submit`; it is only available to reviewer sessions with an active persisted review binding and the bundled reviewer skill metadata requiring review requests.
- Native review submission does not inspect claim leases or expose raw claim tokens. It validates the caller session binding, checks request ids against that binding, and uses `task.Service.RecordRunReview` as the sole verdict authority.
- Native task execution profile tools are model-facing as `task_execution_profile_get`, `task_execution_profile_set`, and `task_execution_profile_delete`, but registered internally as `agh__task_execution_profile_get`, `agh__task_execution_profile_set`, and `agh__task_execution_profile_delete`.
- Native execution profile tools are task-toolset tools and delegate all profile reads/mutations to `task.Service`; they do not bypass active-run mutation checks, normalization, audit events, or default profile behavior by writing GlobalDB directly.
- Native review request/read tools are model-facing as `task_run_review_request`, `task_run_review_list`, and `task_run_review_show`, but registered internally as `agh__task_run_review_request`, `agh__task_run_review_list`, and `agh__task_run_review_show`.
- Native review request/read tools are task-toolset tools and delegate to `task.Service.RequestRunReview`, `task.Service.ListRunReviews`, and `task.Service.GetRunReview`; they do not bypass review policy validation or GlobalDB through daemon code.
- Task execution profile HTTP, UDS, and CLI profile surfaces are transport wrappers over `task.Service` profile authority. They do not write GlobalDB directly and inherit active-run mutation rejection, normalization, audit events, and default-profile behavior from task service.
- `agh task profile inspect|update|delete` uses UDS parity with the new HTTP/UDS profile endpoints and supports JSON output for agent manageability. Generated CLI docs are source-derived; do not hand-edit `packages/site/content/runtime/cli-reference/task/profile/`.
- Task-run review HTTP, UDS, and CLI review surfaces are transport wrappers over `task.Service` review authority. They do not write GlobalDB directly and inherit review policy validation, reviewer binding enforcement, idempotent delivery handling, and continuation-run creation from task service.
- `agh task review request|list|show|submit` uses UDS parity with the new HTTP/UDS review endpoints and supports JSON output for agent manageability. Generated CLI docs are source-derived; do not hand-edit `packages/site/content/runtime/cli-reference/task/review/`.
- Review request routing is a daemon composition-root concern. The task service emits a best-effort review-request observer notification after persisted request creation; the daemon ReviewRouter owns runtime session selection/creation because it can see task service authority, task store data, session state, workspace resolution, and agent capability catalogs.
- ReviewRouter no-route diagnostics are persisted as task-service review verdicts with deterministic delivery ids. This keeps downstream API/web/docs surfaces truthful: no-route is a review outcome/diagnostic, not a channel state transition or direct store mutation.
- Review-store UNIQUE constraint handling must use typed `modernc.org/sqlite` error inspection. String matching on constraint names is not acceptable for review persistence or future SQLite constraint classification.
- Task context bundle is exposed as `task.ContextBundle` in Go to avoid package stutter, while the wire field remains `/agent/context.task.bundle`. Downstream TypeScript/web consumers should use generated OpenAPI types instead of defining a duplicate `TaskContextBundle` DTO.
- Reviewer sessions can receive task context without receiving a worker lease. A review-bound context comes from `LookupRunReviewBySession` and contains the reviewed run bundle; it must not imply the reviewer can mutate the worker's token-fenced lease.
- Session-start task context is a shared rendered overlay assembled by `internal/situation`; worker, reviewer, and coordinator runtime paths prepend it to role-specific instructions rather than each creating a bespoke task-context prompt.
- `latest_event_seq` is a read projection over durable `task_events.event_seq`, not a mutable task column or separate migration. Consumers should seed reconnectable task streams from this projection rather than inventing local cursor state.
- Task SSE seed precedence is explicit: a present, trimmed `Last-Event-ID` header overrides `after_sequence`, including a zero value; malformed `Last-Event-ID` values fail parsing instead of silently falling back to query parameters.
- Web data-layer types for task orchestration must be derived from `web/src/generated/agh-openapi.d.ts` through `web/src/lib/api-contract.ts`; do not add hand-written DTO clones. Public exports go through `@/systems/tasks` and follow the existing `adapters -> lib/query-* -> hooks -> components` flow.
- The web orchestration surface is read-truthful: the operator inspects execution profile, reviews, notification cursor diagnostics, and SSE resume seed but never submits review verdicts from web. Verdict authority remains in reviewer-bound `submit_run_review`; the reviews card surfaces a permanent disclaimer reinforcing this. Profile mutation rejects while `tasks.current_run_id` is non-empty; the UI mirrors that by disabling edit/delete during active runs.
