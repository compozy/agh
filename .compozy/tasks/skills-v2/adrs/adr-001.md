# ADR-001: MCP Consent Model — Trust Tiers by Skill Source

- **Status:** Accepted
- **Date:** 2026-04-07
- **Context:** Skills can declare MCP server dependencies in `metadata.agh.mcp_servers`. MCP servers are subprocesses with full system access, so consent is a security concern.

## Decision

Auto-approve MCP servers from bundled, user, additional-root, and workspace skills. Only marketplace skills require explicit user consent before MCP servers are spawned. Rationale: user/additional/workspace skills are placed by the user themselves, so implicit trust is reasonable. Marketplace skills come from external authors and require a consent gate.

Trust tiers:
- **Bundled**: auto-approved (immutable, shipped with binary)
- **User/Additional/Workspace**: auto-approved (user placed them deliberately)
- **Marketplace**: require one-time consent, persisted in config

## Alternatives Considered

1. **Config TOML allowlist for all non-bundled skills** — adds friction for user-created skills with no security benefit since the user authored them.
2. **Separate consent.json file** — new persistence path with no advantage over config.toml for marketplace consent.

## Consequences

- Marketplace consent stored as `allowed_marketplace_mcp` list in `SkillsConfig`
- User/additional/workspace skills with MCP servers work without any additional configuration
- If a marketplace skill is compromised after consent, the user must manually revoke
