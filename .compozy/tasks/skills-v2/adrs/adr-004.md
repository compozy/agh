# ADR-004: Hash-Based Provenance for Marketplace Skills

- **Status:** Accepted
- **Date:** 2026-04-07
- **Context:** Marketplace skills come from external authors. After ClawHavoc (1,184+ malicious skills on ClawHub), provenance verification is critical. Full cryptographic signing requires PKI infrastructure that doesn't exist yet.

## Decision

Implement SHA-256 hash verification combined with `VerifyContent` security scanning. On install: download skill, compute hash, scan content, store hash in skill metadata file. On every load: recompute hash and compare — if changed, re-scan and warn. This closes the time-of-check/time-of-use gap.

No cryptographic signing in this iteration — deferred until PKI infrastructure is available.

## Alternatives Considered

1. **Full cryptographic signing (GPG/age)** — maximum security but requires key distribution infrastructure, author onboarding workflows, and revocation lists. Premature for alpha.
2. **VerifyContent scan only** — simplest but no tamper detection between loads. A modified skill could pass initial scan but be altered after install.

## Consequences

- Each marketplace skill gets a `.agh-meta.json` sidecar with SHA-256 hash, install timestamp, registry source
- Load-time hash comparison detects tampering
- Hash mismatch triggers re-scan + warning log
- Critical VerifyContent findings require an explicit retained quarantine state if the skill must remain visible for re-approval; otherwise the implementation should keep block-on-load semantics until that UX exists. `Enabled = false` alone is not sufficient because current catalog consumers do not filter disabled skills.
- Path to cryptographic signing remains open — hash infrastructure is a subset of what signing requires
