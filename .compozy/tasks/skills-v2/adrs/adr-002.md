# ADR-002: Hybrid Hook Execution Model

- **Status:** Accepted
- **Date:** 2026-04-07
- **Context:** Skills declare lifecycle hooks for session events. The daemon needs a hook execution model that balances performance, extensibility, and the existing Notifier pattern.

## Decision

Use a hybrid approach: built-in lifecycle behaviors (for example memory consolidation and other daemon-native reactions to session events) run as Go callbacks integrated with the existing `session.Notifier` interface. Custom hooks declared in SKILL.md frontmatter run as subprocess commands with JSON stdin/stdout, enforced timeout, and fail-open semantics.

Hook events in this iteration: `on_session_created` and `on_session_stopped`. The `on_prompt_assembly` hook is deferred to a follow-up due to per-message latency concerns.

Execution order: Notifier callbacks first (fast, in-process), then subprocess hooks in hierarchy precedence order (bundled > marketplace > user > additional > workspace), alphabetical within same level.

## Alternatives Considered

1. **Subprocess-only** — simpler mental model but forces built-in behaviors into external processes, adding unnecessary IPC overhead for daemon-native operations.
2. **Go callbacks only** — faster but limits hooks to what the daemon implements natively, breaking the RFC's language-agnostic extensibility promise.

## Consequences

- `session.Notifier` interface unchanged — new built-in hooks register as additional Notifier implementations
- `internal/daemon/notifier.go` needs an explicit post-notifier hook phase; registering subprocess hooks as an ordinary notifier would not preserve the required ordering
- New `HookRunner` component in `internal/skills/` dispatches subprocess hooks
- Subprocess hooks receive `{"session_id", "agent_name", "workspace"}` via stdin, and any structured stdout is captured for logging/future enrichment rather than injected into prompt context in this iteration
- Hook failures are logged as warnings, never block session lifecycle
- `on_prompt_assembly` deferred — reduces risk and avoids per-message latency impact
