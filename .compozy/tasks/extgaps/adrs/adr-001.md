# ADR-001: Use a daemon-owned bundle activation runtime

## Status

Accepted

## Date

2026-04-14

## Context

The extension manifest already supported static resources such as skills, agents, hooks, and MCP servers. The extension gaps work needed to add a way for one installed extension to ship higher-level team or product packages that combine network channels, automation definitions, and bridge presets. Those resources require lifecycle control, persistence, ownership tracking, and reconciliation across daemon boot, extension reload, activation, update, and deactivation.

The existing Host API could create runtime resources dynamically, but that model would make the extension process responsible for materialization timing and ownership, and it would not give operators a stable catalog of packageable resources before activation.

## Decision

Implement bundle activation as a daemon-owned runtime in `internal/bundles`, backed by persisted activation and inventory records in the global SQLite database. Extensions declare bundle specs as static resources; operators activate those specs through daemon APIs; the daemon reconciles package-managed automations, bridge instances, and effective network defaults.

## Alternatives Considered

### Alternative 1: Extension self-materialization through Host API

- **Description**: Let extension processes create automations, bridge instances, and network settings at runtime through existing Host API methods.
- **Pros**: Reuses existing extension action plumbing and avoids a new daemon subsystem.
- **Cons**: Pushes ownership and reconciliation into extension code, weakens operator visibility before activation, and makes boot/reload behavior depend on extension process logic.
- **Why rejected**: The shipped implementation needed deterministic daemon-side activation, persisted inventory, and lifecycle enforcement independent of extension subprocess behavior.

### Alternative 2: Manifest-only expansion without persisted activations

- **Description**: Treat bundles as static declarative data only and expand them in memory on startup.
- **Pros**: Smaller implementation surface and no new persistence tables.
- **Cons**: No explicit activation lifecycle, no per-activation ownership, and no durable bind/default-channel state.
- **Why rejected**: The implemented behavior requires explicit activate/update/deactivate flows and inventory tracking across restarts.

## Consequences

### Positive

- Operators get a stable catalog and explicit activation lifecycle.
- The daemon owns reconciliation, effective network defaults, and package resource inventory.
- Extension disable/uninstall can be guarded against active package usage.

### Negative

- Introduces a new runtime package and new SQLite tables.
- Bundle behavior now depends on daemon boot ordering and reconcile semantics.

### Risks

- Reconcile failures can block activation or extension lifecycle flows.
- Bundle/runtime ownership rules must stay aligned with automation and bridge APIs.

## Implementation Notes

- `internal/daemon/boot.go` boots bundles after extensions and automations, then calls `service.Reconcile(ctx)`.
- `internal/bundles/service.go` owns catalog, preview, activate, update, deactivate, and network settings computation.
- `internal/store/globaldb/global_db_bundles.go` persists activations and activation inventory.
