# ADR-003: Resolve bundle default channels as runtime state instead of mutating daemon config

## Status

Accepted

## Date

2026-04-14

## Context

One of the extension gaps was the inability for a team package to declare channels and optionally align one of them with the daemon's default network channel. The daemon already had a configured `default_channel` in static config. Persisting bundle activation should not rewrite configuration files on disk, but sessions and status payloads still need to see the effective default when a bundle claims it.

## Decision

Keep the configured network default in static daemon config and compute an effective default channel in the bundle runtime. Bundle activations may opt in with `bind_primary_channel_as_default`; the bundle service resolves a single effective default at runtime, exposes both configured and effective values via API payloads, and clears the override when the activation is removed or updated.

## Alternatives Considered

### Alternative 1: Rewrite the daemon config default channel on activation

- **Description**: Persist the bundle's primary channel back into the daemon configuration source.
- **Pros**: Makes the network layer consume one source of truth.
- **Cons**: Couples operational activation with config mutation, risks drift on deactivate, and is incompatible with the delivered operator-driven runtime model.
- **Why rejected**: The implementation explicitly avoided writing activation effects into config files.

### Alternative 2: Keep channels informational only

- **Description**: Expose declared channels in the catalog but never let them affect effective defaults.
- **Pros**: Simpler semantics and no conflict handling.
- **Cons**: Does not close the documented gap around packaging and operational alignment of channels.
- **Why rejected**: The shipped feature includes effective default resolution and conflict detection.

## Consequences

### Positive

- Separates static configuration from runtime activation state.
- Makes the active default visible through daemon and network status payloads.
- Keeps activation/deactivation reversible without config-file edits.

### Negative

- The daemon now has two related channel values: configured and effective.
- Activations that request the default can conflict and must be rejected.

### Risks

- Consumers must read the effective default instead of assuming the configured value is always active.
- Future CLI/operator surfaces must preserve the same conflict rules.

## Implementation Notes

- `internal/bundles/service.go` computes `NetworkSettings` with configured and effective defaults plus declared channels.
- `internal/api/contract/contract.go` extends `NetworkStatusPayload` with configured/effective fields.
- `internal/api/core/bundles.go` and `defaultSessionChannel(...)` consume the runtime-computed value.
