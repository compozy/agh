# BUG-007: Bundle activation bridge payload omits owning extension_name

**Severity:** Medium
**Priority:** P2
**Type:** Functional
**Status:** Open

## Environment

- **Build:** local branch state on 2026-04-15 after `make verify` and `make test-integration`
- **OS:** macOS
- **URL:** `POST /api/bundles/preview`, `POST /api/bundles/activations`, `GET /api/bundles/activations/:id`

## Summary

Live bundle activation responses return bridge payloads with `extension_name: ""` even when the bundle materializes a bridge instance owned by the activating extension. The persisted bridge record is correct under `/api/bridges`, but the bundle activation payload drops that ownership field.

## Reproduction

```bash
AGH_HOME=/tmp/codex-qa-extgaps/home go run ./cmd/agh extension install /tmp/codex-qa-extgaps/fixture/live-bundle-ext -o json
curl -sS -X POST http://127.0.0.1:22123/api/bundles/activations \
  -H 'Content-Type: application/json' \
  -d '{"extension_name":"live-bundle-ext","bundle_name":"team-bundle","profile_name":"default","scope":"global"}'
curl -sS http://127.0.0.1:22123/api/bridges
```

Observed before the fix:

- Bundle activation response bridge payload contains `"extension_name":""`
- `/api/bridges` returns the same materialized bridge with `"extension_name":"live-bundle-ext"`

## Expected

Bundle activation payloads should report the owning bridge extension consistently with the persisted bridge instance so API consumers can correlate bundle output with bridge APIs.

## Root cause

The bundle-to-contract bridge payload mapping drops the resolved extension ownership when building `contract.BundleBridgePayload`, so the response uses the zero-value field instead of the materialized bridge instance's extension name.

## Fix

Not fixed in this QA pass. The payload mapper should populate `extension_name` from the resolved/materialized bridge instance.

## Verification

- Reproduced through live daemon HTTP APIs with a real installed bundle extension
- Confirmed mismatch against `/api/bridges` on the same activation

## Impact

- **Users Affected:** API consumers using bundle activation payloads to discover or manage bridge instances
- **Frequency:** Always for live bridge payloads in the tested bundle flow
- **Workaround:** Query `/api/bridges` or activation inventory and ignore the inline bridge `extension_name`
