# BUG-006: defaultSessionChannel fallback logic may bypass bundle effective default

**Severity:** Medium
**Priority:** P2
**Component:** `internal/api/core/bundles.go:169-187`
**Related:** TC-FUNC-009

## Environment

- Code review of `internal/api/core/bundles.go`
- Function `defaultSessionChannel` at line 169

## Description

The `defaultSessionChannel` function has a three-tier fallback:
1. Explicit channel from request → use it
2. Bundle effective default → use it
3. Config network default channel → use it

However, when the bundle service's `NetworkSettings()` returns successfully but with an empty `EffectiveDefaultChannel`, the function falls through to tier 3 (config default). This is correct.

But there's a subtle issue: if `NetworkSettings()` returns an error, the function returns the error immediately without falling back to the config default. This means a transient bundle service error could break session creation even when the config has a valid default channel.

### Expected Behavior

If the bundle service is unavailable or errors, session creation should gracefully fall back to the configured default channel.

### Actual Behavior

```go
if h != nil && h.Bundles != nil {
    settings, err := h.Bundles.NetworkSettings(ctx)
    if err != nil {
        return "", err  // <-- Hard failure, no fallback
    }
    ...
}
```

## Impact

- Session creation fails if bundle service has a transient error
- Only affects sessions where no explicit channel is specified
- May cause user-facing errors during reconciliation

## Suggested Fix

Log the error and fall through to tier 3 instead of returning the error:
```go
settings, err := h.Bundles.NetworkSettings(ctx)
if err == nil && strings.TrimSpace(settings.EffectiveDefaultChannel) != "" {
    return strings.TrimSpace(settings.EffectiveDefaultChannel), nil
}
// Fall through to config default
```

## Workaround

Always specify an explicit channel when creating sessions.
