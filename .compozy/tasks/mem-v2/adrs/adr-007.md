# ADR-007: Daily-Log Retention Policy

## Status

Accepted

## Date

2026-05-04

## Context

mem-v2 introduces per-scope `daily/YYYY-MM-DD.md` logs (ADR-005, P10 in `analysis.md` §3). These daily files capture short-term episodic content from extractor runs and ad_hoc agent notes that are too granular for curated entries. The dreaming worker reads daily logs to synthesize cross-day patterns into curated knowledge.

Without explicit retention, these files grow unbounded. With three scopes × N workspaces × 365 days/year, disk usage and dreaming-worker read latency become real concerns within months.

Source-verified competitor patterns (`analysis_session-ledger-retention.md` §Q5):

- **OpenFang** — 1 MB hard cap with **silent drop** on overflow (`kernel.rs:451-475`). No rotation, no warning. **Bug, not feature.**
- **OpenClaw** — no file cap (`flush-plan.ts:120`); dreaming worker distills via `daily-ingestion.json` checkpoints. Daily files grow indefinitely.
- **Claude Code memdir** — daily logs uncapped (`memdir.ts:335`); only `MEMORY.md` index is capped (200 lines / 25 KB at `memdir.ts:35-38`).
- **paperclip / PARA** — Hot (7d) / Warm (8-30d) / Cold (30+d) tiering; **never hard-delete, only supersede.** Access-count decay tiers move records between Hot / Warm / Cold without losing data.
- **Codex** — no daily logs (rollouts only); consolidation pipeline reads N days back via `state/runtime/memories.rs:128` 30-day window.

The convergent pattern: **never hard-delete by default; cold-archive after a configurable threshold; impose size caps with rotation to prevent silent data loss.**

## Decision

Adopt the following retention policy for `<scope>/memory/daily/YYYY-MM-DD.md` files:

- **Per-file caps**:
  - Size: **1 MiB** (`max_bytes = 1048576`)
  - Lines: **5,000** (`max_lines = 5000`)
- **Rotation on overflow**: when either cap is reached during a write, rotate to `daily/YYYY-MM-DD.<seq>.md` (`<seq>` starts at 0; new file replaces the active append target). The current day's primary file is `daily/YYYY-MM-DD.md`; rotated files keep the original filename plus `.<seq>` infix. **No silent drops** — operators are guaranteed every write lands somewhere.
- **Dreaming read window**: dreaming worker reads the **last 7 days** of daily logs across all rotations within that window. Aligned with paperclip's Hot tier (7d) — captures recent learning without burning excess context on stale entries.
- **Cold-archive threshold**: after **30 days**, daily files are moved to `<scope>/memory/_system/archive/YYYY-MM/daily/<filename>` (Codex/paperclip Warm→Cold transition). Files in archive are **not read by dreaming**, **not indexed in catalog**, **not queried by recall**, but **preserved on disk** for forensic / replay / cold-retrieval.
- **Hard-delete threshold**: **never by default** (`hard_delete_days = 0`). Operators must explicitly purge via CLI: `agh memory daily purge --older-than 365d --dry-run` (paperclip "never hard-delete" pattern).
- **Safety valve**: per-scope archive dir hard-capped at **1 GiB** (`max_archive_bytes = 1073741824`). When exceeded, the oldest-first cold-deletion runs and emits a `op=archive_purged` event with the deleted file count and total bytes freed. Threshold tunable.

All values are config keys (see Implementation Notes); the defaults above ship with v2.

## Alternatives Considered

### Alternative 1: OpenFang silent-drop (1 MB cap, no rotation)

- **Description**: Cap at 1 MiB; subsequent writes to the same day fail silently or overwrite oldest entries.
- **Pros**: Simplest enforcement.
- **Cons**: Silent data loss is a bug, not a feature. Operators have no signal that their day's content was truncated.
- **Why rejected**: Violates AGH's truthful-UI / agent-manageability posture (SD-007, SD-011). Every write must be auditable.

### Alternative 2: OpenClaw uncapped (no file cap, no rotation)

- **Description**: Daily files grow indefinitely; dreaming worker handles distillation.
- **Pros**: No code complexity around rotation.
- **Cons**: A noisy day (1000+ extractor decisions, agent burst) could produce a 50-MB daily file. Dreaming worker reads suffer; operator `cat` of the file becomes painful.
- **Why rejected**: The dreaming worker is downstream of file size; pre-bounding the file makes the system more predictable.

### Alternative 3: paperclip strict tiering (Hot 7d / Warm 8-30d / Cold 30+d, never delete)

- **Description**: Three explicit storage tiers with movement triggered by access-count decay.
- **Pros**: Maximum data preservation. Access patterns drive tier transitions.
- **Cons**: paperclip's tiering is access-count-based — requires per-record read tracking, which adds complexity. Time-based tiering (chosen) gives 80% of the benefit at 20% of the cost.
- **Why rejected**: AGH adopts paperclip's "never hard-delete" principle but uses simpler time-based tier transitions instead of access-count.

### Alternative 4: Compaction-then-delete

- **Description**: Run dreaming worker on daily files older than N days; if dreaming succeeded, delete the original daily file.
- **Pros**: Saves disk by not keeping pre-distilled content.
- **Cons**: Loses ability to re-distill with improved future dreaming logic. Loses forensic record of what was actually said vs what dreaming inferred. "We can't see why dreaming made that decision because we deleted the input."
- **Why rejected**: Dreaming output is a lossy projection of the daily input. Keeping the input enables re-runs with better algorithms; deleting it forfeits that capability.

## Consequences

### Positive

- No silent data loss (rotation guarantees every write lands).
- Dreaming worker reads have predictable size (7d × scope × ≤1 MiB per day per scope = bounded).
- Cold-archive preserves forensic surface (`agh memory daily restore <date>` retrieves from archive).
- Disk usage bounded by `max_archive_bytes` safety valve.
- Operator-visible CLI for retention operations (`agh memory daily ls`, `agh memory daily archive`, `agh memory daily restore`, `agh memory daily purge`).
- Aligns with paperclip's "never hard-delete by default" principle while staying simpler than full access-count tiering.

### Negative

- More CLI surface (4 new verbs for daily operations). Mitigated by using existing memory CLI patterns and shared option parsers.
- Cold-archive directory accumulates files indefinitely until `max_archive_bytes` hits. Mitigation: clear documentation; archive purge defaults to oldest-first when triggered.
- Rotation introduces multiple files per day, which complicates date-based queries. Mitigation: `agh memory daily show <date>` concatenates all `<date>.<seq>.md` files in order.

### Risks

- A pathological burst (10,000 extractor decisions in one minute) creates many `<date>.<seq>.md` files within seconds. Mitigation: rotation is sequential, not based on collision; concurrent writes serialize through `Controller.Decide`.
- Operators who set `hard_delete_days > 0` lose data. Mitigation: CLI prompts for confirmation; `--dry-run` is the documented first step; non-zero default is opt-in.
- Cold-archive directory could be on a different filesystem (different mount). Mitigation: `_system/archive/` lives in the same filesystem as the scope root by default; cross-FS archive is configurable but logged as a warning.

## Implementation Notes

- New helper: `internal/memory/daily.AppendWithRotation(ctx, path, content)`. Atomic check-and-rotate before write.
- New helper: `internal/memory/daily.ColdArchive(ctx, scope_root, threshold_days)`. Walks daily/, moves files older than threshold to `_system/archive/YYYY-MM/daily/`.
- Daemon-managed daily task: runs daily archive sweep + safety-valve check at 03:00 local time (configurable via `[memory.daily] sweep_hour`).
- Config keys:

  ```toml
  [memory.daily]
  max_bytes          = 1048576              # 1 MiB per file
  max_lines          = 5000
  rotate_format      = "{date}.{seq}.md"    # daily/2026-05-04.0.md, etc
  dreaming_window    = 7                    # days dreaming worker reads
  cold_archive_days  = 30
  hard_delete_days   = 0                    # 0 = never; explicit CLI to purge
  max_archive_bytes  = 1073741824           # 1 GiB safety valve per scope
  sweep_hour         = 3                    # local hour for daily housekeeping (0-23)
  archive_path       = "_system/archive"    # relative to scope root
  ```

- New CLI verbs:
  - `agh memory daily ls [--scope X] [--workspace Y]` — list daily files including rotations
  - `agh memory daily show <date> [--scope X] [--workspace Y]` — concatenated content
  - `agh memory daily archive [--older-than 30d] [--dry-run]` — manual cold-archive trigger
  - `agh memory daily restore <date> [--scope X]` — pull from archive back to daily/
  - `agh memory daily purge --older-than <duration> [--dry-run]` — explicit hard-delete

- All emit canonical dotted events into `memory_events.op` (peer-review B5): `memory.daily.rotated`, `memory.daily.archived`, `memory.daily.restored`, `memory.daily.purged`, and **`memory.daily.archive_purged`** (safety-valve eviction; emitted whenever `max_archive_bytes` triggers oldest-first cold-deletion). The safety-valve event makes the "never hard-delete by default" exception visible per peer-review OC5.
