# ADR-009: Write Controller — Hybrid Rule-First with LLM-as-Tiebreaker

## Status

Accepted

## Date

2026-05-04

## Context

ADR-001 places curated Markdown as authoritative for memory content; ADR-008 places `MemoryProvider` as the extension surface; ADR-005 reserves `_system/` for machine-managed artifacts. The remaining structural question is: when a candidate fact arrives at the controller (extracted from a turn, written via CLI, proposed via native tool, surfaced by the dreaming worker), **who decides** ADD / UPDATE / DELETE / NOOP / REJECT and **how**?

`analysis_write-controller.md` verified competitor approaches in source code:

- **Mem0** (canonical SaaS reference) — pure LLM-mediated. Two LLM calls per write batch (extract + decide). K=5–10 neighbors. Structured JSON `{event, id, text}`. NOOP is "the most undervalued operation and arguably the strongest indicator of a mature memory system" (`memory-engineering-agents.md:304-314`).
- **Hermes** — agent-as-controller. The LLM (the agent itself) picks ADD/REPLACE/REMOVE in-context. Runtime guards are deterministic only: exact dedup (`memory_tool.py:243`), 13 threat-pattern regex, char-budget cap, atomic-replace + lock. Hermes' own analysis admits this fails at scale: *"MEMORY.md becomes contradictory at hundreds of lines; users must hand-edit periodically"* (`analysis_hermes.md:610`).
- **OpenClaw** — two-tier deterministic. Rule-based flush trigger (`agent-runner-memory.ts:656-657`) → silent agent turn writes content → rule-based promotion gates (`short-term-promotion.ts:23-62`) with weighted score (`freq=0.24, rel=0.30, div=0.15, rec=0.15, cons=0.10, concept=0.06`). LLM never *decides* what stays; LLM only *writes* content.
- **Letta MemFS** — agent-as-controller with git commits per write.
- **Zep / Graphiti** — bi-temporal: never DELETE, mark old fact `valid_to=now`, ADD new. Hours of async ingest.

Critical observation: **no production runtime uses pure-LLM-mediated decisions**. Mem0 is SaaS, optimizing for benchmark accuracy with model cost amortized at the product layer. Runtimes (Hermes, OpenClaw, Letta) keep decision-making deterministic or agent-driven and pay the contradiction-detection cost in operator hand-editing or at-scale brittleness.

AGH constraints:

- **Local-first** — every write must work offline.
- **Agent-manageable** — every decision must be JSON-serializable, audit-visible via CLI/HTTP/UDS.
- **Latency-sensitive** — CLI scripting and web UI writes must be fast.
- **Greenfield** — no migration tax; pick the shape that survives v2 → v3 without redesign.
- **Two-touch rule** — third change to the same area opens a new TechSpec.

Three strategies on the table:
1. Pure LLM-mediated (Mem0).
2. Pure deterministic rules (Hermes-style).
3. Hybrid: rules first, LLM as tiebreaker only when rules find ambiguity.

The user selected option 3.

## Decision

Adopt **Hybrid (rule-first, LLM-as-tiebreaker), single-step. Default verdict is NOOP.**

> **⚠ Superseded by Post-Approval Refinement (Round-1 B1).** The cosine-based algorithm and the `0.72/0.88/0.92` thresholds described below were superseded after Codex peer review. See the Post-Approval Refinement section at the end of this ADR for the **canonical Slice 1 algorithm** (lexical/entity-only, no embeddings, no cosine thresholds, no `Embedding` field on `Candidate`). The original algorithm is retained here for traceability only.

### Decision algorithm (SUPERSEDED — see Post-Approval Refinement)

```
Decide(candidate):
  if threat_scan/size/scope/origin policy fails: → REJECT
  hash = blake2b(normalize(candidate.content))
  if hash already in store: → NOOP (exact dup, O(1))
  topK = vectorStore.search(candidate.embedding, k=cfg.top_k)
  if topK[0].similarity >= near_dup_threshold (0.92): → NOOP (near dup)
  if no entity slot overlap AND topK[0].similarity < ambiguity_low (0.72): → ADD (genuinely novel)
  if entity slot match + attribute contradicts + similarity >= update_threshold (0.88): → UPDATE(target=topK[match])
  if entity slot match + no attribute change: → NOOP (attribute_match_no_change)
  if ambiguity_low <= similarity < ambiguity_high (0.72..0.88): → llmDecide(candidate, topK)
  else: → NOOP (default safe)

llmDecide(candidate, topK):
  budget = 1 LLM call, 250 ms timeout, top_k=5 in prompt
  output = strict JSON schema {op, target_id?, confidence, reason}
  on schema-invalid OR timeout OR error: → NOOP (fallback)
  return validated decision
```

### Mode configuration

`[memory.controller] mode` is a hot-reloadable enum:
- `mode = "hybrid"` (default) — full algorithm above
- `mode = "rules"` — ambiguity band collapses to NOOP (no LLM call ever)
- `mode = "llm"` — rule engine becomes a pre-validator only (REJECT/NOOP-on-exact-dup), every other path escalates to LLM

`mode = "llm"` is a strict subset of `mode = "hybrid"` (set ambiguity band to `[0, 1]`); picking hybrid forecloses no future option.

### Latency budget

| Path | Frequency | Latency |
|---|---|---|
| Hash hit / sim ≥ 0.92 / clear ADD path | ~80% | <5 ms |
| Entity-slot match (clear contradiction or no contradiction) | ~15% | <10 ms |
| Ambiguity band → LLM call | ~5% | 150–300 ms |

Aggregate: **P95 ≈ 50 ms, P99 ≈ 300 ms**. Pure-LLM baseline P95 ≈ 400 ms.

### Audit + reversibility

Every `Decision` is persisted to a `memory_decisions` SQLite write-ahead log **before** the corresponding store mutation lands. Schema includes `RuleTrace` (always populated), `LLMTrace` (nil unless escalated), `prior_content` (for UPDATE/DELETE rollback), `prompt_version`, `decided_at`.

`agh memory decisions list/show/revert` exposes the log to operators. Decisions are JSON-serialised and surfaced in HTTP/UDS at parity (SD-011).

### Single write path invariant

The controller is the **only** path for writes into the memory store. `internal/memory/store.go` accepts only `(Decision, Candidate)` pairs — never raw `Add`. Direct curated-file writes by the operator are detected by the daemon-owned file watcher (ADR-001 reconciliation path) and re-funnelled through `Decide()` as `Origin = file`; if the observed bytes already match an unapplied WAL row's `post_content_hash`, replay stamps `applied_at` rather than inventing a second decision.

## Alternatives Considered

### Alternative 1: Pure LLM-mediated (Mem0 model)

- **Description**: Every write batch triggers two LLM calls — extraction + decide. Top-K=5–10 neighbors. Structured JSON output.
- **Pros**: Highest dedup/contradiction quality on benchmarks. Single uniform decision path. Natural fit for ADD/UPDATE/DELETE/NOOP semantics.
- **Cons**: 200–500 ms LLM call per write. Mandatory model dependency on the write path (breaks offline / constrained envs). Cost: haiku × N agents × M writes/day accumulates. Failure modes: LLM hallucinates `target_id`; JSON schema drift; prompt injection in candidate content; timeout half-applies the write. Inverts AGH's truth-direction (model decides what the runtime stores).
- **Why rejected**: Local-first requirement plus latency budget make uniform LLM cost unacceptable. Mem0's accuracy lead is a SaaS property — they amortize the cost at the product layer.

### Alternative 2: Pure deterministic rules

- **Description**: Pure-Go rule engine: cosine similarity threshold + exact-name match + content scan. No LLM in the path.
- **Pros**: Sub-millisecond decisions. Synchronous, predictable, replayable. No model dependency. Trivially auditable. Hermes proves it works for hand-curated, char-bounded memory; OpenClaw proves it works for promotion gates.
- **Cons**: Cannot detect semantic contradictions reliably (e.g. "I prefer email" vs "I prefer phone" share insufficient surface form for cosine to flag). Rules over-trigger UPDATE on benign paraphrase; under-trigger on different phrasings of the same change. Heavy dependency on embedding quality. Hermes' own analysis admits flat-markdown contradictions are the failure mode at scale.
- **Why rejected**: AGH must scale beyond Hermes' ≤2200-char ceiling. Hand-editing memory periodically is not an acceptable operator burden.

### Alternative 3: Async LLM-mediated (eventual consistency)

- **Description**: Write returns immediately with provisional decision; LLM controller decides asynchronously; store reconciled later. Mem0 ships this with 100–500 ms eventual consistency window.
- **Pros**: No latency on the write path.
- **Cons**: "Memory not yet readable" window — the next agent turn may not see a fact that was just written. Breaks expectations from CLI scripting and turn-to-turn agent reasoning. Requires speculative store + reconciliation logic that's brittle.
- **Why rejected**: Sync write path is simpler and matches operator expectations. Async is a v3 optimization if benchmarks justify it.

### Alternative 4: Mem0g-style three-LLM-call (extract + decide + conflict-detect)

- **Description**: Add a third LLM call (conflict detector / entity-extractor / relations generator) before the write, flagging contradictions explicitly.
- **Pros**: Highest accuracy in class (+36% temporal accuracy reported by Mem0g paper).
- **Cons**: 300 ms – 3 s p95. Three model dependencies. Adds graph schema (entities + relations + edges) that AGH does not have in v2.
- **Why rejected**: Defer to v3 if benchmarks show v2 misses too many contradictions. Mem0g requires graph memory which is parking-lotted in `analysis.md` §5.

## Consequences

### Positive

- Sub-millisecond on the dominant path. CLI scripting feels native; web UI writes feel instant.
- LLM cost burns only on the genuinely-ambiguous middle band (~5% of writes), not on every write.
- Local-first preserved: `mode = "rules"` is a single config flag away — no LLM dependency for offline / strict-determinism deployments.
- Fully agent-manageable: every Decision is JSON-serializable; `agh memory decisions list/show/revert` exposes the log; HTTP/UDS at parity.
- Reversible: UPDATE/DELETE store `prior_content`; `agh memory decisions revert <id>` rolls back N steps.
- Composable with `mode = "llm"` (LLM-only) and `mode = "rules"` (deterministic-only) — three operational profiles via config.
- Extensible: rule signals (entity slot, FTS surface form, content hash), LLM model, prompt template are config keys. Extensions can register additional prefilter rules via a hook.
- LLM-only is a strict subset of hybrid (escalate every non-trivial case); picking hybrid forecloses no future option (two-touch rule preserved).

### Negative

- Two code paths to maintain (rule engine + LLM bridge). Mitigated by the LLM bridge already existing in `internal/tools` (native-tool LLM dispatch).
- ~~Requires careful tuning of the ambiguity-band thresholds (0.72 / 0.88 / 0.92).~~ **Superseded by Post-Approval Refinement (B1):** Slice 1 has no cosine thresholds. Entity-slot ambiguity is the LLM trigger; the only "tuning" knob is whether to escalate on multiple slot matches — binary, not continuous.
- ~~Embedding pipeline becomes mandatory on the write path.~~ **Superseded by Post-Approval Refinement (B1):** Slice 1 has no embedding pipeline on the write path. Embeddings re-enter only in Slice 3.
- Threshold values need empirical validation. **Updated:** the entity-slot rules need eval data (false-update rate, false-NOOP rate); see open concern OC1 in TechSpec.

### Risks

- LLM call fails or schema drifts → NOOP fallback drops a write that should have been ADD. Mitigation: `agh memory decisions list --reason="llm_fallback"` surfaces these for operator inspection; eval harness baseline catches regressions.
- A pathological agent floods the controller with paraphrased candidates → escalation rate spikes → latency degrades. Mitigation: `[memory.controller.policy] max_writes_per_min = 60` rate limit per scope; circuit-breaker pattern on the LLM bridge.
- ~~Threshold drift across embedding models~~ **Superseded by B1:** no embeddings in Slice 1. Re-introduced when vector graduates in Slice 3.
- Contradiction missed by entity-slot rules (no slot declared, surface forms differ enough to not collide on FTS). Mitigation: dreaming worker (ADR-010) re-evaluates entries periodically and surfaces contradictions for operator review; Slice 3 vector + LLM ranker tightens the gap.

type Op uint8

    OpUpdate
    OpDelete

    Embedding   []float32         // pre-computed; controller does NOT call embedding

type Decision struct {
    ID            string         // ULID
    CandidateHash string         // blake2b
    Op            Op
    Targets       []string       // existing memory IDs

    Decide(ctx context.Context, c Candidate) (Decision, error)

### Prompt template (LLM tiebreaker)

Stored at `internal/memory/prompts/decide.v1.tmpl`. Versioned. Prompt schema enforces strict JSON output `{op, target_id?, confidence, reason}` with strict-zod-equivalent Go validation. Schema-invalid → NOOP fallback.

### Persistence

New SQLite table `memory_decisions` in the per-workspace DB (ADR-003) for workspace + agent-workspace decisions; in the global DB for global + agent-global decisions. Numbered migration in registry (`agh-schema-migration` skill mandatory).

```sql
CREATE TABLE memory_decisions (
    id                TEXT PRIMARY KEY,           -- ULID
    candidate_hash    TEXT NOT NULL,
    idempotency_key   TEXT NOT NULL UNIQUE,
    frontmatter_hash  TEXT NOT NULL,
    workspace_id      TEXT,                       -- nullable for global decisions
    scope             TEXT NOT NULL CHECK (scope IN ('global','workspace','agent')),
    agent_name        TEXT,
    agent_tier        TEXT,
    op                TEXT NOT NULL CHECK (op IN ('noop','add','update','delete','reject')),
    targets           TEXT NOT NULL DEFAULT '[]', -- JSON array of memory IDs
    target_filename   TEXT NOT NULL,
    frontmatter       TEXT NOT NULL DEFAULT '{}',
    post_content      TEXT,
    post_content_hash TEXT,
    prior_content     TEXT,
    confidence        REAL NOT NULL,
    source            TEXT NOT NULL CHECK (source IN ('rule','llm')),
    rule_trace        TEXT NOT NULL,              -- JSON
    llm_trace         TEXT,                       -- JSON; nullable
    reason            TEXT,
    prompt_version    TEXT,
    applied_at        INTEGER,
    decided_at        INTEGER NOT NULL            -- unix ms
);
CREATE INDEX idx_decisions_workspace ON memory_decisions(workspace_id, decided_at);
CREATE INDEX idx_decisions_op ON memory_decisions(op, decided_at);
CREATE INDEX idx_decisions_unapplied ON memory_decisions(applied_at) WHERE applied_at IS NULL;
```

## Post-Approval Refinement (Codex peer-review B1 + B3 incorporation, 2026-05-04)

Two refinements after peer review.

### B1 — Slice 1 controller is lexical/entity-only (no embeddings)

The original algorithm used cosine similarity with thresholds `0.72/0.88/0.92` against an embedding store. Codex peer-review B1 surfaced that the TechSpec explicitly defers the embedding pipeline (no sqlite-vec, no provider/cache/cgo work) to Slice 3 — making the cosine-based algorithm non-implementable as drafted.

**Refinement adopted:** Slice 1 controller is fully **deterministic + lexical**. The algorithm becomes:

```
Decide(candidate):
  if policy violation (size, scope, threat scan, WHAT_NOT_TO_SAVE): → REJECT
  hash = blake2b(normalize(candidate.content))
  if hash already in store: → NOOP
  if frontmatter.entity != "" AND frontmatter.attribute != "":
      slot_targets = catalog.LookupBySlot(scope, entity, attribute)
      if 0 → ADD; if 1 same content → NOOP; if 1 different content → UPDATE; if >1 → llmDecide
  else:
      fts_hits = ftsSearch(scope, content, top_k=5)
      if no overlap above surface-form threshold: → ADD
      if exact-name slug collision: → UPDATE(target = first match)
      else: → llmDecide

llmDecide: 1 LLM call, 250 ms timeout, JSON {op, target_id?, confidence, reason}, NOOP fallback.
```

**No cosine, no embeddings, no `0.72/0.88/0.92` thresholds in Slice 1.** The `Embedding []float32` field is removed from `contract.Candidate`. Cosine similarity returns in Slice 3 alongside vector retrieval; that's when the threshold-based ambiguity band re-enters the design.

The `mode = hybrid | rules | llm` config-key behavior is preserved:
- `hybrid` (default): runs the algorithm above
- `rules`: ambiguity escalations collapse to NOOP
- `llm`: deterministic path becomes a pre-validator only

### B3 — Decision row carries replay material

The original `memory_decisions` row stored `candidate_hash` + `prior_content`. Codex peer-review B3 surfaced that this is insufficient for deterministic crash replay — without the post-mutation content, a crash between decision-write and file-write leaves an unreplayable state.

**Refinement adopted:** `memory_decisions` adds `post_content`, `post_content_hash`, `target_filename`, `frontmatter`, `idempotency_key`, and `applied_at`. Replay walks `WHERE applied_at IS NULL` and applies each row idempotently (skip when on-disk content already matches `post_content_hash`). See ADR-001's same-date refinement and the TechSpec §Data Models / `memory_decisions` schema.

`contract.Decision` is updated to carry the same fields. `Controller.Decide` populates them before returning; the store's `ApplyDecision` reads them when persisting.
