# ADR-002: Three Scopes with Agent Two-Tier (Read Precedence agent-workspace ▸ agent-global ▸ workspace ▸ global)

## Status

Accepted

## Date

2026-05-04

## Context

AGH agents can be defined in two locations today (`internal/config/agent.go:131-169`):

- **Global**: `$AGH_HOME/agents/<name>/AGENT.md`
- **Workspace-local**: `<workspace>/.agh/agents/<name>/AGENT.md` (when present)

The existing AGH memory subsystem only ships two scopes (`global`, `workspace`) and has **no agent scope**, despite RFC 001 promising `agent-local overriding all` (`internal/CLAUDE.md` Memory & Skills Runtime). `analysis_agh-current.md` §3 names this as a documented gap.

The mem-v2 design must answer: when a globally-defined agent (`pedro-reviewer` at `$AGH_HOME/agents/pedro-reviewer/`) runs in workspace A and learns something, where does that memory live? If it lives at `<workspace-A>/.agh/agents/pedro-reviewer/memory/`, the agent starts blank in workspace B — losing cross-workspace user preferences. If it lives at `$AGH_HOME/agents/pedro-reviewer/memory/`, the agent leaks workspace-private context across workspaces.

Source-verified competitor patterns (`analysis_agent-scope-dilemma.md`):

- **OpenClaw** (C1 — memory follows agent definition): each `agentId` owns its world (`packages/memory-host-sdk/src/host/config-utils.ts:270-290`). Multi-agent dir validation (`config.multi-agent-agentdir-validation.test.ts:8-22`). No workspace-vs-global split.
- **Claude Code** (C5 — three sub-modes per agent): frontmatter declares `memory: 'user' | 'project' | 'local'` (`tools/AgentTool/agentMemory.ts:12-65`). Implements **project→user snapshot bootstrap** (`loadAgentsDir.ts:262-294`) — direct evidence single-tier strands operators.
- **Hermes**: profile-aware `HERMES_HOME` (`hermes_cli/main.py:94-163`). Single-user, single-tier. Avoids the question.
- **Codex**: no agent memory at all; sub-agents excluded from extractor (`memories/write/src/start.rs:33`). Codex flags this as a known weak spot (`analysis_codex.md` §20.2).

Five candidate architectures evaluated in `analysis_agent-scope-dilemma.md` §4. The right answer must:

- Honor RFC 001's "agent-local overriding all" without leaking workspace context across workspaces.
- Match AGH's existing dual agent-DEFINITION discovery symmetrically.
- Provide a home for both cross-workspace user preferences AND workspace-private project knowledge per agent.

## Decision

Adopt **three scopes with agent two-tier** (Candidate C3):

- **Three scopes**: `global`, `workspace`, `agent`.
- **Agent scope is two-tier**:
  - **agent-global** at `$AGH_HOME/agents/<agent>/memory/` (cross-workspace baseline)
  - **agent-workspace** at `<workspace>/.agh/agents/<agent>/memory/` (workspace override)

**Read precedence** (deeper wins, shadow-by-id never merge silently):

```
agent-workspace ▸ agent-global ▸ workspace ▸ global
```

**Default write tier for `--scope agent`**: `agent-workspace`. Cross-workspace promotion is an explicit operation: `agh memory promote --to agent-global <filename>`.

**API shape**: scope is a 3-value enum (`global | workspace | agent`); when `--scope agent`, an optional `--agent-tier {workspace, global}` selects the tier (default `workspace`). CLI / HTTP / UDS / native tool surfaces all expose the same shape.

**Default `AgentDef` binding**: when `AgentDef.Memory.Scope` is omitted, the runtime defaults to `scope = agent`, `agent_tier = workspace` for both globally-defined and workspace-defined agents. This is the least-leakage default. Sessions without an active workspace reject `agent-workspace` writes with `memory.scope.workspace_required` rather than silently routing them to `global`.

**Sub-agent inheritance**: a sub-agent forks from its parent's curated snapshot at session boot. The sub-agent's controller is `Mode = ReadOnly` (defense in depth, P12 in `analysis.md` §3). Parent / session-end extraction may process the sub-agent's final evidence through the controller into the parent agent's tier.

## Alternatives Considered

### Alternative 1: C1 — Memory follows agent definition

- **Description**: Global agent → global memory. Workspace agent → workspace memory. OpenClaw shape.
- **Pros**: Single-tier, simple. No promote verb. No precedence ambiguity.
- **Cons**: Can't coexist with workspace context isolation. A globally-defined agent learning workspace-private knowledge in workspace A leaks it to workspace B.
- **Why rejected**: AGH's project rules explicitly reject silent cross-workspace context leakage. OpenClaw only avoids the issue because OpenClaw assumes single-user single-tenant.

### Alternative 2: C2 — Memory always workspace-bound

- **Description**: Regardless of where the agent is defined, memory lives at `<workspace>/.agh/agents/<agent>/memory/`. Same agent in two workspaces starts blank in each.
- **Pros**: Zero leakage by default. Simpler than C3 (one tier per agent).
- **Cons**: Cross-workspace user preferences require manual `agh memory promote` every time a useful feedback is learned. Operator friction for power-users with one agent across many workspaces. Claude Code's project→user bootstrap is direct evidence single-tier strands operators.
- **Why rejected**: Claude Code's snapshot-bootstrap mechanism exists *because* single-tier was insufficient in production. Adopting C2 would require AGH to invent the same workaround later.

### Alternative 3: C4 — Type-routed regardless of agent

- **Description**: `feedback`/`user` types → global by default; `project`/`reference` → workspace. Agent definition has zero effect on memory routing.
- **Pros**: Conceptually clean — type drives scope.
- **Cons**: Loses agent-private memory entirely. A `code-reviewer` agent and a `pedro-style` agent would write `feedback_*.md` to the same global pool — collision-prone, unclear ownership.
- **Why rejected**: AGH agents are first-class identity boundaries. Losing per-agent memory breaks the "agent-local overriding all" RFC 001 promise.

### Alternative 4: C5 — Per-agent declared scope (Claude Code style)

- **Description**: Each `AgentDef.memory.scope = "user" | "project" | "local"` declares its own scope, mapping to one of three storage locations.
- **Pros**: Maximum operator control. Matches Claude Code shape.
- **Cons**: Adds cognitive overhead — operators must reason about per-agent scope choice. Doesn't compose well with workspace-vs-global agent definition (4-cell matrix: agent definition × declared scope = confusing).
- **Why rejected**: C3 covers C5's flexibility with simpler defaults. C5's per-agent scope can be added later as syntactic sugar over C3 if needed.

## Consequences

### Positive

- Cross-workspace user preferences live at `agent-global`; workspace-private project knowledge lives at `agent-workspace`. Each tier has a clear purpose.
- Symmetric with existing dual agent-DEFINITION discovery — no new conceptual model for operators.
- `internal/CLAUDE.md` "agent-local overriding all" is honored: `agent-workspace` is the deepest read layer.
- Default binding is conservative (`agent-workspace`); cross-workspace durability requires explicit operator action.
- Shadow-by-id (deeper-wins, audited) prevents silent merge contradictions.

### Negative

- 4 read layers vs 3 — slightly more complex precedence resolution. Mitigated by clear shadow-by-id rule and audit events.
- New `agh memory promote --to agent-global` verb. Required to honor agent-manageability (SD-011).
- Catalog rows need `scope_owner` (who owns the row: `'global' | 'workspace' | 'agent'`) AND `agent_name` (when `scope_owner = 'agent'`) AND `agent_tier` (when `scope_owner = 'agent'`).

### Risks

- Operators may write to the wrong tier accidentally — e.g., a workspace-bound feedback that should have been agent-global. Mitigation: extractor sub-agent classifies tier in the same prompt that classifies type; structured CLI flag is mandatory (no implicit defaulting beyond what `AgentDef.memory.scope` declares).
- Sub-agent forks reading parent's two-tier snapshot need clear semantics for which tier dominates when parent and child run in different workspaces (rare but possible). Mitigation: sub-agent inherits parent's resolved snapshot, not a re-resolution; resolution happens once at parent session boot.

## Implementation Notes

- Migration delta atomic per greenfield rule (`internal/CLAUDE.md`):
  - `internal/memory/types.go::Scope` enum extended to `('global', 'workspace', 'agent')`.
  - New `internal/memory/types.go::AgentTier` enum `('workspace', 'global')`.
  - `memory_catalog_entries` schema migration: add `scope_owner TEXT NOT NULL`, `agent_name TEXT`, `agent_tier TEXT`. Numbered migration in registry; no `EnsureSchema`.
  - `internal/config/agent.go::AgentDef` gains `Memory.Scope` (string) and `Memory.AgentTier` (string); empty values resolve to `agent-workspace`.
  - CLI: `agh memory write --scope agent [--agent-tier workspace|global]`.
  - HTTP/UDS body field: `scope: "agent"`, `agent_tier: "workspace"`.
  - Native tool: `agh__memory_propose({scope: "agent", agent_tier: "workspace", ...})`.
  - Extension Host API: `Write(scope, agent_tier?, ...)`.
