# ADR-011: Recall Pipeline — Deterministic-First with Optional Vector + LLM Ranker (Slice 1 Deterministic Only)

## Status

Accepted

## Date

2026-05-04

## Context

Recall is the read path for memory: given a query (next user turn, agent reasoning step, or explicit search), produce the top-K curated memory entries that should be injected into the prompt context.

The pipeline has six concerns:
1. **Candidate generation** — how do we produce the initial set of N candidates from potentially thousands of stored entries?
2. **Score fusion** — when multiple candidate sources exist (full-text, vector, metadata), how do we combine ranks?
3. **Re-ranking** — should an LLM (or a small model) re-rank the candidates for precision?
4. **Scope precedence** — multi-scope candidates (`agent-workspace ▸ agent-global ▸ workspace ▸ global`, ADR-002) need shadow-by-id resolution before injection.
5. **Filtering** — `alreadySurfaced` (don't re-inject what was injected this session), `_system/` exclusion (ADR-005), staleness banner (per `analysis.md` §3 P10).
6. **Packaging** — stable cached header for prompt-cache stability, freshness banner, structured `<system-reminder>` blocks per the convention used by Claude Code.

Source-verified competitor patterns:

- **Hermes** (`analysis_hermes.md` §2.5) — FTS5 unicode61 + FTS5 trigram twin index. Two-stage: cheap retrieval (FTS5 candidates) → LLM summarization at read time. **No vector backend in built-in path.** Honcho plugin adds vector via `MemoryProvider` extension.
- **Claude Code** (`analysis_claude-code.md` §3) — `findRelevantMemories.ts` runs a forked Sonnet ranker over a frontmatter manifest of stored entries (≤200 files), returns ≤5 filenames. No vector. No FTS — direct file scan + LLM ranking.
- **OpenClaw** (`analysis_openclaw.md`) — hybrid FTS5 + sqlite-vec via `node:sqlite` builtin. Default `vector=0.7/text=0.3` weighted merge. `minScore=0.35`, `maxResults=6`. Hash-keyed embedding cache.
- **goclaw** (`analysis_goclaw-paperclip-multica.md` §1.5) — pgvector + tsvector hybrid. RRF (Reciprocal Rank Fusion, k=60). Per-user 1.2× boost rejected for AGH (ADR-002, no double-counting on top of shadow-by-id).
- **Codex** (`analysis_codex.md` §1) — agents read `memory_summary.md` (≤5K tokens, frozen at session start) + side-query into per-rollout summaries when needed. No FTS+vector hybrid; entirely file-based.

AGH constraints:
- **Local-first** — recall must work offline.
- **Greenfield, no cgo gate** — sqlite-vec is cgo-only on Windows; pure-Go fallback must reach parity if vector ships.
- **Hot path** — recall fires per turn; latency must stay sub-50ms typical.
- **Two-touch rule** — recall design must survive v2 → v3 without rewrite.
- **User pushback** — slice 1 must reach **Hermes-parity** without exceeding scope.

## Decision

Adopt **deterministic-first recall** with optional vector and optional LLM ranker, **deterministic-only in Slice 1**. Vector retrieval and LLM ranker graduate to Slice 3 with explicit eval-driven gating.

### Slice 1 — Deterministic recall pipeline

Single entry point in new package `internal/memory/recall`:

```go
type Recaller interface {
    // Recall returns up to opts.TopK packaged candidates for the query, applying
    // scope precedence + alreadySurfaced filter + staleness annotation.
    Recall(ctx context.Context, query Query, opts RecallOptions) (Packaged, error)
}
```

**Pipeline stages:**

1. **Trivial-message gate** — drop queries with <3 meaningful tokens (stop-list + length check). goclaw `trivial_filter.go` pattern. Returns empty Packaged immediately.

2. **Context-aware query rewrite** — rune-safe `Context: ...\nQuery: ...` formatting (preserves UTF-8 multi-byte boundaries). Codex pattern.

3. **Candidate generation — FTS5 hybrid**:
   - **FTS5 unicode61** index on `memory_chunks.content` for English / Latin scripts
   - **FTS5 trigram** index on the same content for CJK + paraphrase resilience (Hermes pattern)
   - Both queries fire in parallel against the per-workspace catalog DB (ADR-003)
   - Each returns up to `opts.RawCandidates` (default 50) rows with bm25 scores

4. **Score fusion (deterministic)** — merge unicode61 and trigram results by `chunk_id` with weighted formula:
   ```
   score = α·bm25_unicode + β·bm25_trigram + γ·recency + δ·recall_signal
   ```
   - Defaults: `α=0.55, β=0.20, γ=0.15, δ=0.10`
   - Recency: 14-day half-life decay on `mtime_ms` (matches dreaming worker, ADR-007)
   - `recall_signal`: from `memory_recall_signals` table, live in Slice 1 via `SignalRecorder`
   - All four weights are config keys

5. **Scope precedence (shadow-by-id)** — per ADR-002:
   - For each candidate, look up `(scope_owner, agent_name, agent_tier, type, slug)` key
   - Shadowed entries removed; emit `op=shadowed` audit event
   - Order remaining candidates: deeper scope wins ties

6. **`alreadySurfaced` filter** — paths surfaced earlier in same session are filtered out unless `IncludeAlreadySurfaced = true`. Tracked in session-scoped event log (ADR-006).

7. **`_system/` exclusion** — all candidates with `injection = false` (ADR-005) dropped unless `IncludeSystem = true`.

8. **Staleness annotation** — per candidate, compute `age_days = floor((now - mtime) / 1day)`. Add `staleness_banner` field if `age_days > freshness.banner_after_days` (default 1).

9. **Top-K cap** — keep top `opts.TopK` (default 5).

10. **Packaging** — produce `Packaged` containing:
    - One `<system-reminder>` block per scope (least-specific first; deeper scope last for prompt-cache stability — Codex hierarchical-AGENTS.md ordering)
    - **Stable cached header** — precomputed at attachment time, content-hash-keyed; identical headers across turns hit prompt cache
    - Per-entry: id, scope, agent_name (when applicable), title, body, age_days, staleness_banner, why_recalled (debug field with rule trace)

**No vector. No LLM ranker. No paraphrase escalation. No cross-encoder.**

### Slice 3 — Optional vector + optional LLM ranker (deferred)

When ship gates pass (eval harness shows deterministic recall precision <60% on LoCoMo-class evals OR operator demand surfaces), Slice 3 adds:

- **`sqlite-vec` virtual table** (`memory_chunks_vec`) holding embeddings, behind config `[memory.vector] backend = "sqlite-vec"`. Default off.
- **Pure-Go fallback** — in-process cosine over JSON-stored embeddings — for non-cgo builds. CI matrix adds non-cgo lane.
- **Hybrid retrieval** — fuse FTS5 + trigram + vector via:
  - Default: weighted merge (`α + β + γ + δ + ε·cosine_score`)
  - Alternative: RRF (k=60) behind `[memory.recall] fusion = "rrf"`
- **Embedding pipeline** — `[memory.embedding] model`, `provider`, hash-keyed cache table `memory_embedding_cache`
- **Optional LLM ranker** — `[memory.recall.ranker] enabled = true, model = "haiku", timeout = "200ms"`. Falls back to weighted score on timeout / error. Provider-neutral (not Sonnet-specific — corrects integration map's "Sonnet ranker" wording).

### Read-side data model (Slice 1)

```sql
-- memory_chunks (per-workspace DB, ADR-003)
CREATE TABLE memory_chunks (
    id              TEXT PRIMARY KEY,         -- ULID
    file_id         TEXT NOT NULL,            -- FK to memory_catalog_entries.id
    workspace_id    TEXT,                     -- FK; NULL for global-scope
    scope_owner     TEXT NOT NULL,            -- 'global' | 'workspace' | 'agent'
    agent_name      TEXT,
    agent_tier      TEXT,
    type            TEXT NOT NULL,            -- 'user' | 'feedback' | 'project' | 'reference'
    slug            TEXT NOT NULL,
    content         TEXT NOT NULL,
    content_hash    TEXT NOT NULL,
    start_line      INTEGER NOT NULL,
    end_line        INTEGER NOT NULL,
    injection       INTEGER NOT NULL DEFAULT 1,
    mtime_ms        INTEGER NOT NULL,
    indexed_at      INTEGER NOT NULL
);
CREATE INDEX idx_chunks_file ON memory_chunks(file_id);
CREATE INDEX idx_chunks_scope ON memory_chunks(scope_owner, agent_name, agent_tier);

-- FTS5 unicode61
CREATE VIRTUAL TABLE memory_chunks_fts USING fts5(
    content,
    content=memory_chunks,
    tokenize='unicode61'
);

-- FTS5 trigram
CREATE VIRTUAL TABLE memory_chunks_fts_trigram USING fts5(
    content,
    content=memory_chunks,
    tokenize='trigram'
);

-- AI/AD/AU triggers keep FTS in sync (already in current AGH catalog)

-- Recall signals (LIVE in Slice 1 per Post-Approval Refinement / NB4 — populated by SignalRecorder via RecordRecall fire-and-forget on every successful recall; consumed by dreaming fourth gate)
CREATE TABLE memory_recall_signals (
    chunk_id              TEXT PRIMARY KEY,
    recall_count          INTEGER NOT NULL DEFAULT 0,
    last_recalled_at      INTEGER,
    recall_score          REAL NOT NULL DEFAULT 0,
    freshness_started_at  INTEGER NOT NULL,
    promoted_at           INTEGER,
    promotion_run_id      TEXT,
    last_score_update_at  INTEGER NOT NULL
);

CREATE INDEX idx_signals_unpromoted ON memory_recall_signals(promoted_at, recall_score)
    WHERE promoted_at IS NULL;
```

### Configuration keys (Slice 1)

```toml
[memory.recall]
top_k                = 5
raw_candidates       = 50
fusion               = "weighted"     # weighted | rrf (rrf reserved for slice 3)
include_already_surfaced = false
include_system           = false

[memory.recall.weights]
bm25_unicode = 0.55
bm25_trigram = 0.20
recency      = 0.15
recall_signal = 0.10

[memory.recall.freshness]
banner_after_days = 1                 # show "this memory is N days old" banner

[memory.recall.fts]
unicode61_remove_diacritics = 2       # SQLite default
trigram_case_sensitive      = false

# Reserved for slice 3:
# [memory.vector]   backend = "off"
# [memory.recall.ranker]   enabled = false
```

All keys hot-reloadable.

## Alternatives Considered

### Alternative 1: Hybrid FTS+vector in Slice 1, LLM ranker in Slice 3

- **Description**: Add sqlite-vec + pure-Go fallback + weighted-merge fusion to Slice 1. Defer only the LLM ranker.
- **Pros**: Better paraphrase recall than Hermes built-in. Matches goclaw + Mem0 baseline.
- **Cons**: cgo build complexity on Windows (CI matrix needs non-cgo lane). Embedding pipeline must be production-ready in Slice 1 (provider, model, cache, hash-keying). Pure-Go fallback is extra code. Slice 1 scope grows substantially.
- **Why rejected**: User explicitly chose Slice 1 to reach Hermes-parity, not Hermes-superiority. Hermes ships FTS5+trigram only in built-in path (Honcho plugin adds vector). Deterministic-only meets parity; vector is a Slice 3 graduation.

### Alternative 2: Full hybrid (FTS+vector+LLM ranker) in Slice 1

- **Description**: Ship everything in Slice 1 — FTS5 + trigram + sqlite-vec + LLM ranker.
- **Pros**: Slice 1 surpasses every competitor in recall accuracy from day 1.
- **Cons**: Slice 1 already has 4 Eixos (spine + dreaming v2 + provider ABC + Hermes engineering parity); adding vector + ranker makes it 5+ parallel work areas. Risk of slice 1 slipping. cgo + LLM dependency on the recall path.
- **Why rejected**: Two-touch rule and slice deliverability favor a tighter Slice 1 with clear graduation gates.

### Alternative 3: Direct file scan + Sonnet ranker (Claude Code pattern)

- **Description**: Skip FTS entirely. Scan all curated files in scope, build a frontmatter manifest, pass to Sonnet ranker, return top-5 filenames.
- **Pros**: Simplest implementation. No DB indexing complexity.
- **Cons**: Scales to ≤200 files (Claude Code's documented limit). AGH targets thousands of curated entries across years of use. Forces Sonnet (or equivalent) on every recall — high latency + model cost. Not provider-neutral.
- **Why rejected**: Doesn't scale. AGH's posture is FTS5+catalog from day 1.

### Alternative 4: Reciprocal Rank Fusion (RRF, k=60) as default fusion

- **Description**: Use RRF instead of weighted-merge as the default fusion strategy. Common in IR literature; goclaw uses RRF with vector + tsvector.
- **Pros**: Parameter-free aggregation; robust across heterogeneous score scales.
- **Cons**: RRF only outperforms weighted-merge meaningfully when the source rankings are independent and well-calibrated. With unicode61+trigram both being lexical (highly correlated), weighted-merge gives operators tunable knobs (recency, recall_signal). RRF would need to be reserved for slice 3 when vector enters the mix.
- **Why deferred**: Reserved as `fusion = "rrf"` config option for slice 3 when vector lands. Not needed for slice 1's two-source lexical fusion.

## Consequences

### Positive

- Slice 1 ships at clear Hermes-parity with no cgo dependency.
- Pure-deterministic pipeline is replayable + auditable + offline-capable.
- All four ranking weights are config keys; operators tune without code changes.
- Eval harness (slice 1 QA proofs) measures precision baseline before vector enters the system.
- Hot-path latency: P95 sub-30 ms typical (FTS5 query + score fusion + scope shadow + packaging on a 10K-entry catalog).
- Stable cached header preserves prompt-cache hit rate across turns.
- Trigram twin index handles CJK + paraphrase recall without vector.

### Negative

- Lexical-only retrieval misses semantic paraphrase recall. ("I prefer email" vs "send via SMTP" won't match without vector.) Mitigated by trigram giving partial paraphrase resilience; full coverage waits for slice 3.
- Slice 1 has no LLM ranker, so precision-over-recall tuning (Claude Code Sonnet pattern) can't be replicated yet. Mitigated by tight `top_k = 5` default plus weighted recency.
- ~~Recall_signal weight is 0 in slice 1 (the table exists but no path populates it)~~ **Superseded by Post-Approval Refinement (B2 + NB4):** `recall_signal` is live in Slice 1 with weight δ=0.10 contributing. The `SignalRecorder` writes via fire-and-forget on every successful recall; the dreaming fourth gate reads via the unpromoted partial index; `MarkPromoted` flips `promoted_at` after synthesis.

### Risks

- Eval harness reveals deterministic recall is insufficient before slice 3 ships. Mitigation: `[memory.recall]` config keys allow operators to tune weights manually; `agh memory recall trace` exposes rule trace so operators see why a recall was made; emergency LLM-ranker injection can be a slice 1.5 patch if needed.
- FTS5 trigram tokenizer compiled-in availability — older sqlite versions may lack trigram. Mitigation: AGH bundles its own sqlite (`internal/store` dependency); minimum version pinned with trigram support.
- Per-workspace DB (ADR-003) means recall queries must run against the right DB. Mitigation: `Recaller` accepts `WorkspaceID` resolved upstream; cross-scope recall (global + workspace) runs two parallel queries (one per DB) and merges.

## Implementation Notes

- New package: `internal/memory/recall`. Owns `Recaller` interface, FTS5 pipeline, score fusion, scope shadow resolver, packaging.
- Reuses existing `internal/store` helpers (`ExecuteWrite` jittered retry from Hermes pattern).
- FTS5 triggers (AI/AD/AU) keep `memory_chunks_fts` and `memory_chunks_fts_trigram` in sync with `memory_chunks` writes.
- Schema migration (numbered): `memory_chunks` + `memory_chunks_fts` + `memory_chunks_fts_trigram` + `memory_recall_signals` tables added in slice 1 catalog migration. `internal/store/migrations/` ships the file.
- Eval harness scaffolds slice 1 QA proofs:
  - Recall precision on synthetic transcripts (target: ≥60% top-5)
  - Stale fact correction (memory updated → next recall returns updated)
  - Contradictory memory handling (shadow-by-id keeps deeper scope; older entry not injected)
  - Scope isolation (agent-A memory does not leak into agent-B recall)
  - Token budget (packaged Packaged stays under 25 KB top-of-prompt cap)
  - Long-session continuity (recall maintains relevant entries across 100+ turns)
- `agh memory search` and `agh memory recall trace` ship in slice 1.
- Codegen co-ship: contract types for `RecallRequest` / `RecallResult` / `Packaged` cross `internal/api/contract`; `agh-contract-codegen-coship` skill applies.

## Post-Approval Refinement (Codex peer-review B2 incorporation, 2026-05-04)

The original `memory_recall_signals` table was labelled "slice 1: stub schema; slice 2 wires writes." Codex peer-review B2 surfaced that this contradicts Eixo 2's dependency on live signals (the dreaming worker's fourth gate reads `recall_score`, the promotion path flips `promoted_at`).

**Refinement adopted:** `memory_recall_signals` is **live in Slice 1**. The schema gains `promoted_at INTEGER` (nullable; non-null when consumed by a dreaming run) and `promotion_run_id TEXT` (links back to `memory_consolidations.id` for forensics) and `last_score_update_at INTEGER NOT NULL` (idempotency barrier for racing recall paths). A partial index on `(promoted_at, recall_score) WHERE promoted_at IS NULL` supports the fourth-gate query in O(log N).

The write path in Slice 1:

1. `Recaller.Recall()` returns `Packaged`.
2. Fire-and-forget goroutine updates each surfaced chunk: `recall_count += 1`, `last_recalled_at = now`, `recall_score = ema(prev, similarity)`, `last_score_update_at = now`. Idempotent compare-and-swap on the barrier.
3. Failure to update is logged but does not bubble — recall surface latency preserved.

The read path (dreaming worker fourth gate) and the promotion path (`MarkPromoted(chunk_ids[], run_id)`) both ship in Slice 1.

This replaces the previous "slice 1 stub / slice 2 live" plan with a single Slice 1 deliverable. Eixo 2 (dreaming v2) was always part of Slice 1 per ADR-012; the schema/code now matches.
