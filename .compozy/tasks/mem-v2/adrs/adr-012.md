# ADR-012: Slice 1 Fat Scope — Single TechSpec with Four Eixos

## Status

Accepted

## Date

2026-05-04

## Context

The mem-v2 redesign was scoped via `analysis.md` §5 into six slices:

- **Slice 1**: spine + dreaming v2 + MemoryProvider ABC + Hermes engineering parity (the "fat" slice)
- **Slice 2**: real compactor + pre-compaction flush
- **Slice 3**: optional vector + optional LLM ranker
- **Slice 4**: external memory providers
- **Slice 5**: AGH Network memory federation
- **Slice 6**: KG / bi-temporal / advanced

Codex peer review (`_codex_peer_response.md`) explicitly flagged: *"is Slice 1 too fat to ship in one cycle?"* The concern is real — Slice 1 covers four parallel work areas (Eixos):

- **Eixo 1 — Spine** (ADR-001 + ADR-002 + ADR-003 + ADR-004 + ADR-005 + ADR-006 + ADR-009 + ADR-011 + parts of ADR-010)
- **Eixo 2 — Dreaming v2** (ADR-007 + extension of existing `internal/memory/dream.go` + new signals + 4-component scoring + promotion gates + dedicated dreaming agent)
- **Eixo 3 — MemoryProvider ABC** (ADR-008 — 10-hook lifecycle, single active provider, bundled local provider as reference)
- **Eixo 4 — Hermes engineering parity** (FTS5 trigram twin index, jittered retry helper, SSE `<memory-context>` scrubber, `parent_session_id` lineage, atomic write pattern, pre-write content scan, char caps, action-cue header)

Plus the extractor (ADR-010 Mode A).

Three options were considered:

1. **Keep fat** — single TechSpec covering all four Eixos, ships together.
2. **Split 1A (foundation) + 1B (intelligence)** — Eixos 1+3+4 in 1A; Eixo 2 (dreaming) in 1B.
3. **Split 1A (spine only) + 1B (everything else)** — Eixo 1 in 1A; Eixos 2+3+4 in 1B.

Each option trades TechSpec deliverability against competitive parity timing. The user's stance throughout the design pipeline has been explicit: *"AGH cannot ship behind Hermes — that's why everything goes in slice 1."*

## Decision

**Keep Slice 1 fat. Single TechSpec covers all four Eixos.**

### Rationale

1. **Competitive parity is non-negotiable from day 1.** Splitting dreaming or provider ABC into a follow-up slice means AGH ships at less than Hermes parity for an indeterminate window. The integration map (`analysis_integration-map.md`) and the mem-v2 audit (`analysis_agh-current.md`) both establish that Hermes' moat is the provider plugin model + the curator + the engineering polish — not any single piece. Splitting them out means Hermes-parity shipping in two acts, which forfeits the "leading not catching up" stance the user took explicitly in `analysis.md` §1.

2. **The four Eixos are tightly coupled, not independent layers.**
   - Eixo 1's spine (controller + event log + frozen snapshot) is the scaffolding everything else hangs from.
   - Eixo 2's dreaming worker writes its outputs to `_system/dreaming/` (ADR-005, defined in spine), reads `memory_recall_signals` (slice 1 schema in spine), and uses the extractor's DLQ pattern (ADR-010, also in spine).
   - Eixo 3's provider ABC must respect the controller's invariants (ADR-009: single write path), the scope precedence (ADR-002), the frozen snapshot (ADR-001).
   - Eixo 4's engineering wins (trigram FTS, retry helper, SSE scrubber) are infrastructure that the spine + dreaming + provider all depend on. Shipping spine without retry helper means racing against `BEGIN IMMEDIATE` contention from day 1.
   - Splitting creates artificial integration-test boundaries between layers that operate as one unit at runtime.

3. **AGH already paid for the hardest part.** `dream.go` (456 LOC) and `consolidation/runtime.go` (464 LOC) are the dreaming scaffolding; `internal/hooks` is the typed dispatch infrastructure; `memory_operation_log` is the event-log shape. The "fat" slice is mostly extension and reconciliation, not greenfield invention. Splitting forces re-establishing the same code surface twice.

4. **Two-touch rule preservation.** A split (1A foundation, 1B intelligence) creates two TechSpecs touching the same files (`internal/memory/types.go`, `internal/memory/store.go`, `internal/memory/dream.go`, `internal/memory/consolidation/`). The third structural change to those files would then trigger the two-touch rule — slice 2's compactor work would already be a third touch. Better to consolidate into one TechSpec with one round of reconciliation.

5. **Operational reality.** A TechSpec at 800–1200 lines is normal for AGH (the autonomy `_techspec.md` is well above that). The QA proof set will be ~12 named scenarios covering the cross-cutting interactions. `cy-create-tasks` will decompose the TechSpec into ~25–35 implementable tasks; the dependency graph naturally serializes the build order even when the spec is broad.

6. **Risk mitigation, not avoidance.** Codex peer review's concern is legitimate, but the right response is *risk-mitigated execution*, not scope-narrowing:
   - Per-Eixo task slicing in `_tasks.md` lets the build proceed in dependency order (Eixo 1 → 4 → 3 → 2, roughly).
   - QA proofs are scoped per Eixo plus cross-Eixo invariants.
   - `make verify` is the gate at every commit, not at the end of the slice.
   - Codex peer review on the TechSpec draft (opt-in step in `cy-create-techspec`) provides an external pressure test before tasks are generated.

## Alternatives Considered

### Alternative 1: Split 1A (foundation) + 1B (intelligence)

- **Description**: Eixos 1 + 3 + 4 + extractor in 1A. Eixo 2 (dreaming) in 1B.
- **Pros**: 1A focused on platform/infrastructure (objectively testable); 1B focused on intelligence layer; smaller blast radius per delivery.
- **Cons**: Forfeits Hermes-parity until 1B ships. Two TechSpecs touching the same packages triggers the two-touch rule on the third structural change. Extra QA cycle before the system is competitive.
- **Why rejected**: User explicitly chose competitive parity from day 1; forfeiting dreaming is the most expensive concession against Hermes.

### Alternative 2: Split 1A (spine only) + 1B (everything else)

- **Description**: Eixo 1 in 1A. Eixos 2+3+4+extractor in 1B.
- **Pros**: 1A is the smallest TechSpec that proves the architecture (Codex peer-review's preferred shape).
- **Cons**: 1A ships dramatically less competitive (no provider ABC, no dreaming, no Hermes engineering wins). Looks like AGH is "behind" until 1B ships. Three structural touches on the same files (1A creates, 1B extends, slice 2 extends again) conflicts with two-touch rule.
- **Why rejected**: Same competitive concern as Alternative 1, amplified.

## Consequences

### Positive

- Slice 1 ship is the moment AGH reaches at-or-better-than-Hermes — clean inflection point in the product narrative.
- All four Eixos work in their final shape from day 1; no integration-test seams between artificial slice boundaries.
- ADR set (1–11) is internally consistent — every cross-Eixo invariant is captured before tasks generate.
- Two-touch rule preserved for Slice 2's compactor work.
- One QA cycle proves the cross-cutting invariants (frozen snapshot survives dreaming write, controller is the only write path including extractor and provider, recall respects scope precedence even with provider override).
- Canonical events: see TechSpec §Monitoring for the full enumeration; `memory_events.op` CHECK is kept aligned 1:1 (peer-review N10 — earlier "12 typed events" wording was stale).

### Negative

- TechSpec size grows. Mitigated: AGH precedent (`autonomous/_techspec.md`) is comparable.
- Tasks list is long (~25–35 tasks). Mitigated: dependency graph in `_tasks.md` naturally orders the build.
- QA proofs cover ~12 cross-cutting scenarios. Mitigated: each scenario maps to a single Eixo's deliverable plus one cross-Eixo invariant.
- Slip risk on any single Eixo blocks the whole slice. Mitigated: per-Eixo independent test surface; `make verify` gates every commit; Codex peer review on the draft catches structural issues before tasks generate.

### Risks

- A late-discovered cross-Eixo conflict (e.g., extractor's bounded queue interacts badly with provider's `OnMemoryWrite` hook) requires reopening the TechSpec. Mitigation: per-Eixo design captured in ADRs 1–11; `cy-spec-peer-review` invocation after the draft surfaces these.
- Operator pressure to ship spine alone (deploy a "preview" of mem-v2). Mitigation: rejected — there is **no `[memory.v2] enabled` feature flag** (peer-review N2 + user directive). Memory v2 ships as the memory subsystem in one atomic migration per the greenfield-delete rule. If operators want to roll back, they roll back the binary.
- Per-Eixo work parallelizes poorly if the team is small. Mitigation: dependency graph in `_tasks.md` exposes the critical path; non-critical-path Eixos can sequence after the spine without slipping the whole slice.

## Implementation Notes

- Single TechSpec at `.compozy/tasks/mem-v2/_techspec.md` covers all four Eixos.
- ADR section in the TechSpec lists ADR-001 through ADR-012.
- `_tasks.md` (generated by `cy-create-tasks` after TechSpec approval) groups tasks by Eixo for legibility, but dependencies are explicit per task.
- QA pair (`qa-report` + `qa-execution`) closes the task list per `cy-tasks-tail-qa-pair`.
- Web/Docs Impact populated via `cy-web-docs-impact` for every backend task.
- Codex peer review (opt-in via `cy-spec-peer-review`) runs after the TechSpec draft is approved and saved, before tasks are generated.
- **No feature flag.** The current memory subsystem is replaced atomically. Rollback = revert the binary.
- Feature parity check at slice 1 end (per `analysis.md` §13 step 6): explicit pass against `analysis_hermes.md` checklist; every Hermes capability either present in AGH or explicitly marked deferred-with-reason.
