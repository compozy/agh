# ADR-003: Per-Workspace Catalog Database

## Status

Accepted

## Date

2026-05-04

## Context

AGH today persists all memory state in a single global SQLite database at `$AGH_HOME/agh.db` (`internal/daemon/boot.go:285-288`, opened during boot). The catalog (`memory_catalog_entries`), operation log (`memory_operation_log`), session metadata, and observability spine all share this DB.

For mem-v2 with three scopes (`global`, `workspace`, `agent` — see ADR-002), keeping everything in one DB creates several problems:

- **Lock contention** under multi-workspace use. SQLite's `BEGIN IMMEDIATE` serializes writers across the entire DB. A workspace A write blocks unrelated workspace B writes.
- **Atomic event-log + catalog updates** (per ADR-001) require both writes in the same transaction. With one global DB, a workspace mutation invalidates global-scope read snapshots.
- **Backup / migration / deletion** at workspace granularity is non-trivial when state is interleaved.
- **Workspace removal cleanup** requires deleting every catalog row matching `workspace_id = X` — surgical and error-prone.

Empirical references:

- **Hermes** uses one global `state.db` (`~/.hermes/state.db`) but with FTS5 + jittered retry (`_execute_write` 15-retry loop, 20-150 ms jitter, `analysis_hermes.md` §2.3). Lock contention is mitigated, not eliminated.
- **OpenClaw** ships **per-agent** SQLite (`~/.openclaw/memory/<agentId>.sqlite`) — partitioning by agent rather than workspace.
- **Codex** uses a single global `state DB` for leases and signals; rollouts are JSONL files outside the DB.

AGH's posture is local-first, observable, and `BEGIN IMMEDIATE` write helpers (Hermes pattern). Partitioning by workspace makes both write throughput and operational hygiene better.

## Decision

Adopt **two-DB layout** partitioned by scope ownership:

- **`$AGH_HOME/agh.db` (global DB)** — owns:
  - Global-scope memory rows (`memory_catalog_entries WHERE scope_owner = 'global'`)
  - Global-scope event log (`memory_events WHERE scope_owner = 'global'`)
  - Cross-workspace tables: sessions index, observability spine, agent identity registry, agent-global memory rows (`scope_owner = 'agent' AND agent_tier = 'global'`)
  - Agent-global event log

- **`<workspace>/.agh/agh.db` (per-workspace DB)** — created lazily on first workspace-scope write. Owns:
  - Workspace-scope memory rows (`scope_owner = 'workspace'`)
  - Agent-workspace memory rows (`scope_owner = 'agent' AND agent_tier = 'workspace'`)
  - Workspace event log (workspace + agent-workspace operations)
  - Workspace recall signals, consolidation runs, dreaming history

Both DBs share the same schema (numbered migrations registry, ADR-008 forthcoming on schema migration). Both use the Hermes-style `_execute_write` helper with `BEGIN IMMEDIATE` + jittered retry. WAL mode + periodic checkpoint.

A **workspace-id resolver** maps `workspace_id` (UUID, ADR-004) → `<workspace>/.agh/agh.db` path. The composition root (`internal/daemon/boot.go`) opens DB handles lazily; the in-process registry caches connections per workspace_id.

## Alternatives Considered

### Alternative 1: Single global DB (current AGH shape)

- **Description**: Keep `$AGH_HOME/agh.db` as the only catalog DB. Add `workspace_id` and `agent_name` columns; query with `WHERE workspace_id = ?`.
- **Pros**: Simpler. One DB handle. One migration path. One backup target.
- **Cons**: Lock contention scales with active workspace count. Atomic event + catalog updates lock the whole DB. Workspace deletion = surgical row removal. Backup granularity is all-or-nothing.
- **Why rejected**: AGH targets long-running daemons with many concurrent workspaces. Lock contention is real, not theoretical, under that load profile. `BEGIN IMMEDIATE` per workspace gives substantially better write throughput.

### Alternative 2: Per-agent DB (OpenClaw shape)

- **Description**: Partition by agent (`~/.openclaw/memory/<agentId>.sqlite`).
- **Pros**: Maximum partition granularity.
- **Cons**: Workspace-scope memory has no natural home (which agent owns it?). Cross-agent recall requires multi-DB joins. AGH's three-scope model puts agent **inside** workspace via two-tier (ADR-002), not above it.
- **Why rejected**: Doesn't compose with ADR-002's three-scope model.

### Alternative 3: One DB per scope (3 DBs)

- **Description**: Three DB files: `global.db`, `<workspace>/.agh/workspace.db`, `<workspace>/.agh/agent.db`.
- **Pros**: Maximum scope separation.
- **Cons**: Workspace + agent rows almost always update together (write to workspace + dreaming worker promotes a row to agent-workspace tier). Splitting them across DBs forces two-phase commits or eventual consistency. Operationally more complex.
- **Why rejected**: Workspace + agent state are tightly coupled in operations; sharing a workspace DB is the natural boundary.

## Consequences

### Positive

- Workspace write throughput scales with workspace count (no global lock contention for workspace-local mutations).
- Atomic event log + catalog row update happens in one `BEGIN IMMEDIATE` per workspace (`<workspace>/.agh/agh.db`).
- Workspace deletion = `rm -rf <workspace>/.agh/` removes all workspace + agent-workspace state in one operation.
- Backup granularity is per-workspace: `tar <workspace>/.agh/` ships all workspace state.
- Global state isolation: workspace-scope mutations never invalidate global-scope read snapshots.
- Symmetric with the filesystem layout (Markdown files are co-located with their owning DB).

### Negative

- Two DB handles to manage. Mitigated by the `internal/store` shared helpers and a per-workspace connection cache.
- Cross-DB queries (e.g., "list all my feedback memories across all workspaces") require iterating per-workspace DBs. Mitigated by the global DB owning a `workspace_id_index` for cross-workspace observability queries.
- Migration runs per DB. Mitigated by the migration registry being shared and the bootstrap routine running migrations on every DB it opens.

### Risks

- Inconsistent migration state between global and per-workspace DBs (e.g., daemon upgraded but workspace DB not yet migrated because the workspace hasn't been touched since upgrade). Mitigation: every DB-open path runs migrations to head before any read or write; daemon refuses to operate on a DB whose schema is ahead of the daemon's binary version (rollback safety).
- Connection cache leaks if a workspace is moved or deleted mid-session. Mitigation: `workspace_id` resolver invalidates cached handles on workspace removal; `OnWorkspaceDeleted` hook closes the handle.

## Implementation Notes

- New helper: `internal/store/agh.OpenWorkspace(ctx, workspace_id)` returns `*sql.DB` for `<workspace>/.agh/agh.db`. Lazy-create on first call. Caches by `workspace_id`.
- Migration registry (`internal/store/migrations/`) gets a per-DB driver: applies migrations to whichever DB is being opened.
- `internal/store/globaldb` (existing) keeps owning the global DB. New: `internal/store/workspacedb` owns workspace DB schema + helpers.
- Shared helpers: `store.ExecuteWrite(ctx, db, fn)` wraps `BEGIN IMMEDIATE` + 15-retry jittered loop + WAL checkpoint. Used by both DBs.
- Cross-DB observability: global DB owns a `workspace_index` table mapping `workspace_id → realpath, last_seen_at`; updated whenever a workspace DB is opened.
