# ADR-001: Hybrid Escopado as Memory Source-of-Truth Model

## Status

Accepted

## Date

2026-05-04

## Context

AGH memory v2 must pick one of three structural shapes for "what is the source of truth for curated memory":

1. **Pure event-sourced log** ŌĆö `memory_events` append-only is authoritative; Markdown files and SQLite catalog are projections rendered from the log. Codex peer-review (`gpt-5.5 xhigh`) recommended this in the first synthesis pass.
2. **Markdown authoritative** ŌĆö files on disk are the truth; SQLite catalog is a derived index (current AGH shape).
3. **Hybrid escopado por tipo** ŌĆö different layers have different authorities: curated semantic memory = Markdown; signals / audit / operations = event log; FTS5 / scoring = derived catalog.

The choice cascades into schema design, controller responsibilities, projection rules, replay semantics, agent-edit ergonomics, and the "what does an operator see when they `cat MEMORY.md`?" UX.

Empirical reference: Hermes, Claude Code, and Codex were each verified against source code (`analysis_hermes.md`, `analysis_claude-code.md`, `analysis_codex.md`). **None of the three uses pure event-sourced for curated memory.** All three are hybrid:

- **Hermes** ŌĆö `~/.hermes/memories/{MEMORY.md, USER.md}` is **authoritative Markdown**; `~/.hermes/state.db` is SQLite for session memory + FTS5 indexes.
- **Claude Code** ŌĆö `memdir/` and `CLAUDE.md` are **authoritative Markdown**; recall is direct file read + Sonnet ranker; no separate catalog DB for curated memory.
- **Codex** ŌĆö `~/.codex/memories/` is a **git repo of authoritative Markdown** (handbook + summary + skills + rollout summaries); `~/.codex/sessions/...rollout-*.jsonl` is **authoritative JSONL** for transcripts; SQLite is `state DB` for leases / signals only.

## Decision

Adopt **hybrid escopado por tipo**:

- **Curated semantic memory** (`user`/`feedback`/`project`/`reference` types) ŌĆö **Markdown files on disk are authoritative.** Layout under `<scope>/memory/{MEMORY.md, <type>_<slug>.md, daily/, _system/}`.
- **Audit / observability** ŌĆö `memory_events` append-only table is authoritative for canonical dotted events only. It records controller outcomes, recall traces, daily retention, provider lifecycle, and other observable transitions.
- **Recovery WAL** ŌĆö `memory_decisions` is authoritative for crash replay of curated-file mutations. Decision-before-mutation persists the bytes needed to re-apply an ADD/UPDATE/DELETE deterministically.
- **Live signal / runtime state** ŌĆö `memory_recall_signals` is authoritative for recall feedback state; `memory_consolidations` is authoritative for dreaming gate/runtime state.
- **Derived catalog** ŌĆö `memory_catalog_entries` + FTS5 + trigram are a projection of the Markdown files. Read-only from controller perspective (writes happen via reindex on file mutation). Provides search, ranking, and metadata queries.

The Markdown files are git-friendly, human-editable, agent-readable. The event log gives audit / observability / replay. The catalog gives fast retrieval. Each layer has one authority; cross-layer consistency comes from the controller mediating writes.

## Alternatives Considered

### Alternative 1: Pure event-sourced log

- **Description**: `memory_events` is the single source of truth. Markdown files and catalog are projections rendered from event replay. Human edits enter as `human_edit` events via git hook or CLI.
- **Pros**: Replay, rollback, audit, causal ordering for free. Aligns with SD-009 (strong observability) and the existing operation log.
- **Cons**: More projection code. Human-edited Markdown becomes either an input event or a projection (not both) ŌĆö needs explicit reconcile semantics. None of Hermes / Claude Code / Codex does this for curated memory.
- **Why rejected**: Empirical evidence from the three best systems shows hybrid wins. Pure event-sourced would put AGH on a path no production system has validated for *curated* memory. Codex peer-review proposed this from architectural-purity reasoning, not from competitor evidence.

### Alternative 2: Markdown authoritative + catalog derived (current AGH)

- **Description**: Keep current shape. Files are truth; catalog is derived; `memory_operation_log` is audit-secondary, not authority.
- **Pros**: Smaller blast radius. Git-diffable. Human and agent both read files directly. No new persistence model.
- **Cons**: Split-brain documented in Claude Code (`analysis_claude-code.md` ┬¦12) and AGH today (`analysis_agh-current.md` ┬¦16) ŌĆö files / catalog / operation log can disagree after crash, edit, or reindex. Replay / rollback become add-on features, not native capabilities. Loses the audit-first posture (SD-009).
- **Why rejected**: AGH's existing operation log is too good to relegate to "secondary." Promoting it to first-class authority (for operations, not for curated content) gives AGH something Claude Code does not have: structured observability of every memory mutation.

## Consequences

### Positive

- Authority for each layer is unambiguous ŌĆö no silent split-brain.
- Markdown stays git-friendly + human-editable + agent-readable.
- Operation log is the audit + observability + replay surface (SD-009 alignment).
- Catalog is a pure projection ŌĆö can be rebuilt from Markdown + event log without data loss.
- Aligns AGH with the production-validated shape of Hermes, Claude Code, and Codex.

### Negative

- Four persistence layers (files / event log / decision WAL / catalog, plus live-signal tables) means each layer must have a single clear owner: rebuild catalog from files; replay curated writes from `memory_decisions`; recompute observability from runtime events only where needed.
- Human-edited curated files do not bypass the controller. The file watcher turns external add/update/delete into repair-only `Origin=file` decisions so the WAL and event log catch up deterministically.

### Risks

- Operators who edit Markdown files outside AGH's CLI may create temporary file/catalog drift. Mitigation: the daemon-owned file watcher observes curated files only, routes the change through `Controller.Decide()` with `actor_kind=file`, and reindexes the resulting committed state. There is no separate non-canonical watcher event type.
- Event log growth ŌĆö append-only logs need rotation. Mitigation: per-scope event log, with archival to `_system/archive/` after configurable retention (matches daily-log pattern).

## Implementation Notes

- Migration: Slice 1 lifts `memory_operation_log` to the authoritative observability surface; rename to `memory_events` with the canonical dotted event taxonomy defined in the TechSpec.
- Controller pattern: `Controller.Decide(ctx, candidate) (Decision, error)` ŌĆö every mutation flows through here, regardless of caller (CLI / HTTP / UDS / native tool / extractor / dreaming worker).
- Catalog stays read-only at the controller boundary ŌĆö controller writes to disk + decision WAL + event log atomically; catalog re-projects on the next read or via explicit `agh memory reindex`.
- The `BEGIN IMMEDIATE` write helper (Hermes pattern, ADR-003) wraps every event-log + catalog update so they commit together.

## Post-Approval Refinement (Codex peer-review B3 incorporation, 2026-05-04)

The original "Decision-before-mutation" invariant (┬¦Consequences / Positive #4) implied that the event log alone could reconstruct curated state after a crash. Codex peer-review B3 surfaced that `memory_decisions` as originally drafted (storing only `candidate_hash` + `prior_content`) lacks the post-mutation content needed to deterministically replay an ADD or UPDATE that crashed before the file landed.

**Refinement adopted:** `memory_decisions` rows now store the full replay material ŌĆö `post_content`, `post_content_hash`, `target_filename`, `frontmatter`, `idempotency_key` ŌĆö so that:

1. Decision-before-mutation persists *enough bytes* to re-derive the file state on boot.
2. Daemon-boot replay walks `memory_decisions WHERE applied_at IS NULL ORDER BY decided_at`, applies each row idempotently (skip when on-disk file already has matching `post_content_hash`), and the curated tree converges.
3. Replay is the QA-proof gate: `TestStore_ReplayUnappliedDecisions_ReconstructsState` asserts replay produces the same state as crash-free execution.

The hybrid escopado model is preserved: **Markdown remains authoritative for content semantics; the event log + decision WAL together are authoritative for operations and recovery.** The TechSpec ┬¦"Safety Invariants" #2 is updated accordingly.
