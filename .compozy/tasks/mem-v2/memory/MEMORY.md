## Shared Decisions

- Memory frontmatter uses canonical YAML `agent` for the agent name while JSON/API payloads use `agent_name`; no legacy YAML `agent_name` compatibility field remains in the contract.
- `contract.Header.Provenance` is optional (`*Provenance`) so existing header/list responses remain valid until later write-controller tasks populate provenance.
- Future Memory v2 DB writers should use `store.ExecuteWrite` for serialized `BEGIN IMMEDIATE` writes instead of adding ad hoc write-transaction helpers.
- Future workspace DB consumers should open memory workspace databases through `internal/store/workspacedb`, which resolves the durable `workspace_id` and opens `<workspace>/.agh/agh.db`.
- Future controller/WAL tasks must persist `post_content` and `post_content_hash` for add/update `memory_decisions`; replay uses those fields as the durable authority and does not reconstruct content from derived indexes.
- Future controller/extractor/dream tasks should load prompt assets from `internal/memory/prompts` by explicit `Name*` and `VersionV1`; do not embed policy prose inline.
- Future write-controller tasks should call `scan.Candidate`/`scan.Content` before persistence and persist only redaction-safe scan rule IDs/reasons from `Result.RuleHits` or `Result.Reason`.
- Future Memory v2 write callers should enter through `Store.ProposeWrite`, `Store.ProposeDelete`, or `Store.ApplyDecision`; `Store.Write`/`Store.Delete` are raw storage primitives and must not become new public runtime bypasses.
- Decision idempotency keys include candidate hash, op, targets, target filename, post hash, frontmatter hash, and prompt version; keep this composition stable unless a later contract task intentionally changes WAL semantics.
- Scanner rejection telemetry now stores redaction-safe rule traces with sample byte counts in `RuleHit.Details`; do not persist raw rejected content in audit reasons or event summaries.
- Future recall consumers should call `Store.Recall` / `internal/memory/recall` and consume `memcontract.Packaged`; do not add independent ranking or packaging logic in prompt, provider, HTTP/UDS, CLI, or web surfaces.
- Recall identity shadowing is `(type, slug)` with deeper scope precedence: global < workspace < agent-global < agent-workspace. Blocks are packaged least-specific first after top-K selection.
- Recall query triviality is intentional: fewer than three meaningful lexical tokens skip recall unless the query is a sufficiently long non-ASCII token. Tests expecting recall should use non-trivial queries.
- `memory_recall_signals` now carries live dreaming-ready fields (`recall_score`, freshness barrier, promotion columns, score-update barrier) and is updated on every non-empty recall without bubbling write failures to callers.
- Daemon composition registers the bundled local provider via `internal/extension.MemoryProviderRegistry` and adapts `memory.Store` through `internal/memory/provider/local/memstore`; provider implementations must stay free of controller, recall, and concrete store internals.
- Provider-name, provider-tool, and reserved built-in-tool collisions must fail registration without mutating active provider selection. Collision telemetry is the global event summary type `memory.provider.collision`.
- Future prompt/situation/daemon consumers should use `memory.SnapshotService` through `Assembler.PromptStartupSection` or equivalent composition seams; do not reload indexes or re-render memory blocks independently.
- `SnapshotService.InvalidateNextBoot` is intentionally generation-only: it affects subsequent captures and must not mutate active session prompt text.
- Future sub-agent wiring should pass the parent `FrozenSnapshot` as `ParentSnapshot`; child sessions must inherit that rendered section with `ControllerMode=read_only` instead of resolving child-private memory.
- Future recall prompt consumers should use `memory.RenderRecallPromptSection` for `memcontract.Packaged` output to avoid duplicated entry caps, headers, scope labels, and freshness formatting.
- Future observe/health/public route tasks should use `Store.ListMemoryEventSummaries` and the observer `MemoryEventSource` seam instead of re-querying `memory_events` independently.
- Per-workspace observability DBs are read authorities only when `<workspace>/.agh/agh.db` already exists; read-only health/observe paths must not create empty workspace DBs.
- Any SSE/log-facing surface that may carry prompt or memory payloads must call `sse.ScrubMemoryContextString` or `sse.ScrubMemoryContextBytes` before emitting data.
- Future extractor consumers should use the `session.message_persisted` hook payload and must not tail session DBs or stream deltas to discover extraction triggers.
- Extractor outputs must enter through `Store.ProposeCandidate` / `ProposalSink.ProposeCandidate`; direct writes from extractor code are a controller bypass and should be rejected in review.
- Extractor failure files live under `_system/extractor/failures/`; prompt/recall/catalog code must continue treating `_system` as non-injectable.
- Future dreaming/config/daemon tasks should preserve the Time -> Sessions -> Lock -> Signal order. `ShouldRun` stays side-effect free for Time/Sessions; `Run` owns the lock-protected signal gate and anti-thrash lock stamp on empty candidate sets.
- Dreaming promotions use `Store.ProposeCandidate` with `OriginDreaming`; the controller intentionally bypasses multi-source surface ambiguity for dreaming-origin syntheses while preserving exact, filename, and entity-slot rules.
- Dreaming runtime artifacts live under `_system/dreaming/`; failures live under `_system/dream/failures/`. Neither path should be exposed to prompt-facing recall/index surfaces.
- `consolidation.NewSessionSpawner` now defaults generic dream config to the dedicated `dreaming-curator` agent name, while still honoring explicit non-default dream agents until task 13 finalizes config semantics.
- Daemon composition owns `internal/memory/extractor.Runtime` lifecycle and calls `Drain`/`Close` during shutdown. `Close` must remain safe for boot-only lifecycles where the extractor consumer was never started.
- Provider-backed prompt snapshots now pass both `workspace_id` and `workspace_root`; the bundled local provider needs the root to resolve workspace/agent workspace MEMORY.md files while preserving workspace_id as the durable catalog identity.
- Session ledger projection must stay forensic-only: `internal/sessions/ledger.Materializer` reads `events.db`, writes `ledger.jsonl`, and must not become a live session writer or alternate recall/source authority.
- Ledger idempotence is content/checksum based: identical reruns skip, while an existing different `ledger.jsonl` returns `ErrLedgerExists` to preserve forensic immutability.
- Ledger paths are `<sessions>/<workspace_id>/<session_id>/ledger.jsonl` for workspace-bound sessions and `<sessions>/_unbound/<session_id>/ledger.jsonl` for unbound session records; unsafe path segments are rejected before opening the event store.
- Memory v2 config lives in the existing `[memory]` tree; do not add a `[memory.v2] enabled` flag or a parallel compatibility config namespace in later tasks.
- `memory.dream.agent` defaults to `dreaming-curator` and no longer inherits `[defaults].agent`; later daemon/docs/UI work should preserve that dedicated agent identity unless a task explicitly changes the config contract.
- `memory.workspace.toml_path` is informational/read-facing and validation-locked to `<workspace>/.agh/workspace.toml`; tool/settings writers should not offer it as a mutable path.
- Later public contract, CLI, web, and docs tasks should consume the generated settings contract shape rather than inventing separate Memory v2 config models.
- Public Memory v2 contracts live in `internal/api/contract/memory.go`; route handlers in task 16 and CLI/native-tool adapters in later tasks should consume these DTOs rather than re-declaring transport-local shapes.
- Public selectors use `workspace_id` and `agent_name`; do not reintroduce old path-style `workspace` fields or legacy two-scope-only payloads in HTTP, UDS, CLI, web, or docs work.
- Public decision/LLM trace payloads intentionally omit replay/raw fields such as `post_content`, `prior_content`, and raw LLM responses. Those remain durable WAL/internal audit material only.
- The OpenAPI source is codegen-ready for the v2 route surface before task 16 implements every route. Generated public site API reference currently filters to registered HTTP/UDS routes so docs stay truthful until route parity lands.
- Web generated consumers should continue deriving Memory v2 shapes through `web/src/lib/api-contract.ts` (`OperationResponse`, `OperationRequestBody`, `OperationQuery`, `OperationPath`) rather than hand-maintained DTO mirrors.
- CLI, native-tool, web, and docs work after task 16 should target the final route family (`POST /api/memory/search`, `POST /api/memory`, `PATCH /api/memory/:filename`, `POST /api/memory/dreams/trigger`, etc.) and must not reintroduce legacy `GET /api/memory/search`, `PUT /api/memory/:filename`, or `POST /api/memory/consolidate`.
- Transport parity tests must not import one transport package from another; use the neutral `internal/api/testutil` Memory v2 route-key contract to avoid package boundary violations.
- Native/builtin Memory v2 tools are now the final Slice 1 set: `agh__memory_list`, `agh__memory_show`, `agh__memory_search`, `agh__memory_propose`, and `agh__memory_note`. Do not reintroduce `agh__memory_read` or `agh__memory_history`.
- Native memory proposals and ad-hoc notes must route through `Store.ProposeWrite` / `Store.ProposeDelete` with `OriginTool`; native tools should not call raw write/delete helpers directly.
- Extension Host API recall should use the active/default `MemoryProvider` first and fall back to `Store.Recall` only for absent registry/provider or `ErrNotImplemented`; avoid adding a second manual recall scorer.
- `agh__memory_note` currently uses a generated controller-backed ad-hoc filename because proposal validation forbids slash-separated `_system/ad_hoc` paths. Do not bypass the controller to force `_system` materialization.
- Web knowledge code addresses memories by `KnowledgeSelector { scope, workspaceId?, agentName?, agentTier? }`; later web/docs/QA tasks should reuse this selector instead of reintroducing positional `(scope, workspaceID)` parameters or filename-prefix scope derivation.
- Web mutations (`useDeleteMemory`, `useEditMemory`) are controller-backed: the UI keeps the failed mutation marker visible until the user changes selection/scope/search to surface daemon errors inline. Don't auto-clear on settle.
- `useMemorySearch` is the only path the Knowledge page uses for explicit recall; do not reintroduce client-side text filtering as a substitute for `POST /api/memory/search`.
- Route page components in `web/src/routes/**` should be exported as named exports alongside `Route = createFileRoute(...)` so vitest specs can import the component directly instead of casting `Route as any`. The `vi.mock("@tanstack/react-router")` factory still substitutes `createFileRoute` at runtime, so adding the named export does not affect router behavior.
- Web settings surfaces should source every config field from generated `OperationResponse`/`OperationRequestBody` types and only render mutable controls for fields the backend actually accepts. Daemon-managed paths and validation-locked surfaces (e.g. `inbox_path`, `dlq_path`, `ledger_root`, `unbound_partition`, `daily.rotate_format`, `daily.archive_path`, `workspace.toml_path`, `controller.policy.allow_origins`) ship as read-only inputs rather than tempting operators to mutate them.
- The Memory v2 dreaming action is a `dream trigger` (renamed from `consolidate`); web copy and hook handler names should follow that hard cut (`handleTriggerDream`, `isTriggeringDream`, "Dream triggered" / "Dream not triggered" messages, `dream-trigger` test id).
- `SettingsDecimalInput` lives in `web/src/systems/settings/components/` for bounded fractional fields (recall weights, dream gate scores, dream scoring weights). Use it instead of forcing decimals through `SettingsNumberInput` (integer-only) or raw `<Input type="number">`.
- Future session-inspector consumers must source memory state from `useSessionLedger` (via the generated `getMemorySessionLedger` operation) and feed it through `InspectorMemoryState`. Do not recreate `InspectorMemoryDoc[]` or render arbitrary `event.payload` JSON; only contract-approved redaction-safe ledger event fields (`sequence`, `event_type`, `emitted_at`) should appear. Treat `SessionLedgerUnavailableError` (404) as the truthful "not yet materialized" empty state.

## Shared Learnings

- Generated OpenAPI, the TypeScript SDK contract, and `web/src/generated/agh-openapi.d.ts` now expose the Memory v2 `agent` scope enum from the contract package; web knowledge formatting has a minimal agent-scope label/order fallback.
- The contract package currently imports only standard-library packages (`context`, `encoding/json`, `errors`, `fmt`, `log/slog`, `strings`, `time`).
- The global DB now has `memory_events`; observability should read memory summaries from this table rather than legacy `memory_operation_log`.
- `make verify` initially failed for task 05 because `Controller.Decide` exceeded `funlen`; extracting preparation/rejection/write helpers fixed the lint issue without changing rule order.
- `internal/memory` coverage can sit close to the 80% floor; focused WAL tests should include error-path behavior, not only happy-path decisions.
- `scripts/check-test-conventions.py` is absent in this repo even though `agh-test-conventions` references it; use focused Go tests, race tests, coverage, and `make verify` as the available guardrails until the script is restored.
- CJK recall should be exercised through the trigram chunk FTS path; the current fixture proves Japanese substring retrieval via `Store.Recall`.
- The bundled local provider package is intentionally contract-clean; concrete Store integration lives in `provider/local/memstore`. Keep this split when task 19 wires daemon composition.
- Snapshot freshness/cap behavior is now covered with real memory documents and deterministic file mtimes; keep future tests behavior-based rather than snapshot-string exact.
- Memory event summaries intentionally expose empty `ActorID` for `memory_events`; the current Memory v2 event schema has `actor_kind` but no `actor_id`.
- `Observer.QueryEvents` currently fans in memory DBs only for non-session-scoped queries. Later public route tasks should add explicit workspace-aware session fan-in only when the transport contract can pass the workspace authority safely.
- `<memory-context>` may arrive as literal XML-like tags or as JSON-escaped `\u003c...\u003e` in SSE payloads; tests should cover both encodings.
- `internal/session` async hook tests can be timing-sensitive under heavy package-load runs. If a full affected-package run times out in an existing async streaming test, rerun the targeted test before changing production behavior.
- Dreaming syntheses are multi-source by design; regular controller surface ambiguity would turn them into noop decisions unless `OriginDreaming` keeps the consolidation path distinct.
- `internal/memory` coverage is exactly 80.0% after task 11; future memory tasks should add meaningful behavior/error tests with any new code to avoid falling below the package floor.
- The new `internal/sessions/ledger` package is coverage-sensitive because it is small; keep behavior coverage for idempotence, unsafe paths, unbound layout, and lifecycle seam above the 80% floor.
- Expanding Memory v2 settings payloads requires generated web and SDK consumers plus web fixtures to move in the same change; partial backend-only contract changes fail typecheck quickly.
- `internal/config` is coverage-sensitive after default/validation expansion; keep meaningful validation branch tests when adding new config keys.
- Memory contract changes can affect web Knowledge consumers even before the dedicated web UI tasks; keep generated OpenAPI types, fixtures, handlers, and route hooks in sync with route/response renames.
- Spec tests should assert hard-cut legacy Memory operations are absent, not just that new operations exist, because old verbs can otherwise remain in OpenAPI unnoticed.
- Avoid parallel `make bun-*` invocations; concurrent Mage runs can race on `mage_output_file.go` and produce false failures. Sequential reruns are the reliable validation path.
- Avoid parallel Mage-backed Make targets such as `make codegen-check` with other Make gates; task 16 reproduced the same `mage_output_file.go` race and the sequential rerun passed.
- `internal/extension` broad `-race` runs should match the gate profile with `-parallel=4`; default higher parallelism can saturate SQLite migrations and produce context-deadline false failures on macOS.
- Task 19 verification showed `internal/daemon -race -parallel=4` is the fastest useful preflight for daemon wiring changes; it caught lifecycle/order bugs before the final `make verify`.
- Avoid assigning a typed nil concrete provider into an interface shutdown field. It makes `target != nil` true and turns disabled-memory shutdown into a bogus provider call.
- Web route page components must use `export function FooPage() { ... }` for direct vitest imports — `export { FooPage };` after the function declaration fails rolldown's parse pass during `make web-build` even though tsc is happy. Confirmed during task 22 build verification.
- Oxlint's `no-useless-fallback-in-spread` flags `...(maybe ?? {})` patterns; spread the optional with `...maybe?.field` instead of substituting an empty-object fallback.
- The Slice 1 hard-cut left some current-tense identifiers in place by design: `memory.global_dir` is still a real `[memory]` key (`internal/config/config.go:173`), `memory.consolidated` is still the automation `ActivationEnvelope.Kind` emitted by `internal/automation/trigger.go:1043`, and `memory_consolidations` is still the dreaming runtime catalog table (`internal/memory/catalog.go:340`). Future docs/QA tasks should treat these as truthful runtime surfaces, not legacy bleed-through, and reserve the negative-context wording for `[memory.v2]`/`memory_read`/`memory_history`/`agh memory read`/`agh memory consolidate`/`PUT /api/memory*`/`GET /api/memory/search` only.
- Site `bun run typecheck`, `bun run test`, and `bun run build` each implicitly run `bun run generate:openapi`, `bun run source:generate`, and `bun run content:generate` via `pretypecheck`/`pretest`/`prebuild`. Generated `runtime/api-reference/*.mdx`, `.source/`, and `.velite/` are build artifacts — never commit them.
- `runtime-docs-truth.test.ts` deliberately contains the literal forbidden patterns (e.g. `[memory.v2]`, `memory_read`, `memory_history`) inside `expect(...).not.toMatch(...)` regex bodies. Any doc-truth scan over `packages/site/lib` will surface those occurrences; treat them as test infrastructure, not docs leakage.
- `make cli-docs` regenerates every Fumadocs CLI page in unpadded markdown-table form; `make verify` re-runs `bun run format` (`bunx oxfmt`) which re-pads the tables to the committed canonical form. After running `make cli-docs`, always run `make verify` (or at least the `bun-lint` stage) before assessing drift — only diffs that survive `oxfmt` represent real regenerator changes.
- Review round 001 proved two Memory v2 invariants future changes must preserve: recall signal writes stay off the request path through a bounded `SignalRecorder`, and explicit root memory tool writes suppress extractor work for the same turn while sub-agent tool writes fail closed with `memory_subagent_write_denied`.
