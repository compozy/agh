# BUG-008: Transcript query could reuse a recorder while its database was closing

## Status

Fixed

## Severity / Priority

- Severity: High
- Priority: P1

## Scope

- Scenario: ACP mock crash and invalid-frame runtime E2E.
- Surface: Session transcript/query recorder lifecycle.
- CLI/Web parity affected: Indirectly, transcript read surfaces depend on this recorder.
- Browser tool: Not involved.
- Live provider/LLM involved: No.
- Environment: Runtime E2E final-postfix QA run.

## Behavioral Impact

Crash-path transcript reads could observe `database is closed` after an agent process failure, preventing the runtime from projecting the failure and transcript state operators need for diagnosis.

## Reproduction

1. Run runtime E2E crash or invalid-frame ACP mock scenarios.
2. Query the transcript shortly after the process failure.
3. Observe transcript query failure from a recorder whose SQLite handle is already closing.

## Expected

Transcript queries wait for active recorder finalization and reopen a stable on-disk database when needed.

## Actual

`openQueryRecorder` could return an active recorder while shutdown/finalization was racing, leading to a closed database error.

## Root Cause

The session query path did not wait for finalization before using a recorder that was transitioning from active process ownership to on-disk query ownership.

## Fix

`internal/session/query.go` now waits for finalization before returning a query recorder and reopens the on-disk session database after process teardown.
