# BUG-009: Bridge ingress returned before the routed prompt became observable

## Status

Fixed

## Severity / Priority

- Severity: High
- Priority: P1

## Scope

- Scenario: Telegram bridge ingress runtime and browser flows.
- Surface: Extension Host API bridge ingress, session prompt availability.
- CLI/Web parity affected: Yes, bridge route counts and delivery health depend on the routed session prompt.
- Browser tool: Playwright bridge E2E.
- Live provider/LLM involved: No.
- Environment: Runtime and web E2E final-postfix QA run.

## Behavioral Impact

Bridge ingress could return before the session prompt was durably available or drained, causing route/health assertions and transcript checks to race the runtime.

## Reproduction

1. Run the bridge ingress E2E scenario against the Telegram reference extension.
2. Trigger a bridge ingress update that should reuse/create a route.
3. Observe intermittent timeout waiting for the routed prompt/transcript state.

## Expected

Host API bridge ingress returns only after prompt submission has crossed the runtime availability boundary.

## Actual

The API could return after enqueue/submission while the prompt was still not observable to the route/transcript checks.

## Root Cause

`submitBridgePrompt` did not wait for prompt availability/drain after submitting the bridge prompt into the session runtime.

## Fix

`internal/extension/host_api_bridges.go` now waits for prompt availability after submission so the bridge Host API response aligns with the observable runtime state.
