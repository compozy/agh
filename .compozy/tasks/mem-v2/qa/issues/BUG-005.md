# BUG-005: Memory extractor failed queued post-stop turns for provider-backed sessions

## Status

Fixed

## Severity

P1 / High

## Scenario

TC-SCEN-002 and TC-INT-004: provider-backed sessions must preserve forensic ledger and Memory v2 extraction behavior through normal stop boundaries. Queued extractor turns created from durable `session.message_persisted` events must not fail only because the parent session was stopped after the provider response completed.

## Root Cause

The Memory v2 extractor runtime intentionally queues durable `session.message_persisted` events asynchronously. The daemon extractor implementation spawns an internal child session for extraction using the regular `session.Manager.Spawn` path. That path required the parent session to be `active` for all spawn roles, so a final queued extractor range could fail after a normal provider-session stop even though the work originated from durable parent transcript events and still needed the parent lineage.

`AutoStopOnParent` also did not match extractor semantics: the extractor child has its own TTL and explicit stop path, and should finish draining a captured turn even when the parent has already completed.

## Fix

- `session.SpawnOpts` now has an internal `AllowStoppedParent` escape hatch restricted to the `memory-extractor` spawn role and rejected when combined with `AutoStopOnParent`.
- `Manager.Spawn` still rejects stopped parents for normal agent/API spawned sessions.
- The daemon memory extractor now uses `session.SpawnRoleMemoryExtractor`, enables `AllowStoppedParent`, and does not request `AutoStopOnParent`.
- Regression coverage verifies normal stopped-parent spawns still fail while daemon memory-extractor spawns can preserve lineage from a stopped parent.
