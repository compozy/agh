# BUG-007: Permission fixtures used deny-all while expecting operator approval

## Status

Fixed

## Severity / Priority

- Severity: Medium
- Priority: P2

## Scope

- Scenario: Permission approval parity and browser session lifecycle.
- Surface: ACP mock fixtures and browser E2E session approval.
- CLI/Web parity affected: Yes, approval flows must be deterministic across HTTP/UDS/Web tests.
- Browser tool: Playwright session-onboarding flow.
- Live provider/LLM involved: No.
- Environment: Task 26 final-postfix QA run and browser daemon-served E2E lane.

## Behavioral Impact

Fixtures that expected an `allow-once` or `allow-always` operator decision were configured with `deny-all`, which auto-rejected permission requests before the UI approval could reach the active process.

## Reproduction

1. Run `make test-e2e-web`.
2. In `web/e2e/session-onboarding.spec.ts`, submit the browser lifecycle prompt.
3. Click "Allow Once" after the permission prompt appears.
4. The approval endpoint returns `409` because the pending permission was already resolved by deny-all auto-rejection.

## Expected

Fixtures that assert an operator approval decision use an interactive permission mode, leaving the request pending until the test/operator responds.

## Actual

The fixture auto-rejected, leaving the browser with a stale visible prompt and no pending permission to approve.

## Root Cause

`browser_session_lifecycle_fixture.json` declared `permissions: deny-all` while its permission step declared `expect_decision: allow-once`. Earlier runtime permission fixtures had the same mismatch for `allow-always` flows.

## Fix

Changed approval-expecting fixtures to `approve-reads`, including the browser lifecycle fixture and the permission/runtime fixture set touched during task 26.
