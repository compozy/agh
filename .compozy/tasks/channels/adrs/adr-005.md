# ADR-005: Hybrid Channel Substrate with Extension-Based Platform Adapters

## Status

Accepted

## Date

2026-04-10

## Context

The initial channel-adapters design assumed a 100% extension-based model where every messaging concern, including routing, delivery, and credential access, lived entirely inside extension subprocesses. Follow-up review against AGH's extension protocol, automation boundaries, and competitor architectures showed that channel integrations are load-bearing infrastructure rather than a thin integration edge.

OpenFang, GoClaw, and Hermes all keep channel routing and delivery as core runtime concerns. OpenClaw is the closest to a plugin-friendly model, but even there the Gateway owns the channel matrix and health surface. In AGH, the risk of a pure extension model is higher because the extension protocol would become responsible for too many invariants at once: routing identity, delivery semantics, backpressure, credential mediation, and automation egress.

## Decision

AGH will adopt a hybrid architecture for channels:

1. The daemon gains a small core channel substrate in `internal/channels/`.
2. The core substrate owns registry, routing, delivery-target resolution, credential binding, and the negotiated delivery-stream contract.
3. Platform-specific adapters (Telegram, Slack, Discord, etc.) remain extension subprocesses.
4. Extensions are responsible for platform transport, authentication handshakes with the upstream platform, formatting, media handling, and platform-specific UX.

This keeps channel-specific SDK and protocol code out of the daemon while moving system-level invariants into the core runtime where they can be validated and observed consistently.

## Alternatives Considered

### Alternative 1: 100% Extension-Based Channels

- **Description**: Keep all channel concerns in extension subprocesses and expose only generic Host API methods from the daemon.
- **Pros**: Maximal extension independence, no new core package, strong extension-first purity.
- **Cons**: Pushes routing, stream semantics, and credential mediation into an RPC boundary that is not designed to own all of them.
- **Why rejected**: Too much load-bearing behavior would move into protocol glue. Review showed this is both riskier than competitor practice and misaligned with AGH's current extension protocol.

### Alternative 2: Core-Only Channel Subsystem

- **Description**: Build channels as daemon-owned Go packages under a full internal subsystem, with platform adapters compiled into the binary.
- **Pros**: Simplest operational model, fewer RPC edges, easier end-to-end testing.
- **Cons**: Breaks AGH's extension-first strategy and couples platform SDK churn directly to daemon releases.
- **Why rejected**: Over-corrects in the opposite direction. Platform adapters are still a good fit for extension subprocesses.

## Consequences

### Positive

- Keeps load-bearing invariants in the daemon where they can be validated and observed.
- Preserves extension-based platform adapters and language flexibility.
- Aligns better with competitor practice without abandoning AGH's extension architecture.
- Gives automation, CLI, API, and web UI a stable core registry to integrate with.

### Negative

- Adds a new core subsystem that did not exist in the original design.
- Requires a clearer boundary between generic channel governance and platform-specific behavior.
- Increases the amount of daemon-side design work before implementation can begin.

### Risks

- The substrate may grow too large if platform-specific concerns leak into the core.
- Extension authors may still find the delivery contract too rigid if it is poorly scoped.

## Implementation Notes

- `internal/channels/` should stay narrow and governance-focused.
- The first reference adapter remains Telegram, but it now integrates against the core substrate rather than owning its own local routing database.
