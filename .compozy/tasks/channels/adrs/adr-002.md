# ADR-002: Persistent Session Per Chat with Serial Queue

## Status

Superseded by ADR-006

## Date

2026-04-10

## Context

Channel adapters must route incoming messages to ACP sessions. Two decisions are interleaved: (1) how to map a chat/channel identifier to a session, and (2) how to handle concurrent messages in the same chat.

Competitor analysis shows:
- **GoClaw**: SessionKey rotation per day, serial queue per chat
- **Hermes**: Persistent session per chat, ThreadPoolExecutor with 4 workers, serial per chat
- **OpenClaw**: Per-channel queue policy (queue/reject/interrupt)

## Decision

**Session routing**: One ACP session per unique chat identifier (e.g., Telegram `chat_id`, Slack `channel_id`), reused across messages. The extension maintains a local mapping table (`chat_id → session_id`). If the session is stopped or expired, a new one is created automatically. This provides conversation continuity — the agent remembers prior context within the session.

**Concurrency**: Each chat identifier has a serial FIFO queue managed by the extension. If the agent is processing a message, subsequent messages wait in the queue. This prevents race conditions and ensures predictable ordering.

Both decisions are implemented entirely inside the extension subprocess — no daemon-side routing or queuing.

## Alternatives Considered

### Alternative 1: New Session Per Message

- **Description**: Each incoming message creates a fresh ACP session
- **Pros**: Simplest, no state management
- **Cons**: No conversation continuity, agent cannot reference prior messages
- **Why rejected**: Users expect chat-like continuity. Cross-session memory recall is not a substitute for in-context conversation history.

### Alternative 2: Daily Session Rotation (GoClaw-style)

- **Description**: New session created each day, reused within the day
- **Pros**: Bounds session context growth, natural breakpoint
- **Cons**: Arbitrary cutoff, loses context at midnight
- **Why rejected**: Persistent sessions with AGH's existing context compaction (handled by ACP agents) are a better fit.

### Alternative 3: Interrupt Policy (OpenClaw-style)

- **Description**: New message cancels current processing via `sessions/stop`
- **Pros**: Responsive to user intent changes
- **Cons**: Loses work in progress, complex state management, may frustrate users
- **Why rejected**: Serial queue is simpler and safer for the default behavior. Interrupt can be added as a future option.

## Consequences

### Positive

- Conversation continuity within a chat
- Predictable message ordering — no race conditions
- Simple mental model for users and extension authors
- Extension owns all routing state — daemon stays stateless about channels

### Negative

- Long-running agent tasks block subsequent messages in the same chat
- Extension must persist the routing table (chat_id → session_id) to survive restarts
- No built-in way to "cancel" a long-running task from the chat

### Risks

- Session context grows unbounded over long conversations. Mitigation: ACP agents handle their own context compaction. Extension can optionally rotate sessions based on token count or time.

## Implementation Notes

- Routing table stored in a local SQLite database within the extension's data directory
- On extension restart, the routing table is reloaded and existing sessions are verified via `sessions/status`
- Queue is a simple Go channel per chat_id, drained by a single goroutine
