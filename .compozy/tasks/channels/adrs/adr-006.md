# ADR-006: Core-Owned Channel Registry, Scoped Instances, and Policy-Driven Routing

## Status

Accepted

## Date

2026-04-10

## Context

The previous design stored `chat_id -> session_id` mappings inside each extension and treated session reuse as an extension-local concern. That model is too narrow for multi-account channels, threads/topics, group conversations, workspace boundaries, and automation delivery targets.

AGH already moved automation toward explicit `global` and `workspace` scope. Channel integrations need the same clarity. They also need a canonical place to answer operational questions such as: which instances exist, which workspace owns them, what routing policy applies, what delivery target is valid, and which session currently backs a conversation.

## Decision

The daemon will own a persisted channel registry with three primary models:

1. `channel_instances` — one record per configured channel instance, scoped as `global` or `workspace`.
2. `channel_secret_bindings` — explicit bindings from a channel instance to the secrets it may use.
3. `channel_routes` — the canonical mapping from a daemon-built routing key to an ACP session.

Routing keys use a fixed base plus policy-driven dimensions:

- fixed base: `scope`, `workspace_id?`, `channel_instance_id`
- policy-controlled dimensions: `peer_id`, `thread_id`, `group_id`

The core, not the extension, constructs the routing key according to the instance's routing policy and owns the `routing key -> session_id` mapping.

## Alternatives Considered

### Alternative 1: Extension-Local Routing State

- **Description**: Each adapter persists its own local `chat_id -> session_id` table.
- **Pros**: Keeps the core small and lets each platform choose its own identity model.
- **Cons**: Makes automation delivery, UI introspection, and operational continuity inconsistent across adapters.
- **Why rejected**: The daemon needs canonical routing ownership to make channels a first-class system surface.

### Alternative 2: Fully Fixed Routing Key

- **Description**: Always include every known dimension in the routing key regardless of platform or policy.
- **Pros**: Maximum isolation and deterministic routing behavior.
- **Cons**: Over-fragments sessions and ignores that some platforms or deployments want coarser scoping.
- **Why rejected**: Too rigid. Routing policy must be configurable per channel instance.

### Alternative 3: Global-Only Registry

- **Description**: Keep all channel instances and routes global to the daemon.
- **Pros**: Simpler schema and fewer scoping rules.
- **Cons**: Conflicts with workspace-scoped automation and creates avoidable multi-project leakage.
- **Why rejected**: Workspace scope is already an approved pattern in related systems and must carry through here.

## Consequences

### Positive

- Gives the daemon a canonical identity and routing surface for channels.
- Supports both global and workspace-owned channel instances.
- Enables automation, CLI, API, and web UI to resolve delivery targets consistently.
- Keeps session continuity intact across extension restarts.

### Negative

- Adds new persistent models and lifecycle transitions to the daemon.
- Requires clear policy semantics for when `peer`, `thread`, and `group` participate in routing.
- Moves some complexity from the adapter into the core registry.

### Risks

- Misconfigured routing policy could merge or split conversations incorrectly.
- Workspace scoping could become confusing if channel instances are moved or copied between scopes.

## Implementation Notes

- Registry APIs should expose routing policy as a typed config surface, not as opaque JSON blobs whenever possible.
- The first version should include strong validation on scope/workspace combinations and routing-policy completeness.
