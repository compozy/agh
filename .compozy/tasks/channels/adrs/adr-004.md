# ADR-004: Minimal Credential Vault for Extension Secrets

## Status

Superseded by ADR-008

## Date

2026-04-10

## Context

Channel adapters need bot tokens, API keys, and OAuth credentials. The gap analysis identifies "credential vault" as a missing security fundamental across AGH. Three options: environment variables (existing pattern), TOML config fields, or a proper vault with encryption.

Environment variables work but are visible in `/proc/PID/environ`, cannot be rotated without restart, and lack audit trail. Channel adapters are a forcing function for proper secret management because bot tokens are high-value credentials that persist across daemon restarts.

## Decision

Implement a minimal credential vault in `internal/vault/`:

- **Storage**: SQLite table in global DB, values encrypted with AES-256-GCM
- **Key derivation**: Master key derived from a user-provided passphrase via Argon2id, stored as a hash for verification
- **Host API**: `vault/get` and `vault/set` methods for extensions
- **CLI**: `agh vault set <name> <value>`, `agh vault list`, `agh vault delete <name>`
- **Scoping**: Secrets scoped by name. Extensions declare required secrets in manifest and can only access secrets granted via capability scoping.
- **No rotation, no audit log, no multi-tenant** — these are future enhancements

The vault is a daemon component (not an extension) because extensions cannot securely store their own encryption keys.

## Alternatives Considered

### Alternative 1: Environment Variables Only

- **Description**: Use existing `{{env:TOKEN}}` manifest templates
- **Pros**: Works now, zero implementation effort
- **Cons**: Visible in process environment, no encryption at rest, requires restart to change, no scoping per extension
- **Why rejected**: Insufficient for a daemon that runs continuously with multiple extensions accessing different secrets.

### Alternative 2: Vault as Separate TechSpec

- **Description**: Channel adapters use env vars as fallback, vault implemented later
- **Pros**: Unblocks channel adapters immediately
- **Cons**: Creates a temporary pattern (env vars) that must be migrated. Extensions written against env vars may never migrate.
- **Why rejected**: The vault is small enough (~400 LOC) to include in this spec, and having it from day one sets the right pattern for all extensions.

### Alternative 3: External Vault Integration (HashiCorp Vault, etc.)

- **Description**: Integrate with external secret managers
- **Pros**: Enterprise-grade, supports rotation and audit
- **Cons**: External dependency, complex setup, overkill for local-first daemon
- **Why rejected**: AGH is single-binary local-first. External dependencies contradict this principle. Can be added as an extension later.

## Consequences

### Positive

- Secrets encrypted at rest (AES-256-GCM)
- Extensions access secrets via Host API without knowing storage details
- Single CLI command to set secrets — no file editing
- Pattern established for all future extensions that need credentials

### Negative

- Master key management adds complexity (user must provide passphrase on first use)
- No automatic rotation in v1
- Vault is a new daemon component that must be maintained

### Risks

- Master key loss means secrets are unrecoverable. Mitigation: clear warning in CLI, option to reset vault (destroys all secrets).
- Encryption overhead per access. Mitigation: negligible for key-value lookups (< 1ms).

## Implementation Notes

- SQLite schema: `CREATE TABLE vault_secrets (name TEXT PK, encrypted_value BLOB, created_at TEXT, updated_at TEXT)`
- Master key stored as Argon2id hash in `vault_meta` table
- Daemon prompts for passphrase on first boot if vault has secrets but no unlocked key
- `--vault-passphrase` flag or `AGH_VAULT_PASSPHRASE` env var for non-interactive unlock
- Host API: `vault/get {name}` returns `{value}` (plaintext), requires `vault.read` capability
- Host API: `vault/set {name, value}` stores encrypted, requires `vault.write` capability
