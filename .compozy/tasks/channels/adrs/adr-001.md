# ADR-001: 100% Extension-Based Channel Adapters

## Status

Superseded by ADR-005

## Date

2026-04-10

## Context

AGH needs channel adapters (Telegram, Slack, Discord, etc.) to close a critical gap — all 4 competitors have them, AGH has none. Three architectural options exist: built-in `internal/channel/` package, hybrid (thin internal interface + extension adapters), or 100% extension-based (each adapter is a standalone extension subprocess).

The extension system is fully implemented with subprocess lifecycle, capability checking, Host API, and resource registration. The ext-architecture TechSpec already lists "Channel adapters" as a future seam with integration via extension capability + Host API `sessions/create`. The gap analysis warns against "in-process channel adapters at scale" as an OpenClaw anti-pattern.

## Decision

Each channel adapter is a standalone extension subprocess with its own `extension.toml` manifest. No new `internal/` package for channel logic. The daemon provides:

1. New Host API methods for session event streaming (`sessions/subscribe`)
2. A `channel.adapter` capability contract for documentation and capability checking
3. A minimal credential vault (`internal/vault/`) for secure token storage

Channel extensions use existing Host API methods (`sessions/create`, `sessions/prompt`, `sessions/events`) plus the new `sessions/subscribe` for real-time event streaming. All session routing, message formatting, platform API interaction, and concurrency management live inside the extension subprocess.

## Alternatives Considered

### Alternative 1: Built-in `internal/channel/`

- **Description**: Channel adapters compiled into the binary as Go packages
- **Pros**: Simplest to develop, compile-time type safety, no IPC overhead
- **Cons**: Violates AGH's extension-first architecture, grows binary with every adapter, tightly couples platform SDKs to daemon
- **Why rejected**: Contradicts the core philosophy of "minimal core + extensions". Every new platform would require a daemon release.

### Alternative 2: Hybrid (thin internal/ + extension)

- **Description**: Small `internal/channel/` package defines `ChannelAdapter` interface and session routing; adapters implement as extensions
- **Pros**: Compile-time type safety for the contract, shared routing logic
- **Cons**: Adds a new internal package for what is essentially a Host API contract, creates coupling between daemon and extension API surface
- **Why rejected**: The Host API already provides the contract surface. An internal package would duplicate what capability declarations and Host API method schemas already express.

## Consequences

### Positive

- Zero daemon code changes per new channel — community can ship adapters independently
- Process isolation prevents a buggy adapter from crashing the daemon
- Each adapter can be written in Go or TypeScript (via extension SDK)
- Follows the validated pattern from ext-architecture TechSpec
- Adapters can be installed/updated without daemon restarts

### Negative

- IPC overhead (~100-500us per JSON-RPC call) vs in-process function calls
- Extension authors must understand the Host API contract
- No compile-time enforcement of the channel adapter contract

### Risks

- Host API surface may be insufficient for complex adapters. Mitigation: the `sessions/subscribe` method covers real-time streaming; additional methods can be added incrementally.
- Extension development experience may be poor without good SDK/examples. Mitigation: reference Telegram extension serves as template.

## Implementation Notes

- Channel extensions declare `capabilities.provides = ["channel.adapter"]` in their manifest
- Extensions require `session.read`, `session.write`, and `vault.read` security capabilities
- The `channel.adapter` capability is documentation-only in v1 — no capability service methods (AGH → extension) are defined for it
