# ADR-007: Negotiated Channel Delivery Stream for Real-Time Outbound Messaging

## Status

Accepted

## Date

2026-04-10

## Context

The previous design proposed `sessions/subscribe` plus a daemon-initiated `channel/on_event` push. Review showed this conflicts with AGH's extension protocol in two ways:

1. The current protocol does not define `channel/on_event` as a negotiated daemon-to-extension method.
2. Raw ACP/session events are the wrong abstraction for channel delivery because adapters need delivery-oriented semantics rather than internal runtime events.

Polling `sessions/events` or reusing hooks would avoid the protocol change, but both approaches degrade real-time UX and overload abstractions not meant for channel delivery.

## Decision

AGH will add a negotiated channel delivery stream as an extension service in the extension handshake. The daemon will project session output into a delivery-oriented stream rather than pushing raw ACP events.

The stream contract will:

- be negotiated during `initialize`
- be scoped to channel-capable extensions
- guarantee ordering per routing key
- use bounded queues per channel instance and per routing key
- allow delta coalescing under pressure while preserving `start`, `final`, and `error`
- require request/response acknowledgements so the adapter can report remote message IDs for progressive edits

Hooks remain the observability and policy surface. The delivery stream exists only for conversational outbound delivery.

## Alternatives Considered

### Alternative 1: `sessions/subscribe` + `channel/on_event`

- **Description**: Add a Host API subscription method and push raw session events to the adapter.
- **Pros**: Reuses the idea of session-level subscriptions, keeps the adapter close to the raw runtime stream.
- **Cons**: Conflicts with the current extension protocol and leaks internal runtime semantics into channel delivery.
- **Why rejected**: Wrong abstraction and wrong place in the protocol.

### Alternative 2: Polling `sessions/events`

- **Description**: Adapter repeatedly fetches persisted events and reconstructs delivery behavior.
- **Pros**: Minimal protocol changes.
- **Cons**: Adds latency, wastes work on idle sessions, and makes high-quality streaming difficult.
- **Why rejected**: Incompatible with the real-time conversational UX chosen for v1.

### Alternative 3: Hooks for Streaming Delivery

- **Description**: Use hook dispatch for `message.*` and `turn.*` as the delivery mechanism.
- **Pros**: Reuses an existing daemon-to-extension path.
- **Cons**: Hooks are for observability and policy, not per-conversation delivery with backpressure guarantees.
- **Why rejected**: It mixes two separate responsibilities and would create fragile semantics.

## Consequences

### Positive

- Brings the new channel surface into the protocol the right way: negotiated and typed.
- Lets adapters implement progressive message delivery without understanding raw ACP internals.
- Makes ordering, queueing, and recovery semantics explicit and testable.

### Negative

- Requires updates to handshake, protocol docs, SDK, and runtime negotiation.
- Introduces a new delivery-specific abstraction the daemon must maintain.

### Risks

- Poorly chosen event granularity could make the stream either too chatty or too lossy.
- Restart and recovery semantics could become confusing if not tightly specified.

## Implementation Notes

- The first version should focus on `message`, `turn`, and terminal delivery signals only.
- Replay after reconnect should use resumable snapshots, not raw delta history.
