# ADR-003: Session Event Subscription via Host API

## Status

Superseded by ADR-007

## Date

2026-04-10

## Context

Channel adapters need real-time event streaming to relay agent responses to messaging platforms. When a user sends a message on Telegram, the extension creates/prompts a session, then must receive message deltas, tool calls, and completion events as they happen to send typing indicators and incremental responses.

Competitor analysis:
- **Hermes**: Direct callback-based streaming with buffering (500+ chars or 2s idle)
- **OpenFang**: Fully-formed responses via bridge handle (no streaming)
- **GoClaw**: Event bus with per-run subscriptions for real-time delta delivery

AGH's extension protocol is bidirectional JSON-RPC over stdio. The daemon can send requests to the extension (already used for `execute_hook` and `health_check`).

## Decision

Add a `sessions/subscribe` Host API method. The extension calls it with a `session_id` to register interest. The daemon then pushes events to the extension via a daemon→extension JSON-RPC request method `channel/on_event` whenever events occur on that session. The extension responds with `{}` to acknowledge.

This combines GoClaw's subscription model with AGH's existing bidirectional JSON-RPC pattern.

Event types pushed: `message.start`, `message.delta`, `message.end`, `tool.pre_call`, `tool.post_call`, `turn.start`, `turn.end`, `session.stopped`.

The extension is responsible for buffering, chunking, and platform-specific formatting internally.

## Alternatives Considered

### Alternative 1: Hook-Based (observe events)

- **Description**: Extension registers hooks for `message.delta`, `message.end`, etc.
- **Pros**: Works with existing hook system, no new Host API method
- **Cons**: Hooks are broadcast to all extensions for all sessions — extension must filter by session_id. Hook dispatch overhead for high-frequency events (message.delta). Hooks are designed for cross-cutting concerns, not per-session subscriptions.
- **Why rejected**: Wrong abstraction level. Hooks are for policies and transformations, not for data streaming to a specific subscriber.

### Alternative 2: Polling sessions/events

- **Description**: Extension periodically calls `sessions/events` with offset/cursor
- **Pros**: Simplest to implement, no new protocol
- **Cons**: Introduces polling latency (100ms-1s), wastes resources on idle sessions, cannot support typing indicators
- **Why rejected**: Real-time delivery is essential for good chat UX.

### Alternative 3: Fully-formed responses only (OpenFang-style)

- **Description**: `sessions/prompt` returns only when complete
- **Pros**: Simplest protocol
- **Cons**: No streaming, no typing indicators, poor UX for long-running tasks
- **Why rejected**: Users expect progressive responses in messaging apps.

## Consequences

### Positive

- Real-time event delivery using existing bidirectional JSON-RPC
- Extension controls which sessions it subscribes to (no broadcast overhead)
- Extension handles buffering/chunking/formatting internally (platform-specific)
- Pattern is reusable for any extension that needs session event streaming

### Negative

- New Host API method (`sessions/subscribe`) and daemon→extension method (`channel/on_event`) to implement
- Daemon must maintain subscription state per extension
- High-frequency events (message.delta) generate significant JSON-RPC traffic

### Risks

- Subscription state cleanup on extension crash. Mitigation: daemon cleans subscriptions when extension process exits.
- High event throughput overwhelming the extension. Mitigation: extension can unsubscribe, daemon drops events if extension response is slow.

## Implementation Notes

- `sessions/subscribe` params: `{session_id: string}`, result: `{subscribed: true}`
- `sessions/unsubscribe` params: `{session_id: string}`, result: `{unsubscribed: true}`
- `channel/on_event` params: `{session_id, event_type, timestamp, data}` — fire-and-forget semantics (daemon does not block on response)
- Subscription is cleaned up automatically when: extension process exits, session stops, or explicit unsubscribe
- Requires `session.read` security capability
