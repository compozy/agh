# ADR-008: Bound Secret Injection per Channel Instance

## Status

Accepted

## Date

2026-04-10

## Context

The previous design added a minimal vault and exposed `vault/get` and `vault/set` directly to extensions. That gives any extension with the right capability a broad secret-reading surface and forces channel adapters to reason about vault internals.

Channel adapters do need secrets, but only the secrets explicitly attached to the channel instance they are serving. They do not need arbitrary secret discovery. A daemon that already owns instance identity and lifecycle is the right place to resolve bindings and inject only the permitted credentials.

## Decision

AGH will use bound secret injection per channel instance:

1. Secrets remain stored in daemon-owned secure storage.
2. Each `channel_instance` declares explicit `channel_secret_bindings`.
3. On adapter startup, the daemon resolves only those bindings and injects them into the extension runtime.
4. Extensions do not receive arbitrary `vault/get` access in v1.

This makes secret access instance-scoped rather than extension-scoped and removes vault browsing from the adapter contract.

## Alternatives Considered

### Alternative 1: Arbitrary `vault/get` Host API Access

- **Description**: Extensions can ask the daemon for named secrets at runtime.
- **Pros**: Flexible, easy to reason about for simple integrations.
- **Cons**: Too broad a trust surface and too much coupling between adapters and vault internals.
- **Why rejected**: Violates least privilege and creates avoidable secret-discovery risk.

### Alternative 2: Environment Variables Only

- **Description**: Keep secrets outside the daemon and inject via environment templates.
- **Pros**: Minimal implementation effort.
- **Cons**: Weak operational story for rotation, instance-level scoping, and always-on daemons.
- **Why rejected**: Not sufficient for a first-class channel subsystem.

### Alternative 3: Mixed Static Bindings plus Runtime Secret Reads

- **Description**: Static credentials are injected, but extensions can still read some instance-scoped secrets dynamically.
- **Pros**: Helps for refresh-token style flows.
- **Cons**: Complicates the v1 contract before the baseline binding model is proven.
- **Why rejected**: Premature complexity. Instance-scoped token refresh can be designed later on top of the binding model.

## Consequences

### Positive

- Enforces least privilege per channel instance.
- Keeps vault internals out of adapter logic.
- Aligns naturally with a registry-owned channel lifecycle.

### Negative

- Requires the daemon to own one more piece of startup orchestration.
- Makes dynamic secret refresh a follow-up design problem rather than a built-in v1 feature.

### Risks

- Operators may expect direct secret mutation flows that no longer exist for extensions.
- Secret-binding UX must be clear in CLI/API/UI or the system will feel opaque.

## Implementation Notes

- The first version should support launch-time binding only.
- Secret-binding names should be stable and human-readable, even if the underlying vault references are opaque.
