# ADR-001: Multi-Source RegistrySource Interface

## Status

Accepted

## Date

2026-04-13

## Context

AGH needs remote extension discovery and installation. The skills system already integrates with ClawHub (OpenClaw ecosystem), and extensions currently only support local-path install. We need an architecture that supports multiple registry backends (ClawHub, GitHub, future AGH Registry) without coupling the core to any single provider.

The existing `marketplace/clawhub/` client in `internal/skills/` proves the pattern works but is skill-specific and not reusable for extensions.

## Decision

Define a `RegistrySource` Go interface with methods for `Search`, `Info`, `Download`, and `Close`. Each backend (ClawHub for skills, GitHub for both, future AGH Registry) implements this interface. A `MultiRegistry` aggregator iterates over configured sources in priority order. `MultiRegistry` does NOT implement `RegistrySource` — it is a higher-level aggregator that adds `CheckUpdate()` as a convenience method using `Info()` + local version comparison.

The interface lives in `internal/registry/` as a new package, shared by both skills and extensions. The existing `internal/skills/marketplace/clawhub/` client (which implements `marketplace.Registry` at `clawhub/client.go:34`) is refactored to implement `RegistrySource`.

```go
type RegistrySource interface {
    Name() string
    Search(ctx context.Context, query string, opts SearchOpts) ([]Listing, error)
    Info(ctx context.Context, slug string) (*Detail, error)
    Download(ctx context.Context, slug, version string) (*DownloadResult, error)
    Close() error
}
```

Design rationale:
- **`Download` returns `*DownloadResult`** (not `io.ReadCloser`) to carry metadata: content length (for progress bars), server-resolved version, optional server-side checksum, and content type. This matches the existing `marketplace.SkillArchive` pattern which returns `Slug`, `Version`, and `Data` together.
- **No `CheckUpdate` in the interface** — update checks are identical across backends (fetch latest version, compare locally). `MultiRegistry` implements this once using `Info()` + `versionIsNewer()`.
- **`Close()` for lifecycle** — HTTP clients hold connection pools that need cleanup.

## Alternatives Considered

### Alternative 1: Plugin-based registry (extensions as registry providers)

- **Description**: Each registry backend runs as an AGH extension subprocess exposing the registry API via Host API RPC.
- **Pros**: Maximum extensibility, third parties can add registries without modifying AGH core.
- **Cons**: Circular dependency (need extensions to install extensions), significant complexity for alpha, requires daemon running to search.
- **Why rejected**: Over-engineered for current needs. The interface approach allows adding plugin-based registries later without breaking changes.

### Alternative 2: Config-driven URL only

- **Description**: Single configurable base URL in TOML. All registries must speak the same HTTP protocol.
- **Pros**: Simplest possible implementation, one code path.
- **Cons**: Cannot support GitHub (different API), forces all registries to be API-compatible, no room for protocol evolution.
- **Why rejected**: Too rigid. GitHub and ClawHub have fundamentally different APIs.

## Consequences

### Positive

- Clean separation between registry protocol and backend implementation
- New backends added by implementing one interface, no core modifications
- CLI and daemon code depend only on the interface, not concrete backends
- Existing ClawHub client is preserved and wrapped, maintaining OpenClaw ecosystem parity

### Negative

- New `internal/registry/` package adds to the package count
- Aggregating results from multiple sources adds latency (mitigated by concurrent queries)

### Risks

- Interface may need to evolve as new backends reveal different capabilities — mitigated by keeping the interface minimal and using options structs for extensibility

## Implementation Notes

- `MultiRegistry` queries all enabled sources concurrently and merges results using priority-based dedup (later sources override earlier, following the existing `overlaySkill()` pattern in `internal/skills/registry.go:502-515`)
- Each source is configured in `config.toml` under `[skills.marketplace]` (existing) and new `[extensions.marketplace]` (consistent naming)
- The `Download` method returns `*DownloadResult` with the tar.gz stream plus metadata — caller wraps in `io.LimitReader` and handles extraction/verification
- ClawHub is skills-only — its API only exposes `/skills/*` endpoints. Extensions use GitHub as primary remote source.
- The `Installer` accepts a `Downloader` interface (not `*MultiRegistry`) for testability
