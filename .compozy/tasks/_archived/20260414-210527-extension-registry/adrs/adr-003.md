# ADR-003: tar.gz Archive as Universal Distribution Format

## Status

Accepted

## Date

2026-04-13

## Context

Extensions and skills need a distribution format for remote install. ClawHub already serves skills as tar.gz archives. GitHub Releases support arbitrary binary attachments. We need a format that works across all registry sources without requiring source-specific extraction logic.

## Decision

Use tar.gz as the universal distribution format for both skills and extensions. The archive must contain the manifest file (`extension.toml`/`SKILL.md`) at the root level of the archive. The extraction pipeline is:

1. Download tar.gz stream from `RegistrySource.Download()`
2. Extract to a temporary directory
3. Validate manifest exists at root
4. Run security checks (checksum + VerifyContent)
5. Move to install target directory
6. Register in SQLite (extensions) or provenance sidecar (skills)

For GitHub sources, the download path uses GitHub Releases API to fetch the archive asset.

## Alternatives Considered

### Alternative 1: Git clone (shallow)

- **Description**: Clone the repository directly for GitHub-sourced extensions.
- **Pros**: Always gets latest, familiar to developers, includes git history for debugging.
- **Cons**: Requires git on the system, much slower than archive download, two completely different install flows (clone vs extract), no versioning control without tags.
- **Why rejected**: Dual install paths add complexity. Archive download is simpler and faster.

### Alternative 2: OCI registry (container images)

- **Description**: Use OCI artifacts as the distribution format, leveraging container registry infrastructure.
- **Pros**: Industry standard (Helm charts, WASM modules), built-in signing and verification, content-addressable.
- **Cons**: Massive dependency overhead (OCI client libraries), requires users to understand OCI concepts, overkill for alpha.
- **Why rejected**: Disproportionate complexity for the current stage. Can be added as a `RegistrySource` later if demand warrants.

## Consequences

### Positive

- Single extraction pipeline for all registry sources
- ClawHub compatibility preserved (already uses tar.gz)
- GitHub Releases work naturally (upload tar.gz as release asset)
- Simple implementation using Go stdlib `archive/tar` and `compress/gzip`
- No external dependencies required

### Negative

- GitHub sources must package releases as tar.gz (not just raw repo snapshots)
- No delta updates — full archive download every time

### Risks

- Large extensions may have slow downloads — mitigated by progress reporting in CLI
- Archive corruption during download — mitigated by checksum verification post-extraction

## Implementation Notes

- Extraction uses `archive/tar` + `compress/gzip` from Go stdlib
- Maximum archive size limit: 50MB (configurable) to prevent abuse
- Temporary directory for extraction uses `os.MkdirTemp` with cleanup on failure
- Symlinks in archives are followed only within the archive root (no directory traversal)
