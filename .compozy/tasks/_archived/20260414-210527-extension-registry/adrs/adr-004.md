# ADR-004: Reuse Existing SQLite extensions Table for Remote Install Tracking

## Status

Accepted

## Date

2026-04-13

## Context

Extensions installed from remote registries need persistent metadata tracking: which registry they came from, their slug, version, and when they were installed. The existing `extensions` table in `globaldb` already tracks name, version, source (as enum), enabled state, manifest path, checksum, capabilities, and actions.

Skills use a different approach: `.agh-meta.json` provenance sidecar files alongside the SKILL.md directory (constant `sidecarFileName` in `internal/skills/provenance.go:17`).

## Decision

Extend the existing `extensions` table with three new columns for remote install tracking:

- `registry_slug` (TEXT, nullable) ŌĆö the registry-specific identifier (e.g., `compozy/oauth-bridge`)
- `registry_name` (TEXT, nullable) ŌĆö which registry source (e.g., `clawhub`, `github`)
- `remote_version` (TEXT, nullable) ŌĆö the version string from the remote registry

For skills, maintain the existing `.agh-meta.json` provenance sidecar approach ŌĆö it's already implemented and tested.

This avoids duplicating persistence logic and leverages the existing `ExtensionInfo` type with minimal additions.

## Alternatives Considered

### Alternative 1: Separate `registry_installs` table

- **Description**: A new table with FK to `extensions`, storing registry-specific metadata.
- **Pros**: More normalized, cleaner separation of concerns.
- **Cons**: Requires joins for common queries, adds migration complexity, duplicates the existing pattern without clear benefit.
- **Why rejected**: Three nullable columns on an existing table is simpler than a new table with foreign key relationships.

### Alternative 2: Sidecar files for extensions too

- **Description**: Follow the skills pattern ŌĆö `.agh-meta.json` alongside the extension directory.
- **Pros**: No database schema changes, consistent with skills.
- **Cons**: Not queryable (can't list all remote extensions without scanning filesystem), breaks the existing pattern where extensions are DB-first.
- **Why rejected**: Extensions are already DB-centric. Mixing filesystem and DB provenance would create two sources of truth.

## Consequences

### Positive

- Minimal schema change (three nullable columns)
- All extension metadata in one place ŌĆö `List()` and `Get()` return complete info
- `source` enum already has `SourceMarketplace` ŌĆö remote installs use this existing value
- No new tables, no FK complexity

### Negative

- Nullable columns are slightly less clean than a normalized schema
- Skills and extensions use different provenance storage (sidecar vs DB) ŌĆö but this matches their existing patterns

### Risks

- Schema migration on existing databases ŌĆö mitigated by the greenfield alpha policy (no backward compatibility needed)

## Implementation Notes

- Schema is defined inline as a string literal in `internal/store/globaldb/global_db.go:92-102` ŌĆö modify that var block directly (no separate schema.go file)
- Alpha greenfield policy: recreate table with new columns, no migration needed
- `ExtensionInfo` struct gains `RegistrySlug`, `RegistryName`, `RemoteVersion` fields
- `source` = `"marketplace"` indicates remote install; `SourceMarketplace` at `internal/extension/capability.go:72` already exists with restricted security ceiling
- `registry_name` specifies which registry backend (e.g., `"clawhub"`, `"github"`)
- Query for update checks: `SELECT * FROM extensions WHERE registry_slug IS NOT NULL`
- Existing `Registry.Uninstall()` (`registry.go:84-101`) is DB-only ŌĆö does NOT delete the extension directory. The CLI `remove` handler must do both: `os.RemoveAll(dir)` + `Registry.Uninstall(name)`
