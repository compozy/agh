# ADR-004: Assert Through Domain-Specific Product Surfaces

## Status

Accepted

## Date

2026-04-16

## Context

The original narrow E2E proposal leaned heavily on session transcript goldens. That is still useful for ACP session behavior, but it is not sufficient for a system-wide E2E strategy. Different domains expose different truths:

- sessions expose transcripts and events
- network exposes message logs, audit logs, peers, channels, and inbox state
- automation exposes jobs, triggers, and runs
- tasks expose task and task-run records
- bridges expose health, routes, secret bindings, and provider-side effects
- environments expose persisted metadata and tool-host outcomes
- browser E2E exposes navigation, visible state, streaming, approval UI, and hydration behavior

Using one assertion surface for every domain would either under-test the system or create noisy, misleading goldens.

## Decision

Use domain-specific product surfaces as the primary assertion mechanism.

Examples:

- session-oriented scenarios use transcripts, selected event checks, and session state
- network scenarios use channel messages, audit snapshots, API projection, and CLI visibility
- automation scenarios use run records, session linkage, and downstream transcript state
- task scenarios use task records, task-run records, and attached session references
- bridge scenarios use health snapshots, delivery state, and mocked provider call logs
- environment scenarios use persisted sandbox metadata and tool-host side effects
- browser scenarios use UI-visible outcomes and route state, not protocol-level goldens

The browser lane does not repeat low-level daemon assertions unless a second read path is needed to explain a visible failure.

## Alternatives Considered

### Alternative 1: Transcript-only goldens

- **Description**: Treat session transcript output as the main golden surface for the whole system.
- **Pros**: Stable for prompt-centric session scenarios.
- **Cons**: Hides important truth for tasks, bridges, network, environments, and browser behavior.
- **Why rejected**: The broader system has multiple first-class outputs.

### Alternative 2: Raw event logs only

- **Description**: Persist low-level event dumps as the single truth source.
- **Pros**: Maximum detail.
- **Cons**: Too noisy and brittle; hard to review; not always aligned with product behavior.
- **Why rejected**: E2E should focus on meaningful product surfaces.

### Alternative 3: Browser-only assertions for browser E2E

- **Description**: Never read back daemon state from a browser scenario.
- **Pros**: Strictly user-visible testing.
- **Cons**: Harder to diagnose failures where the visible symptom needs one authoritative backend read.
- **Why rejected**: A second read path is acceptable when it clarifies failure without duplicating protocol assertions.

## Consequences

### Positive

- Assertions match the domain being tested.
- Failures are easier to explain with the right artifacts.
- Browser E2E stays focused on user-visible value.

### Negative

- More assertion helpers and artifact capture paths are needed.
- Scenario authors need to choose surfaces deliberately rather than defaulting to one golden format.

### Risks

- Teams may over-assert and make scenarios noisy. Mitigate by documenting the primary surfaces per domain and keeping checks minimal but sufficient.

## Implementation Notes

- Artifact manifests should list the domain surfaces captured by each scenario.
- Transcript goldens remain valid for session-centric flows; they are no longer the universal pattern.
