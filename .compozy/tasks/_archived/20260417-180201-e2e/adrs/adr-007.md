# ADR-007: No Current E2E Lane Uses Real LLM Providers

## Status

Accepted

## Date

2026-04-17

## Context

The current E2E matrix now covers:

- runtime E2E against the real daemon, stores, network runtime, extension processes, and deterministic ACP mock agents
- browser E2E against the daemon-served SPA, using the same deterministic ACP mock agents
- nightly or credentialed infrastructure scenarios such as Daytona-backed environments

What it does **not** cover is any canonical flow executed against a real model provider such as Claude, OpenAI, or Gemini. The deterministic suite intentionally proves daemon integration and product-surface behavior, not provider quality or provider-specific schema compatibility.

Without an explicit ADR, the task pack risks overstating what “E2E confidence” means and blurring two different goals:

- deterministic integration confidence for AGH runtime behavior
- provider-backed product confidence for real LLM behavior, cost, and flake boundaries

## Decision

Document that no current E2E lane exercises real LLM providers.

In practice:

- PR-required runtime and browser lanes remain deterministic
- current nightly coverage may include heavier infrastructure, but not implicit real-provider claims
- any future real-provider verification must ship as a distinct, credentialed lane with its own cost, flake, and ownership model

The deterministic ACP mock suite remains authoritative for runtime integration behavior, not provider correctness.

## Alternatives Considered

### Alternative 1: Treat deterministic mock coverage as implicit provider coverage

- **Description**: let the current E2E suite stand in for real-provider confidence.
- **Pros**: simpler messaging.
- **Cons**: false confidence; cannot detect provider schema drift, tool-call differences, or quality regressions.
- **Why rejected**: it misstates what the current suite actually proves.

### Alternative 2: Fold real-provider flows into the default E2E lanes now

- **Description**: add credentialed provider runs into PR-required or existing nightly lanes immediately.
- **Pros**: more realistic product signal.
- **Cons**: higher cost, flake, secrets handling, and unclear scope ownership during this hardening task.
- **Why rejected**: it is a separate initiative from making the deterministic suite honest and reliable.

## Consequences

### Positive

- The E2E claim is accurate and reviewable.
- Deterministic failures remain diagnosable and cheap enough for regular use.
- Any future provider lane can be designed intentionally instead of being smuggled into the existing suite.

### Negative

- Real-provider regressions remain outside the current default confidence envelope.
- Product stakeholders may need a separate initiative if they want ongoing provider-backed acceptance tests.

### Risks

- Teams may forget that deterministic coverage has a ceiling. Mitigate by referencing this ADR from the techspec and task pack whenever E2E scope is described.

## Implementation Notes

- Keep provider references explicit in docs: deterministic suite today, credentialed provider lane later if desired.
- Do not weaken the deterministic suite by trying to mix provider variability into its default assertions.
