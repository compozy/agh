# ADR-005: Keep PR-Required E2E on Shipped Surfaces and Use Tiered Execution

## Status

Accepted

## Date

2026-04-16

## Context

The codebase already contains some implemented or partially implemented seams that are not yet exposed through a real shipped workflow. It also contains important runtime domains whose web surface is incomplete today. At the same time, some scenarios are expensive or credential-dependent:

- Daytona-backed environments
- provider-specific external integrations
- broader restart and recovery matrices
- combined multi-domain flows with external dependencies

If the PR-required lane tries to prove everything at once, it will become slow, fragile, and misleading. If it ignores missing product surfaces, it will misrepresent what the product actually exposes today.

## Decision

Use tiered execution and scope PR-required E2E to shipped, externally reachable surfaces.

In practice:

- **PR-required runtime E2E** covers real daemon/operator flows that exist today
- **PR-required browser E2E** covers complete operator journeys on the in-scope web surfaces that exist today: workspace onboarding, session chat and approval, network, automation, and bridges
- **Daemon-only E2E for now** covers domains such as tasks when there is no real web workflow yet
- **Nightly or credentialed E2E** covers Daytona and external-provider scenarios
- **Future product gaps** are documented explicitly instead of being silently treated as covered

This means the absence of a web tasks route is not a blocker for task E2E coverage. Tasks remain a runtime E2E responsibility until the web product surface exists. Other shipped SPA surfaces that are not part of this agentic-runtime initiative, such as skills and knowledge, stay outside the PR-required browser lane unless the scope is revised.

## Alternatives Considered

### Alternative 1: Put every scenario in the PR-required lane

- **Description**: Run the full system matrix on every pull request.
- **Pros**: Maximum immediate signal.
- **Cons**: Expensive, noisy, and unrealistic for credentialed or unstable dependencies.
- **Why rejected**: It would slow normal development without improving signal quality proportionally.

### Alternative 2: Browser E2E only for smoke checks

- **Description**: Keep the browser lane cheap and relegate meaningful work to runtime E2E.
- **Pros**: Lower browser test cost.
- **Cons**: Undersells the shipped SPA and fails to prove that operators can complete real work in the UI.
- **Why rejected**: Existing web surfaces deserve complete operator-journey coverage.

### Alternative 3: Pretend missing web flows already exist

- **Description**: Require browser E2E for tasks or other domains without a real web route yet.
- **Pros**: Forces eventual product parity.
- **Cons**: Turns the techspec into a product roadmap dependency and misstates current reality.
- **Why rejected**: The techspec should be honest about what is shipped now.

## Consequences

### Positive

- The PR-required lane stays meaningful and maintainable.
- Browser E2E proves real product workflows where the web surface exists.
- Runtime E2E continues to cover important domains that do not yet have UI.

### Negative

- The overall strategy is more layered and takes more explanation.
- Some desirable combined or credentialed flows move to later tiers.

### Risks

- Teams may confuse “daemon-only for now” with “less important.” Mitigate by documenting daemon-only domains explicitly and promoting them to browser E2E when the product surface exists.

## Implementation Notes

- Keep tasks in runtime E2E until the web app exposes a real task workflow.
- Keep heavier Daytona and external-provider scenarios out of the default PR lane.
- Name CI lanes clearly so “runtime,” “web,” and “nightly” responsibilities are obvious.
