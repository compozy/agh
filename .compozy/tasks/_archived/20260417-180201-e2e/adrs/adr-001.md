# ADR-001: Mock ACP Through a Temporary Agent Definition

## Status

Superseded by ADR-006

## Date

2026-04-16

## Context

This ADR is retained as historical context for the original mock-agent proposal.
The shipped implementation replaced the planned Node/TypeScript driver with the
Go driver formalized in ADR-006.

AGH already knows how to resolve an agent definition, merge it with provider defaults, and start the resulting command through the ACP runtime. The original proposal introduced extra seams such as custom launchers or daemon-level driver injection, but those seams are not required for a subprocess daemon test. The daemon can already be pointed at a different ACP-capable command by setting an explicit `command` in an agent definition.

The E2E goal is to cover the real daemon startup path, agent resolution path, ACP subprocess boundary, and persisted session behavior. The narrowest way to do that is to make the daemon believe it is launching a normal agent.

## Decision

Build a test-only Node/TypeScript subprocess named `mock-acp`, then configure it as a normal AGH agent definition written into the isolated test home for each integration runtime.

The temporary agent definition:

- has a stable test-only name such as `mock-coder`
- sets an explicit `command` pointing to `node <abs>/internal/testutil/acpmock/driver/dist/index.js --fixture <path>`
- keeps a normal AGH `provider` field so validation and agent resolution still follow the standard path

The daemon continues to use `session.NewManager()`, `acp.New()`, and the regular startup path with no new production seams.

## Alternatives Considered

### Alternative 1: Add custom ACP launcher injection for the daemon

- **Description**: Thread a custom `Launcher` or `AgentDriver` through daemon construction so tests can replace the process command at runtime.
- **Pros**: Centralized test seam; no temp agent files.
- **Cons**: Adds test-driven indirection to production startup; larger blast radius; unnecessary for a subprocess daemon test.
- **Why rejected**: The existing agent definition mechanism already solves the problem with less code and lower risk.

### Alternative 2: Use `session.WithDriver` in daemon subprocess tests

- **Description**: Construct the daemon with a custom in-memory driver.
- **Pros**: Full control over events; avoids Node subprocesses.
- **Cons**: Not available once the daemon is running as an external process; skips the real subprocess boundary that ACP tests should cover.
- **Why rejected**: It is a good seam for in-process tests, not for daemon subprocess tests.

## Consequences

### Positive

- Keeps the production ACP runtime unchanged.
- Exercises the same agent resolution and subprocess startup path used in normal operation.
- Avoids a test-only daemon option that production code would need to carry forever.

### Negative

- Tests must create and manage temporary agent definitions.
- The Node mock driver path must be rendered correctly on every platform we support in integration CI.

### Risks

- Broken temp agent rendering can fail the whole suite before the prompt flow starts. Mitigate with focused unit tests in `internal/testutil/acpmock`.
- If the config validation rules change, the temp agent helper must change with them. Mitigate by keeping the helper small and close to existing config tests.

## Implementation Notes

- Shared helper package: `internal/testutil/acpmock/`
- Node driver workspace: `internal/testutil/acpmock/driver/`
- Scenario fixtures and goldens: `internal/testutil/acpmock/testdata/`
