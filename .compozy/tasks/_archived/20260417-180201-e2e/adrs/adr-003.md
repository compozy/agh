# ADR-003: Run Cross-System Runtime E2E From the Composition Root

## Status

Accepted

## Date

2026-04-16

## Context

The flows this techspec wants to prove do not stay inside a single package:

- sessions join channels
- network messages become prompt turns
- automation creates sessions or delegates tasks
- bridges ingest events and deliver replies
- extensions call Host API methods
- environments own tool hosts and sandbox behavior

Those paths are wired together in daemon boot and runtime composition, not inside any one subsystem package. Package-local integration tests remain valuable, but they are not the right place to prove the whole system behaves correctly when composed.

## Decision

Place the main cross-system runtime E2E scenarios at the daemon composition root.

Concretely:

- use `internal/daemon/*_integration_test.go` for flows that require multiple runtime domains together
- reuse package-level helpers and harnesses where useful
- keep transport-specific follow-up checks in `internal/api/httpapi` and `internal/api/udsapi` only when the composition-root scenario is not enough

This keeps the most important E2E proofs anchored to the place where the real product wiring happens.

## Alternatives Considered

### Alternative 1: Keep every scenario package-local

- **Description**: Put network flows under `internal/network`, bridge flows under `internal/bridges`, automation flows under `internal/automation`, and so on.
- **Pros**: Each package owns its own tests.
- **Cons**: Cross-domain flows become fragmented and hard to reason about; many tests would need to recreate daemon boot wiring implicitly.
- **Why rejected**: The product behavior we care about is compositional.

### Alternative 2: Add a new top-level Go `test/e2e` tree

- **Description**: Create a standalone top-level suite outside the current package layout.
- **Pros**: One obvious directory for E2E tests.
- **Cons**: Drifts from current repo testing conventions and duplicates existing harness patterns.
- **Why rejected**: The composition root already exists in `internal/daemon`.

### Alternative 3: Put everything under API integration tests

- **Description**: Make HTTP and UDS integration suites the only runtime E2E layer.
- **Pros**: Uses existing public transports.
- **Cons**: Cross-domain runtime orchestration gets buried in transport harnesses; some flows are easier to prove at daemon level and then supplement with transport-specific parity checks.
- **Why rejected**: Composition-root scenarios should lead, with transport suites supplementing them.

## Consequences

### Positive

- The main runtime E2E suite reflects real daemon boot and wiring.
- Cross-domain failures are easier to reason about.
- Existing package-level integration suites keep their role without pretending to be the whole system.

### Negative

- `internal/daemon` tests become broader and require careful fixture design.
- Some assertions will still need follow-up transport-level coverage elsewhere.

### Risks

- The daemon harness may grow into a second framework. Mitigate by keeping it focused on boot, seeding, and public-surface reads rather than reproducing domain logic.

## Implementation Notes

- Put shared boot and artifact helpers under `internal/testutil/e2e/`.
- Keep transport-specific scenarios in `internal/api/httpapi` and `internal/api/udsapi` narrow and explicit.
