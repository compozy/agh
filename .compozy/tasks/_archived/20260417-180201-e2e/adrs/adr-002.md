# ADR-002: Separate Runtime and Browser E2E Lanes

## Status

Accepted

## Date

2026-04-16

## Context

AGH ships as more than a daemon API. It also ships an embedded web application that operators use for session chat, network inspection, automation management, bridge management, workspace setup, and related workflows. At the same time, many of the most important correctness properties live below the browser:

- AGH Network v0 semantics
- automation dispatch and delegated task runs
- bridge ingress and delivery
- extension subprocess behavior
- environment and sandbox restrictions

Treating daemon/runtime tests as the only E2E proof is too narrow for the shipped product. Treating browser automation as the only E2E proof is also wrong, because the browser should not be the source of truth for protocol and orchestration semantics.

## Decision

Split E2E into two coordinated lanes:

- **Runtime E2E**
  - the authoritative proof for daemon/runtime correctness
  - boots the real daemon and real runtime surfaces
  - exercises protocol, orchestration, persistence, and cross-domain behavior

- **Browser E2E**
  - the authoritative proof for shipped operator workflows in the SPA
  - runs Playwright against the daemon-served UI
  - verifies meaningful end-user journeys on the in-scope web surfaces that exist today

The runtime lane remains the system-of-record for low-level correctness. The browser lane proves that operators can complete real work through the web app without duplicating protocol-level assertions already owned by runtime E2E.

For this spec, the in-scope browser surfaces are workspace onboarding, session chat and approval, network, automation, and bridges. Other shipped SPA surfaces such as skills and knowledge remain outside this initiative unless the spec is expanded later.

## Alternatives Considered

### Alternative 1: Daemon-only E2E

- **Description**: Keep all end-to-end validation below the browser and rely on existing Vitest coverage for the UI.
- **Pros**: One harness; lower CI cost.
- **Cons**: Does not validate the shipped SPA, route wiring, streaming UI behavior, approval UX, SSE-driven health, or reload/hydration behavior.
- **Why rejected**: The product includes the browser, so daemon-only coverage is not a complete E2E story.

### Alternative 2: Browser-only E2E

- **Description**: Push all validation through Playwright and the web app.
- **Pros**: One user-facing surface; simple to explain.
- **Cons**: Low-level protocol failures become harder to diagnose; browser tests become too broad and fragile; many runtime surfaces do not have web flows today.
- **Why rejected**: The daemon/runtime is where the core system semantics live.

### Alternative 3: Browser smoke only

- **Description**: Keep browser tests limited to route loads and basic clicks.
- **Pros**: Cheap to run and maintain.
- **Cons**: Too weak to justify the E2E label for the web surface.
- **Why rejected**: The browser lane should prove complete operator workflows on existing web surfaces.

## Consequences

### Positive

- The techspec becomes honest about the shipped product boundary.
- Runtime and browser failures are easier to localize.
- The browser lane can stay focused on journeys while the runtime lane owns protocol truth.

### Negative

- Two E2E harnesses and CI lanes must be maintained.
- Some scenarios need seeded runtime state so the browser can observe a real downstream effect.

### Risks

- Teams may accidentally duplicate the same assertions in both lanes. Mitigate by making runtime E2E own protocol and persistence truth, and browser E2E own user-visible outcomes.

## Implementation Notes

- Playwright should run against the daemon-served UI, not only the Vite proxy.
- Vitest remains for unit, component, adapter, and `jsdom` integration tests; it does not carry the E2E claim.
