# ADR-001: Markdown with YAML Frontmatter as Unified Artifact Format

## Status

Accepted

## Date

2026-03-31

## Context

Roles are currently stored as TOML files with all fields (including multi-line `system_prompt`) encoded in TOML syntax. Playbooks are raw Markdown files with no structured metadata — the name is derived solely from the filename. The planned skills system uses Markdown with YAML frontmatter following the AgentSkills specification. Having three different serialization formats for similar catalog artifacts creates inconsistency in the codebase and forces each artifact type to maintain its own parser.

Markdown with YAML frontmatter is the industry standard for content-with-metadata files (used by Hugo, Jekyll, Astro, AgentSkills spec, etc.) and cleanly separates structured metadata from unstructured content.

## Decision

Adopt Markdown with YAML frontmatter as the unified file format for roles, playbooks, and (future) skills. Role files change from `.toml` to `.md`. The YAML frontmatter carries structured metadata fields, and the Markdown body carries the content (system prompt for roles, strategy content for playbooks).

## Alternatives Considered

### Alternative 1: Keep TOML for Roles, Add Frontmatter Only to Playbooks

- **Description**: Maintain the current TOML format for roles and only add YAML frontmatter to playbooks.
- **Pros**: No migration needed for roles; roles keep strict schema enforcement via TOML.
- **Cons**: Two different formats for similar artifacts; no shared parser; TOML multi-line strings are awkward for large prompts.
- **Why rejected**: Inconsistency across artifact types increases cognitive load and code duplication.

### Alternative 2: TOML Frontmatter Instead of YAML

- **Description**: Use TOML as the frontmatter language (delimited by `+++`) for all artifacts.
- **Pros**: Consistent with the existing config.toml format; reuses BurntSushi/toml dependency.
- **Cons**: TOML frontmatter is non-standard; no ecosystem tooling supports it; conflicts with AgentSkills spec which mandates YAML.
- **Why rejected**: Non-standard format with no tooling support; misaligned with skills spec.

## Consequences

### Positive

- Unified format across all catalog artifacts (roles, playbooks, skills).
- Markdown body avoids YAML/TOML escaping issues for large multi-line prompts.
- Aligns with AgentSkills specification and industry conventions.
- Enables shared tooling (parser, linters, IDE support).

### Negative

- Breaking change for any existing role `.toml` files on disk.
- Roles lose strict TOML schema enforcement (unknown keys detection) — mitigated by YAML struct validation.

### Risks

- YAML parsing edge cases with special characters in frontmatter values. Mitigation: only structured short fields go in frontmatter; long content stays in Markdown body.

## Implementation Notes

- File extension changes from `.toml` to `.md` for roles.
- Playbooks keep `.md` extension but gain frontmatter.
- The `parseCatalogFileName` function needs updated suffix parameters for roles.
- Draft/versioned naming pattern remains: `{name}.md`, `{name}.draft.md`, `{name}.draft_v{N}.md`.
