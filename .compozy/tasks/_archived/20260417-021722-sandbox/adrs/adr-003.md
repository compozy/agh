# ADR-003: Session-Scoped Sandbox with Copy-on-Start / Collect-on-Stop Sync

## Status

Accepted

## Date

2026-04-15

## Context

When agents run in remote sandboxes, the workspace filesystem must be synchronized between the local canonical copy and the sandbox runtime. The sync strategy must account for:

1. **Multi-agent scenarios**: AGH's network layer allows multiple agents in a channel to reference the same workspace. Each agent runs in its own session.
2. **Mixed mode**: Some agents may be local, others remote — they see different filesystem views.
3. **Terminal edits**: Agents modify files via terminal commands (not just ACP `fs/write`), so the ToolHost abstraction alone does not capture all changes.
4. **Network design**: AGH's channel model is loosely-coupled messaging (NATS). There is no filesystem coordination at the network level. Agents coordinate explicitly via messages, not implicit sync.

The \_idea.md proposed three sync modes: `none`, `session-bidirectional`, and `turn-bidirectional`.

## Decision

Adopt a **one-sandbox-per-session** topology with **session-bidirectional sync** as the default:

- **Start**: Snapshot the local workspace into the sandbox (local → remote) using tar streams over the provider transport.
- **Stop**: Collect changes from the sandbox back to the local workspace (remote → local) using tar streams over the provider transport.
- **Crash**: Best-effort collect on crash recovery.
- **During session**: No automatic sync. ToolHost handles individual ACP file operations.
- **Divergence**: Last-write-wins at the file level. No merge, no conflict detection.

The mental model: "AGH snapshots the workspace into a sandbox, the agent works there, AGH collects results back." This is a **copy-on-start / collect-on-stop** pattern.

Sync mode is configurable per environment profile:

- `session-bidirectional` (default) — snapshot in, collect out
- `none` — no automatic sync (sandbox starts empty or from snapshot image)

`turn-bidirectional` is documented as a future option but not implemented in alpha.

Tar streaming is part of the decision, not an optional optimization. Provider SDK file APIs remain available for point operations (`ToolHost` reads/writes, stat/list behavior), but directory sync must avoid one remote API call per file. For Daytona alpha, this means tar-over-SSH for both sync directions.

## Alternatives Considered

### Alternative 1: Turn-Bidirectional Sync

- **Description**: Sync at every turn boundary (after each prompt/response cycle).
- **Pros**: More up-to-date filesystem state. Better for long sessions with many turns.
- **Cons**: Sync overhead at every turn. Partial-state bugs if sync fails mid-turn. Complex failure handling. Uncertain value in alpha with no usage data.
- **Why rejected**: Premature complexity. No evidence that session-boundary sync is insufficient. Can be added later with real usage data.

### Alternative 2: Watch-Based Real-Time Sync

- **Description**: File watcher (like Mutagen/rsync-watch) monitors changes in real-time on both sides.
- **Pros**: Best UX — changes appear immediately on both sides.
- **Cons**: Extremely complex. Conflict resolution required. Performance overhead. Dependency on external tools. Overkill for alpha.
- **Why rejected**: Massive complexity for uncertain value. Not aligned with AGH's explicit-coordination network design.

### Alternative 3: Shared Sandbox for Same-Workspace Sessions

- **Description**: Multiple sessions referencing the same workspace share one sandbox.
- **Pros**: No divergence — all agents see the same files.
- **Cons**: Implicit coupling — one agent's destructive action affects others. Breaks loose-coupling design. Complex lifecycle (who owns the sandbox? when does it stop?).
- **Why rejected**: Violates AGH's explicit-coordination principle. If agents need to see each other's edits, they coordinate via network messages.

## Consequences

### Positive

- Simple mental model: snapshot in, work, collect out.
- Session owns its sandbox lifecycle — clean teardown, no cross-session coordination.
- Consistent with AGH's loose-coupling network design.
- Last-write-wins matches current local behavior (no filesystem locking today either).

### Negative

- Multiple agents editing the same workspace concurrently will have divergent views.
- Last-write-wins can silently overwrite changes from a concurrent session.
- No turn-level freshness — local workspace state is stale during the session.

### Risks

- Large workspaces may have slow initial sync. Mitigation: tar-over-SSH is the primary directory sync path, with configurable excludes for generated dependency/build output.
- Malicious or malformed tar archives may attempt path traversal. Mitigation: reject absolute paths, `..`, and entries that escape the destination after symlink evaluation.
- Crash recovery may fail to collect changes. Mitigation: best-effort sync with logged warnings; sandbox persists (Daytona archive) for manual recovery.

## Implementation Notes

- Sync is triggered by `Provider.SyncToRuntime()` (on start) and `Provider.SyncFromRuntime()` (on stop/crash).
- `SyncReason` enum (`start`, `stop`, `crash`) allows providers to adjust behavior per reason.
- For Daytona: use tar-over-SSH for directory sync. Use `FileSystemService.UploadFile()`, `DownloadFile()`, and `ListFiles()` only for point operations and diagnostics.
- Sync scope: workspace `RootDir` + `AdditionalDirs`. Respect profile exclude patterns plus safe defaults for generated dependency/build output. Do not exclude `.git` by default because agents often need repository metadata.
- Extraction safety is provider-independent and belongs in the shared sync service: reject absolute paths, `..` traversal, symlink escapes, and unsupported file modes.
