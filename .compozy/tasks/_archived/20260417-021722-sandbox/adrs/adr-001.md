# ADR-001: Daemon-Native Environment Providers with Extension Hooks

## Status

Accepted

## Date

2026-04-15

## Context

AGH needs to support remote execution environments (Daytona, E2B) alongside the current local subprocess model. The key architectural question is whether environment providers should be implemented as extensions (out-of-process subprocesses communicating via JSON-RPC) or as daemon-native subsystems (in-process, wired at boot).

AGH already has an extension model used by bridges, memory backends, and skills. Extensions run as supervised subprocesses with JSON-RPC over stdio, which works well for loosely-coupled, async operations like message delivery and hook execution.

However, environment providers have fundamentally different coupling requirements:

- **Launcher** must return `io.WriteCloser`/`io.ReadCloser` streams for ACP JSON-RPC — these cannot cross a JSON-RPC process boundary.
- **ToolHost** serves file IO and terminal operations on the ACP hot path — every `fs/read`, `fs/write`, and terminal creation goes through the provider.
- **Sync** operations must coordinate with session lifecycle at the Go interface level.
- **Recovery** must reconcile partially-created remote resources after daemon crashes or provider timeouts, using daemon-owned identity and provider-specific state.

## Decision

Environment providers are **daemon-native subsystems** (in-process), not extensions. They live in `internal/environment/{provider}/`, are wired through the daemon composition root via dependency injection, and follow the same patterns as other core subsystems (workspace resolver, session manager, ACP driver).

Provider lifecycle is daemon-owned end to end: the session manager allocates an `EnvironmentID`, persists `SessionEnvironmentMeta`, and invokes providers with enough identity for idempotent create/reattach/destroy. Remote providers must tag or label external resources with AGH identity so daemon restart reconciliation can find resources even when a create call succeeded remotely but failed locally before writing `InstanceID`.

To preserve extensibility, environment lifecycle hooks are exposed to the extension system:

- `environment.prepare` — before sandbox creation
- `environment.ready` — after sandbox is ready
- `environment.sync.before` — before workspace sync
- `environment.sync.after` — after sync completes
- `environment.stop` — before sandbox teardown

Host API methods are added for extensions to query/interact with environments:

- `environment/list`, `environment/info`, `environment/exec`

These extension-facing seams layer onto the canonical extensibility runtime from `.compozy/tasks/extensibility-parity/_techspec.md`. Environment lifecycle hooks participate in the shared hook taxonomy and are routed by the resource-backed `hook.binding` runtime, while `environment/list`, `environment/info`, and `environment/exec` are operational Host APIs registered through `internal/extension/surfaces` and the daemon-computed capability/grant model. They are not modeled as desired-state resources.

## Alternatives Considered

### Alternative 1: Full Extension Model

- **Description**: Environment providers as extension subprocesses with `environment.provider` capability, communicating via JSON-RPC.
- **Pros**: Consistent with existing extension architecture. Provider isolation. Independent development.
- **Cons**: Cannot return Go `io.ReadWriteCloser` streams across process boundary. Adds latency to every ACP tool call. Requires complex new Host API surface for streams and tool hosting.
- **Why rejected**: Fundamental architecture mismatch — extensions are designed for loosely-coupled async operations, not synchronous hot-path stream management.

### Alternative 2: Hybrid Extension + In-Process

- **Description**: Launcher/ToolHost in-process, but provider discovery and health uses the extension manager.
- **Pros**: Leverages existing extension lifecycle management.
- **Cons**: Awkward split — an "extension" that doesn't actually run as a subprocess. Confuses the extension model semantics.
- **Why rejected**: Creates a misleading abstraction. Better to follow established patterns (workspace resolver, session manager) for daemon-native subsystems.

## Consequences

### Positive

- Zero latency overhead for ACP tool calls — provider is in-process.
- Clean Go interface contracts (Launcher, ToolHost, Provider) with compile-time verification.
- Consistent with how AGH already handles workspace resolution and session management.
- Extension hooks and operational Host APIs provide extensibility without compromising the hot path.
- Remote lifecycle recovery can use local metadata plus provider labels without requiring extension subprocess state.

### Negative

- Provider code must be compiled into the AGH binary (increases binary size).
- Provider SDK dependencies (Daytona Go SDK, SSH library) become daemon dependencies.
- Adding a new provider requires a code change and rebuild, not just installing an extension.

### Risks

- Provider SDK instability could affect daemon stability. Mitigation: isolate provider failures with context cancellation and error boundaries.
- Binary size growth with multiple provider SDKs. Mitigation: build tags to exclude unused providers.

## Implementation Notes

- Package structure: `internal/environment/` for core types, `internal/environment/local/` for local provider, `internal/environment/daytona/` for Daytona provider.
- Daemon composition root (`internal/daemon/daemon.go`) wires the provider registry and injects it into the session manager.
- Provider selection follows the cascade: `Workspace.EnvironmentRef` → `Config.Defaults.Environment` → implicit `local`.
- Remote providers receive daemon-owned `EnvironmentID` and must label/tag resources with `agh_session_id` and `agh_environment_id`.
- `SessionEnvironmentMeta.ProviderState` stores provider-specific reattach data. It is not exposed as extension-owned desired state.
- Hook payload/event definitions still live in `internal/hooks`, but delivery authority comes from the canonical hook-binding runtime rather than a sandbox-specific registry.
- Environment boot reconcile runs after the canonical resource-runtime boot/reconcile phase when both features are enabled.
