# ADR-002: SSH as Primary Transport for Daytona, with Provider-Internal Transport Abstraction

## Status

Accepted

## Date

2026-04-15

## Context

AGH communicates with ACP agents via JSON-RPC over stdin/stdout — clean bidirectional byte streams with no terminal processing. When an agent runs inside a Daytona sandbox, AGH needs a remote transport that preserves this clean byte-stream contract.

Research revealed that Daytona's Go SDK offers three process execution models:

1. **ExecuteCommand** — synchronous fire-and-collect, no streaming stdin. Not suitable for ACP.
2. **Sessions** — command-oriented, half-duplex. Not suitable for continuous bidirectional JSON-RPC.
3. **PTY** — bidirectional WebSocket-backed terminal. Adds echo, line discipline, ANSI escapes that corrupt JSON-RPC framing.

None provide native non-PTY stdio streams. However, further research found two viable alternatives:

- **SSH non-PTY**: Daytona exposes SSH access (`ssh <token>@ssh.app.daytona.io`). Without `-t`, SSH provides clean stdio pipes — the same pattern used by `git push/pull`, `rsync`, and `scp`. Go's `golang.org/x/crypto/ssh` provides `session.StdinPipe()`/`session.StdoutPipe()` matching AGH's existing `io.WriteCloser`/`io.ReadCloser` interfaces.
- **WebSocket relay sidecar**: A small Go bridge binary (~200 LOC) inside the sandbox that spawns the agent with clean `os/exec` pipes and relays over WebSocket through Daytona's preview URLs. Same pattern as VS Code Remote, Gitpod, and Coder.

## Decision

Use **SSH non-PTY as the primary transport** for Daytona in alpha. Implement a **provider-internal transport abstraction** (Design B) that allows swapping SSH for WebSocket sidecar later without changing the provider's public interface.

The SSH transport owns two distinct stream classes:

- **ACP process stream**: one long-lived non-PTY SSH session for the agent's stdin/stdout JSON-RPC stream.
- **Bulk sync streams**: short-lived non-PTY SSH sessions used by the provider sync service to send or receive tar archives at session boundaries.

These stream classes share token management and connection setup, but they must not share a single SSH session or multiplex tar payloads onto the ACP JSON-RPC stream.

The transport interface lives inside the Daytona provider package:

```go
// internal/sandbox/daytona/transport.go
type transport interface {
    Dial(ctx context.Context, sandbox sandboxInfo) (conn io.ReadWriteCloser, err error)
}
```

For alpha, only `sshTransport` is implemented. `wsTransport` is added when production hardening requires reconnection, health checks, or reduced dependency on the SSH gateway.

Token refresh is part of the transport contract. The Daytona provider persists token expiry in `SessionSandboxMeta`, refreshes at 50% of the token lifetime, and retries one failed SSH connection after forcing a fresh token. Repeated auth failures are surfaced as provider errors.

## Alternatives Considered

### Alternative 1: WebSocket Sidecar Directly

- **Description**: Skip SSH, go straight to the production-grade WebSocket relay pattern.
- **Pros**: Better reconnection semantics. Proven at scale (VS Code, Gitpod). Bridge binary evolves to handle sync and health.
- **Cons**: Requires building and baking a bridge binary into sandbox images. More moving parts. Speculative complexity for alpha with zero users.
- **Why rejected**: YAGNI. SSH gives a working transport with minimal delta from the local implementation. The internal transport interface preserves the migration path.

### Alternative 2: PTY with `stty raw -echo` Workaround

- **Description**: Use Daytona's native PTY with terminal processing disabled.
- **Pros**: No additional transport needed. Uses SDK directly.
- **Cons**: Fragile — terminal artifacts can corrupt JSON-RPC framing. Initial shell prompt noise. Signal byte injection risk.
- **Why rejected**: Workaround, not a designed-for use case. Unacceptable risk for a protocol transport.

### Alternative 3: Shared Transport Interface Across Providers

- **Description**: A `TransportBackend` interface in `internal/sandbox/` shared between Daytona and E2B.
- **Pros**: Theoretical reuse across providers.
- **Cons**: Providers have fundamentally different auth, lifecycle, and connection models. SSH for Daytona vs Connect Protocol for E2B — the "reuse" is illusory.
- **Why rejected**: Over-engineering. Each provider's transport is an implementation detail of its Launcher.

## Consequences

### Positive

- SSH maps 1:1 to AGH's existing `io.WriteCloser`/`io.ReadCloser` interface — minimal code delta.
- `golang.org/x/crypto/ssh` is battle-tested Go extended standard library.
- No sidecar binary, no image customization required for alpha.
- Internal transport interface enables clean migration to WebSocket when evidence warrants it.

### Negative

- Depends on Daytona's SSH gateway supporting non-PTY sessions (needs validation).
- SSH token rotation required (default 60-minute expiry).
- SSH adds a network hop through Daytona's gateway.
- Rich interactive terminal UX still needs a separate process/PTY design. The ACP transport must remain non-PTY even if a future terminal surface uses WebSocket or a provider-native PTY API.

### Risks

- Daytona SSH gateway may force PTY allocation. Mitigation: validate with a simple test before implementation. If blocked, fall back to WebSocket sidecar.
- Token expiry during long sessions. Mitigation: persist token expiry, refresh at 50% of the expiry window, and retry one failed connection with a fresh token.
- Bulk sync competes with ACP stream resources. Mitigation: use separate short-lived SSH sessions for tar transfer and keep the ACP session dedicated to JSON-RPC.

## Implementation Notes

- SSH validation test: `echo '{"jsonrpc":"2.0"}' | ssh <token>@ssh.app.daytona.io cat` — if output matches input exactly, non-PTY is confirmed.
- Token management: `POST /api/sandbox/{id}/ssh-access?expiresInMinutes=60` → persist `expires_at`, rotate at 50% of the expiry window, and retry once on auth failure.
- The `Launcher` interface returns `Handle` with `Stdin()`/`Stdout()` — the SSH session pipes fulfill this directly.
- Tar sync uses the same transport package but opens separate SSH sessions. It does not flow through `Launcher.Handle`.
