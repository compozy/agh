# BUG-003: Teams delivery test swapped API factory before async init settled

**Severity:** Medium
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** 014df557-dirty
- **OS:** macOS
- **Browser:** N/A
- **URL:** N/A

## Summary

The Teams runtime delivery test could trip the race detector because it replaced `runtime.apiFactory` immediately after `initialize`, while the provider was still using the original factory inside asynchronous initialization.

## Reproduction

```bash
make test-integration
```

Observed before the fix:

- The race detector reported concurrent access to `runtime.apiFactory` during `TestRuntimeDeliveriesCallTeamsAPI`.

## Expected

The test should only replace the API factory after asynchronous initialization has completed and the provider has recorded a ready state.

## Root cause

The test mutated shared runtime state before the provider's background initialization path finished reading it.

## Fix

Updated the Teams runtime delivery test to wait for the provider `stateMarker` to report readiness before swapping in the fake API factory used by the delivery assertions.

## Verification

- `go test -race ./extensions/bridges/discord ./extensions/bridges/teams ./extensions/bridges/slack ./extensions/bridges/gchat -count=3`
- `make test-integration`
- `make verify`

## Impact

- **Users Affected:** developers running the integration gate
- **Frequency:** sometimes
- **Workaround:** rerun the integration suite until timing lines up
