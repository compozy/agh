# BUG-002: Bridge stop paths accessed route state without synchronization

**Severity:** High
**Priority:** P1
**Type:** Crash
**Status:** Fixed

## Environment

- **Build:** 014df557-dirty
- **OS:** macOS
- **Browser:** N/A
- **URL:** N/A

## Summary

The integration gate exposed a real data race in bridge shutdown logic. `stop()` could read and mutate route/batcher state while asynchronous reconciliation was still replacing route entries, which is undefined behavior outside the race detector and a latent runtime risk.

## Reproduction

```bash
make test-integration
```

Observed before the fix:

- The race detector reported concurrent access between `stop()` and background initialization/reconciliation in the Discord provider.
- Slack and GChat contained the same shutdown pattern and the same latent race shape.

## Expected

Provider shutdown must take ownership of route and batcher state under the provider mutex before draining or closing batchers.

## Root cause

`stop()` collected batchers from `p.routes` while route replacement could still occur on another goroutine. The route map needed to be walked only while holding the provider mutex.

## Fix

Reworked Discord, Slack, and GChat shutdown paths to gather batchers under lock, clear the stored batcher references in-place, release the mutex, and then close the copied batchers afterward.

## Verification

- `go test -race ./extensions/bridges/discord ./extensions/bridges/teams ./extensions/bridges/slack ./extensions/bridges/gchat -count=3`
- `make test-integration`
- `make verify`
