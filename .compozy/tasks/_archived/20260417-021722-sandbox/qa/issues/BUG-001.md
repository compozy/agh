# BUG-001: Async bridge readiness made delivery tests race the auth probe

**Severity:** Medium
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** 014df557-dirty
- **OS:** macOS
- **Browser:** N/A
- **URL:** N/A

## Summary

The broad unit gate failed intermittently in bridge runtime delivery tests because the test code sent outbound deliveries immediately after `initialize`, while the providers were still completing their asynchronous `afterInitialize` readiness/auth probe.

## Reproduction

```bash
make verify
```

Observed before the fix:

- `TestRuntimeDeliveriesCallTelegramBotAPI` could observe `sendMessage` before the expected `getMe` auth probe.
- The same timing assumption existed in the Slack and WhatsApp runtime delivery tests.

## Expected

Runtime delivery tests should wait until provider initialization reaches the ready state before asserting API call ordering or dispatching outbound deliveries.

## Root cause

The providers return from `initialize` before their background `afterInitialize` work finishes. The tests assumed initialization was synchronous and raced the readiness path.

## Fix

Updated the Telegram, Slack, and WhatsApp runtime delivery tests to wait for the recorded `stateMarker` to reach a ready state before sending deliveries or asserting call order.

## Verification

- `go test -race ./extensions/bridges/telegram ./extensions/bridges/slack ./extensions/bridges/whatsapp -run "TestRuntimeDeliveriesCall(TelegramBotAPI|SlackAPI|WhatsAppGraphAPI)$" -count=5`
- `go run gotest.tools/gotestsum@latest --format pkgname -- -count=1 -race -parallel=4 ./...`
- `make verify`

## Impact

- **Users Affected:** developers running the unit gate
- **Frequency:** sometimes
- **Workaround:** rerun the test suite until timing lines up
