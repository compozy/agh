# ADR-002: SKILL.md as Native Format (AgentSkills Spec)

## Status

Accepted

## Date

2026-03-31

## Context

AGH needs a file format for skill definitions. The ecosystem has converged on two formats: SKILL.md (YAML frontmatter + Markdown body, standardized by AgentSkills/OpenClaw) and skill.toml (TOML manifest, used by OpenFang). The AgentSkills spec has been adopted by 30+ tools including Claude Code, Codex, Cursor, VS Code Copilot, Gemini CLI, and many others.

## Decision

Use **SKILL.md with YAML frontmatter** as the single native format, following the AgentSkills specification exactly. AGH-specific metadata lives under `metadata.agh` in the frontmatter. No secondary format, no auto-conversion layer.

```yaml
---
name: code-review
description: Review code for quality and best practices.
metadata:
  agh:
    tags: [quality, review]
---
```

## Alternatives Considered

### Alternative 1: skill.toml + SKILL.md Compatibility Layer

- **Description**: Native TOML format (Go-idiomatic) with auto-conversion from SKILL.md, following the OpenFang approach.
- **Pros**: TOML is standard in Go ecosystem. More structured than YAML.
- **Cons**: Requires two parsers. Conversion can lose nuances. Community skills are SKILL.md — the TOML format would be AGH-only.
- **Why rejected**: The ecosystem has standardized on SKILL.md. Introducing a proprietary format fragments the ecosystem without meaningful benefit.

### Alternative 2: Dual Native (Both Formats Equally Supported)

- **Description**: Support both SKILL.md and skill.toml as first-class formats.
- **Pros**: Maximum compatibility.
- **Cons**: Doubles parsing, validation, and documentation surface. Two ways to do the same thing.
- **Why rejected**: Unnecessary complexity. SKILL.md covers all use cases.

## Consequences

### Positive

- 100% compatibility with community skills from ClawHub, GitHub, and other AgentSkills-compliant tools
- Skills authored for AGH work in Claude Code, Codex, Cursor, etc. and vice versa
- Single parser to maintain
- AGH-specific extensions via `metadata.agh` namespace without breaking compatibility

### Negative

- YAML frontmatter parsing requires a YAML library (new dependency)
- YAML has well-known gotchas (unquoted colons, implicit type coercion)

### Risks

- Malformed YAML from other tools. Mitigation: lenient parsing with fallback, following the AgentSkills client implementation guide.

## Implementation Notes

Parse frontmatter between `---` delimiters. Required fields: `name`, `description`. All other fields optional. Follow lenient validation: warn on issues, skip only if description missing or YAML unparseable.
