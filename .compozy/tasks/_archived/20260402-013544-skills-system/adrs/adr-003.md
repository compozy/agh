# ADR-003: System Prompt Injection via CLI Access Pattern

## Status

Accepted

## Date

2026-03-31

## Context

AGH orchestrates 4 different agent drivers (Claude Code, Codex, Pi, OpenCode), each with different configuration mechanisms. Skills need to be accessible to all agents without modifying driver configurations or touching user config files. Three integration strategies were considered: MCP server, config file modification, and system prompt injection.

## Decision

Use a **dual-channel approach**:

1. **System prompt injection**: The kernel injects an XML skill catalog into the system prompt of every agent. The catalog lists available skills (name + description) and instructs the agent to use `agh skill view <name>` to load full instructions.

2. **CLI commands**: The AGH binary exposes `agh skill view <name>` and `agh skill view <name> --file <path>` commands. Agents invoke these via their Bash/shell tool to load skill content on demand.

No MCP server. No config file modification. No driver changes.

## Alternatives Considered

### Alternative 1: Embedded MCP Server

- **Description**: AGH runs an MCP server (stdio) that exposes skill tools. Each driver configured to connect to it via mcpServers config.
- **Pros**: Standard protocol. Clean tool namespacing (`mcp__agh-skills__*`).
- **Cons**: Only Claude Code supports MCP via CLI flag (`--mcp-config`). Codex, Pi, OpenCode require config file modification. Adds IPC complexity.
- **Why rejected**: Requires modifying driver configs, which risks conflicting with user settings. Unnecessary infrastructure for prompt-only skills.

### Alternative 2: Native Driver Config Sync

- **Description**: Copy SKILL.md files into each driver's native skill directory (e.g., `.claude/skills/`, `.codex/skills/`).
- **Pros**: Skills appear natively in each tool. Slash commands work within each IDE.
- **Cons**: Risk of overwriting user's existing skills. Different formats per driver. Requires adapter per driver.
- **Why rejected**: Conflict risk with user-installed skills. Maintenance burden of per-driver adapters.

### Alternative 3: File-Read Activation (Direct Path)

- **Description**: Include SKILL.md file paths in the catalog. Agents use their Read tool to load files directly.
- **Pros**: No CLI command needed. Simple.
- **Cons**: Agents read raw files including frontmatter. No kernel control over what's returned. No security scanning at access time. No resource listing.
- **Why rejected**: Less control. The CLI command allows the kernel to strip frontmatter, scan content, and include resource listings.

## Consequences

### Positive

- Zero driver modification — works with all current and future drivers
- Zero config file conflicts — never touches user configs
- Kernel controls access — can scan, filter, and format responses
- Portable — any driver that has shell/Bash access can use `agh skill view`
- Progressive disclosure — catalog in prompt (tier 1), full content via CLI (tier 2), resources via CLI (tier 3)

### Negative

- Agents must have Bash/shell tool access to invoke `agh skill view`
- Slightly less ergonomic than native slash commands within each IDE

### Risks

- An agent without Bash access cannot load skills beyond the catalog. Mitigation: all 4 current drivers have Bash/shell tools.

## Implementation Notes

The `agh skill view` command returns content wrapped in `<skill_content>` XML tags with a `<skill_resources>` listing. This structured output helps agents distinguish skill content from other command output.
