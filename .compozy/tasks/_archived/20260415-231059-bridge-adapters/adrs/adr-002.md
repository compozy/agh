# ADR-002: Hardened Webhook + REST Provider Communication

## Status

Accepted

## Date

2026-04-15

## Context

Bridge providers need to both receive messages from platforms and send messages back. Platforms offer different communication patterns:

- **Telegram**: Bot API with getUpdates (long-polling) or webhooks + sendMessage REST
- **Slack**: Events API (webhooks) or Socket Mode (WebSocket) + Web API REST
- **Discord**: Interactions API (webhooks) or Gateway (WebSocket) + REST API
- **WhatsApp**: Cloud API webhooks + REST API

AGH's bridge direction keeps providers isolated from daemon HTTP handling, so providers own their inbound surface. Prior research from GoClaw, Hermes, and OpenClaw also showed that production bridge endpoints need more than signature verification: request guards, dedup, batching, and actionable error handling matter in practice.

## Decision

All providers use the **webhook + REST** pattern with hardened ingress:

1. **Inbound**: Each provider subprocess runs its own HTTP server to receive platform webhooks. The provider validates method/content-type/body size, enforces rate and in-flight limits, verifies signatures, applies adapter-local dedup and optional batching, maps the payload to a typed bridge event, and calls the daemon Host API with an explicit `bridge_instance_id`.

2. **Outbound**: When the daemon sends `bridges/deliver`, the provider uses the platform's REST or GraphQL API to post/edit/delete messages and other v1-supported interaction effects.

3. **Webhook URL**: Webhook addressing is provider and bridge-instance configuration, not a static manifest concern. The operator configures the platform to send webhooks to the provider's public URL.

4. **Error handling**: The shared SDK classifies provider failures into actionable classes such as `auth`, `rate_limit`, `timeout`, `transient`, and `permanent`, and maps them to retry/backoff or status transitions such as `Degraded` and `AuthRequired`.

## Alternatives Considered

### Alternative 1: Long-polling for Telegram

- **Description**: Use Telegram's `getUpdates` API instead of webhooks
- **Pros**: no public URL needed, simpler local development
- **Cons**: higher latency, polling overhead, inconsistent provider pattern
- **Why rejected**: webhook is the production pattern; long-polling can be added later as explicit provider config

### Alternative 2: Daemon-proxied webhooks

- **Description**: Daemon exposes a single webhook endpoint and routes to providers via JSON-RPC
- **Pros**: one public endpoint, provider does not need an HTTP server
- **Cons**: daemon becomes a routing bottleneck, needs platform-specific parsing, absorbs webhook defense complexity, tighter coupling
- **Why rejected**: violates the extension isolation principle

## Consequences

### Positive

- each provider fully owns its platform protocol and ingress hardening
- shared SDK can enforce consistent webhook defense, dedup, batching, and error classification
- providers can be developed and tested independently with platform-specific tooling

### Negative

- each provider needs a public URL for webhooks
- more processes listen on ports in production
- webhook URL configuration remains a platform/operator concern

### Risks

- webhook delivery failures or platform retries can amplify load if ingress is under-defended
- mitigate with shared request guards, adapter-local dedup, daemon dedup, and classified recovery actions
