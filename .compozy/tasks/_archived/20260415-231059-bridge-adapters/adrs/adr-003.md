# ADR-003: Bridge V1 Scope Instead of Full Chat-SDK Parity

## Status

Accepted

## Date

2026-04-15

## Context

The `.resources/chat` TypeScript SDK implements rich platform features across eight adapters. That breadth is useful as product reference material, but AGH's current bridge transport does not yet model every interaction family required for full parity.

Trying to force full parity in v1 would either require a much larger daemon/bridge protocol redesign up front or push critical features into opaque metadata blobs, weakening portability and testability.

## Decision

Implement a narrower but production-credible bridge v1 for all eight providers.

Bridge v1 required scope:

- inbound and outbound text messages
- attachments
- thread-aware routing and targets
- streaming textual delivery
- message edit/delete semantics
- auth and health reporting
- per-instance provider configuration
- DM policy enforcement

Bridge v1 typed optional interaction scope:

- `command`
- `action`
- `reaction`

Out of scope for v1:

- modal lifecycle orchestration
- ephemeral delivery portability
- typing indicators
- approval UI
- two-lane reasoning versus answer streaming
- credential rotation pools

## Alternatives Considered

### Alternative 1: Full parity immediately

- **Description**: Implement every feature family from chat-sdk in the first bridge release
- **Pros**: maximum feature ambition and closer short-term product marketing parity
- **Cons**: requires a larger bridge protocol redesign, reopens daemon/runtime scope, and encourages opaque metadata shortcuts if rushed
- **Why rejected**: bridge v1 should be strong, typed, and implementable, not broad but structurally underspecified

### Alternative 2: Text-only bridge substrate

- **Description**: Keep bridge v1 to plain inbound/outbound messaging and defer everything else
- **Pros**: minimal initial protocol surface
- **Cons**: too weak for Slack, Discord, GitHub, and Linear workflows; misses high-value interactive flows
- **Why rejected**: commands, actions, and reactions are important enough to model explicitly now

## Consequences

### Positive

- the bridge protocol stays typed and coherent
- all eight providers still land in scope
- high-value interactive flows are modeled without overcommitting to every UI family
- future bridge protocol expansion has a cleaner base

### Negative

- some chat-sdk features move to later bridge revisions
- provider docs must clearly distinguish v1-required versus deferred capabilities
- rich platform-native UI remains provider-specific until later protocol revisions

### Risks

- Teams, GitHub, and Linear still have complex provider-specific flows even under the narrower scope
- mitigate with focused provider conformance tests and explicit per-provider capability docs
