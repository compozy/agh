# ADR-004: Foundation-First Build Order

## Status

Accepted

## Date

2026-04-09

## Context

The redesign involves 5 work streams: design tokens, sidebar/layout, session chat update, Skills page, and Knowledge page. These have natural dependencies — you can't build pages without tokens, and pages need the sidebar to navigate to them.

## Decision

Build in strict dependency order (Foundation → Layout → Pages):

1. **Design tokens + fonts** — Replace styles.css, update shadcn theme mapping. No dependencies.
2. **Custom sidebar + app shell** — Icon rail, panel, workspace selector, agent list, nav items. Depends on step 1.
3. **Session/chat view update** — Redesign existing session page to match Paper. Depends on steps 1-2.
4. **Skills page** (Installed + Marketplace) — New route, system, three-panel layout. Depends on steps 1-2. Backend: skills HTTP endpoints.
5. **Knowledge page** — New route, system, three-panel layout. Depends on steps 1-2. Backend: memory HTTP endpoints.

Steps 3, 4, and 5 can run in parallel after step 2 completes.

## Alternatives Considered

### Alternative 1: Pages in Parallel from Start

- **Description**: Build tokens + sidebar, then all pages simultaneously
- **Pros**: Faster total calendar time
- **Cons**: More merge coordination, shared component conflicts
- **Why rejected**: Parallel execution is possible for steps 3-5 but the foundation (1-2) must be sequential

### Alternative 2: User-Facing First

- **Description**: Session chat redesign first, then Skills, then Knowledge, sidebar last
- **Pros**: Most-used feature gets updated first
- **Cons**: Working on session page without new tokens/sidebar means rework, sidebar is prerequisite for navigation to new pages
- **Why rejected**: Cannot navigate to new pages without sidebar nav items

## Consequences

### Positive

- Each step produces a working, visually consistent app
- Clear dependency chain with no circular blocks
- Steps 3-5 parallelizable once foundation is complete

### Negative

- Steps 1-2 are sequential blockers — no progress on pages until sidebar is done

### Risks

- **Risk**: Token replacement breaks existing session page — Mitigation: update session components as part of step 1, visual verification
- **Risk**: Custom sidebar takes longer than expected — Mitigation: keep sidebar scope tight, match Paper exactly without overengineering

## Implementation Notes

- Step 1 deliverable: styles.css replaced, all existing components updated, `make web-lint && make web-typecheck` passes
- Step 2 deliverable: sidebar matches Paper, navigation to empty Skills/Knowledge routes works
- Steps 3-5 deliverable: each page matches its Paper artboard, data flows from backend
