# ADR-001: Full Replace of Design Token System

## Status

Accepted

## Date

2026-04-09

## Context

The current web app uses OKLCH color space, Geist Variable + Bricolage Grotesque fonts, gradients, texture overlays, box-shadows, and `color-mix()` — all of which conflict with the DESIGN.md specification. DESIGN.md mandates hex colors, Inter + JetBrains Mono fonts, and a flat depth model with zero shadows. The Paper designs confirm these requirements across all 17 artboards.

The current token system (`--ds-canvas`, `--ds-panel-base`, etc.) uses 30+ OKLCH-based custom properties in `styles.css`, and components reference them via `color:var(--ds-*)` inline styles and utility classes like `ds-texture-canvas`, `ds-panel`, etc.

## Decision

Perform a full replacement of the design token system in a single pass:

1. Replace all OKLCH color values with hex values from DESIGN.md
2. Replace Geist Variable + Bricolage Grotesque with Inter + JetBrains Mono
3. Remove all gradient backgrounds, texture overlays (`.ds-texture-canvas`), box-shadows, and `color-mix()` expressions
4. Rename CSS custom properties to match DESIGN.md token names (`--color-canvas`, `--color-surface`, etc.)
5. Update all component styles to reference the new tokens

## Alternatives Considered

### Alternative 1: Incremental Migration

- **Description**: Keep current tokens, add new DESIGN.md tokens in parallel, migrate components page by page
- **Pros**: Lower risk per change, pages stay functional during migration
- **Cons**: Two competing token systems create confusion, doubles CSS size, temptation to leave old tokens indefinitely
- **Why rejected**: Greenfield alpha policy — no compat layers, no migration code

### Alternative 2: New Tokens Only for New Pages

- **Description**: Keep current session page as-is, apply DESIGN.md tokens only to Skills/Knowledge pages
- **Pros**: Existing session page keeps working untouched
- **Cons**: Visual inconsistency between pages, divergent styling approaches, blocks eventual unification
- **Why rejected**: Creates technical debt on day one, contradicts greenfield alpha philosophy

## Consequences

### Positive

- Single, authoritative token system matching DESIGN.md and Paper designs
- All pages visually consistent from the start
- No dead CSS or unused font imports

### Negative

- Every existing component needs style updates in one batch
- Potential for regressions in session page during migration

### Risks

- **Risk**: Session chat view regression — Mitigation: visual diff testing before/after, component-level review
- **Risk**: shadcn component styling conflicts — Mitigation: update shadcn theme variables (`--primary`, `--border`, etc.) to map to new tokens

## Implementation Notes

- Remove `@fontsource-variable/geist` and `@fontsource/bricolage-grotesque` imports
- Add `@fontsource-variable/inter` and `@fontsource/jetbrains-mono` imports
- Delete all `.ds-texture-*`, `.ds-panel` CSS classes and their `::before`/`::after` pseudo-elements
- Map shadcn theme variables to new DESIGN.md tokens (e.g., `--primary: #E8572A`, `--border: #3A3A3C`)
