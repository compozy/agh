# ADR-003: Full Systems Architecture for Skills and Knowledge

## Status

Accepted

## Date

2026-04-09

## Context

The Paper design introduces two new major pages: Skills (Installed + Marketplace views) and Knowledge/Memory. These require:
- Backend HTTP endpoints for CRUD operations
- Frontend systems (adapters, hooks, components) following the app-renderer-systems pattern
- The backend already has `internal/skills/` and `internal/memory/` packages but no HTTP endpoints

The question was whether to build UI shells with mock data or wire up complete data layers from day one.

## Decision

Build complete systems from day one — both backend HTTP endpoints and frontend systems modules:

1. **Backend**: Add HTTP endpoints in `internal/api/httpapi/` for Skills and Knowledge resources
2. **Frontend**: Create `src/systems/skill/` and `src/systems/knowledge/` following the existing systems pattern (adapters → lib → hooks → components)
3. Use TanStack Query with polling (`refetchInterval`) for data freshness — no SSE for these pages
4. Keep SSE exclusively for session chat streaming

## Alternatives Considered

### Alternative 1: UI Shells with Mock Data

- **Description**: Build all pages with static/mock data, wire real APIs later
- **Pros**: Unblocks frontend from backend, faster initial visual progress
- **Cons**: Mock data diverges from real API shapes, double work (mock → real), delays integration testing
- **Why rejected**: Full-stack TechSpec scope was chosen — we want real data flowing end-to-end

### Alternative 2: Progressive (Real for Existing, Mock for New)

- **Description**: Real data for sessions/agents/workspaces, mock for Skills/Knowledge
- **Pros**: Pragmatic middle ground
- **Cons**: Inconsistent data layer patterns, still defers integration
- **Why rejected**: With backend packages already existing, the marginal effort to add HTTP endpoints is small

## Consequences

### Positive

- End-to-end data flow from day one, catching integration issues early
- Consistent systems architecture across all domains
- Real data in dev environment from the start

### Negative

- Backend and frontend must advance in lockstep
- More total work in each build step

### Risks

- **Risk**: Backend endpoint development blocks frontend — Mitigation: define API contract types first, implement adapters against the contract, backend fills in handlers
- **Risk**: Skills/Memory packages may need API surface changes — Mitigation: review existing package APIs before defining HTTP endpoints

## Implementation Notes

- Skills endpoints: `GET /api/skills` (list), `GET /api/skills/:name` (detail), `POST /api/skills/:name/enable`, `POST /api/skills/:name/disable`
- Knowledge endpoints: `GET /api/memories` (list), `GET /api/memories/:id` (detail), `DELETE /api/memories/:id`
- TanStack Query: `staleTime: 30s`, `refetchInterval: 60s` for list views
- Frontend adapters use typed error classes per existing convention
