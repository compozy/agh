# ADR-003: Canonical Tool Sets per Agent Type

## Status

Accepted

## Date

2026-03-31

## Context

AGH supports five agent types (master, worker, advisor, reviewer, researcher) and four drivers (Claude, Codex, OpenCode, Pi). Each driver has its own tool naming convention but maps to seven kernel canonical tools: `read`, `write`, `edit`, `bash`, `grep`, `glob`, `list`. The `StartOpts.Tools` field exists but is never populated, so all agents receive unrestricted tool access regardless of their type.

## Decision

Define **canonical tool sets per agent type** in the prompt package. A new `ToolsForAgentType(agentType string) []string` function returns the allowed kernel tool names for each type. These are populated into `StartOpts.Tools` during agent spawn, and each driver translates them to driver-specific names using existing translation maps.

| Agent Type | Allowed Tools | Rationale |
|---|---|---|
| master | `read`, `grep`, `glob`, `list`, `bash` | Read context + `agh` CLI (bash restricted via prompt) |
| worker | `read`, `write`, `edit`, `bash`, `grep`, `glob`, `list` | Full implementation capability |
| advisor | `read`, `grep`, `glob`, `list` | Read-only consultation |
| reviewer | `read`, `bash`, `grep`, `glob`, `list` | Read + verification (tests/lint/build) |
| researcher | `read`, `grep`, `glob`, `list` | Strictly read-only |

## Alternatives Considered

### Alternative 1: Per-Role Tool Configuration

- **Description**: Add an `AllowedTools []string` field to `RoleConfig` so each role definition specifies its own tool set.
- **Pros**: Maximum flexibility — custom roles can have custom tool sets.
- **Cons**: Every role file must define tools. Missing or incorrect tool definitions create security gaps. Higher maintenance burden.
- **Why rejected**: Agent types already establish clear capability tiers. Per-role override can be added later if needed, but type-level defaults cover all current use cases.

### Alternative 2: Driver-Level Enforcement Only

- **Description**: Let each driver decide tool restrictions based on the agent type string, without kernel-level canonical sets.
- **Pros**: Each driver can optimize for its own capabilities.
- **Cons**: Inconsistent enforcement across drivers. No central source of truth for what each agent type can do. Each driver must independently implement the same logic.
- **Why rejected**: Canonical sets at the kernel level ensure consistency and make the policy visible and testable in one place.

## Consequences

### Positive

- Single source of truth for agent capabilities per type
- All four drivers already support `StartOpts.Tools` translation — no driver changes needed
- Type-level toolsets are simple, testable, and auditable
- Master agent physically cannot use `write`/`edit` tools

### Negative

- No per-role customization (all workers get the same tools regardless of role specialization)
- Adding a new canonical tool requires updating the type toolsets

### Risks

- If a future driver introduces tools outside the canonical seven, they won't be filtered. Mitigation: the canonical set covers all current tools; new tools should be added to the canonical list and assigned to appropriate types.
