# ADR-001: Runtime + Prompt Dual Enforcement for Role Boundaries

## Status

Accepted

## Date

2026-03-31

## Context

The AGH master/supervisor agent writes code directly instead of delegating to worker agents. The `StartOpts.Tools` field exists in the kernel but is never populated during agent spawn, giving all agents unrestricted tool access. The master template contains coordination guidance but no explicit prohibitions, so the underlying LLM treats role boundaries as suggestions rather than constraints.

Reference implementations (Claude Code coordinator mode, Hermes delegate tool) solve this with dual enforcement: prompt-level MUST NOT constraints combined with runtime tool filtering that physically removes capabilities from agents.

## Decision

Enforce role boundaries through **both** runtime tool filtering (populating `StartOpts.Tools` per agent type) **and** prompt-level MUST NOT constraints in the master template. Neither mechanism alone is sufficient — prompt constraints can be ignored by the LLM, and tool filtering alone doesn't guide the agent toward correct delegation behavior.

## Alternatives Considered

### Alternative 1: Prompt-Only Enforcement

- **Description**: Rewrite master template with MUST NOT constraints and delegation protocol without changing runtime behavior.
- **Pros**: Zero runtime changes, fast to implement.
- **Cons**: LLMs can override prompt instructions, especially under complex goals. No hard guarantee that the supervisor won't write code.
- **Why rejected**: Does not provide defense in depth. Claude Code and Hermes both use runtime enforcement as the primary mechanism, with prompts as guidance.

### Alternative 2: Full Tool Isolation (No Bash for Master)

- **Description**: Remove all implementation tools including Bash from the master, requiring a separate IPC mechanism for `agh` CLI commands.
- **Pros**: Absolute enforcement — master literally cannot execute anything.
- **Cons**: Requires significant refactoring of the transport layer. The `agh` CLI is the primary coordination mechanism and runs via Bash.
- **Why rejected**: Disproportionate engineering cost. Bash with prompt guardrails ("ONLY for `agh` commands") is the same pattern Claude Code uses for its coordinator mode.

## Consequences

### Positive

- Master agent is physically prevented from using write/edit tools by driver-level enforcement
- Prompt constraints guide delegation behavior and teach the workflow
- Consistent with proven patterns from Claude Code and Hermes
- Uses existing `StartOpts.Tools` infrastructure — no new runtime abstractions needed

### Negative

- Bash access for master means it could theoretically write files via shell commands, though prompt constraints discourage this
- Each new agent type requires a toolset definition to be maintained

### Risks

- If a driver does not honor `StartOpts.Tools`, enforcement degrades to prompt-only. Mitigation: all four drivers (Claude, Codex, OpenCode, Pi) already implement tool translation and filtering.
