# ADR-004: Master Template Rewrite with Coordinator Pattern

## Status

Accepted

## Date

2026-03-31

## Context

The current master template (`internal/prompt/templates/master.md`) provides coordination guidance but lacks explicit behavioral constraints. It says "you own coordination" but never prohibits direct implementation. When paired with unrestricted tool access, the master agent defaults to executing tasks directly instead of delegating to specialized agents.

Claude Code's coordinator system prompt demonstrates a proven pattern: explicit MUST NOT constraints, a structured delegation protocol (Research → Synthesis → Implementation → Verification), anti-patterns with examples, a continue-vs-spawn decision framework, and a worker capabilities table.

## Decision

Rewrite the master template with the following structure, adapted from Claude Code's coordinator prompt for AGH's multi-agent architecture:

1. **ROLE** — Identity as coordinator, not implementer
2. **MUST NOT** — Explicit prohibitions (write files, run builds, modify git, etc.)
3. **MUST** — Explicit obligations (delegate to workers, verify with evidence)
4. **BASH RESTRICTIONS** — "ONLY for `agh` commands"
5. **COMMANDS AVAILABLE** — Existing `agh` CLI reference
6. **AGENT CAPABILITIES** — Table of tools per agent type
7. **DELEGATION PROTOCOL** — Research → Synthesis → Implementation → Verification
8. **WRITING AGENT PROMPTS** — Anti-patterns and best practices
9. **CONTINUE VS SPAWN** — Decision framework
10. **SELF-IMPROVEMENT** — Can create new roles via `agh roles create`
11. **BOOT SEQUENCE** — Existing boot steps
12. **RULES** — Existing coordination rules
13. **ERROR HANDLING** — Existing error handling

## Alternatives Considered

### Alternative 1: Minimal Patch (Add MUST NOT Only)

- **Description**: Keep existing template, prepend a MUST NOT section.
- **Pros**: Minimal change, low risk.
- **Cons**: Template remains unstructured. Missing delegation protocol, anti-patterns, and capabilities table. Master doesn't learn HOW to delegate effectively, only that it shouldn't implement.
- **Why rejected**: Preventing bad behavior is necessary but not sufficient. The master also needs positive guidance on delegation patterns, synthesized from proven coordinator prompts.

### Alternative 2: Copy Claude Code Coordinator Prompt Verbatim

- **Description**: Adapt the Claude Code coordinator prompt directly, replacing Agent/SendMessage/TaskStop with `agh` equivalents.
- **Pros**: Proven at scale, comprehensive.
- **Cons**: Claude Code's coordinator is designed for a different model (parent spawns children, all Claude). AGH has multi-driver, multi-model workgroups with NATS messaging, blackboard state, and recursive workgroup structure.
- **Why rejected**: AGH's coordination model is richer (workgroups, advisors, researchers, hooks, blackboard). The template must teach AGH-specific patterns, not Claude Code's simpler parent/child model.

## Consequences

### Positive

- Master agent has clear behavioral boundaries with explicit prohibitions
- Delegation protocol teaches effective coordination patterns
- Anti-patterns prevent common mistakes (lazy delegation, skipping synthesis)
- Agent capabilities table gives master complete awareness of what each type can do
- Self-improvement section enables runtime role creation

### Negative

- Larger system prompt (more tokens consumed per master agent)
- Template is more complex to maintain
- Changes to agent type toolsets must be reflected in the capabilities table

### Risks

- Over-constraining the master could make it too rigid for novel situations. Mitigation: the template teaches principles, not rigid procedures. The master retains coordination autonomy within its constraints.
