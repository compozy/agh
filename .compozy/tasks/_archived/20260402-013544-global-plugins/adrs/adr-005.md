# ADR-005: Zero Workdir Pollution via CLI Flags and Environment Variables

## Status

Accepted

## Date

2026-04-01

## Context

Beyond hook files, AGH drivers also generate runtime configuration files in the user's working directory:

- **Codex**: `.codex/AGENTS.md` (system prompt)
- **OpenCode**: `opencode.json` (model, prompt, tools, permissions)
- **Pi**: `.pi/SYSTEM.md` (system prompt), `.pi/AGENTS.md` (runtime context)

These files are generated every session and never cleaned up. Each target CLI supports alternative mechanisms to receive this configuration:

- **Codex**: `-c 'developer_instructions="..."'` flag
- **OpenCode**: `OPENCODE_CONFIG_CONTENT` environment variable (inline JSON)
- **Pi**: `--system-prompt "..."` and `--append-system-prompt "..."` flags

## Decision

Eliminate all per-session file writes to user's working directory by passing runtime configuration via CLI flags and environment variables native to each CLI.

| Driver | Before (file) | After (flag/env) |
|--------|---------------|-------------------|
| Claude | `.agh/settings-*.json` via `--settings` | `--bare --system-prompt` (no settings file) |
| Codex | `.codex/AGENTS.md` | `-c 'developer_instructions="..."'` |
| OpenCode | `opencode.json` | `OPENCODE_CONFIG_CONTENT` env var |
| Pi | `.pi/SYSTEM.md` + `.pi/AGENTS.md` | `--system-prompt` + `--append-system-prompt` |

Combined with ADR-001 (global plugins for hooks), this achieves **zero files** written to the user's working directory.

## Alternatives Considered

### Alternative 1: Write to AGH-Managed Directory Instead

- **Description**: Write the same files to `~/.agh/agents/<name>/` and point each CLI to that location.
- **Pros**: Minimal code change. Stops polluting user's project.
- **Cons**: Still generates files every session. Not all CLIs support custom config directory paths. Adds cleanup responsibility.
- **Why rejected**: CLI flags and env vars are simpler, require no cleanup, and are the intended way to pass config programmatically.

### Alternative 2: Keep File Writes for Drivers Without Flag Support

- **Description**: Only eliminate files where CLI flags exist. Keep file writes for CLIs that don't have equivalent flags.
- **Pros**: Lower risk per driver.
- **Cons**: All four CLIs have been confirmed to support non-file alternatives. Keeping files for some creates inconsistency.
- **Why rejected**: All drivers can be migrated. No reason for partial approach.

## Consequences

### Positive

- User's project directory is never modified by AGH
- No cleanup logic needed (no files to clean up)
- Each driver uses the CLI's native programmatic interface
- Claude's `--bare` flag ensures complete isolation from user's local settings

### Negative

- Long system prompts passed via CLI args may hit shell argument length limits on some systems (typically 2MB+ on modern systems, unlikely to be reached)
- `OPENCODE_CONFIG_CONTENT` has known precedence bugs (Issue #11628) where `.opencode` directory configs can override it

### Risks

- Shell argument limits for system prompts. Mitigation: modern systems support 2MB+ arguments. System prompts are typically under 100KB.
- OpenCode config precedence bug. Mitigation: monitor upstream issue #11628. Fallback to `OPENCODE_CONFIG` env var (file path) if needed.
