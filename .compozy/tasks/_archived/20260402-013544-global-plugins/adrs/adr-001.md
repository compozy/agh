# ADR-001: Global Plugins Over Per-Session File Generation

## Status

Accepted

## Date

2026-04-01

## Context

Each AGH driver (Claude, Codex, OpenCode, Pi) generates hook configuration files in the user's working directory every time an agent starts. These files (`.agh/settings-*.json`, `.codex/hooks.json`, `.pi/extensions/agh-hook.ts`) pollute the user's project directory, are never cleaned up automatically, and are regenerated identically on every session. The content is static except for the agent name, which can be passed dynamically via environment variables.

All four target CLIs (Claude Code, Codex, OpenCode, Pi) support global plugin/extension directories that are loaded automatically at startup. This provides a natural mechanism for permanent, install-once hook integrations.

## Decision

Replace per-session hook file generation with **permanent global plugins/extensions** installed once per machine. Each driver gets a plugin tailored to its CLI's native plugin system:

- **Claude**: Formal plugin via local marketplace (`.claude-plugin/` structure), installed at user scope.
- **Codex**: Global `~/.codex/hooks.json` + shell forwarder script (Codex has no programmatic plugin API for hooks).
- **OpenCode**: TypeScript plugin at `~/.config/opencode/plugins/agh-hook.ts` (auto-discovered).
- **Pi**: TypeScript extension at `~/.pi/agent/extensions/agh-hook.ts` (auto-discovered).

Plugins use environment variable gating (`AGH_AGENT_NAME` presence) to activate only when running under AGH orchestration. Plugin source files are embedded in the AGH binary via `go:embed` and extracted by `agh install`.

## Alternatives Considered

### Alternative 1: Inline CLI Arguments

- **Description**: Pass hook config as CLI arguments (e.g., Claude's `--settings '{json}'` inline, Codex's `-c` flag).
- **Pros**: No files at all, not even in global directories. Zero installation step.
- **Cons**: Not all CLIs support inline hook config (Codex hooks.json has no CLI flag). Shell escaping of complex JSON is fragile. Does not leverage native plugin ecosystems.
- **Why rejected**: Incomplete coverage (Codex has no inline hook mechanism for PostToolUse events). Plugin approach is more robust and maintainable.

### Alternative 2: Keep Per-Session Files But Write to Temp Directory

- **Description**: Continue generating hook files per session but write them to `~/.agh/agents/<name>/` instead of the user's workdir.
- **Pros**: Stops polluting user's project. Minimal code change.
- **Cons**: Still generates files every session. Requires cleanup logic. Does not leverage native plugin ecosystems. Needs each CLI to be pointed at a non-standard config path.
- **Why rejected**: Does not solve the root problem (unnecessary repeated file generation). Global plugins are the idiomatic approach for each CLI.

### Alternative 3: HTTP/SSE Event Streaming for All Drivers

- **Description**: All drivers use a streaming approach (like OpenCode's SSE) instead of command-based hooks.
- **Pros**: Eliminates all hook files. Real-time event consumption.
- **Cons**: Requires major rearchitecture of Claude, Codex, and Pi drivers from PTY to stream consumers. Some CLIs don't support JSON streaming mode in interactive use.
- **Why rejected**: Disproportionate engineering cost. Plugin approach achieves the same goal with minimal driver changes.

## Consequences

### Positive

- Zero files written to user's working directory for hooks
- Plugins installed once, work forever across all sessions and projects
- Leverages native plugin systems of each CLI (idiomatic integration)
- Environment variable gating ensures zero overhead when CLI is used outside AGH
- `agh install` / `agh uninstall` provides clean lifecycle management

### Negative

- Requires `agh install` as a setup step before first use
- Claude plugin requires marketplace scaffolding (more complex than a simple file)
- Codex hooks.json merge requires careful handling of existing user hooks

### Risks

- CLI updates may change plugin APIs or directory conventions. Mitigation: plugins are simple (just forward events) and embedded in the binary, so they can be updated with `agh install`.
- User may have conflicting hooks in global config. Mitigation: Claude uses isolated plugin system; Codex uses merge with AGH-specific markers; OpenCode and Pi use separate files.
