# ADR-004: Remove BuildHookConfig from AgentDriver Interface

## Status

Accepted

## Date

2026-04-01

## Context

The `AgentDriver` interface requires every driver to implement `BuildHookConfig(agentName, hookEndpoint) (*HookConfig, error)`. With the migration to global plugins, no driver needs to generate hook configuration at runtime:

- **Claude**: Hooks come from global plugin (user scope).
- **Codex**: Hooks come from global `~/.codex/hooks.json`.
- **OpenCode**: Already returns empty `HookConfig{}` (uses SSE stream).
- **Pi**: Hooks come from global extension.

The `HookConfig` struct (`ConfigContent`, `ConfigPath`, `ScriptContent`, `ScriptPath`) exists solely to describe files that should be written to disk — a pattern being eliminated.

## Decision

Remove `BuildHookConfig()` from the `AgentDriver` interface and delete the `HookConfig` struct. The project is alpha — no backward compatibility required.

Keep `ParseHookEvent(rawPayload []byte) (*HookEvent, error)` in the interface — each driver still needs to normalize incoming hook payloads from its CLI's format to the kernel's canonical `HookEvent`.

## Alternatives Considered

### Alternative 1: Keep Method, Return Empty

- **Description**: All drivers return `&HookConfig{}`. Method stays in interface as a placeholder.
- **Pros**: No interface change. Future drivers could use it.
- **Cons**: Dead code. Every driver has an unused method. Misleading interface contract.
- **Why rejected**: YAGNI. The project is alpha. If a future driver needs per-session hook config, the method can be re-added.

### Alternative 2: Replace with HookMode Enum

- **Description**: Replace `BuildHookConfig()` with `HookRequirements() HookMode` returning `HookModePlugin`, `HookModeSSE`, etc.
- **Pros**: Self-documenting. Kernel knows how each driver receives hooks.
- **Cons**: Adds abstraction for information that is currently unused by the kernel. The kernel does not branch on hook delivery mechanism.
- **Why rejected**: Over-engineering. The kernel treats all drivers uniformly after hook events arrive.

## Consequences

### Positive

- Cleaner `AgentDriver` interface (7 methods → 6 methods)
- Removes `HookConfig` struct and all file-writing logic from drivers
- Eliminates dead code paths in `Start()` methods
- `UnimplementedDriver` has one fewer stub to maintain

### Negative

- If a future driver needs per-session hook files, the method must be re-added
- All four drivers must be updated simultaneously (interface change)

### Risks

- None — the removal is part of a larger change that already modifies all drivers.
