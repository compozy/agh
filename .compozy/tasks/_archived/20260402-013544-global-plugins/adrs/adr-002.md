# ADR-002: Hybrid Plugin Strategy (Formal Plugins + File Merge)

## Status

Accepted

## Date

2026-04-01

## Context

The four target CLIs have different plugin/extension systems with varying levels of isolation:

- **Claude Code**: Formal plugin system with marketplace, plugin.json manifest, scope-based installation. Hooks in plugins are isolated from user's settings.json.
- **Codex**: No programmatic plugin API for hooks. Hooks live in `hooks.json` files that are additive across discovery paths. No isolation mechanism.
- **OpenCode**: TypeScript plugins auto-discovered from `~/.config/opencode/plugins/`. Each plugin is a separate file — natural isolation.
- **Pi**: TypeScript extensions auto-discovered from `~/.pi/agent/extensions/`. Each extension is a separate file — natural isolation.

The question is whether to use formal plugin systems where available or to use a uniform file-copy approach across all drivers.

## Decision

Use a **hybrid strategy**: formal plugin isolation where the CLI supports it, and file-based installation with merge logic where it does not.

- **Claude**: Formal plugin via local marketplace. Hooks are scoped to the plugin and do not touch the user's `~/.claude/settings.json`.
- **Codex**: Merge AGH hooks into `~/.codex/hooks.json` with identifiable markers. Remove only AGH entries on uninstall.
- **OpenCode**: Copy standalone `.ts` file to `~/.config/opencode/plugins/`. No merge needed — separate file.
- **Pi**: Copy standalone `.ts` file to `~/.pi/agent/extensions/`. No merge needed — separate file.

## Alternatives Considered

### Alternative 1: Uniform File Copy for All

- **Description**: Skip Claude's plugin system. Write hooks directly into `~/.claude/settings.json` like Codex.
- **Pros**: Simpler implementation. Same pattern everywhere.
- **Cons**: Modifying `~/.claude/settings.json` risks corrupting user's existing hooks, MCP servers, and env vars. Claude's plugin system exists precisely to avoid this.
- **Why rejected**: Claude's plugin system provides proper isolation. Using it is the correct approach.

### Alternative 2: Formal Plugins for All

- **Description**: Create proper plugin packages for every CLI.
- **Pros**: Maximum isolation everywhere.
- **Cons**: Codex has no plugin API for hooks — its plugin system only bundles skills, MCP servers, and app mappings. Cannot register hook handlers programmatically.
- **Why rejected**: Codex's architecture makes this impossible for hook registration.

## Consequences

### Positive

- Each driver uses the most appropriate mechanism for its CLI
- Claude hooks fully isolated from user settings
- OpenCode and Pi hooks are naturally isolated (separate files)
- Codex merge is manageable (hooks.json is a simple structure)

### Negative

- Two different installation patterns to maintain (plugin install vs file copy/merge)
- Codex merge logic must handle edge cases (missing file, malformed JSON, existing AGH entries)

### Risks

- Codex hooks.json format may change. Mitigation: the format is simple and stable; AGH entries are tagged for easy identification.
