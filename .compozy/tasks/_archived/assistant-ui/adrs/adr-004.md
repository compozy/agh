# ADR-004: Unify Transcript Replay on the AI SDK `UIMessage` Shape

## Status

Accepted

## Date

2026-04-21

## Context

`GET /api/sessions/:id/transcript` currently returns a `transcript.Message[]` payload defined in `internal/transcript/transcript.go`. The shape is AGH-specific: flattens assistant content + thinking into a single record, embeds tool results as a `ToolResult` sub-struct, and carries AGH-only fields (`ThinkingComplete`, `ToolError`). The live wire at `POST /api/sessions/:id/prompt` emits a completely different shape: AI SDK UI Message Stream frames (parts with `text-delta`, `reasoning-delta`, `tool-input-*`, `tool-output-*`).

Today, `web/src/systems/session/lib/transcript-mapper.ts` translates `transcript.Message` into the custom `UIMessage` frontend type, and `live-message-mapper.ts` translates AI SDK frames into the same custom type. Two mappers, two drift surfaces.

Assistant-ui's `ThreadHistoryAdapter.withFormat("ai-sdk/v6")` expects AI SDK `UIMessage[]` directly. Any other shape requires a translation layer on the frontend â€” exactly the layer the migration aims to delete.

## Decision

Migrate `GET /api/sessions/:id/transcript` to emit AI SDK `UIMessage[]` directly. Delete the frontend-visible `transcript.Message` contract and its frontend mapper. `transcript.Message` / `transcript.ToolResult` remain as internal Go implementation types and feed a new `transcript.ToUIMessages` projection.

## Alternatives Considered

### Alternative 1: Keep `transcript.Message` on the wire, translate on the client

- Description: leave `/transcript` as is, feed through a client-side adapter inside the history adapter.
- Pros: backend untouched.
- Cons: re-introduces the transcript mapper we want to delete; two representations of a message on the frontend; drift surface between live and replay; conflicts with the greenfield rule in `CLAUDE.md`.
- Why rejected: contradicts the cleanup goal and adds ongoing maintenance.

### Alternative 2: Keep both shapes behind a content-type negotiation

- Description: serve `transcript.Message` or `UIMessage[]` based on `Accept` header or a query flag.
- Pros: no hard break for any fictional third-party consumer.
- Cons: AGH has no third-party consumer yet; greenfield explicitly forbids dual paths; adds backend branching.
- Why rejected: no consumer requires it, and it violates the zero-legacy-tolerance rule.

### Alternative 3: Replace `transcript.Message` entirely, including internals

- Description: rewrite `internal/transcript` around AI SDK `UIMessage` as the only in-memory shape.
- Pros: one shape end-to-end.
- Cons: couples the internal event replay implementation to the AI SDK schema; harder to re-target if we later switch wire formats.
- Why rejected: keep AI SDK shape at the edge. Internal types can stay AGH-native.

## Consequences

### Positive

- One message shape on the frontend (`UIMessage`) shared by live and replay.
- Deletes `transcript-mapper.ts`, a frontend `UIMessage` type alias, and related test code.
- `ThreadHistoryAdapter.withFormat` integrates without a custom adapter.
- Aligns with the greenfield rule from `CLAUDE.md` and `web/CLAUDE.md`.

### Negative

- Hard break for the frontend transcript consumer â€” must ship with the frontend migration in the same series of PRs.
- `internal/transcript` gains a projection function with its own tests.
- `openapi/agh.json` regenerates with a different transcript schema.

### Risks

- Risk: part IDs drift between replay and live, causing assistant-ui to deduplicate or reorder messages incorrectly.
  - Mitigation: compute replay part IDs deterministically from stored event IDs; add a test that captures a live turn, replays it, and asserts identical part IDs and ordering.
- Risk: the projection misorders reasoning vs text or mispairs tool-call + tool-result.
  - Mitigation: co-locate `transcript.ToUIMessages` with a table-driven test covering interleaved reasoning, multi-tool-call turns, and error terminations.

## Implementation Notes

- Introduce `transcript.ToUIMessages(events []store.Event) ([]uiMessage, error)` in `internal/transcript/transcript.go`.
- Delete the `TranscriptMessage` field (or equivalent) from `internal/api/contract/contract.go`; add a `UIMessage` contract type mirroring the AI SDK schema.
- Regenerate `openapi/agh.json`.
- On the frontend, delete `web/src/systems/session/lib/transcript-mapper.ts` and all `UIMessage`/`UIMessageRole` custom definitions in `web/src/systems/session/types.ts`; import `UIMessage` from `ai` instead.
