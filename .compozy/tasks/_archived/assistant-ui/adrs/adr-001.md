# ADR-001: Adopt `assistant-ui` with AI SDK `useChatRuntime`

## Status

Accepted

## Date

2026-04-21

## Context

The AGH web frontend currently renders chat with ~2,500 LOC of hand-rolled React in `web/src/systems/session` — custom message bubbles, markdown renderer, thinking collapsible, tool-call cards, streaming buffer, live/history merger, and a TanStack-virtualized chat viewport. The backend at `POST /api/sessions/:id/prompt` already emits the AI SDK v6 UI Message Stream protocol (`x-vercel-ai-ui-message-stream: v1` header; frames for `text-delta`, `reasoning-delta`, `tool-input-start`, `tool-input-available`, `tool-output-available`, `data-agh-permission`, `data-agh-event`, `finish`). The frontend already uses `@ai-sdk/react::useChat()` with `DefaultChatTransport`.

Assistant-ui is a React library that composes a full chat surface (Thread, Message, Composer, ActionBar primitives plus shadcn-installable themed components) on top of a runtime abstraction. It ships three runtime adapters: `LocalRuntime` (client-side inference), `ExternalStoreRuntime` (bring-your-own-state), and `useChatRuntime` / `useAISDKRuntime` (wrap AI SDK's `useChat`).

We need a concrete reduction in custom UI code without changing the wire protocol, the persistence model, or the single-binary daemon.

## Decision

Adopt `assistant-ui` as the chat renderer and wire it through `useChatRuntime` from `@assistant-ui/react-ai-sdk`.

- The AI SDK `useChat` instance continues to consume `POST /api/sessions/:id/prompt` with `AssistantChatTransport`.
- `useChatRuntime` wraps that instance and exposes an `AssistantRuntime` that assistant-ui primitives consume.
- The thread surface is installed via the shadcn CLI into `web/src/components/assistant-ui/`, then themed with AGH tokens from `DESIGN.md`. We own the copied files.

## Alternatives Considered

### Alternative 1: `ExternalStoreRuntime` adapter

- Description: write a custom runtime that projects our own event store into `ThreadMessageLike[]`.
- Pros: zero dependency on AI SDK on the client; full control over message shape.
- Cons: we already translate ACP to AI SDK frames in Go; a second translation on the client is redundant and reintroduces the live/replay mapping we want to delete.
- Why rejected: duplicates work that `useChatRuntime` does for free.

### Alternative 2: Keep the custom renderer

- Description: do nothing; continue maintaining the 2.5K LOC of bespoke chat UI.
- Pros: no migration risk.
- Cons: recurring cost for markdown, virtualization, streaming buffering, tool-call pairing, scroll-locking, and every new AI SDK frame type. Large surface that is not AGH-specific.
- Why rejected: the alpha budget is not well spent on rebuilding a general-purpose chat renderer.

### Alternative 3: Run a Node sidecar with a server-side AI SDK provider

- Description: wrap ACP as a `LanguageModelV3` provider (such as the published `@mcpc-tech/acp-ai-provider`) in a TS process and talk to it from the frontend via AI SDK.
- Pros: reuses AI SDK provider ecosystem.
- Cons: violates AGH's single-binary constraint, flattens ACP permission/plan/mode semantics into prompt-shaped strings, and adds a deployment hop.
- Why rejected: architectural mismatch with ACP and with the AGH daemon model.

### Alternative 4: Use `useAISDKRuntime` directly without `useChatRuntime`

- Description: call `useChat` ourselves and hand its return value to `useAISDKRuntime`.
- Pros: slightly thinner wrapper; gives access to advanced `AssistantChatTransport` customization.
- Cons: we lose the built-in remote thread list, the history adapter glue, and the memoization that makes session switching safe.
- Why rejected: `useChatRuntime` is the sanctioned integration path for AI SDK v6; we only drop to `useAISDKRuntime` if we later need custom session list behavior.

## Consequences

### Positive

- Deletes the hand-rolled chat renderer, streaming buffer, live/history merger, and virtualization code.
- Uses an upstream library that evolves alongside AI SDK v6 with reasoning, tool-call status, attachments, and branch picking already covered.
- Keeps the wire protocol and the Go daemon untouched.
- Unlocks future features (branch switching, edit-message, voice, attachments) without additional custom code.

### Negative

- Pulls assistant-ui and a larger set of shadcn components into the bundle (~120-180 KB gz after typical UI blocks).
- Thread layout is opinionated; non-linear layouts would require heavier customization.
- We now depend on assistant-ui release cadence; minor releases have tightened adapter contracts in the past (see ADR-004 on `withFormat`).

### Risks

- Risk: tool-renderer props differ from our current `UIMessage`-based components.
  - Mitigation: introduce thin `makeAssistantToolUI` wrappers that adapt `ToolCallContentPartProps` to the existing renderer APIs.
- Risk: custom data parts (`data-agh-permission`, `data-agh-event`) are not rendered by default.
  - Mitigation: register each via `makeAssistantDataUI` (see ADR-003).
- Risk: session switching leaks SSE connections.
  - Mitigation: key `SessionChatRuntimeProvider` on `sessionId` so the subtree remounts.

## Implementation Notes

- Install `@assistant-ui/react` and `@assistant-ui/react-ai-sdk` via `bun add` only.
- Copy Thread / Composer / Markdown / Reasoning / ToolFallback via the assistant-ui shadcn CLI into `web/src/components/assistant-ui/`.
- `SessionChatRuntimeProvider` owns the `useChatRuntime` instance and is rendered at the session route level.
- Tool renderers stay inside `web/src/systems/session/components/tool-renderers/` and are wrapped with `makeAssistantToolUI` at registration time.
