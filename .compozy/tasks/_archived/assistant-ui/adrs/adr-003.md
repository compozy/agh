# ADR-003: Render AGH-Specific Flows as AI SDK Data Parts via `makeAssistantDataUI`

## Status

Accepted

## Date

2026-04-21

## Context

AGH emits two custom SSE frames outside the standard AI SDK set: `data-agh-permission` (tool approval prompts) and `data-agh-event` (fallback for ACP events without a canonical AI SDK mapping). Assistant-ui's runtime renders known parts (`text`, `reasoning`, `tool-*`) by default and ignores unknown data parts unless a consumer registers a renderer.

AI SDK v6 also ships a native human-in-the-loop flow (`requires-action` status on tool parts plus `interrupt`/`resume`) that could, in principle, replace `data-agh-permission`. Adopting it requires changes in `prompt.go`, the ACP translation layer, and the session-store / approve endpoint.

We need a rendering path for `data-agh-permission` that works immediately and does not force a backend refactor in the same phase.

## Decision

Render AGH-specific flows on the frontend via `makeAssistantDataUI`. `data-agh-permission` gets a dedicated `AghPermissionUI` that renders the existing `PermissionPrompt` component. `data-agh-event` remains on the wire for observability and is read by the inspector pane without being rendered in the thread viewport.

Do not migrate permission to AI SDK v6 `interrupt`/`resume` in this phase. Reassess once a second AGH-specific flow needs HITL semantics.

## Alternatives Considered

### Alternative 1: Adopt AI SDK v6 `tool-approval-request` / `interrupt` / `resume`

- Description: model permissions as a tool part that transitions to `requires-action`; emit `interrupt` and accept `resume` on approval.
- Pros: canonical AI SDK shape; participates in assistant-ui's default tool rendering.
- Cons: requires rewriting `prompt.go` permission emission, the `POST /approve` endpoint, and the session-store bookkeeping simultaneously with the frontend migration; scope creep.
- Why rejected for now: expands the blast radius of an already sizable migration. Revisit after phase 1 stabilizes.

### Alternative 2: Custom inline rendering inside `MessagePrimitive.Parts`

- Description: inspect part types inline and render permission UI where it appears in the message flow.
- Pros: no runtime registration; inline control.
- Cons: couples permission UI to the generic message part renderer; harder to theme and reuse; duplicates logic every time the thread is used.
- Why rejected: less composable than the sanctioned `makeAssistantDataUI` path.

### Alternative 3: Keep permission handling in Zustand state, outside assistant-ui

- Description: parse `data-agh-permission` out of the stream and render a modal via Zustand, leaving assistant-ui unaware.
- Pros: mirrors current code.
- Cons: re-introduces the mapper we are trying to delete; breaks the runtime/state contract that makes session switching reliable.
- Why rejected: contradicts the whole point of the migration.

## Consequences

### Positive

- Permission flow is rendered through the assistant-ui runtime tree, reusing its lifecycle and switching.
- No backend change required in phase 1.
- `data-agh-event` keeps working as observability telemetry for the inspector.

### Negative

- Temporarily diverges from the AI SDK v6 idiomatic HITL path.
- Adds a small schema surface (`AghPermissionData`) that must stay in sync between Go and TypeScript.

### Risks

- Risk: schema drift between `prompt.go` permission payload and the TS `AghPermissionData` schema.
  - Mitigation: validate with Zod on the frontend; add an integration test that replays a captured permission payload through `AghPermissionUI`.
- Risk: ergonomic friction if users expect permission prompts to participate in branch switching or history editing.
  - Mitigation: document that permissions are point-in-time events; revisit when moving to native interrupts.

## Implementation Notes

- `AghPermissionUI` is registered once under `AssistantRuntimeProvider` and renders nothing by itself.
- The permission data shape is defined in `web/src/systems/session/lib/permission-parts.ts` with a Zod schema that mirrors the Go payload.
- The approve action continues to POST `/api/sessions/:id/approve`.
- `data-agh-event` has no `makeAssistantDataUI` registration and is consumed exclusively by the inspector pane.
