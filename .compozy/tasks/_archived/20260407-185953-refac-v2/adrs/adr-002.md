# ADR-002: Make internal/api/contract the Canonical Shared API Contract

## Status

Accepted

## Date

2026-04-06

## Context

The codebase currently has a partially consolidated transport layer in `internal/apicore`, but the CLI still defines its own request and response DTOs in `internal/cli/client.go`. This creates a split-brain contract surface between daemon transports and CLI consumers.

In the broad `refac-v2` reorganization, the API layer will become an explicit subtree. That makes contract ownership a first-order decision: either the shared DTOs live beside the core handlers or they live in a dedicated package consumed by all transports and clients.

## Decision

Create `internal/api/contract` as the canonical home for shared API request/response DTOs.

- `api/contract` will define transport-agnostic request and response shapes used by CLI, `api/core`, `api/httpapi`, and `api/udsapi`.
- `api/core` will own shared handler logic, parsing, and transport-facing behavior, but not the canonical DTO definitions.
- Transport-specific envelopes, such as AI SDK streaming payloads that are not part of the general daemon contract, may remain in transport packages.

## Alternatives Considered

### Alternative 1: Keep payload ownership in api/core

- **Description**: Move CLI to import `api/core` payloads directly and keep DTOs in the same package as shared handlers.
- **Pros**: Fewer packages and simpler initial migration.
- **Cons**: Couples clients to server-core behavior and mixes contract ownership with handler implementation.
- **Why rejected**: The CLI should depend on a contract package, not on a handler-oriented core package.

### Alternative 2: Split stable DTOs and internal envelopes across multiple packages immediately

- **Description**: Create separate public contract, internal streaming envelope, and transport contract packages.
- **Pros**: Maximum conceptual purity.
- **Cons**: Higher up-front complexity and more migration overhead.
- **Why rejected**: The current code only clearly justifies one shared contract package plus transport-local exceptions.

## Consequences

### Positive

- Gives CLI and transports one source of truth for the daemon API surface.
- Reduces duplicated DTO maintenance and drift risk.
- Makes future API evolution more explicit.

### Negative

- Introduces another package into the API subtree.
- Requires careful separation between shared contract types and transport-only payloads.

### Risks

- Over-eager migration could move transport-specific streaming details into the shared contract.
- Mitigation: keep AI SDK SSE shapes and other transport-only envelopes local unless they are consumed by multiple clients.

## Implementation Notes

- Existing `internal/apicore/payloads.go` content should be evaluated type-by-type and moved into `internal/api/contract` only when it represents a shared daemon contract.
- `internal/cli/client.go` becomes the first major consumer to migrate to the new package.
