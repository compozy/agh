# ADR-001: Adopt a Broad Package-Graph Reorganization for Refac V2

## Status

Accepted

## Date

2026-04-06

## Context

The current AGH codebase already completed several tactical refactors such as `apicore` extraction, oversized file splits, and helper deduplication. The remaining findings are no longer isolated cleanup tasks: they cluster around package boundaries that still mix concerns, especially across API transport, persistence, transcript replay, frontmatter parsing, and dream consolidation.

The `refac-v2` effort is intended to produce a target architecture, not only a list of local cleanups. A path-preserving plan would reduce short-term import churn, but it would also keep large package boundaries in place and weaken the purpose of a second-generation refactor plan.

## Decision

`refac-v2` will target a broad package-graph reorganization with these end-state moves:

- Reorganize the API layer under an `internal/api/` subtree.
- Extract cross-package parsing into `internal/frontmatter`.
- Extract transcript replay into `internal/transcript`.
- Move dream orchestration out of `daemon` into `internal/memory/consolidation`.
- Reduce broad packages by introducing narrower domain and infrastructure boundaries where the current code already demonstrates distinct responsibilities.

The TechSpec will describe the target architecture explicitly, then sequence implementation in phases.

## Alternatives Considered

### Alternative 1: Preserve current package paths and refactor internally

- **Description**: Keep package layout mostly unchanged and limit work to file splits, helper extraction, and interface narrowing.
- **Pros**: Smaller diffs, lower import churn, easier incremental merges.
- **Cons**: Retains oversized package boundaries and leaves the architectural target implicit.
- **Why rejected**: `refac-v2` is intended to define a cleaner end-state architecture, not only an incremental cleanup queue.

### Alternative 2: Reorganize only the API subtree

- **Description**: Limit package moves to transport-related code and keep transcript, memory, and persistence mostly where they are.
- **Pros**: High ROI in one hotspot, lower total migration scope.
- **Cons**: Leaves other major boundary problems unresolved and fragments the architectural direction.
- **Why rejected**: The observed problems span multiple domains, not only API transport.

## Consequences

### Positive

- Creates an explicit target architecture for follow-on task generation.
- Converts recurring package-boundary discussions into concrete implementation intent.
- Aligns the refactor plan with the remaining high-impact findings in `refac-v2`.

### Negative

- Increases import churn and migration sequencing complexity.
- Requires stronger verification discipline during each structural phase.

### Risks

- Large package moves can obscure behavioral regressions if phases are too broad.
- Documentation can drift from the live tree if the plan ignores already-landed refactors.
- Mitigation: use phased cutovers, same-phase bridge removal when necessary, and mandatory verification gates.

## Implementation Notes

- The plan must reflect the current repository state, including the fact that `internal/apicore` already exists and `httpapi`/`udsapi` are already split by concern.
- The TechSpec should call out which proposed moves are new versus already partially implemented.
