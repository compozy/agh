**Date**: 2026-04-06

## Executive Summary

The CLI package is the largest in the codebase by file count (26 files, ~9.4K LOC including tests). It is well-structured for a flat package, with clear file-per-domain separation (session.go, daemon.go, workspace.go, etc.) and thorough test coverage. The dependency injection via `commandDeps` is sound.

However, several structural issues emerge at scale:

| Severity | Count | Description |
|----------|-------|-------------|
| P1       | 3     | Oversized client.go (types + transport), skill.go doing direct FS/DB work, triple output-format duplication |
| P2       | 5     | CLI importing `store`/`workspace`/`skills` directly, 25-method DaemonClient interface, stream handler duplication, integration test sharing harness code |
| P3       | 4     | Minor helpers, dead assignment, small code smells |

**Total findings**: 12

**Top 3 highest-impact opportunities**:
1. Extract `client.go` types into a shared API contract package (moderate effort, high impact)
2. Move skill resolution logic out of CLI into a skill service (moderate effort, high impact)
3. Deduplicate the output bundle pattern with generics or code generation (significant effort, medium impact)

### Finding 1: `client.go` mixes transport, DTO types, and query builders (P1, B)

**Smell**: Large Class / Divergent Change
**Location**: `internal/cli/client.go` (781 lines)
**Impact**: High -- any change to API contracts requires editing this file, as does any change to transport/SSE logic

`client.go` contains three distinct responsibilities in one file:

1. **DTO types** (lines 57-297): `SessionRecord`, `AgentRecord`, `WorkspaceRecord`, `ObserveEventRecord`, `HealthStatus`, `DaemonStatus`, `MemoryReadRecord`, `MemoryWriteRequest`, etc. -- 20+ exported types.
2. **HTTP/UDS transport** (lines 298-700): `unixSocketClient` struct, `doJSON`, `doSSE`, `doRequest`, `decodeSSE` -- the actual client implementation.
3. **Query value builders** (lines 701-780): `sessionListValues`, `sessionEventValues`, `observeEventValues`, `memoryValues`, `readAPIError`.

**Key concern**: These DTO types represent the daemon API contract. They are defined **only** in the CLI package, meaning the UDS API server has no shared contract with its client. The server serializes `session.SessionInfo` directly with `json` tags, and the CLI defines its own `SessionRecord` struct separately. This creates a fragile implicit coupling -- if a server-side JSON tag changes, the CLI silently breaks.

**Recommended action**: **(B) Package-level split**
- Create `internal/api/contract` package containing all shared DTO types (`SessionRecord`, `AgentRecord`, etc.) -- co-located inside the `api/` subtree, not as standalone `apitypes/` at root
- Both `api/udsapi`, `api/httpapi`, and `cli` import from this shared package
- Keep transport (`unixSocketClient`, SSE decoder) in `cli/` or extract to `internal/udsclient`

> **UPDATED (2026-04-06)**: Contract package lives at `api/contract` per Codex review (avoids split-brain with two centers of gravity for the API subsystem).

**Effort**: Moderate

### Finding 2: `DaemonClient` interface is too large -- 25 methods (P2, D)

**Smell**: Interface Segregation Principle violation
**Location**: `internal/cli/client.go:28-54`
**Impact**: Medium -- every test stub must implement all 25 methods (see `helpers_test.go` stubClient at 220 lines)

```go
type DaemonClient interface {
    DaemonStatus(ctx context.Context) (DaemonStatus, error)
    ListSessions(ctx context.Context, query SessionListQuery) ([]SessionRecord, error)
    CreateSession(ctx context.Context, request CreateSessionRequest) (SessionRecord, error)
    // ... 22 more methods
}
```

The `stubClient` in tests is 220 lines of boilerplate with one `func` per method, each checking if the corresponding `Fn` field is non-nil.

**Recommended action**: **(D) Inline fix** -- Split into domain-scoped interfaces consumed where needed:
- `SessionClient` (8 methods: CRUD + prompt + events + stream + history)
- `WorkspaceClient` (5 methods: CRUD + detail)
- `AgentClient` (2 methods: list + get)
- `ObserveClient` (3 methods: events + stream + health)
- `MemoryClient` (5 methods: list + read + write + delete + consolidate)
- `DaemonStatusClient` (1 method)

Each subcommand file would accept only the interface it needs. The concrete `unixSocketClient` would still implement all of them.

**Effort**: Moderate (mechanical but touches many files)

### Finding 3: `skill.go` does direct FS and database work -- bypasses daemon (P1, B)

**Smell**: Feature Envy / Coupling
**Location**: `internal/cli/skill.go` (920 lines, largest source file)
**Impact**: High -- the skill commands are the only CLI commands that bypass the daemon entirely

The skill commands import `skills`, `skills/bundled`, `store`, `workspace`, and `config` directly and perform:
- Skills registry creation and loading (line 279-286)
- Workspace resolution by opening the SQLite database directly (lines 334-367)
- Walking filesystem directories for resources (lines 485-556)
- Reading files from bundled `fs.FS` and on-disk paths (lines 558-626)

Every other CLI command goes through the `DaemonClient` interface (UDS API). Skills are the exception -- they call into internal packages directly.

This means:
- Skills work without the daemon running (potentially intentional)
- The CLI imports `store.OpenGlobalDB` -- a heavy dependency for a CLI package
- CLI is tightly coupled to the `workspace.Resolver` implementation details

**Recommended action**: **(B) Package-level split** or **(C) Extraction**
- If offline skill access is intentional: extract skill resolution logic into `internal/skills/resolver` or similar, so the CLI calls a thin service layer instead of doing raw DB+FS work
- If not intentional: add skill endpoints to the daemon API and route through `DaemonClient` like every other command
- Either way, `loadSkillCommandContext` (lines 263-304) and `resolveRegisteredSkillWorkspace` (lines 334-367) should not live in the CLI package

**Effort**: Significant

### Finding 4: Triple output format duplication in bundle functions (P1, D)

**Smell**: Duplicated Code / DRY violation
**Location**: Every `*Bundle()` function across session.go, workspace.go, agent.go, observe.go, memory.go, daemon.go, skill.go, whoami.go
**Impact**: Medium-high -- 15+ bundle functions, each with identical structural pattern

Every bundle function follows the same pattern:

```go
func fooBundle(item FooRecord) outputBundle {
    return outputBundle{
        jsonValue: item,
        human: func() (string, error) {
            return renderHumanSection("Foo", []keyValue{
                {Label: "ID", Value: stringOrDash(item.ID)},
                // ...
            }), nil
        },
        toon: func() (string, error) {
            return renderToonObject("foo", []string{"id", ...}, []string{
                item.ID,
                // ...
            }), nil
        },
    }
}
```

The `human` and `toon` closures are nearly identical -- they just map the same fields with different renderers. Adding a new output format requires touching 15+ functions.

**Recommended action**: **(D) Inline fix** -- Introduce a declarative field-mapping approach:

```go
type fieldSpec struct {
    Label string  // human label
    Key   string  // toon/json key
    Value string  // rendered value
}

func objectBundle(name string, jsonValue any, fields []fieldSpec) outputBundle { ... }
func tableBundle(name string, jsonValue any, headers []fieldSpec, rowFn func(item) []string) outputBundle { ... }
```

This would reduce 15+ bundle functions to simple field-spec declarations.

**Effort**: Moderate

### Finding 5: Stream handler duplication between session and observe (P2, C)

**Smell**: Duplicated Code
**Location**: `session.go:295-359` (`streamSessionEvents`) and `observe.go:93-144` (`streamObserveEvents`)
**Impact**: Medium -- two functions with near-identical structure

Both stream handlers:
1. Resolve output format
2. Create JSON encoder
3. Call a stream method with an SSE handler that:
   - Unmarshals event data
   - Switches on output mode (JSON/toon/human)
   - Renders and writes output

The only differences are the event type (`SessionEventRecord` vs `ObserveEventRecord`) and the field names for toon rendering.

**Recommended action**: **(C) Extraction** -- Create a generic stream rendering helper:

```go
func streamToOutput[T any](cmd *cobra.Command, streamFn func(SSEHandler) error, decode func(SSEEvent) (T, error), render streamRenderer[T]) error
```

**Effort**: Low-moderate

### Finding 6: CLI imports `daemon` package directly (P2, D)

**Smell**: Architectural boundary violation (potential)
**Location**: `internal/cli/root.go:15-16`, `daemon.go:15-16`
**Impact**: Medium -- the architecture doc says "`daemon/` is the sole composition root"

The CLI imports `daemon` for:
- `aghdaemon.New()` in `commandDeps.newDaemon` (to run foreground mode)
- `aghdaemon.Info`, `aghdaemon.ReadInfo`, `aghdaemon.WriteInfo`, `aghdaemon.RemoveInfo` (to read/check daemon PID file)

The first import (`aghdaemon.New()`) is a legitimate composition root call -- the CLI is starting the daemon in foreground mode. The info file operations are lightweight read-only checks.

**Assessment**: This is **not** a violation. The CLI needs to start the daemon (foreground mode) and check its status (PID file). The `daemon.Info`/`ReadInfo` types are essentially config/file utilities. However, the daemon info file operations could be extracted to avoid the CLI depending on the full `daemon` package.

**Recommended action**: **(D) Inline fix** (low priority) -- Consider extracting `daemon.Info` / `ReadInfo` / `WriteInfo` to a lightweight `daemoninfo` package if `daemon` grows heavier.

**Effort**: Trivial

### Finding 10: `commandDeps.withDefaults()` is a 60-line nil-check chain (P3, D)

**Smell**: Long Function / Conditional Complexity
**Location**: `internal/cli/root.go:130-191`
**Impact**: Low -- the function is straightforward but verbose

The `withDefaults()` method is a chain of 16 nil-checks, each setting a default. This is the standard functional options fallback pattern and is correct, but could be made more concise with a generic helper:

```go
func setDefault[T comparable](field *T, defaultValue T) {
    var zero T
    if *field == zero {
        *field = defaultValue
    }
}
```

**Recommended action**: **(D) Inline fix** -- Optional cleanup, low priority.

**Effort**: Trivial

## Group-Level Summary: Should CLI Be Split Into Subpackages?

### The Key Question

With 26 files, CLI is the largest package. Should it become `cli/session/`, `cli/daemon/`, `cli/workspace/`, etc.?

### Recommendation: **No subpackage split -- but extract the client layer**

**Arguments against subcommand subpackages:**

1. **File-per-domain already works well.** Each subcommand domain has exactly one source file and one test file (`session.go`/`session_test.go`, `daemon.go`/`daemon_wait_test.go`, etc.). Go tooling navigates this cleanly.

2. **Shared infrastructure is deep.** All subcommands share `commandDeps`, `outputBundle`, `writeCommandOutput`, `clientFromDeps`, `loadRuntimeContext`, and the format helpers. Splitting into subpackages would require either a `cli/internal/shared` package or duplicating these utilities.

3. **26 files is manageable.** The Go standard library has packages with 50+ files. The real issue is not file count but responsibility mixing -- which is addressed by the extractions below.

4. **Test stubs are monolithic.** The `stubClient` in `helpers_test.go` spans all domains. Splitting would require either a shared test package or duplicating the stub.

`acp`, `memory`, `observe`, `procutil`, `session`, `skills`, `skills/bundled`, `store`, `testutil`, `udsapi`, `version`, `config`, `daemon`, `workspace`

| Principle | Assessment |

| **SRP** | Mostly good at file level. `client.go` and `skill.go` violate SRP at the package level by mixing transport/types/domain-logic. |
