# ADR-003: Reuse the Task Runtime for Detached Harness Work and Policy-Based Synthetic Reentry

## Status

Accepted

## Date

2026-04-17

## Context

The harness architecture needs detached async execution that can:

- persist execution state durably
- survive daemon restarts
- support idempotency and recovery
- optionally wake the owning session through synthetic reentry

Earlier design work proposed a separate `BackgroundRun` entity. Follow-up review
against the codebase showed that AGH already has a strong durable async
substrate:

- run lifecycle in `internal/task/types.go`
- run persistence in `internal/store/globaldb/global_db_task.go`
- boot/runtime integration in `internal/daemon/task_runtime.go`

That existing substrate already provides run status, attempts, idempotency,
timestamps, session linkage, and recovery patterns. Introducing a second
persisted async runtime for harness work would duplicate state-machine,
recovery, and observability concerns.

## Decision

Reuse the existing task runtime for detached harness work.

The harness architecture will not introduce a separate persisted
`BackgroundRun` entity.

Detached harness work is represented through task/task-run records plus
daemon-owned harness metadata and policy. Synthetic wake-up behavior is derived
from task-run completion through daemon-owned reentry logic.

## Alternatives Considered

### Alternative 1: Separate global `BackgroundRun` runtime entity

- **Description**: Create a dedicated harness-specific persistent async runtime
  model in global storage.
- **Pros**: Clean harness-specific vocabulary and direct alignment with some
  competitor terminology.
- **Cons**: Duplicates durable run lifecycle, idempotency, recovery, and
  session-bridge concerns already solved in the task runtime.
- **Why rejected**: The existing task runtime is already the stronger
  persistence substrate.

### Alternative 2: Event-only background completion with no durable run model

- **Description**: Emit completion events and wake sessions without a durable
  execution record.
- **Pros**: Smallest implementation surface.
- **Cons**: Weak recovery, poor observability, and no strong idempotent state
  model.
- **Why rejected**: Detached work needs durable lifecycle tracking.

## Consequences

### Positive

- Avoids introducing a second persisted async state machine.
- Reuses existing recovery, idempotency, and session bridge behavior.
- Keeps detached harness execution aligned with the daemon’s current runtime
  model.

### Negative

- Harness async semantics now depend on task-domain concepts and metadata.
- The bridge between harness policy and task execution must be documented
  carefully.

### Risks

- Task-domain concepts may leak too directly into harness language.
- Mitigation: keep a daemon-owned harness bridge that maps task-run lifecycle
  into harness semantics and synthetic reentry.

## Implementation Notes

- Use explicit task/run origin metadata for detached harness work.
- Reuse task-run idempotency and recovery rather than inventing parallel fields.
- Synthetic reentry must use a dedicated event/turn model and must not be
  persisted as `EventTypeUserMessage`.
- Policy-based wake-up still applies; detached work completion is always
  observable, but only some completions produce session reentry.
