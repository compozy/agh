# ADR-004: Defer Coordinator-Grade Orchestration Contracts from the Harness Architecture Phase

## Status

Accepted

## Date

2026-04-17

## Context

There is strong local research for richer multi-agent orchestration in
`docs/ideas/orchestration/multi-agent-patterns-analysis.md`, including:

- coordinator mode
- immutable handoffs
- workflow-level observability
- circuit breakers
- compensation patterns

Those ideas are relevant to AGH, but they sit at a higher coordination layer
than the harness architecture being specified here.

The current harness architecture already includes significant scope:

- harness context resolution
- startup prompt architecture
- turn augmentation
- synthetic reentry
- detached async execution on task infrastructure
- observability and verification

Pulling coordinator/planner/reviewer contracts into the same architecture phase
would conflate harness runtime foundations with full multi-agent workflow
orchestration.

## Decision

Keep specialized coordinator/planner/reviewer orchestration contracts out of
scope for the current harness architecture phase.

The TechSpec must still reference orchestration research as follow-up context
and preserve extensibility points, but it does not introduce coordinator-grade
runtime contracts in this document.

## Alternatives Considered

### Alternative 1: Include coordinator-grade orchestration in the same TechSpec

- **Description**: Design worker, coordinator, reviewer, and planner contracts
  together with the harness architecture.
- **Pros**: More complete long-term picture.
- **Cons**: Significantly expands scope and mixes foundational harness work with
  workflow orchestration.
- **Why rejected**: The current work should land a stable harness substrate
  first.

### Alternative 2: Ignore orchestration research entirely

- **Description**: Keep the harness document narrow and omit the broader
  orchestration context.
- **Pros**: Simpler document.
- **Cons**: Risks future drift and duplicated design effort.
- **Why rejected**: The orchestration analysis is strategically important and
  should remain an explicit follow-up boundary.

## Consequences

### Positive

- Keeps the harness architecture focused on runtime foundations.
- Preserves a clean follow-up path for orchestration-specific design.
- Reduces the risk of hard-coding premature multi-agent workflow semantics.

### Negative

- Coordinator-grade semantics require a later dedicated spec.
- Some readers may still expect richer workflow behavior than this phase
  delivers.

### Risks

- The harness architecture could accidentally make choices that constrain later
  orchestration work.
- Mitigation: preserve extensibility points explicitly, especially around
  synthetic reentry, detached async execution, and harness policy resolution.

## Implementation Notes

- Reference the orchestration analysis in the TechSpec as follow-up context.
- Do not introduce new coordinator/planner/reviewer runtime types here.
- Treat later orchestration work as a separate architectural program layered on
  top of the harness foundation.
