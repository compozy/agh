# ADR-002: Extend Existing Prompt Assembly and Turn Augmentation Seams with Staged Composition

## Status

Accepted

## Date

2026-04-17

## Context

AGH already has two useful harness seams:

- startup prompt assembly through `PromptAssembler`, `PromptProvider`, and
  `ComposedAssembler`
- turn-time prompt augmentation through `PromptInputAugmenter`

These seams are visible in:

- `internal/daemon/composed_assembler.go`
- `internal/daemon/boot.go`
- `internal/daemon/daemon.go`
- `internal/session/manager.go`
- `internal/session/manager_prompt.go`
- `internal/memory/recall.go`

The revised harness architecture needs:

- daemon-owned startup section selection
- ordered turn augmentation
- explicit budgets and failure policy
- preservation of existing integration seams where possible

An immediate hard break of the session-layer augmentation interface would add
migration risk without a strong architectural payoff.

## Decision

Keep the existing seam split:

- startup prompt composition remains daemon-owned and assembler-based
- turn-time augmentation remains on the current session seam initially

Evolve the implementation in stages:

1. Add daemon-owned startup section selection on top of the existing assembler
   chain.
2. Introduce ordered augmentation through daemon-side composition over the
   current single augmenter seam.
3. Only promote augmentation to a richer session-layer interface later if the
   composed seam proves insufficient.

No new top-level harness prompt builder replaces these seams.

## Alternatives Considered

### Alternative 1: Build a brand-new prompt policy stack

- **Description**: Replace assembler and augmenter seams with a new unified
  policy component.
- **Pros**: Conceptually clean reset.
- **Cons**: Larger refactor, unnecessary migration risk, and duplication of
  working integration points.
- **Why rejected**: The current split between startup and turn-time context is
  already correct.

### Alternative 2: Immediately break the session augmenter API

- **Description**: Replace `PromptInputAugmenter` at once with a fully typed
  turn augmentation interface.
- **Pros**: Cleaner final API from day one.
- **Cons**: Bigger migration surface through `session.Manager`, memory recall,
  tests, and daemon composition.
- **Why rejected**: The daemon can compose augmenters first and defer the API
  break until it is justified.

## Consequences

### Positive

- Reuses proven startup and turn-time seams.
- Lowers migration risk for the first implementation passes.
- Keeps startup and per-turn behavior clearly separated.

### Negative

- The architecture will temporarily rely on composition over a thinner session
  API.
- The final richer augmentation contract may still be needed later.

### Risks

- Volatile context may leak back into startup assembly if boundaries are not
  explicit.
- Augmenter ordering and failure behavior may remain implicit unless encoded in
  the TechSpec and tests.

## Implementation Notes

- Add section metadata, predicates, and budgets on the startup path.
- Make `agh-network` an explicit startup-section concern rather than an
  undocumented inline exception.
- Introduce deterministic augmenter ordering, aggregate budget handling, and
  explicit failure semantics in the daemon-side composite.
- Preserve “store original input, dispatch augmented input” semantics for real
  user and network turns.
