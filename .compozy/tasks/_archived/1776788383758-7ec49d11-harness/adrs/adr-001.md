# ADR-001: Resolve Harness Behavior from Durable Session Context and Turn Origin

## Status

Accepted

## Date

2026-04-17

## Context

The original harness draft centered behavior around an internal `HarnessProfile`
enum. Follow-up review against the AGH codebase showed that this collapses two
distinct runtime axes that already exist today:

- durable session context such as `session.Type`, channel presence, and
  workspace binding
- per-turn provenance through `TurnSource`

Those axes are visible in:

- `internal/session/session.go`
- `internal/session/interfaces.go`
- `internal/session/manager_prompt.go`

Treating `interactive`, `network`, `background`, and `worker` as one
foundational enum creates ambiguous combinations and weakens the mapping to the
current runtime model. The harness architecture still needs an internal way to
resolve prompt sections, augmenters, detached execution rules, and synthetic
reentry behavior, but that policy should be derived from the runtime state
rather than stored as a primary mode enum.

## Decision

Resolve harness behavior from:

- durable session context
- turn origin
- additional runtime signals such as channel presence and synthetic reentry
  metadata

Introduce a daemon-owned resolver that maps those inputs into a
`ResolvedHarnessPolicy`. Any remaining “profile” terminology is only a derived
internal label for diagnostics, not the foundational architecture primitive.

No new user-facing configuration surface is introduced.

## Alternatives Considered

### Alternative 1: Foundational `HarnessProfile` enum with hybrid resolution

- **Description**: Keep `interactive`, `network`, `background`, and `worker` as
  the main internal model and layer turn-time behavior on top.
- **Pros**: Simple vocabulary and easy early communication.
- **Cons**: Collapses durable and per-turn state into one axis, produces
  ambiguous combinations, and drifts away from the code that already separates
  session type from turn source.
- **Why rejected**: The enum is a weaker model than the runtime signals AGH
  already has.

### Alternative 2: Fully user-declared profiles

- **Description**: Expose harness mode selection directly in config or agent
  definitions.
- **Pros**: Explicit operator control.
- **Cons**: Pushes harness complexity to users and conflicts with the agreed
  internal-only direction for this architecture phase.
- **Why rejected**: The runtime should infer behavior first.

## Consequences

### Positive

- Aligns the harness model with the current session and prompt runtime.
- Avoids overloading one enum with multiple concerns.
- Makes room for an explicit resolution matrix in the TechSpec.

### Negative

- Requires a more explicit policy matrix in design and tests.
- “Profile” becomes a secondary term rather than a simple top-level primitive.

### Risks

- Resolution logic may drift across codepaths if not centralized.
- Mitigation: keep resolution in one daemon-owned component and encode the
  matrix in tests.

## Implementation Notes

- Reuse existing session/runtime signals instead of introducing a new persisted
  `session_shape` field unless later evidence requires it.
- Expand turn provenance to cover synthetic reentry explicitly.
- Treat any profile label as derived output from the resolver, not as source of
  truth.
