# ADR-002: Frontmatter Catalog Injection for Roles and Playbooks

## Status

Accepted

## Date

2026-03-31

## Context

The master agent needs to know which roles and playbooks are available for delegation and coordination. Currently, the master template instructs the agent to run `agh roles list` and `agh playbooks list` on boot, which costs API turns and delays autonomous operation. The skills system already solves this problem for skills by injecting a catalog of frontmatter metadata into the system prompt, with full content accessible via `agh skill view`.

## Decision

Inject a **frontmatter catalog** (name, type, description, domain, tags) for roles and playbooks into the master agent's system prompt, following the same progressive disclosure pattern as the skills system. The master sees a summary immediately and uses `agh roles get <name>` / `agh playbooks get <name>` via CLI to load full definitions when needed.

## Alternatives Considered

### Alternative 1: CLI-Only Discovery

- **Description**: Keep the current approach where the master runs `agh roles list` and `agh playbooks list` on boot.
- **Pros**: No prompt changes needed. Master discovers at runtime.
- **Cons**: Wastes API turns on discovery. Master starts with zero awareness of available resources. Delays autonomous operation.
- **Why rejected**: The supervisor should begin with full awareness of its coordination surface. API turn cost for discovery is avoidable.

### Alternative 2: Full Content Injection

- **Description**: Inject the complete role definitions and playbook content into the system prompt.
- **Pros**: Master has everything immediately, zero CLI calls needed.
- **Cons**: System prompt bloat. A project with 10 roles and 5 playbooks could add thousands of tokens of content that may not be relevant to the current goal.
- **Why rejected**: Wastes prompt context. The frontmatter-only approach provides sufficient awareness for delegation decisions, and full content is available on demand.

## Consequences

### Positive

- Master knows immediately what roles and playbooks exist without spending API turns
- Consistent pattern with skills catalog (same progressive disclosure: catalog in prompt, full content via CLI)
- Minimal prompt size — only frontmatter metadata, not full content

### Negative

- Requires new catalog rendering functions (`BuildRoleCatalog`, `BuildPlaybookCatalog`)
- Catalogs only injected for master type agents — other types don't need them

### Risks

- If many roles/playbooks exist, the catalog section could become large. Mitigation: catalog entries are one line each (name + description), so even 50 roles would be ~50 lines.
