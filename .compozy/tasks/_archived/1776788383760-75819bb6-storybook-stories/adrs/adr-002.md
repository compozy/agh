# ADR-002: MSW + Shared Decorators for System Stories

## Status

Accepted

## Date

2026-04-17

## Context

Components under `web/src/systems/<name>/components` are not pure — they consume `TanStack Query` hooks, read router context, pull from Zustand stores, and post to `/api/**` endpoints proxied by Vite to the AGH daemon. Rendering them inside Storybook without a data layer either crashes (missing QueryClientProvider) or produces empty skeletons with no way to exercise loading/success/error states.

## Decision

The web Storybook instance enables **Mock Service Worker** (`msw` + `msw-storybook-addon`) and exposes three global decorators in `web/.storybook/preview.ts`:
1. `QueryClientProvider` with a story-scoped `QueryClient` that disables retries and sets `staleTime: Infinity`.
2. `RouterProvider` stub backed by `createMemoryHistory` so components using `useRouter` / `Link` render without navigation.
3. MSW worker initialization (`loader: [mswLoader]`) and a root-level `parameters.msw.handlers` with domain handlers imported from each system's `mocks/`.

Per-story overrides are possible via `parameters.msw.handlers = [...]`, enabling loading, success, and error state stories from the same component.

## Alternatives Considered

### Alternative 1: Prop Stubs Without MSW

- **Description**: Refactor components to accept data via props; wrap with empty `QueryClientProvider` only.
- **Pros**: No network layer; cleanest separation.
- **Cons**: Requires invasive refactor across ~40 components; many are view-models that own their own queries by design. Would contradict the app-renderer-systems pattern.
- **Why rejected**: Breaks existing architecture purity for the sake of Storybook.

### Alternative 2: QueryClient with setQueryData

- **Description**: Prefill the query cache per story with `queryClient.setQueryData(key, fixture)`.
- **Pros**: Zero runtime, deterministic success states.
- **Cons**: Cannot exercise loading, fetching, or error states; duplicates query keys across stories; breaks as soon as a component refetches or invalidates.
- **Why rejected**: Incomplete state coverage.

## Consequences

### Positive

- One global setup; system stories read realistic data without per-component scaffolding.
- Loading/error/empty stories fall out naturally via `http.get(...).delay(...)` and `http.get(...).error(500)`.
- Fixtures are reusable by Vitest (unit/integration) and future Playwright e2e via the same handlers.

### Negative

- Adds `msw@^2`, `msw-storybook-addon@^2`, and a service-worker bootstrap to the web Storybook build.
- First-run requires `msw init public/` to publish `mockServiceWorker.js`.

### Risks

- Handler drift from the real `/api` contract. Mitigation: fixtures are typed via the same OpenAPI-generated types used by the app (`web/src/adapters` reads the same contract).
- Storybook serves the worker from `public/`; ensure Vite does not process it.

## Implementation Notes

- Install at `web/` workspace: `bun add -D msw msw-storybook-addon`.
- Run once: `bunx msw init web/public --save`.
- `web/.storybook/preview.ts` wires `initialize({ onUnhandledRequest: "bypass" })` and exports `loader: [mswLoader]`.
- Each system owns `web/src/systems/<name>/mocks/{handlers.ts,fixtures.ts}` (see ADR-004).
