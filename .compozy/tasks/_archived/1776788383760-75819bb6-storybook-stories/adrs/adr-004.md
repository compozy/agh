# ADR-004: Per-System Mocks Directory

## Status

Accepted

## Date

2026-04-17

## Context

With MSW in place (ADR-002), we need a convention for where MSW handlers and fixtures live. Options ranged from a single `web/src/mocks/` tree, inline per-story mocks, or per-system folders colocated with the domain they describe.

## Decision

Each system owns its mocks:
```
web/src/systems/<name>/
├── mocks/
│   ├── handlers.ts       # MSW handlers (http.get, http.post, …)
│   ├── fixtures.ts       # Typed sample payloads
│   └── index.ts          # Barrel
├── components/
│   └── stories/
```

The web Storybook `preview.ts` imports each system's `mocks` barrel and composes the default handler set:
```ts
import { handlers as sessionHandlers } from "@/systems/session/mocks";
import { handlers as agentHandlers } from "@/systems/agent/mocks";
// …
export const parameters = { msw: { handlers: [...sessionHandlers, ...agentHandlers, …] } };
```

Per-story overrides replace handlers locally. Fixtures are typed against the adapter types already exported by each system to prevent contract drift.

Base UI layers (`packages/ui/src/components`, `web/src/components/ui`) do not need mocks — their stories are pure render with `args`.

## Alternatives Considered

### Alternative 1: Central web/src/mocks/

- **Description**: One `web/src/mocks/handlers.ts` + domain fixture files.
- **Pros**: One import path.
- **Cons**: Violates the app-renderer-systems rule that each system owns its full surface (adapters, lib, hooks, components). Cross-system edits collide on one file.
- **Why rejected**: Breaks architectural pattern.

### Alternative 2: Inline per-story

- **Description**: Each story declares its own `parameters.msw.handlers` with literal fixtures.
- **Pros**: Maximal locality.
- **Cons**: Duplicates fixtures; changing an API shape touches every story.
- **Why rejected**: Duplication.

## Consequences

### Positive

- Fixtures live next to the adapters/types that define the contract.
- Other tooling (Vitest, Playwright) can import the same `fixtures`/`handlers` without relocation.
- Deleting a system removes its mocks atomically.

### Negative

- One new subfolder per system (eight folders initially).
- Public system barrel (`systems/<name>/index.ts`) must decide whether to re-export mocks — default: no, test-only.

### Risks

- Fixtures could import production code and inflate the stories bundle. Mitigation: fixtures import types only, not adapters or hooks.

## Implementation Notes

- `agent`, `automation`, `bridges`, `daemon`, `knowledge`, `network`, `session`, `skill`, `workspace` each gain a `mocks/` folder. Systems without external HTTP (none currently) still declare a no-op `handlers: HttpHandler[] = []` for symmetry.
- Fixture file exports named constants (e.g. `sessionFixture`, `sessionListFixture`) — no default exports.
