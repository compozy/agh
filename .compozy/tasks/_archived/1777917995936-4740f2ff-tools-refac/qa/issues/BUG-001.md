# BUG-001: Hosted MCP binary validation rejected the same AGH executable on case-insensitive macOS paths

## Status

Fixed and re-verified.

## Severity

High.

## Scenario

Task 13 real-scenario QA attempted to bind the hidden hosted MCP transport through `agh tool mcp --session <id> --bind-nonce <nonce>` against the isolated daemon.

## Expected

The daemon accepts the hosted MCP bind when the peer process is the same AGH executable that the daemon launched/injected into the ACP session.

## Actual

The bind failed before the MCP handshake:

```text
error: mcp: hosted MCP binary validation failed: peer executable mismatch
```

## Root Cause

`HostedService.validatePeer` compared the normalized expected executable path and peer executable path as strings. On the QA host, the daemon was started from `/Users/pedronauck/dev/...`, while Darwin peer inspection reported the same executable under the case-preserved path `/Users/pedronauck/Dev/...`. On a case-insensitive filesystem these are the same file, but the string comparison rejected the peer.

## Fix

`internal/mcp/hosted.go` now compares executable identity by normalized path first and then by `os.SameFile` when paths differ. This preserves fail-closed behavior for different files while accepting the same executable reached through a different filesystem spelling or hardlink.

Regression coverage was added in `internal/mcp/hosted_test.go` using an alternate filesystem path to the same executable.
