# ADR-002: Expand the Task Domain for Paper-Parity Semantics

## Status

Accepted

## Date

2026-04-17

## Context

The Paper create flow, list cards, and board views rely on task concepts that are not first-class in the current domain:

- `priority`
- a true `draft` state
- `max_attempts` as task-level execution policy

Today, `priority` only fits inside `metadata`, `save draft` does not map to a durable draft lifecycle, and attempts are properties of individual runs rather than task configuration. Full parity now requires these semantics to exist in the backend, not only in UI presets.

## Decision

Expand the core task domain in v1 to add first-class support for:

- task priority
- durable draft semantics
- task-level maximum attempts

These capabilities must be represented in domain types, validation, storage, contracts, and OpenAPI-generated frontend types. UI templates may still provide presets, but the underlying semantics must be real task capabilities.

## Alternatives Considered

### Alternative 1: Keep semantics in metadata and UI conventions

- **Description**: Store `priority` in metadata, treat draft as a client-only state, and keep attempt policy outside the task model.
- **Pros**: Lower backend churn; faster to wire up the initial UI.
- **Cons**: Produces ambiguous semantics, weak filtering, and brittle client logic. Makes the Paper UI a façade instead of a faithful product surface.
- **Why rejected**: The user chose full parity now and asked to expand the domain.

### Alternative 2: Add only priority as first-class

- **Description**: Promote `priority`, but leave `draft` and `max_attempts` as secondary concerns.
- **Pros**: Smaller schema and API changes.
- **Cons**: Leaves the create flow and execution policy semantically incomplete.
- **Why rejected**: It would still require frontend-only workarounds for draft and retry policy.

## Consequences

### Positive

- Aligns backend semantics with the designed UI.
- Enables reliable filtering, badges, sorting, and create/edit behavior.
- Reduces metadata abuse and client-side special cases.

### Negative

- Requires storage, contract, validation, and status-lifecycle changes.
- Increases migration surface across manager logic and read models.

### Risks

- Draft lifecycle can conflict with current ready/blocked reconciliation rules. Mitigation: define draft as an explicit non-runnable task state with clear transition rules into runnable states.
- Task-level attempt policy can drift from run-level history. Mitigation: keep run attempt history authoritative while treating `max_attempts` as immutable or patchable task configuration.

## Implementation Notes

- Add task-level fields rather than encoding them indirectly in metadata.
- Extend task queries and summaries so list and kanban surfaces can filter and display these fields without detail fetches.
- Revisit task status transitions to support draft creation and draft publication explicitly.
