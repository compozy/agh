# ADR-005: Visual parity via Playwright snapshots over Storybook stories

## Status

Accepted

## Date

2026-04-18

## Context

ADR-002 forbids coexistence of old and new. That removes the ability to "A/B compare" in the running app. The only way to catch regressions during a rewrite of this scope is either:

- Manual screenshot review per PR (subjective, slow, inconsistent).
- Automated visual regression.

The project already has:

- Vitest + Testing Library unit tests (104 story files + matching tests).
- Playwright for e2e (`web/e2e/`).
- Storybook 10 for both `packages/ui` and `web/`.
- Tests that run in CI via `make verify` + `make test-integration`.

No visual regression tool is wired in yet.

## Decision

**Use Playwright's built-in `toHaveScreenshot()` to snapshot each Storybook story and each live route in `web/`. Snapshots live in the repo (as `.png`s next to the test file). CI compares on every PR. Differences block merge until reviewed.**

Scope:

- **`packages/ui` primitives:** every story gets a snapshot. Every variant in every primitive. Baseline is generated on first Phase 1 merge.
- **`web/` routes:** every route + view state (empty, loading, populated, error) gets a snapshot. Baseline is generated at the start of the domain's rollout phase (ADR-004).
- **Animation handling:** animated stories are snapshotted with `prefers-reduced-motion: reduce` forced; live routes with animation are snapshotted at a settled state.
- **Update workflow:** `pnpm test:visual --update` regenerates baselines; PRs that intentionally change visuals include the baseline updates in the same commit, with before/after screenshots in the PR body.
- **CI gating:** visual diff > threshold fails CI. Threshold is 0.1% pixel difference per screenshot (Playwright default is looser; we tighten because dark canvas diffs stand out).

## Alternatives Considered

### Alternative 1: Chromatic (hosted)

- **Description:** Chromatic runs visual regression against Storybook stories, hosted, with a UI for reviewing diffs.
- **Pros:** Best DX for visual review; team can approve diffs in a web UI; publishes a live Storybook per PR.
- **Cons:** Paid for private projects above free tier; third-party dep on an external service; adds another account to manage. Snapshots live off-repo, which complicates offline dev.
- **Why rejected for now:** The project already has Playwright; adding Chromatic adds cost + vendor dep before we know we need it. Reconsider if review throughput becomes a bottleneck.

### Alternative 2: Manual screenshot diffs per PR

- **Description:** Every PR author attaches before/after screenshots; reviewer eyeballs.
- **Pros:** Zero infra.
- **Cons:** Inconsistent, easy to miss subtle drifts (wrong divider color, off-by-2px padding), unscalable across ~200 files.
- **Why rejected:** Doesn't catch what humans miss — especially color-mix and tint regressions.

### Alternative 3: Vitest + @testing-library snapshots only

- **Description:** Use snapshot tests on rendered HTML + computed styles.
- **Pros:** Fast, already in the test pipeline.
- **Cons:** Serialized HTML snapshots are brittle; they catch class-name changes (noise) but miss actual pixel regressions in a theme-heavy visual system.
- **Why rejected:** Wrong granularity for the failure mode we care about.

## Consequences

### Positive

- **Regression gate** — no visual drift merges unnoticed.
- **Review faster** — reviewers trust that snapshots cover the pixel-level story, can focus on logic / structure.
- **Lives in the repo** — snapshots are versioned with the code; offline devs have everything.

### Negative

- **Snapshot storage** — hundreds of PNGs in the repo. Use git LFS if the tree crosses a size budget (decided during Phase 1 if needed, not pre-emptively).
- **CI time** — headless Chromium per story + per route adds minutes. Parallelize within Playwright (workers) and shard across CI jobs if it becomes the slowest step.
- **Platform fragility** — font rendering differs between macOS / Linux / CI runners. Pin CI to one runner image for snapshot generation. Devs regenerating locally must match that image's output or use `--update` flag with CI-generated baselines.

### Risks

- **Risk:** Flaky snapshots due to timing (animations, SSE content).
- **Mitigation:** Storybook stories mock SSE deterministically; route stories use fixed fixtures; `motion` runs with `reducedMotion="always"` under Playwright.
- **Risk:** Baseline churn drowns review signal.
- **Mitigation:** Each PR groups baseline updates in one commit (`chore: update visual baselines for <domain>`), separate from logic commits; reviewer can scan diffs with a thumbnail tool.

## Implementation Notes

- Add a `make test-visual` target that runs Playwright against the Storybook dev server + the dev server of `web/`.
- Snapshot root dirs: `packages/ui/src/components/stories/__snapshots__/` and `web/e2e/__snapshots__/`.
- CI matrix: one GitHub Actions job for visual tests, pinned to `ubuntu-22.04` with a fixed Chromium version (Playwright bundles its own — this stays consistent).
