# ADR-003: Frozen Snapshot Memory Injection With Dream-Only Extraction

## Status

Accepted

## Date

2026-04-04

## Context

Memory context needs to reach the agent somehow. Two orthogonal decisions: (1) when is memory loaded into the session — once at start, or refreshed each turn? (2) how are new memories extracted from sessions — per-turn background extraction, manual agent writes, or periodic bulk consolidation?

Claude Code and Hermes both use "frozen snapshot" injection (load once at session start, immutable during session). Claude Code additionally runs a background "forked agent" after each turn to extract memories. The old AGH project used manual agent writes + periodic dream consolidation.

## Decision

**Injection: Frozen snapshot.** Memory is loaded once during `Manager.Create()` via `PromptAssembler.Assemble()`. The assembled prompt is immutable for the session's lifetime. Mid-session memory writes update disk immediately but don't change the running session's context. Changes are visible on next session start or resume.

**Extraction: Dream-only + manual.** No per-turn automatic extraction. Agents write memories manually via `agh memory write` CLI commands during sessions. Dream consolidation periodically reads session event databases in bulk and synthesizes memories. This matches the proven pattern from the old project.

## Alternatives Considered

### Alternative 1: Dynamic reload on each prompt

- **Description**: Reload memory on every `Manager.Prompt()` call
- **Pros**: Agent always sees latest memories
- **Cons**: Re-reads files each turn; may break ACP prompt cache; adds latency; memory changes mid-conversation could confuse agent reasoning
- **Why rejected**: Performance cost and potential for reasoning instability outweigh freshness benefits.

### Alternative 2: Per-turn background extraction (Claude Code pattern)

- **Description**: After each prompt turn, spawn lightweight extraction step
- **Pros**: Higher fidelity — catches memories agent didn't explicitly save
- **Cons**: Adds token cost per turn; adds latency; requires a separate extraction prompt; complex to implement
- **Why rejected**: Dream consolidation catches the same patterns in bulk with lower total cost. Per-turn extraction is a Claude Code optimization for their specific use case (single-agent, high-volume).

### Alternative 3: Hybrid with explicit refresh endpoint

- **Description**: Frozen by default + `POST /sessions/:id/refresh-context` to reload mid-session
- **Pros**: Best of both worlds
- **Cons**: Adds API surface; unclear when to call refresh; may break agent reasoning
- **Why rejected**: Added complexity for an edge case. If users need fresh memory, starting a new session is the natural path.

## Consequences

### Positive

- Predictable prompt behavior — agents don't see context shifts mid-conversation
- Preserves prompt cache efficiency (important for ACP agents)
- Simpler implementation — load once, forget
- Lower operational cost — no per-turn extraction token spend

### Negative

- Memories written during a session aren't visible until next session
- Dream consolidation has latency (gate thresholds must pass before extraction runs)
- Agents must explicitly write memories they want to persist — may miss implicit patterns

### Risks

- **Missed patterns** — Mitigated by dream consolidation reading full session transcripts (not just agent-written memories)
- **Stale context** — Mitigated by session resume reloading memory; long-running sessions can be restarted
