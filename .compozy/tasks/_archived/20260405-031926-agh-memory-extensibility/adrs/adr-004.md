# ADR-004: Four-Type Memory Taxonomy — Drop Procedural, Defer to Skills Phase

## Status

Accepted

## Date

2026-04-04

## Context

The issue proposed adding "procedural" as a 5th memory type for learned workflows and tool patterns. Market research showed procedural memory is "consistently overlooked" across competing frameworks, suggesting differentiation potential.

However, AGH Phase 2 includes a dedicated skills system. Procedural memory ("learned patterns from experience") and skills ("taught capabilities") overlap significantly. Committing to a procedural memory format now risks conflicting with the skills format designed later.

The existing 4-type taxonomy (user, feedback, project, reference) from cc-memory is proven and can capture learned patterns via `project` or `feedback` types.

## Decision

Keep the proven 4-type taxonomy: `user`, `feedback`, `project`, `reference`. Do not add `procedural` as a memory type. Dream consolidation can still extract patterns from session transcripts — they get stored as `project` or `feedback` type memories.

When the skills phase arrives, it will define its own format for explicitly taught capabilities and can pull from memory as a source for "skill candidates."

## Alternatives Considered

### Alternative 1: Add procedural as 5th type

- **Description**: New `procedural` MemoryType with natural language body describing learned patterns
- **Pros**: Explicit tagging of learned procedures; differentiator vs competitors; dream can target procedural extraction
- **Cons**: Unclear how it differs from a `project` memory that describes a workflow; may conflict with skills format
- **Why rejected**: The boundary between "learned procedure" and "skill" is too blurry to commit to a format now. Better to let skills phase define this boundary with full context.

### Alternative 2: Separate skills/ directory now

- **Description**: Create `~/.agh/skills/` alongside `~/.agh/memory/` with a different format
- **Pros**: Clean separation; skills are explicitly different from memories
- **Cons**: Pre-commits to a skills format before the skills phase; adds complexity; YAGNI
- **Why rejected**: Premature. Skills phase needs its own design process informed by real usage of the memory system.

## Consequences

### Positive

- Keeps the proven, portable 4-type taxonomy
- No format conflict risk with future skills phase
- Simpler implementation — no new memory type parsing or validation
- Dream consolidation still captures patterns (just not tagged as "procedural")

### Negative

- Lose the explicit "procedural" differentiator from the issue
- Dream prompt doesn't have a specific extraction target for procedures
- Harder to query "show me all learned patterns" without a dedicated type

### Risks

- **Lost patterns** — Mitigated by dream consolidation extracting patterns as project/feedback memories
- **Skills phase divergence** — Mitigated by having memory as a data source for skills discovery
