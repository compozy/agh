# ADR-001: Interleaved Extensibility — Build Memory With Intentional Seams, Defer Formal Plugin System

## Status

Accepted

## Date

2026-04-04

## Context

AGH v2 aims to become a highly extensible agent operating system. The memory system is the first Phase 2 feature, with skills and agent network to follow. The question is whether to design a formal plugin/extensibility architecture before building memory, or build memory first and extract extensibility patterns afterward.

Market research shows that the best plugin architectures (VS Code, Terraform, Docker) were extracted from working features, not designed in advance. AGH v2 already has extensibility-friendly patterns: interfaces defined where consumed, functional options, Notifier pattern, and daemon as sole composition root.

The council debated this and identified that team memory design is the key strategic decision — it naturally seeds the extensibility story without requiring a formal plugin system.

## Decision

Build the memory system with intentional extension seams but without a formal plugin architecture. Specifically:

1. **Port memdir** as `internal/memory/` with minimal changes from cc-memory
2. **Build minimal prompt assembly** as an interface (`PromptAssembler`) that injects memory indexes into agent system prompts
3. **Adapt dream consolidation** to spawn ephemeral ACP sessions with a trusted internal session type
4. **Team memory** uses workspace-scoped memory files with `agent_name` in frontmatter
5. **Cross-agent consolidation** is inherent — dream reads all session event databases regardless of agent type
6. **Extract formal extensibility patterns** after implementing memory + skills (2+ concrete features)

## Alternatives Considered

### Alternative 1: Extensibility Foundation First

- **Description**: Design a full plugin/extension registry, lifecycle hooks, and capability system before building memory
- **Pros**: Every subsequent feature (skills, network) benefits from the foundation; consistent patterns from day one
- **Cons**: High upfront cost with speculative design; no concrete use cases to validate extension points; risk of over-engineering
- **Why rejected**: Premature abstraction. Without memory + skills as concrete features, extension points would be speculative. History shows frameworks designed before features don't match reality.

### Alternative 2: Pure Port With No Extensibility Consideration

- **Description**: Port cc-memory exactly as-is, adapt minimally to v2, no thought about future extensibility
- **Pros**: Fastest path to working memory; minimal design decisions
- **Cons**: May hardwire patterns that block skills/network integration; team memory design without forward-thinking creates technical debt
- **Why rejected**: Team memory and prompt assembly are genuinely new work in v2 — they deserve intentional design even without a full plugin system.

## Consequences

### Positive

- Ships working memory system quickly by reusing proven cc-memory code
- Interfaces (PromptAssembler, SessionType) provide natural seams for future extension
- Cross-agent consolidation emerges naturally from agent-agnostic dream design
- Concrete patterns from memory inform the formal extensibility design when skills arrives

### Negative

- Prompt assembly will likely need redesign when skills arrives (bounded cost)
- No formal plugin registry means early harness builders must use Go interfaces directly
- Team memory as workspace files is simpler but less queryable than a store-based approach

### Risks

- **Prompt assembly redesign** — Mitigated by defining it as an interface from day one; implementation can be swapped
- **Team memory evolution** — Mitigated by including agent_name/session_id metadata in frontmatter for future queryability
- **Dream permission model** — Mitigated by introducing SessionType enum (user, dream, system) with explicit permission defaults
