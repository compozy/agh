# ADR-002: PromptAssembler Interface in session/ With memory/ Implementation

## Status

Accepted

## Date

2026-04-04

## Context

AGH v2 has no prompt assembly — agents receive the raw AGENT.md prompt via `acp.StartOpts`. Memory needs to inject MEMORY.md indexes into this prompt before the agent starts. The question is where this assembly logic lives: `session/`, a new `agent/` package, a new `prompt/` package, or `memory/` itself.

The v2 architecture has 11 packages with strict boundaries. Adding packages requires justification. The `session.Manager` already orchestrates the flow from `loadAgent()` → `loadConfig()` → `driver.Start()`, making it the natural insertion point for prompt assembly.

## Decision

Define `PromptAssembler` as an interface in `session/interfaces.go`, injected into Manager via functional option `WithPromptAssembler()`. The `memory/` package provides the concrete implementation. `Manager.Create()` calls `assembler.Assemble()` between loading the agent definition and calling `driver.Start()`.

The assembled prompt replaces `AgentDef.Prompt` in the `acp.StartOpts` passed to the driver.

## Alternatives Considered

### Alternative 1: New internal/prompt/ package

- **Description**: Dedicated package for prompt assembly with its own types
- **Pros**: Clean separation of prompt concerns; reusable across future features
- **Cons**: Adds a package for what is currently a single function call; YAGNI when only memory uses it
- **Why rejected**: Insufficient complexity to justify a new package. Can be extracted later when skills and network context also need prompt injection.

### Alternative 2: New internal/agent/ package

- **Description**: Package owning agent resolution + prompt assembly
- **Pros**: Groups agent-related logic (definition, resolution, assembly)
- **Cons**: Agent definition parsing already lives in `config/`; would split agent concerns across two packages
- **Why rejected**: Creates confusion about where agent logic lives. The current split (config/ for definitions, session/ for lifecycle, acp/ for protocol) is clearer.

### Alternative 3: memory/ package owns assembly directly

- **Description**: Memory provides `BuildContext()` returning a string; Manager concatenates it
- **Pros**: Simplest; no new interface
- **Cons**: Session becomes tightly coupled to memory's output format; harder to add skills/network context later
- **Why rejected**: Lacks extensibility seam. When skills arrives, Manager would need to know about both memory and skills output formats.

## Consequences

### Positive

- Zero new packages — uses existing `session/` boundary
- Interface allows future prompt contributors (skills, network) to be composed via the same seam
- Memory implementation is fully decoupled from session lifecycle
- Functional option injection follows established v2 patterns

### Negative

- Prompt assembly will need redesign when skills arrives (interface may need to accept multiple assemblers or a chain)
- Single `Assemble()` call may become a bottleneck if multiple prompt sections need coordination

### Risks

- **Interface evolution** — Mitigated by starting minimal (single method) and evolving when concrete needs emerge
- **Ordering concerns** — When multiple prompt sections exist, injection order matters. Deferred to skills phase.
