## Shared Decisions
- Coordinator config keys established for later tasks: `enabled`, `agent_name`, `provider`, `model`, `default_ttl`, `max_children`, and `max_active_per_workspace`.
- Built-in coordinator defaults are conservative: auto-start disabled, agent identity `coordinator`, TTL `2h`, max children `5`, and exactly one active coordinator per workspace.
- Provider/model remain empty in built-in defaults so resolution can fall through to workspace/global config and then agent/provider defaults.
- Raw `claim_token` is confined to command inputs and the synchronous agent claim response; read/list/detail/SSE/channel DTOs expose `claim_token_hash` only.
- `ClaimNextRun` is the authoritative next-work primitive for Task 09+; scheduler/coordinator code should wake, sweep, and create work, but not bypass the task service claim transaction.
- Task-run capability requirements are durable exact-match rows in `task_run_required_capabilities` and `task_run_preferred_capabilities`, not JSON metadata, so Task 08 can claim transactionally without parsing blobs.
- Task 02 intentionally registered contract/OpenAPI surfaces before behavior; later handler/CLI tasks should consume these DTOs rather than add ad-hoc response structs.
- Autonomy hook matcher correlation fields are nested under `matcher.autonomy` in generated/resource-facing contracts; future hook declarations should put `task_id`, `run_id`, `workflow_id`, `coordination_channel_id`, coordinator session, spawn lineage, spawn role, and release reason matchers there.
- Agent context endpoint/CLI work should use `RuntimeDeps.AgentContext` rather than rebuilding situation facts in transport handlers.
- Agent-facing identity must flow through `internal/agentidentity`; future `me`, `ch`, `task`, and `spawn` verbs should not parse or trust env/header identity directly.
- Manual user/dream/system sessions are first-class root lineage rows (`root_session_id = session_id`, no parent, depth 0). Coordinator sessions are root rows with `session_type = coordinator` and future TTL; spawned sessions require parent/root/depth and future TTL.
- Spawn requests inherit the caller parent workspace/channel in MVP; cross-workspace spawn is rejected in session policy, and reaper lease release goes through `task.Service.ReleaseSessionRunLeases` instead of direct database writes.

## Shared Learnings
- `RuntimeDeps.CoordinatorConfig` is the daemon-facing resolver boundary for later coordinator bootstrap; it resolves workspace config through the workspace resolver and does not spawn, stop, or prompt sessions.
- The task package consumes only `task.RunHookDispatcher`; daemon `hooksNotifier` adapts that narrow bridge to the hook runtime. Future task-run transition work should co-emit hooks at the authoritative transition call site, not by tailing task event tables.
- Expired task-run lease recovery is an explicit service operation (`RecoverExpiredRunLeases`) intended for boot recovery and scheduler sweeps; stale holders fail token-fenced mutations after recovery.
- Scheduler state is intentionally ephemeral and rebuildable from durable queued/active task runs plus active session snapshots. Future coordinator behavior should create/enqueue work or rely on wake notifications; it must not add scheduler-side claim authority.
- Session lineage data is queryable through globaldb list filters for `SessionType`, `ParentSessionID`, `RootSessionID`, and `SpawnRole`; Task 13 should use those typed filters instead of scanning JSON metadata.
- Spawn lifecycle cleanup emits structured reasons (`ttl_expired`, `parent_stopped`, `orphaned`) and stops children through session manager APIs after releasing active task-run leases through the task service.
- Generated CLI reference leaves are source-owned by Cobra command metadata. Put durable page-specific guidance in command `Long`/`Example` fields before running `make cli-docs`; manual tails on generated pages are removed by regeneration.
- Fixture-backed ACP mock turns match the canonical user message after stripping daemon-owned situation-context prompt augmentation, so autonomy E2E fixtures can keep exact `user_text` matchers while still exercising live Task 04 context injection.
