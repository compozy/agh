# ADR-002: Agent-Facing CLI Before Built-In MCP Tools

## Status

Accepted

## Date

2026-04-25

## Context

AGH already has operator-facing CLI commands and UDS/HTTP APIs, but agents running inside ACP sessions cannot use them ergonomically. Current network delivery guidance builds long `agh network send --session "$AGH_SESSION_ID" --channel ...` command snippets because the CLI does not infer session identity.

The autonomy roadmap also proposes built-in `agh.*` MCP tools. Those tools would be useful eventually, but adding a second agent-facing surface before the CLI/UDS contract stabilizes would duplicate work.

## Decision

Stabilize an agent-facing CLI and UDS/API contract before introducing built-in MCP tools. The CLI will support identity inference from session environment, stable `-o json` output, `-o jsonl` streaming for wait/subscribe commands, and canonical commands such as `agh me context`, `agh ch reply`, `agh ch recv --wait`, and `agh task next/heartbeat/complete/fail/release`. `agh task next` returns the coordination channel for channel-bound runs, and `agh ch` is the operational communication surface for those runs. Avoid aliases such as `done` or `pass` in the MVP so each lifecycle concept has one command name.

Built-in MCP tools may later mirror the same contract, but they are not part of the first autonomy implementation.

## Alternatives Considered

### Alternative 1: Implement MCP tools first

- **Description**: Expose autonomy actions primarily as ACP/MCP tools instead of CLI commands.
- **Pros**: Better typed affordances for tool-capable runtimes.
- **Cons**: Requires a stable tool host policy and duplicates contract work before the CLI is agent-safe.
- **Why rejected**: The existing system already has CLI/UDS surfaces; fixing those first benefits all runtimes and keeps the first contract simple.

### Alternative 2: Keep operator CLI and teach agents command snippets

- **Description**: Continue injecting explicit `agh network send` snippets with session/channel flags.
- **Pros**: Minimal code change.
- **Cons**: Leaves the agent surface brittle, verbose, and error-prone.
- **Why rejected**: This is the current failure mode; autonomy requires direct, identity-aware verbs.

## Consequences

### Positive

- Establishes one stable contract before adding MCP mirrors.
- Reduces prompt guidance noise.
- Makes agent commands usable from any ACP runtime that can execute shell commands.

### Negative

- Tool-native runtimes will wait longer for first-class MCP affordances.
- CLI output schemas must be treated as compatibility contracts.

### Risks

- Risk: CLI commands become too broad and operator/agent concerns mix again.
- Mitigation: keep agent-facing namespaces explicit (`me`, `ch`, `task`, `spawn`) and leave operator commands intact.

## Implementation Notes

Use `internal/api/contract` response types as the source for stable JSON output where possible. Add `jsonl` only for streaming/wait commands. Keep `agh ch` and `agh task` boundaries clear: channel verbs send coordination messages, while task verbs own claim, lease, and terminal state.
