# ADR-011: Generated Contracts and Documentation Co-Ship with Autonomy MVP Steps

## Status

Accepted

## Date

2026-04-26

## Context

The autonomy MVP changes API contract DTOs, task-run projections, session lineage, agent-facing endpoints, CLI verbs, and operator semantics. Even if broad web visibility is deferred, generated web contracts and existing Tasks UI code consume the same DTOs today. If backend tasks change `internal/api/contract` without regenerating OpenAPI and web types, `web-typecheck`, web tests, and Storybook/MSW fixtures can fail after otherwise valid daemon work.

The docs site also describes the current runtime, task, hook, session, network channel, and CLI behavior. Once the autonomy kernel reaches the step 10 demo milestone, concepts such as coordinator-agent, task-run coordination channels, `ClaimNextRun`, task-run leases, safe spawn, `agh me`, `agh ch`, `agh spawn`, and `agh task next/heartbeat/release` must be documented enough for operators and implementers to use the system. Deferring all docs to post-MVP would leave the first demo milestone undocumented and make task handoff ambiguous.

At the same time, the MVP must avoid building dashboards, new web systems, and marketing changes before the kernel contracts stabilize.

## Decision

Every autonomy MVP step that changes `internal/api/contract` must co-ship generated contract updates:

- regenerate `openapi/agh.json`
- regenerate `web/src/generated/agh-openapi.d.ts`
- update affected `web/src/systems/*/types.ts` consumers and Storybook/MSW fixtures
- pass `make web-typecheck` and `make web-test` in the same implementation unit

Read endpoints exposed over HTTP return `claim_token_hash`, never raw `claim_token`. The raw token is returned only in the synchronous claim response on the issuing transport and is not exposed in list/detail read models, SSE streams, logs, or web UI payloads.

Step 10 of the autonomy MVP must also co-ship the minimum operator and documentation surface:

- a small Tasks UI copy/label/disabled-state pass that makes task creation, publish/approval, run enqueue, and coordinator spawn visually distinct
- one web e2e scenario covering ADR-010 manual-first bookends
- a minimum runtime docs set under `packages/site/content/runtime/core/autonomy/`
- CLI reference pages for `agh me`, `agh ch`, `agh spawn`, and new `agh task next/heartbeat/release` verbs
- docs explaining that coordination channels are bound at run enqueue and are conversation surfaces, not task ownership/status authority
- hook/config/session docs updates needed to describe `coordinator.*`, `spawn.*`, and `task.run.*`

Broad web visibility remains post-MVP: no new `web/src/systems/coordinator` module, no autonomy sidebar item, no live lease dashboard, no spawn lineage tree visualization, no idle-agent registry view, no eval/replay UI, no coordinator config GUI, and no marketing site rewrite.

## Alternatives Considered

### Alternative 1: Defer all web and docs work to post-MVP

- **Description**: Implement daemon/CLI contracts first and update web/docs later.
- **Pros**: Keeps kernel tasks smaller.
- **Cons**: Contract changes can break web builds immediately, and the step 10 demo lacks usable docs.
- **Why rejected**: Generated contracts are an integration dependency, not optional presentation work.

### Alternative 2: Build full autonomy UI in the MVP

- **Description**: Add coordinator dashboards, lease/heartbeat visualizations, spawn lineage trees, scheduler views, and config UI alongside the kernel.
- **Pros**: Strong operator visibility from day one.
- **Cons**: Overbuilds before kernel contracts stabilize and violates the phased MVP scope.
- **Why rejected**: Existing Tasks and Sessions surfaces can carry the MVP with labels and generated type fixes.

### Alternative 3: Put this only in the TechSpec without an ADR

- **Description**: Add rows and sequencing notes in `_techspec.md`, but no dedicated ADR.
- **Pros**: Fewer decision files.
- **Cons**: Easy for downstream task decomposition to treat generated contracts and docs as optional polish.
- **Why rejected**: The co-ship rule is a release discipline decision that should stay visible through task decomposition.

## Consequences

### Positive

- Contract-changing backend tasks cannot silently break the web build.
- The step 10 demo milestone is documented and usable.
- Operator UI preserves the manual-first contract from ADR-010.
- Runtime docs preserve the task/channel boundary from ADR-012.
- Broad autonomy UI remains safely deferred.

### Negative

- MVP tasks that touch contracts include small frontend and docs obligations.
- Some PRs become cross-package even when the runtime change is primarily backend.

### Risks

- Risk: docs co-ship slows the coordinator bootstrap milestone.
- Mitigation: keep docs mechanical and scoped to runtime concepts, CLI reference, hooks, and config. Do not rewrite marketing pages.

- Risk: generated contract updates expand frontend scope.
- Mitigation: limit MVP frontend work to generated types, existing Tasks/Sessions consumers, Storybook/MSW fixtures, copy/labels, and e2e coverage.

## Implementation Notes

Use existing web and site workflows. Contract-changing tasks run `make codegen`, `make web-typecheck`, and `make web-test`. Site docs tasks run `cd packages/site && bun run source:generate`, `cd packages/site && bun run typecheck`, and `cd packages/site && bun run test`. Do not add a docs generator for CLI pages in the MVP. Do not add new web routes or systems for coordinator/scheduler visibility until post-MVP web visibility work begins.

`cy-create-tasks` should attach generated contract and docs obligations to the implementation tasks that create the relevant backend contracts, rather than creating one late cleanup task that tries to repair web/docs after the kernel is complete.
