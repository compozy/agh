# ADR-003: Extend Task Runs for Atomic Claim and Lease

## Status

Accepted

## Date

2026-04-25

## Context

AGH has a task state machine and `task_runs` with statuses such as `queued`, `claimed`, `starting`, `running`, and terminal states. It does not have an atomic `ClaimNextRun(criteria)` operation, lease ownership, heartbeat, or lease expiry recovery. This prevents agents from safely pulling ready work and creates risk of double execution or stuck runs.

The analysis considered a separate scheduler-owned durable queue, but that would duplicate task ownership state.

## Decision

Extend the current `task_runs` model and task service/globaldb APIs. `task_runs` remains the durable source of truth for run queueing, ownership, execution, and completion. In this ADR, lease means the ownership lease for a task run, not a sandbox, workspace runtime, or environment lease.

Add fields equivalent to:

- `claim_token`
- `lease_until`
- `heartbeat_at`
- `coordination_channel_id`

Use existing `task_runs.session_id`, `claimed_by_kind`/`claimed_by_ref`, `origin_kind`/`origin_ref`, `queued_at`, and task-level `CreatedBy` as the canonical owner, origin, enqueue, and creation provenance fields. Do not add duplicate actor or timestamp columns for facts already represented by the current task/run model. `coordination_channel_id` is correlation metadata for coordinator/worker communication; it is not an ownership field.

Add capability side tables for claim-time matching:

- `task_run_required_capabilities(run_id, capability_id)`
- `task_run_preferred_capabilities(run_id, capability_id)`

Both side tables must index `(capability_id, run_id)` so `ClaimNextRun` can filter exact required-capability matches without scanning JSON blobs.

Add task-domain APIs equivalent to:

- `ClaimNextRun(ctx, criteria, actor)`
- `ExtendRunLease(ctx, runID, claimToken, ttl, actor)`
- claim-token checks on complete/fail/cancel paths where the caller is an agent claimant
- boot/sweep recovery for expired leases

The scheduler may keep in-memory indexes and wake eligible agents, but it must not introduce a second durable queue table and must not be the direct run claimant in the MVP. Claims are initiated by the owning agent session through the agent task verb or by an explicit operator assignment path that writes the same token-fenced ownership state. `ClaimNextRun` is the canonical next-work primitive; scheduler wakeups are advisory and recovery-oriented only.

`ClaimNextRun` returns the claimed run's `coordination_channel_id` when one exists so the claimant can communicate in the task-bound channel. Channel messages never substitute for claim, heartbeat, complete, fail, or release.

The lease race contract is part of this decision:

- Exactly one active claim token may own a non-terminal run.
- Heartbeat, complete, fail, and release compare the run owner and claim token.
- Stale heartbeat after recovery fails explicitly.
- Late complete after recovery fails explicitly.
- Sweep and heartbeat are serialized by SQLite transaction boundaries.
- Boot recovery runs before the scheduler accepts wake/claim traffic.
- Lease extension is bounded by config so a wedged session cannot heartbeat forever.
- A session may hold at most one active task-run lease in the MVP; the cap is enforced by the claim transaction.

## Alternatives Considered

### Alternative 1: Scheduler-owned durable queue

- **Description**: Keep `task_runs` as history and add a new durable queue owned by `internal/scheduler`.
- **Pros**: Isolates scheduling concerns.
- **Cons**: Duplicates state, creates reconciliation problems, and violates the current task-domain ownership model.
- **Why rejected**: `task_runs` already expresses the run lifecycle and should remain the single source of truth.

### Alternative 2: Claim by explicit run ID only

- **Description**: Keep `ClaimRun(runID)` and require callers to list runs first.
- **Pros**: Smaller API change.
- **Cons**: Race-prone and insufficient for autonomous agents that need atomic "next work" semantics.
- **Why rejected**: Autonomy requires one atomic operation that selects and claims eligible work.

## Consequences

### Positive

- Preserves task-domain ownership.
- Enables safe agent pull loops.
- Gives scheduler and coordinator a reliable fencing primitive.
- Lets manual assignments and autonomous claims share the same safety rules.
- Gives workers the correct coordination channel without adding a second work-discovery path.

### Negative

- Requires schema changes and careful migration-free greenfield updates.
- Complete/fail/cancel paths need claimant validation.

### Risks

- Risk: lease expiry requeues work that is still running.
- Mitigation: require heartbeat renewal and conservative TTL defaults; emit observable events before recovery actions.

- Risk: stale holders complete work after another session has reclaimed the run.
- Mitigation: completion requires the current claim token and claimed session; stale completions fail and emit an observable event.

## Implementation Notes

Use SQLite transactions (`BEGIN IMMEDIATE` or equivalent existing helper) for atomic claim. Order candidates by priority and age, with capability/workspace/channel filters in criteria.

Use compare-and-swap predicates for sweep recovery so recovery only changes the run if the observed claim token and lease state still match.

Index `coordination_channel_id` for channel-to-run correlation, but do not use channel messages as ownership proof.
