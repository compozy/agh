# ADR-005: Configurable Spawn-On-Run-Enqueue Coordinator

## Status

Accepted

## Date

2026-04-25

## Context

The coordinator-agent is needed for semantic orchestration but should not run unnecessarily in idle workspaces or while a user is merely drafting tasks. The user also requires provider/model configurability so workspaces can choose different runtime cost and quality profiles.

The configuration needs a default policy but must allow workspace-specific overrides.

## Decision

The daemon will spawn one coordinator-agent per workspace when a task run is enqueued by a publish, start, or approval action for coordinated execution. The run enqueue path also creates or resolves the run's coordination channel before the coordinator is asked to orchestrate it. The coordinator-agent will use a restricted orchestration tool surface and configurable provider/model/runtime settings.

The coordinator trigger is:

- A task run is enqueued by a user publish/start action, UI start action, automation approval, or equivalent API that means "this task is allowed to run now."
- The durable trigger is the run enqueue boundary, represented today by AGH's `task.run_enqueued` domain event, not task creation.
- The enqueued workspace-scoped coordinated run has a stable `coordination_channel_id`.
- The workspace has no healthy active coordinator session.
- Coordinator auto-start is enabled in resolved config.
- Spawn caps and permission policy allow the coordinator session.

Creating a task does not trigger coordinator startup or create claimable work by itself. User-started sessions do not trigger coordinator startup by themselves. Agent-created tasks inside an existing coordinator workflow inherit that workflow context and do not spawn a second coordinator.

Global-scope task runs do not trigger coordinator auto-spawn in the MVP. They require explicit operator assignment or a future daemon-global coordinator decision. Workspace-scoped runs are the only auto-bootstrap path in the first implementation.

Operator UI and docs must present the trigger as a run-enqueue boundary, not as task creation. Publish, approval, and start are user-facing actions that may lead to run enqueue; the coordinator trigger remains the enqueued coordinated run.

Configuration precedence:

1. Workspace-level autonomy coordinator override
2. Global `[autonomy.coordinator]` defaults
3. Bundled/default coordinator agent definition

The coordinator config may include enabled state, agent name, provider, model, spawn policy, idle TTL, max active coordinators per workspace, and coordinator-specific limits.

## Alternatives Considered

### Alternative 1: Always-on coordinator per workspace

- **Description**: Spawn coordinators for every workspace at daemon boot.
- **Pros**: Low latency on first work.
- **Cons**: Wastes sessions/cost for idle workspaces and adds operational noise.
- **Why rejected**: Spawn-on-coordinated-run-enqueue better fits local-first alpha usage while letting users draft tasks without starting orchestration.

### Alternative 2: Manual coordinator only

- **Description**: Coordinator starts only when a human or automation explicitly requests it.
- **Pros**: Conservative and explicit.
- **Cons**: Undercuts the autonomy goal.
- **Why rejected**: The approved scope requires autonomous orchestration once work appears.

### Alternative 3: Agent definition only

- **Description**: Encode all coordinator settings in a dedicated `AGENT.md`.
- **Pros**: Reuses existing agent loading.
- **Cons**: Mixes runtime policy such as TTL and spawn policy with agent identity.
- **Why rejected**: Runtime policy needs config-level override semantics.

### Alternative 4: User-set orchestration flag at task creation

- **Description**: Require the user or creator to set `orchestration_required` when the task is created.
- **Pros**: Makes the trigger explicit on the task record.
- **Cons**: Confuses task drafting with execution approval and makes users decide orchestration details too early.
- **Why rejected**: Users should be able to create tasks freely. The publish/start/approval action that enqueues a run is the correct execution trigger.

## Consequences

### Positive

- Avoids idle coordinator sessions.
- Lets users draft tasks manually without starting orchestration.
- Lets teams tune coordinator provider/model per workspace.
- Keeps runtime policy separate from the coordinator prompt identity.

### Negative

- Requires new config schema and resolution precedence.
- First orchestration task may pay coordinator cold-start latency.

### Risks

- Risk: workspace overrides drift from safe defaults.
- Mitigation: validate config, enforce hard spawn limits separately, and keep coordinator tools restricted.

- Risk: task creation accidentally triggers coordinator startup.
- Mitigation: only publish/start/approve-execution actions that enqueue a task run can trigger coordinator auto-start.

## Implementation Notes

Add config resolution in the daemon composition layer. Spawned coordinator sessions should be marked with a distinct type or metadata so reapers and observability can identify them. The coordinator trigger should be idempotent per workspace and must not allow a coordinator to spawn another coordinator. Task creation, publish/approval, run enqueue, coordination-channel binding, and run claim must remain separate operations in the API model.
