# ADR-012: Task-Run Coordination Channels

## Status

Accepted

## Date

2026-04-26

## Context

The autonomy MVP has three different coordination planes:

- `task_runs` own durable execution state, claim/lease fencing, and terminal status.
- The coordinator-agent owns semantic orchestration: decomposition, delegation, validation, follow-up tasks, and synthesis.
- Network channels let agents communicate while work is happening.

Without an explicit bridge between task runs and channels, the coordinator could create tasks and spawn workers but leave inter-agent collaboration implicit in private prompts. That would make handoff, debugging, review, and multi-agent synthesis hard to audit.

At the same time, making channels authoritative for task ownership would duplicate the task service and weaken claim/lease guarantees.

## Decision

Every workspace-scoped task run that is enqueued for coordinated execution has one durable coordination channel association.

- The task service records `coordination_channel_id` on the task run at the run-enqueue/start boundary.
- Task creation alone does not create claimable work and does not require a coordination channel.
- The channel may be created lazily, reused for a task/workflow when policy allows, or resolved from an existing workspace channel, but the resulting `coordination_channel_id` must be stable on the run.
- `ClaimNextRun` returns the claimed run's coordination channel metadata so workers know where to communicate.
- `/agent/context` includes the active task/run coordination channel when one exists.
- Channel messages related to coordinated work carry correlation metadata: `task_id`, `run_id`, `workflow_id` when present, `coordination_channel_id`, `message_kind`, and `correlation_id`.
- The MVP message kinds are `status`, `request`, `reply`, `blocker`, `handoff`, `result`, and `review_request`.
- Channels are not an ownership or status authority. Claim, heartbeat, complete, fail, release, and terminal task-run state remain task service operations guarded by claim tokens.
- Raw `claim_token` values must never be sent through channels, persisted in channel messages, logged, or exposed in channel read models.

The coordinator uses the coordination channel for operational communication: status requests, blocker resolution, result exchange, handoffs, review requests, and synthesis context. It uses the task service for durable ownership and status transitions.

The rule is: bind always, speak when useful. A simple one-agent task may have a coordination channel with little or no traffic, but every coordinated run has an auditable place for collaboration when needed.

## Alternatives Considered

### Alternative 1: Let agents choose arbitrary channels

- **Description**: The coordinator and workers choose or create channels through prompt convention.
- **Pros**: Minimal daemon changes.
- **Cons**: No reliable task/run correlation, weak auditability, and hard-to-debug handoffs.
- **Why rejected**: Autonomy needs deterministic correlation between durable work and conversational coordination.

### Alternative 2: Use channel messages as task ownership

- **Description**: Workers accept work and update progress by posting messages in a channel.
- **Pros**: Natural multi-agent conversation model.
- **Cons**: Duplicates `task_runs`, bypasses claim tokens, and makes recovery depend on conversational state.
- **Why rejected**: Durable ownership must remain in the task service with SQLite transactions and lease fencing.

### Alternative 3: No channels in MVP coordination

- **Description**: The coordinator creates tasks and reads task results only; workers never share a task-bound channel.
- **Pros**: Smaller first implementation.
- **Cons**: Produces a silent coordinator/worker system with poor handoff, review, blocker handling, and synthesis traceability.
- **Why rejected**: The MVP already includes local network channels; coordinated work needs a bounded, explicit use for them.

## Consequences

### Positive

- Makes coordinator/worker collaboration auditable without broad network protocol expansion.
- Keeps task ownership, leases, and terminal status in one durable authority.
- Gives workers a deterministic communication target in `ClaimNextRun` and `/agent/context`.
- Creates a clean path for post-MVP memory/session summaries to reference useful channel context.

### Negative

- Adds one durable field to task-run state and one more contract field to agent context/claim responses.
- Requires task enqueue and channel creation/binding to be transactionally coherent enough to avoid orphaned coordinated runs.
- Requires message metadata validation in channel APIs.

### Risks

- Risk: channels become noisy if every state transition is mirrored as chat.
- Mitigation: do not mirror heartbeat or normal lease transitions into channels. Use channels for operational communication only.

- Risk: implementers treat channel `status` messages as authoritative task status.
- Mitigation: tests and docs must prove terminal status changes only through task service APIs.

- Risk: channel creation becomes a hidden source of enqueue failure.
- Mitigation: channel binding should use existing network/channel services with idempotent create-or-resolve behavior and clear errors before run enqueue commits.

## Implementation Notes

Prefer a single `coordination_channel_id` on `task_runs` plus indexed lookup by channel. Do not add a workflow store or channel membership subsystem for the MVP.

Channel message metadata should use typed envelope extension fields if the current network envelope already supports `ext`. If a first-class field is needed, add only the minimal fields listed in this ADR.
