# ADR-008: Memory Provenance Before Rich Memory Scopes

## Status

Accepted

## Date

2026-04-25

## Context

AGH already has global and workspace memory, FTS5 cataloging, prompt recall, and an operation log with `agent_name`. The analysis found that `agent_name` is plumbed but not meaningfully used, and proposed peer/channel/session scopes plus several automatic extraction paths.

Adding every memory scope and extractor at once risks storing noisy coordination chatter before the system has clear provenance and workflow correlation.

## Decision

Start memory autonomy with provenance and session-level usefulness:

- Ensure memory writes and operation history record the real agent/session provenance.
- Add session-scope memory or session summaries for handoff and post-run review.
- Add recall provenance in prompt blocks: file, scope, agent, score, and freshness.
- Add lightweight session-end summaries before broad turn/network extraction.
- Defer full peer/channel memory and network echo/receipt promotion until workflows produce enough signal.

## Alternatives Considered

### Alternative 1: Add all memory scopes immediately

- **Description**: Implement peer, channel, session, and skill scopes plus hook-based extraction in one pass.
- **Pros**: Comprehensive memory model.
- **Cons**: Higher write volume and risk of low-signal memory pollution.
- **Why rejected**: Provenance and session summaries provide immediate value with less complexity.

### Alternative 2: Leave memory unchanged

- **Description**: Keep global/workspace memory only.
- **Pros**: No schema or recall changes.
- **Cons**: Autonomous workflows lack durable handoff summaries and agent attribution.
- **Why rejected**: Provenance is required for trustworthy autonomous coordination.

## Consequences

### Positive

- Improves trust in memory by showing who wrote what.
- Supports coordinator/worker handoffs without broad memory redesign.
- Keeps automatic extraction conservative.

### Negative

- Peer/channel knowledge sharing remains limited initially.
- Some coordinator context may still live in task/session events rather than memory.

### Risks

- Risk: session summaries omit important facts.
- Mitigation: include links to task runs, session events, and touched artifacts so summaries remain auditable.

## Implementation Notes

Prefer append-only summaries for session-scope artifacts. Do not store every network message as memory by default.
