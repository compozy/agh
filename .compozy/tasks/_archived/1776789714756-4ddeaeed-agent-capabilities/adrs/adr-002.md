# ADR-002: Dual Storage Modes Without Merge

## Status

Accepted

## Date

2026-04-19

## Context

Some agents will have a small number of capabilities and benefit from a single catalog file. Others may grow into larger sets of capability definitions where one file per capability is easier to maintain. At the same time, allowing multiple overlapping storage modes or merge semantics would make loading and validation ambiguous.

## Decision

The runtime will support two storage modes for capability catalogs:

- single-file mode
- directory mode

Only one mode may be used for a given agent. The runtime will not merge file mode with directory mode, and it will not merge TOML and JSON within the same directory mode.

## Alternatives Considered

### Alternative 1: Single-file mode only

- **Description**: Require all capabilities to live in one `capabilities.toml` or `capabilities.json`.
- **Pros**: Simplest loader and validation model.
- **Cons**: Becomes unwieldy for larger capability sets.
- **Why rejected**: It optimizes for small agents only.

### Alternative 2: Support mixed file/directory merge

- **Description**: Allow `capabilities.toml` plus `capabilities/` and merge results.
- **Pros**: Maximum flexibility.
- **Cons**: Ambiguous precedence, duplicate handling, and poor authoring ergonomics.
- **Why rejected**: It adds complexity without improving semantic clarity.

## Consequences

### Positive

- Supports both small and large agent catalogs.
- Keeps loader behavior deterministic.
- Makes validation failures easy to explain.

### Negative

- Loader logic is slightly more complex than a single-mode design.
- Authors must choose a mode and stick to it.

### Risks

- Users may accidentally create mixed-mode catalogs.
- Mitigation: hard validation error with clear diagnostics.

## Implementation Notes

- Supported single-file formats: `capabilities.toml`, `capabilities.json`
- Supported directory formats: `capabilities/*.toml`, `capabilities/*.json`
- Directory mode uses self-contained capability files and does not require a `_catalog` manifest.
