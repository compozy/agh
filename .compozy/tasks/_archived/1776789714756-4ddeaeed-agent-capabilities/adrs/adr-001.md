# ADR-001: Explicit Capability Catalogs

## Status

Accepted

## Date

2026-04-19

## Context

AGH agents already have a self-contained directory structure centered on `AGENT.md`, with sidecars such as `mcp.json`. The network protocol already advertises `PeerCard.capabilities`, but the runtime has no explicit local source of truth for those capabilities. Deriving capabilities from tools, MCP servers, or prompt text would create unstable and ambiguous semantics because those inputs describe implementation details rather than outcome-oriented offers.

## Decision

AGH runtime will use explicit capability catalogs stored in the agent directory as the source of truth for agent capabilities. Capability declarations are authored intentionally by the agent author and projected into network discovery from a normalized runtime model.

## Alternatives Considered

### Alternative 1: Infer capabilities from tools, MCPs, or prompt text

- **Description**: Derive capability identifiers and metadata from existing agent configuration and prose.
- **Pros**: No new authoring surface.
- **Cons**: Semantics are ambiguous, brittle, and difficult to validate.
- **Why rejected**: Capabilities must represent stable delegation offers, not implementation leakage.

### Alternative 2: Keep capabilities only as network-time strings

- **Description**: Let network peers advertise raw capability strings without a local runtime catalog.
- **Pros**: Minimal runtime work.
- **Cons**: No structured authoring model, no rich discovery source, and no validation path.
- **Why rejected**: It preserves the current integration gap instead of fixing it.

## Consequences

### Positive

- Creates a clear local source of truth for capability discovery.
- Keeps capability semantics independent from tools, MCP servers, and prompt wording.
- Enables both brief and rich network projection from one runtime model.

### Negative

- Adds new authoring responsibility for agent maintainers.
- Requires new loader and validation logic in the runtime.

### Risks

- Authors may declare vague capabilities that overlap semantically.
- Mitigation: require `id`, `summary`, and `outcome`, and document authoring guidance.

## Implementation Notes

- The runtime catalog belongs to the agent directory, not to `AGENT.md` frontmatter.
- The network projection remains advisory and discovery-oriented.
