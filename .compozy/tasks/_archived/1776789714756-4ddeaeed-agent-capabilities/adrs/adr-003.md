# ADR-003: Soft Outcome-Oriented Capability Model

## Status

Accepted

## Date

2026-04-19

## Context

AGH operates in a non-deterministic, LLM-first environment where agentic delegation is preferred over rigid workflow bureaucracy. A capability needs enough structure for discovery and delegation, but it should not become a strict RPC contract with mandatory typed input/output schemas. It must also avoid exposing internal skills as the public surface.

## Decision

Capabilities will be modeled as structured, outcome-oriented delegation offers. Each capability must define:

- `id`
- `summary`
- `outcome`

Additional fields such as `context_needed`, `artifacts_expected`, `execution_outline`, `constraints`, and `examples` remain optional. `execution_outline` is free-form and descriptive, not a formal workflow language.

## Alternatives Considered

### Alternative 1: Strict RPC-style capability contracts

- **Description**: Require typed input and output schemas for each capability.
- **Pros**: Stronger automation and validation.
- **Cons**: Too rigid for non-deterministic, agentic execution.
- **Why rejected**: It would push the system toward API bureaucracy instead of agentic delegation.

### Alternative 2: Flat capability strings only

- **Description**: Keep only capability identifiers with no structured metadata.
- **Pros**: Minimal model and smallest payloads.
- **Cons**: Weak discovery, poor delegation ergonomics, and no rich capability understanding.
- **Why rejected**: It is too thin to replace lightweight workflow-like delegation.

## Consequences

### Positive

- Preserves agentic flexibility while improving discoverability.
- Supports richer peer selection and delegation without exposing internal implementation details.
- Keeps network projection lightweight for brief discovery and useful for rich discovery.

### Negative

- Capabilities are less machine-rigid than typed RPC contracts.
- Rich discovery remains partly semantic rather than fully schema-driven.

### Risks

- Poorly written capability metadata may still create confusion.
- Mitigation: require the outcome triplet and document authoring guidelines.

## Implementation Notes

- Same-named capabilities across peers are disambiguated operationally by `peer_id + capability_id`.
- Capability discovery on the wire remains advisory and does not imply authorization.
