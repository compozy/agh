# ADR-002: Shared Frontmatter Parser Package

## Status

Accepted

## Date

2026-03-31

## Context

With roles, playbooks, and skills all adopting Markdown with YAML frontmatter, each artifact loader needs to: (1) split the file at `---` delimiters, (2) unmarshal YAML into a typed struct, and (3) return the Markdown body. Duplicating this logic across `internal/config/roles.go`, `internal/config/playbooks.go`, and the future `internal/skills/loader.go` violates DRY and creates divergent parsing behavior.

The project already has `goccy/go-yaml` as a dependency.

## Decision

Create a shared `internal/frontmatter` package that provides a single generic `Parse` function. The function accepts raw file bytes and a destination pointer (any struct with `yaml:` tags), extracts the YAML frontmatter, unmarshals it into the destination, and returns the Markdown body as a string. All artifact loaders delegate frontmatter extraction to this package.

The package also provides a `Format` function for the reverse operation (serializing a struct + body back to Markdown with frontmatter), used by `discovery.go` when writing role and playbook files.

## Alternatives Considered

### Alternative 1: Inline Parsing in Each Loader

- **Description**: Each artifact loader implements its own frontmatter splitting and YAML parsing.
- **Pros**: No new package; each loader controls its own parsing rules.
- **Cons**: Code duplication across three loaders; risk of divergent behavior; harder to test edge cases consistently.
- **Why rejected**: Unnecessary duplication for identical parsing logic.

### Alternative 2: Use a Third-Party Frontmatter Library

- **Description**: Import a Go library like `github.com/adrg/frontmatter` for parsing.
- **Pros**: Battle-tested; handles edge cases.
- **Cons**: Additional dependency for a simple operation (split on `---`, unmarshal YAML); less control over error messages; the project already has `goccy/go-yaml`.
- **Why rejected**: The parsing logic is straightforward (~30 lines); adding a dependency is not justified.

## Consequences

### Positive

- Single source of truth for frontmatter parsing and formatting.
- Consistent error messages and edge case handling across all artifact types.
- Easy to unit test in isolation.
- Future skills loader reuses the same package with zero additional work.

### Negative

- New internal package to maintain (though minimal surface area).

### Risks

- None significant. The package is small, focused, and has no external dependencies beyond `goccy/go-yaml`.

## Implementation Notes

Core API surface:

```go
package frontmatter

// Parse extracts YAML frontmatter into dest and returns the Markdown body.
func Parse(content []byte, dest any) (body string, err error)

// Format serializes a struct as YAML frontmatter followed by the Markdown body.
func Format(meta any, body string) ([]byte, error)
```

- Detects `---` delimiters (first line must be `---`).
- Returns clear error if frontmatter is missing or malformed.
- Uses `goccy/go-yaml` for marshal/unmarshal.
