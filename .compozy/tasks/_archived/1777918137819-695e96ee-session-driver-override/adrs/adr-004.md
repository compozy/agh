# ADR-004: Use Explicit Session Creation Surfaces For Provider Selection

## Status

Accepted

## Date

2026-04-20

## Context

The current web app starts sessions directly from sidebar quick actions and sends only `agent_name` plus workspace context. That flow does not expose enough room for deterministic provider selection, nor does it make the chosen runtime visible before launch.

The feature must support provider selection cleanly in the web app and stay aligned with HTTP/UDS/CLI transport semantics.

## Decision

AGH will expose provider selection through explicit session-creation surfaces:

- The web app will open a create-session dialog for new sessions.
- Direct session-creation transport contracts will add an optional `provider` request field.
- The extension Host API `sessions.create` method will accept the same optional provider override because it is also an explicit session-creation surface.
- Session response payloads will expose the effective `provider`.
- CLI session creation will add a matching optional `--provider` flag.
- Workspace detail payloads will expose the workspace-visible provider catalog needed to render the dialog picker.
- Internal non-interactive creators that do not expose session choice in this feature will continue to pass an empty provider and therefore use the agent default runtime.

## Alternatives Considered

### Alternative 1: Keep quick-create and add an advanced escape hatch

- **Description**: Preserve the current direct-create path and add a second optional provider picker flow.
- **Pros**: Keeps the fastest current interaction intact.
- **Cons**: Splits session creation into two semantics and makes provider selection feel secondary.
- **Why rejected**: The approved UX favors one explicit creation path.

### Alternative 2: Add provider selection inline in the sidebar

- **Description**: Put provider controls next to each agent entry.
- **Pros**: Fast for power users.
- **Cons**: Increases sidebar density and makes the shell harder to scan and test.
- **Why rejected**: Too much UI complexity for a core creation action.

## Consequences

### Positive

- The operator sees and confirms the provider before session start.
- Web, API, and CLI stay aligned on one transport concept.
- Session detail surfaces can show both agent and provider without reconstructing hidden state.

### Negative

- Web session creation becomes one step more explicit than today.
- Existing sidebar tests and create-session hooks will need refactoring.

### Risks

- If the dialog becomes too broad, it may duplicate workspace-selection behavior already owned by the app shell.
- Mitigation: prefill from current app context and keep fields scoped to session creation concerns.

## Implementation Notes

- The dialog can preselect the clicked agent and active workspace while still requiring explicit confirmation.
- The request field name should stay short and consistent with config terminology: `provider`.
- The web picker should source its options from the resolved workspace payload instead of inventing a second provider-discovery path.
