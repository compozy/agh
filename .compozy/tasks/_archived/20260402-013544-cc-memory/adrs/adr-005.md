# ADR-005: MEMORY.md Index Injection Over Full Content Loading

## Status

Accepted

## Date

2026-04-01

## Context

Agent system prompts need access to persistent memory. The question is how much memory content to inject into prompts — the full content of all memory files, just the index, or nothing (requiring agents to explicitly query).

Claude Code loads MEMORY.md (the index) always into context and uses a background LLM call to select up to 5 relevant memories for full loading. Since AGH does not make LLM calls from the kernel, the relevance selection approach is not available.

## Decision

Always inject both MEMORY.md indexes (global + workspace) into agent system prompts as an additional section in the prompt assembler. Full memory file content is NOT loaded into the prompt. Agents read individual memory files on demand via `agh memory read <filename>`.

The indexes are capped at 200 lines / 25KB each (matching Claude Code's limits) to prevent context bloat.

The prompt section includes:
- Global MEMORY.md content
- Workspace MEMORY.md content
- Instructions on the 4-type taxonomy
- What NOT to save guidance
- Staleness policy
- How to read/write memories via CLI commands

## Alternatives Considered

### Alternative 1: Inject Full Memory Content

- **Description**: Load all memory file contents into the system prompt
- **Pros**: Agents have immediate access to all knowledge, no extra CLI calls
- **Cons**: Context bloat — 20 memory files at 1KB each = 20KB of prompt space. Grows linearly with memory count. Irrelevant memories waste context tokens
- **Why rejected**: Unbounded growth makes this unsustainable. The index-only approach keeps prompt size constant regardless of memory count

### Alternative 2: No Auto-Injection

- **Description**: Agents must explicitly run `agh memory list` and `agh memory read` to access memories
- **Pros**: Zero prompt overhead, agents only load what they need
- **Cons**: Agents must know to check for memories. New agents or agents without memory-aware prompts will never access memories. Discovery is lost
- **Why rejected**: Memory is only useful if agents are aware it exists. The index serves as a lightweight discovery mechanism

### Alternative 3: LLM-Based Relevance Selection

- **Description**: Use a background LLM call (via a fast model) to select relevant memories for full injection
- **Pros**: Best of both worlds — agents get full content of relevant memories only
- **Cons**: AGH's kernel does not make LLM calls. Would require a side-channel agent spawn for every prompt assembly, adding latency and complexity
- **Why rejected**: Architectural mismatch — this is a valid future enhancement but requires infrastructure that doesn't exist yet. The index approach is sufficient for v1

## Consequences

### Positive

- Constant prompt size regardless of memory count (bounded by index caps)
- Agents are always aware of what memories exist
- On-demand loading via CLI keeps full content access available
- Matches Claude Code's pattern of MEMORY.md always in context

### Negative

- Agents must make an extra `agh memory read` call to access full memory content
- No automatic relevance filtering — agents see the full index and must judge relevance themselves
- Index entries must be kept concise (~150 chars) to stay within the 200-line cap

### Risks

- Agents may not read memories even when relevant if the index descriptions are too vague. Mitigated by enforcing descriptive one-line hooks in the index format
- Index truncation (>200 lines) means some memories become invisible. Mitigated by dream consolidation pruning stale entries
