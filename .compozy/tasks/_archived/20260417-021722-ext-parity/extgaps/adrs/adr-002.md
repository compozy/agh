# ADR-002: Treat bundle-managed automations and bridges as package-owned resources

## Status

Accepted

## Date

2026-04-14

## Context

Bundle profiles materialize automation jobs, automation triggers, and bridge instances that originate from extension-shipped specifications rather than direct operator CRUD. Those resources must stay attributable to the package that created them so the daemon can reconcile them safely and prevent accidental mutation of the package base specification through generic APIs.

The pre-existing automation and bridge surfaces already supported config-owned and dynamic resources, but they did not distinguish package-managed definitions or bridge instances.

## Decision

Add explicit package ownership markers to the affected domains and preserve the bundle specification as the source of truth. Automation definitions use `JobSourcePackage`. Bridge instances use `BridgeInstanceSourcePackage`. Generic mutation APIs reject direct edits to package-owned base specs, while limited operational overlays remain allowed through the dedicated runtime behavior already implemented.

## Alternatives Considered

### Alternative 1: Materialize package resources as dynamic resources

- **Description**: Reuse the existing dynamic source classification for package-created automations and bridges.
- **Pros**: Minimal schema and type changes.
- **Cons**: Loses source attribution, weakens reconcile safety, and makes read-only enforcement ambiguous.
- **Why rejected**: The implementation needed clear ownership and conflict handling distinct from user-created runtime resources.

### Alternative 2: Keep ownership only in bundle inventory

- **Description**: Track package ownership outside automation and bridge records, without adding source fields.
- **Pros**: Fewer changes inside automation and bridges packages.
- **Cons**: Requires cross-table joins for basic ownership checks and makes existing CRUD code harder to validate correctly.
- **Why rejected**: The shipped code needed first-class ownership checks inside each domain's validation and API handling.

## Consequences

### Positive

- Reconcile logic can target only package-managed resources.
- Generic CRUD surfaces can reject invalid edits with domain-specific errors.
- Ownership is visible in persisted records and domain types.

### Negative

- Adds new source values and read-only rules to two existing subsystems.
- Requires schema expansion for persisted bridge instances.

### Risks

- Future API changes must preserve the distinction between package-owned and operator-owned resources.
- Overlay scope must remain narrow to avoid silent drift from the bundle spec.

## Implementation Notes

- `internal/automation/types.go` exports `JobSourcePackage`.
- `internal/automation/manager.go` exposes `SyncManagedDefinitions(...)` to reconcile daemon-managed sources.
- `internal/bridges/types.go` adds `BridgeInstanceSourcePackage` and `ErrBridgeInstanceReadOnly`.
- `internal/store/globaldb/global_db_bridge.go` persists bridge instance `source`.
