# BUG-005: Reconciliation race condition with concurrent Activate/Deactivate

**Severity:** Medium
**Priority:** P2
**Component:** `internal/bundles/service.go` — `Reconcile()`
**Related:** TC-FUNC-011

## Environment

- Code review of `internal/bundles/service.go`
- Reconciliation is called from Activate, UpdateActivation, and Deactivate

## Description

The `Service.Reconcile()` method is not serialized. While `NetworkSettings` uses `sync.RWMutex` for the settings cache, the actual reconciliation logic (reading activations, syncing automations, syncing bridges, updating inventory) is NOT protected by a mutex.

If two concurrent Activate calls occur:

1. Both read the activation list (possibly seeing different states)
2. Both run `SyncManagedDefinitions` (last writer wins)
3. Both run `syncPackageBridges` (race on bridge insert/update/delete)
4. Both update inventory (last writer wins for each activation)

The `mu` mutex only protects the `settings` field, not the reconciliation process itself.

### Expected Behavior

Reconciliation should be serialized to prevent race conditions between concurrent state mutations.

### Actual Behavior

Concurrent calls can interleave, potentially leading to:

- Missed bridge deletions (bridge exists in both desired maps)
- Inventory inconsistencies
- Automation sync with stale desired state

## Impact

- Low probability in practice (reconciliation is fast and daemon operations are sequential)
- Could cause inconsistent state if CLI and web UI trigger activations simultaneously
- Would fail under `-race` flag if tested with concurrent goroutines

## Suggested Fix

Add a reconciliation mutex (`reconcileMu sync.Mutex`) that serializes the entire `Reconcile()` method body, separate from the settings RWMutex.

## Workaround

Operations are currently serialized by the HTTP handler's sequential request processing. Issue only manifests with truly concurrent requests from different transports.
