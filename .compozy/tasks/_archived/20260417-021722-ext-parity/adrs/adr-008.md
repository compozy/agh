# ADR-008: Confine Raw JSON to the Persistence Boundary and Expose Typed Domain Adapters

## Status

Accepted

## Date

2026-04-15

## Context

The shared extensibility runtime persists heterogeneous resource specs in SQLite and receives generic resource payloads over operator and extension transport boundaries. That persistence and transport boundary naturally carries encoded JSON bytes.

The earlier TechSpec versions leaked `json.RawMessage` through the main store and projector contracts. That would have pushed untyped payload handling into the exact domain call sites that need strong invariants:

- hook binding projection;
- automation and bridge definition projection;
- bundle activation fan-out;
- kind-specific validation and projection logic.

AGH is a Go codebase with explicit package boundaries and a preference for concrete types. Letting raw JSON propagate through domain code would weaken type safety, encourage repeated decode logic, and make projector code harder to reason about and test.

## Decision

Keep a raw persistence kernel inside `internal/resources`, but expose typed adapters everywhere else.

Specifically:

- the SQLite and transport boundary uses internal raw types such as `RawDraft`, `RawRecord`, and a `rawStore`;
- each resource kind owns a `KindCodec[T]` that decodes and validates raw JSON into one concrete Go type and encodes typed specs back to bytes;
- domain packages consume typed `Store[T]`, `Record[T]`, and `TypedProjector[T]` adapters instead of `json.RawMessage`;
- typed `Store[T]` reads still carry `MutationActor`, so source and scope filtering stays enforced at the façade edge rather than only in the raw kernel;
- the reconcile driver still operates through an internal projector adapter boundary, but that adapter is responsible for decoding raw records into typed domain inputs before invoking domain build logic;
- kinds whose build logic depends only on their primary kind register `TypedProjector[T]` through a thin `Projector` adapter;
- kinds with heterogeneous dependencies, such as `bundle.activation`, use package-local adapter code to decode dependency kinds explicitly rather than forcing a one-size-fits-all generic dependency container across the whole runtime.

## Alternatives Considered

### Alternative 1: Keep `json.RawMessage` in all store and projector contracts

- **Description**: Let the shared runtime and domain code all consume raw JSON payloads directly.
- **Pros**: Smallest shared API surface; fewer adapter types.
- **Cons**: Loses type safety in domain logic, encourages repeated parsing, and makes invariants harder to express.
- **Why rejected**: The shared runtime should not erase the type information that Go domain packages rely on.

### Alternative 2: Make the entire runtime generic by kind

- **Description**: Replace the raw kernel entirely with generic `Store[T]` and `Projector[T]` contracts end to end.
- **Pros**: Strong typing everywhere.
- **Cons**: Fights the heterogeneous persistence and transport boundary; makes mixed-kind operations such as snapshots and dependency-aware reconcile awkward.
- **Why rejected**: The runtime still needs a raw kernel because persistence, snapshot application, and transport negotiation are heterogeneous by nature.

### Alternative 3: Use one generic projector API with generic dependency bags

- **Description**: Force every projector, including mixed-kind cases, into a single generic dependency container.
- **Pros**: Uniform shared interface.
- **Cons**: Either reintroduces untyped bags for dependencies or creates an over-general abstraction that is hard to implement and reason about.
- **Why rejected**: Mixed-kind dependency decoding is better handled in package-local adapters close to the domain.

## Consequences

### Positive

- Keeps raw JSON confined to the boundary that actually needs it.
- Preserves concrete typing for validators, stores, and projector logic.
- Makes codec behavior explicit and testable per kind.
- Avoids reparsing the same record multiple times inside one validation or projection pass.

### Negative

- Adds codecs and thin typed adapters per migrated kind.
- Requires clearer registration wiring inside `internal/resources`.

### Risks

- Adapter code can drift from raw-store semantics if tests are weak.
- Mixed-kind projector inputs still need careful package-local design.

## Implementation Notes

- `rawStore` stays internal to `internal/resources`; domain packages should not import or depend on it directly.
- `KindCodec[T]` should be the only shared place that decodes raw bytes into typed specs.
- `Store[T]` adapters should round-trip through the codec and raw store rather than bypassing persistence rules.
- Projector adapter tests should verify that raw records are decoded once per reconcile pass before domain build logic runs.
