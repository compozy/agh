# ADR-006: Use a Topology-Aware Reconcile Driver

## Status

Accepted

## Date

2026-04-15

## Context

The tightened TechSpec moved consistency to full-snapshot reconcile, but that alone was not enough. The design still needed an explicit component that owns when reconcile runs, how repeated triggers collapse, and how inter-kind ordering works at runtime.

Without that component:

- one write that changes many records could trigger reconcile an arbitrary number of times;
- boot-time rebuild order would remain implicit;
- dependent kinds such as bundle-derived state would have no declared runtime ordering;
- the platform would quietly fall back to ad hoc post-write callbacks spread across subsystems.

The architecture needs a named control-plane component rather than another implicit responsibility inside `daemon` or `internal/resources`.

## Decision

Introduce a named reconcile driver owned by the daemon composition root.

The reconcile driver will:

- run one in-flight reconcile per resource kind;
- coalesce repeated triggers for the same kind into one pending rerun after the current pass completes;
- trigger at most one reconcile request per affected kind for each committed mutation transaction;
- run migrated kinds on boot in topology order derived from projector dependencies;
- schedule dependents after successful commit of the source mutation, not before commit;
- maintain a reverse dependency index from `Projector.DependsOn()` for dependent fan-out;
- run each kind under a bounded reconcile timeout and propagate that deadline into projector execution;
- expose `Close(ctx)` so shutdown drains or cancels in-flight work deterministically;
- open a degraded circuit for repeatedly failing kinds so the driver does not busy-loop on permanent errors.

V1 keeps correctness event-bus-free: the reconcile driver invokes projectors directly from the daemon process. It uses dirty-bit coalescing with a small bounded rerun window, not an unbounded timing queue.

## Alternatives Considered

### Alternative 1: Ad hoc post-write callbacks

- **Description**: Each mutation path directly invokes the projector or runtime it thinks is affected.
- **Pros**: Smallest immediate implementation surface.
- **Cons**: Recreates fragmented change propagation and hides ordering rules in call sites.
- **Why rejected**: It preserves the same structural drift the shared runtime is meant to remove.

### Alternative 2: Projector self-scheduling

- **Description**: Let each projector observe mutations and decide when to rerun itself.
- **Pros**: Keeps the central runtime smaller.
- **Cons**: Pushes coordination into every projector and makes ordering cross-cutting and inconsistent.
- **Why rejected**: Scheduling is a control-plane responsibility, not a per-projector concern.

### Alternative 3: Debounce as the primary correctness mechanism

- **Description**: Make reconcile depend on batching writes into timing windows before projector execution.
- **Pros**: Can reduce repeated work under heavy write bursts.
- **Cons**: Adds timing semantics and latency to correctness-critical behavior.
- **Why rejected**: V1 needs deterministic post-commit semantics; the small rerun window is only a bounded coalescing aid, not the correctness model.

## Consequences

### Positive

- Makes reconcile semantics explicit and testable.
- Avoids hidden event-bus behavior while still collapsing redundant work.
- Gives bundle and dependent-family ordering a clear home.
- Prevents one stuck or permanently failing projector from wedging the whole scheduler indefinitely.

### Negative

- Adds one more foundational component before family migration can begin.
- Requires explicit topology metadata from projectors.

### Risks

- Incorrect dependency declarations could still produce bad ordering.
- Overly broad dependent triggering can create unnecessary reruns.
- A poorly implemented single-flight loop can starve heavily updated kinds.
- Timeout and degraded-circuit thresholds that are too aggressive can mask slow but healthy projectors.

## Implementation Notes

- The driver can live under `internal/resources` or another existing package, but it must remain daemon-owned.
- Each migrated projector should declare `DependsOn() []ResourceKind`.
- Contract tests should prove single-flight, bounded coalescing, topology order, timeout propagation, degraded-circuit behavior, and shutdown drain semantics.
- A healthy idle kind should begin reconcile within the TechSpec target window, and sustained write storms must never create more than one pending rerun per kind.
