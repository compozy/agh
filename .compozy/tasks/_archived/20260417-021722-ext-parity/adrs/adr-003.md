# ADR-003: Gate Every Domain Cutover With Contract, Integration, and Reconcile Verification

## Status

Accepted

## Date

2026-04-15

## Context

The selected architecture changes the source of truth for multiple runtime domains. That kind of refactor can appear correct at the unit level while still failing at boot, reconcile time, or subsystem integration boundaries. The user explicitly selected the strictest validation option for each cutover phase.

Because AGH is alpha, the project does not need compatibility scaffolding. That increases the importance of verification: once a family cuts over, the removed path is gone and confidence must come from tests and runtime evidence rather than fallback code.

## Decision

Every family cutover into the shared resource runtime must be blocked on all of the following:

- `make verify`
- resource-runtime contract tests for the new store, validation, authorization, and reconcile behavior
- migrated-domain integration tests proving the target subsystem operates from the new runtime
- boot or reconcile smoke coverage, including benchmark-style checks where startup or rebuild cost is material

No phase is considered complete if only unit tests pass.

## Alternatives Considered

### Alternative 1: Rely on `make verify` only

- **Description**: Treat the existing repo-wide verification gate as sufficient.
- **Pros**: Simple and already standardized.
- **Cons**: Too coarse to prove shared-runtime behavior, projection correctness, or reconcile semantics.
- **Why rejected**: The chosen refactor needs explicit contract and cutover evidence.

### Alternative 2: Add only unit tests around the new runtime

- **Description**: Verify the generic runtime in isolation and assume migrated domains remain correct.
- **Pros**: Fast feedback; simpler tests.
- **Cons**: Misses domain projection failures and boot-time regressions.
- **Why rejected**: The highest-risk failures are at subsystem boundaries.

### Alternative 3: Defer performance or smoke validation until the end

- **Description**: Validate correctness first and examine startup/reconcile cost only after all phases land.
- **Pros**: Less upfront work per phase.
- **Cons**: Performance regressions accumulate unnoticed and become harder to localize.
- **Why rejected**: The user chose strict cutover gates rather than end-loaded validation.

## Consequences

### Positive

- Detects root-cause regressions where they actually surface: at ownership, projection, and runtime rebuild boundaries.
- Keeps each phase shippable without relying on compatibility scaffolding.
- Produces the evidence needed for later task execution and review rounds.

### Negative

- Raises the implementation cost of each migration phase.
- Requires better test fixtures and representative runtime setup than current domain-local tests.

### Risks

- Test suites can become slow if integration scope is not kept targeted per family.
- Smoke and benchmark checks may be noisy if baseline scenarios are not well defined.
- Developers may be tempted to weaken tests rather than fix production behavior; this must be resisted.

## Implementation Notes

- Resource-runtime contract tests should live near the new runtime package and verify canonical ownership, authorization, and reconcile semantics.
- Each family should add integration tests near its runtime package using real SQLite and real projection code.
- Benchmark-style checks can begin as deterministic smoke assertions with bounded timing expectations and evolve into dedicated benchmarks where needed.
