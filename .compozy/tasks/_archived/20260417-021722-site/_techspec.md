# TechSpec: AGH Website & Documentation Site

## Executive Summary

Build a documentation website for AGH at `agh.network` using Fumadocs (Next.js) as the framework. The site serves two linked audiences through two content collections: **AGH Runtime** for operators and **AGH Network Protocol** for implementers. The landing page is a custom Next.js page that presents a balanced runtime + network story, but keeps **Runtime / Get Started** as the primary conversion path for cold visitors. `AGH Network` is used as the public-facing marketing label; `AGH Network Protocol` is reserved for spec and reference contexts. The site lives in `packages/site/` alongside the existing `web/` (unchanged for alpha), sharing design tokens from `packages/ui`. CLI reference is auto-generated from Cobra; API reference is deferred to Wave 2.

**Primary trade-off**: Fumadocs is younger than Astro Starlight or Docusaurus, with a smaller community and fewer examples. We accept this because it runs on Next.js/React — enabling component sharing from `packages/ui` and consistent DESIGN.md theming. Note: Astro Starlight can also render React components via its React integration, but the styling/theming layer still requires Astro-specific adaptation rather than native Tailwind CSS v4 preset sharing.

## System Architecture

### Component Overview

```
web/                       # React SPA (unchanged for alpha)
├── src/
└── package.json           # depends on @agh/ui

packages/
├── ui/                    # Shared design tokens + server-safe components
│   ├── src/
│   │   ├── tokens.css     # DESIGN.md CSS variables (server-safe)
│   │   ├── tailwind-preset.ts  # Shared Tailwind config (server-safe)
│   │   ├── components/    # Server-safe base components only
│   │   └── index.ts       # Public API — explicit server/client exports
│   └── package.json
│
└── site/                  # Fumadocs documentation site
    ├── app/
    │   ├── layout.tsx     # Root layout (dark theme, shared nav)
    │   ├── page.tsx       # Custom landing page
    │   ├── runtime/       # Fumadocs runtime docs route
    │   └── protocol/      # Fumadocs protocol docs route
    ├── content/
    │   ├── runtime/       # Runtime MDX content (~50 pages)
    │   └── protocol/      # Protocol MDX content (~13 pages)
    ├── components/
    │   ├── landing/       # Landing page sections
    │   └── docs/          # Custom MDX components
    ├── source.config.ts   # Two Fumadocs doc sources
    └── package.json       # depends on @agh/ui, fumadocs-*
```

**packages/ui export boundaries** (critical for Vite SPA + Next.js App Router coexistence):

- **Server-safe exports** (tokens.css, tailwind-preset.ts, pure CSS): usable in both web/ and site/ without restrictions
- **Client components** (Button, Badge, Card with `"use client"` directive): usable in both, but must be explicitly marked
- **NOT shared in alpha**: domain-specific components (session list, network panel, etc.) stay in web/. Only design tokens and base UI primitives move to packages/ui.

**Data flow**:

1. MDX content in `content/` is processed by Fumadocs at build time
2. `source.config.ts` defines two doc sources via `defineDocs()`: `runtime` and `protocol`
3. `lib/source.ts` exposes two loaders with `baseUrl: '/runtime'` and `baseUrl: '/protocol'`
4. Landing page at `/` is a custom Next.js page importing from `packages/ui` and `components/landing/`
5. CLI reference MDX is generated by `make cli-docs` (Cobra GenMarkdownTree → post-process)
6. API reference deferred to Wave 2 (hand-maintained or fumadocs-openapi when OpenAPI spec is ready)

**External interactions**:

- Vercel for deployment and preview environments
- Go binary for CLI doc generation

## Implementation Design

### Core Interfaces

Fumadocs source configuration — defines two independent doc trees:

```typescript
// packages/site/source.config.ts
import { defineDocs, defineConfig } from "fumadocs-mdx/config";

export const runtime = defineDocs({
  dir: "content/runtime",
});

export const protocol = defineDocs({
  dir: "content/protocol",
});

export default defineConfig();
```

Fumadocs source loader — exposes page trees to route handlers with `baseUrl`:

```typescript
// packages/site/lib/source.ts
import { loader } from "fumadocs-core/source";
import { runtime, protocol } from "@/.source";

export const runtimeDocs = loader({
  source: runtime.toRuntime(),
  baseUrl: "/runtime",
});
export const protocolDocs = loader({
  source: protocol.toRuntime(),
  baseUrl: "/protocol",
});
```

### Data Models

**Content structure** — MDX frontmatter schema for all pages:

| Field         | Type   | Required | Description                                |
| ------------- | ------ | -------- | ------------------------------------------ |
| `title`       | string | yes      | Page title, used in sidebar and `<title>`  |
| `description` | string | yes      | One-line description, used in meta and TOC |
| `icon`        | string | no       | Lucide icon name for sidebar               |

**Landing page sections** — no database; content is hardcoded in React components:

| Section          | Component              | Content Source                          |
| ---------------- | ---------------------- | --------------------------------------- |
| Hero             | `hero.tsx`             | Outcome-led runtime + network split     |
| Two Pillars      | `two-pillars.tsx`      | Balanced AGH Runtime vs AGH Network     |
| How It Works     | `how-it-works.tsx`     | 3 steps: install, start, create session |

### API Endpoints

Not applicable — the site is a static/SSG site with no API endpoints. The site _documents_ AGH's API but does not expose its own.

## Integration Points

### Go Binary — CLI Reference Generation

- **Purpose**: Extract CLI command metadata for reference docs
- **Method**: `doc.GenMarkdownTree(rootCmd, outputDir)` in a `doc` subcommand
- **Output**: Markdown files in `packages/site/content/runtime/reference/cli/`
- **Post-processing**: Script adds Fumadocs frontmatter, converts `.md` → `.mdx`
- **Trigger**: `make cli-docs`

### Vercel — Deployment

- **Purpose**: Host the site at `agh.network`
- **Method**: Vercel Git integration, auto-deploy on push
- **Preview**: PR preview deployments for doc changes

### packages/ui — Shared Components

- **Purpose**: Visual consistency between app and site
- **Method**: Workspace dependency (`@agh/ui`)
- **Shared**: CSS variables (DESIGN.md tokens), Tailwind preset, base UI components (Button, Badge, Card, etc.)

## Testing Approach

### Unit Tests

- **Landing page components**: Snapshot tests for each section component (hero, two-pillars, etc.) to catch unintended visual regressions
- **Custom MDX components**: Test protocol-specific components (EnvelopeViewer, MessageKindCard) render correctly with sample data
- **Post-processing scripts**: Test CLI markdown → MDX transformation preserves content and adds correct frontmatter

### Integration Tests

- **Build verification**: `turbo run build --filter=packages/site` must complete without errors — this is the primary integration test
- **Link checking**: Run a link checker against the built site to catch broken internal links between runtime and protocol collections
- **Codegen pipeline**: `make cli-docs && make api-spec && turbo run build --filter=packages/site` must succeed end-to-end

## Development Sequencing

### Build Order

1. **Create packages/ui** — extract design tokens (CSS vars, Tailwind preset) from web/src/styles.css into packages/ui. Update web/ to import from @agh/ui. Update turbo.json. No dependencies.
2. **Scaffold packages/site** — `npx create-fumadocs-app`, configure source.config.ts with two doc sources (runtime + protocol), apply DESIGN.md theme from packages/ui. Depends on step 1.
3. **Implement CLI codegen** — add `doc` subcommand to cmd/agh, post-processing script, `make cli-docs` target. Depends on step 2 (output goes to site content dir).
4. **Build landing page** — implement an outcome-led hero, balanced Runtime + AGH Network split, runtime proof, network proof, named comparison, architecture proof, and runtime-first final CTA. Depends on step 2.
5. **Write Wave 1 content** — ~20 MDX pages: Overview (3), Getting Started (3), Core Concepts (3), CLI Reference (4 groups, from step 3 output), Config Reference (2), Protocol Overview + Spec v0 (adapted from RFCs). Depends on steps 2 and 3.
6. **Deploy to Vercel** — configure agh.network, Vercel project, preview deployments, static export via `next build`. Depends on steps 4 and 5.
7. **(Wave 2) API reference codegen** — add `api-spec` command or hand-maintain openapi.json, configure fumadocs-openapi. Deferred — not alpha-critical. Depends on step 2.
8. **(Future) Monorepo standardization** — rename web/ → packages/app/ when the whole repo benefits. Not gated by docs launch.

### Technical Dependencies

- **Go binary must compile** for CLI codegen (step 3)
- **packages/ui tokens must be extractable** before site work begins (step 1 gates step 2)
- **Fumadocs OpenAPI integration** deferred to Wave 2 — needs OpenAPI spec and CORS/proxy strategy for interactive docs

## Operational Considerations

### Deployment Mode

- **Static export** (`output: 'export'` in next.config.mjs) for alpha — produces `out/` directory, deployed as static files to Vercel
- Turbo cache must include `out/**` (not `.next/**`) in outputs for the site workspace
- If interactive API docs require a proxy for non-CORS AGH endpoints, switch to `output: 'standalone'` with Vercel Functions — evaluate in Wave 2

### Search and Indexing

- Fumadocs includes built-in search via Orama (client-side, no external service)
- Search indexes both runtime and protocol collections
- Search results should display which collection a result belongs to (badge: "Runtime" or "Protocol")
- No additional search infrastructure for alpha

### Versioning and URL Strategy

- **Alpha**: no versioning. Runtime docs at `/runtime/*`, protocol at `/protocol/*`
- **Protocol versioning** (future): URL prefix strategy — `/protocol/v0/*`, `/protocol/v1/*` — when protocol versions diverge enough to need separate pages. For now v0 and v1 are additive (same pages, v1 section within)
- **Redirects**: plan for URL evolution from the start. Maintain a `redirects` array in next.config.mjs for any URL changes. Even alpha URLs may be linked externally.
- **i18n**: English-only. Fumadocs i18n structure is available but not configured for alpha.

### Content Ownership and Source of Truth

- **Site content is the canonical user-facing documentation.** RFCs in `docs/rfcs/` are internal specs; site content is adapted from them, not synced.
- **Generated content** (CLI reference) is overwritten on every `make cli-docs` run. Manual edits to generated files will be lost. Add custom prose via separate MDX files that import generated content, not by editing generated files.
- **Protocol spec pages** are adapted from RFCs 003/004. When the RFCs update, the site pages must be manually updated to match. No auto-sync.
- **Empty QMD collections** (`agh-compozy`, `agh-docs`) should be populated after Wave 1 content is written, making docs searchable via QMD.

## Monitoring and Observability

- **Build status**: Vercel deployment status visible in GitHub PR checks
- **Link health**: Periodic link checker in CI to catch broken internal/external links
- **Lighthouse scores**: Target 90+ performance, 100 accessibility, 100 best practices on landing page
- **Analytics**: Vercel Web Analytics for page views and traffic sources (post-launch)

## Technical Considerations

### Key Decisions

1. **Fumadocs over Astro Starlight**: Chosen for React/Next.js alignment, enabling native Tailwind preset sharing and component reuse. Starlight can render React components via its React integration, but theming requires Astro-specific adaptation. Trade-off: smaller Fumadocs community. (ADR-001)
2. **Additive packages, no web/ rename for alpha**: Add packages/ui + packages/site alongside existing web/. Defer full monorepo standardization to avoid unnecessary churn. Trade-off: hybrid layout. (ADR-002)
3. **Two Fumadocs doc sources**: Chosen to give protocol docs distinct sidebar, URL prefix, and page templates. Trade-off: cross-linking requires full paths. (ADR-003)
4. **Auto-generated CLI reference, API deferred**: CLI reference via Cobra GenMarkdownTree for alpha. API reference via fumadocs-openapi deferred to Wave 2 pending OpenAPI spec availability. Trade-off: no interactive API docs at launch. (ADR-004)

### Known Risks

| Risk                                                                  | Likelihood | Mitigation                                                                                    |
| --------------------------------------------------------------------- | ---------- | --------------------------------------------------------------------------------------------- |
| Fumadocs multi-source (two `defineDocs`) is a less-documented pattern | Medium     | Follow Fumadocs examples; pin version; test early in scaffolding step                         |
| packages/ui server/client boundary confusion                          | Medium     | Explicit export map: tokens and preset are server-safe; components marked with `"use client"` |
| Cobra GenMarkdownTree output needs Fumadocs frontmatter               | Low        | Post-processing script adds frontmatter and converts .md → .mdx                               |
| Protocol spec pages drift from source RFCs                            | Medium     | Manual review checklist when RFCs update; no auto-sync by design                              |
| Static export blocks interactive API docs                             | Low        | Evaluate `output: 'standalone'` in Wave 2 if proxy is needed                                  |
| URL changes after external linking                                    | Low        | Redirects array in next.config.mjs from day one                                               |
