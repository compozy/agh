# ADR-002: Monorepo Package Layout

## Status

Accepted (revised — alpha-scoped)

## Date

2026-04-15

## Context

The AGH monorepo currently has `web/` at the root level for the React SPA. Adding a documentation site requires sharing design tokens and UI components. The project uses Turborepo and Bun. A Codex review identified that renaming `web/` to `packages/app/` is unnecessary churn for an alpha docs launch and should be deferred.

## Decision

For alpha, add new packages alongside the existing layout — do not rename `web/`:

- `web/` — the React SPA (unchanged)
- `packages/site` — the Fumadocs documentation site
- `packages/ui` — shared design tokens, Tailwind preset, and server-safe base components extracted from web/

Full monorepo standardization (`web/` → `packages/app/`) is deferred until there is a repo-wide reason to restructure (e.g., adding more apps).

## Alternatives Considered

### Alternative 1: Full restructure now (web/ → packages/app/)

- **Description**: Rename web/ to packages/app/, move all TS packages under packages/\*.
- **Pros**: Clean naming symmetry. All workspaces in one directory.
- **Cons**: Git history churn, CI/Makefile updates, import path changes across all web/ files. Risk of regressions for zero user-facing benefit.
- **Why rejected**: The rename is cosmetic for alpha. The repo already has a hybrid layout (web/, sdk/). Moving only web/ doesn't fully standardize. Defer until the whole repo benefits.

### Alternative 2: Duplicate tokens, no shared package

- **Description**: Copy CSS variables into packages/site. No packages/ui extraction.
- **Pros**: Fastest to set up. Zero impact on web/.
- **Cons**: Token duplication leads to drift. Changes must be made in two places.
- **Why rejected**: DESIGN.md consistency is a hard requirement. Token drift is unacceptable even for alpha.

## Consequences

### Positive

- Zero impact on existing web/ codebase for alpha
- Single source of truth for design tokens via packages/ui
- Turborepo handles build ordering and caching
- Future rename to packages/app/ remains a clean option

### Negative

- Hybrid layout (web/ at root, packages/\* for new code) until full standardization
- packages/ui must have explicit server/client export boundaries (web/ is Vite client-only, site/ is Next.js with RSC)

### Risks

- packages/ui exports used in Next.js Server Components must be server-safe (no `useState`, no browser APIs). Mitigate by marking client components with `"use client"` directive and documenting export boundaries.

## Implementation Notes

- Create `packages/ui/` with: `src/tokens.css` (extracted from web/src/styles.css), `src/tailwind-preset.ts`, server-safe base components
- Update `web/src/styles.css` to import tokens from `@agh/ui`
- Add `packages/site` and `packages/ui` to turbo.json workspace patterns
- Both web/ and packages/site declare `@agh/ui` as a workspace dependency
- Turborepo pipeline: `turbo run build --filter=packages/site` (site outputs `out/**` for static export)
