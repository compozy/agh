# ADR-001: Fumadocs as Documentation Framework

## Status

Accepted

## Date

2026-04-15

## Context

AGH needs a documentation site that hosts both runtime documentation and the AGH Network Protocol specification. The site must match the project's DESIGN.md dark operator aesthetic, support MDX for custom components (protocol envelope viewers, message kind cards), auto-generated reference docs, and deploy to Vercel. The existing web/ app uses React 19, Tailwind CSS v4, and shadcn/ui.

## Decision

Use Fumadocs (https://fumadocs.dev/) as the documentation framework, running as a Next.js app in `packages/site/`.

## Alternatives Considered

### Alternative 1: Astro Starlight

- **Description**: Static-site documentation framework built on Astro. Used by OpenCode.
- **Pros**: Fast builds, built-in Pagefind search, i18n support, popular in OSS docs.
- **Cons**: Different framework (Astro vs React/Next.js) from existing web/ stack. Cannot share React components from packages/ui directly. Theming requires Astro-specific CSS overrides, not Tailwind preset sharing.
- **Why rejected**: Would introduce a second framework into the monorepo. Cannot reuse packages/ui components. Theming divergence from DESIGN.md would be harder to maintain.

### Alternative 2: Docusaurus

- **Description**: React-based documentation framework by Meta.
- **Pros**: Mature, versioning support, large plugin ecosystem, React-based.
- **Cons**: Heavy React dependency (older patterns), slow builds, opinionated theme that fights custom dark themes, no native Tailwind CSS v4 support.
- **Why rejected**: Theming conflicts with DESIGN.md flat depth model. Build performance concerns. Outdated React patterns compared to Next.js App Router.

### Alternative 3: Custom Next.js MDX setup

- **Description**: Build documentation rendering from scratch using next-mdx-remote or contentlayer.
- **Pros**: Maximum control, no framework constraints.
- **Cons**: Significant engineering effort for sidebar, search, TOC, breadcrumbs, navigation. Would reinvent what Fumadocs provides out of the box.
- **Why rejected**: Unnecessary engineering effort when Fumadocs provides the needed primitives with full customization.

## Consequences

### Positive

- Same React/Next.js stack as the existing app — shared component library works natively
- Fumadocs' source configuration supports multiple content collections (runtime + protocol)
- Built-in OpenAPI integration for auto-generated API reference
- Full Tailwind CSS support for DESIGN.md theming
- Vercel deployment with zero configuration

### Negative

- Fumadocs is younger than Starlight/Docusaurus — smaller community, fewer examples
- Next.js SSG/SSR adds complexity compared to pure static site generators

### Risks

- Fumadocs API surface may change between versions. Mitigate by pinning versions.

## Implementation Notes

- Install via `npx create-fumadocs-app` then customize theme to match DESIGN.md tokens.
- Use two Fumadocs source collections: `runtime` and `protocol` with separate sidebars.
- Landing page is a custom Next.js page (`app/page.tsx`), not a Fumadocs docs page.
