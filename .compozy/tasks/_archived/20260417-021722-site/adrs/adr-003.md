# ADR-003: Two-Collection Content Architecture for Runtime and Protocol Docs

## Status

Accepted

## Date

2026-04-15

## Context

AGH's documentation serves two distinct audiences with different needs: runtime users (install, configure, use AGH) and protocol implementers (build AGH Network compatible software). The protocol specification uses normative language (MUST/SHOULD/MAY) and needs distinct visual treatment. The site uses Fumadocs which supports multiple content collections.

## Decision

Use two separate Fumadocs doc sources via `defineDocs()` — `runtime` and `protocol` — each with its own sidebar, URL prefix, and page templates. Both sources live in the same Next.js app. `source.config.ts` defines the sources; `lib/source.ts` exposes loaders with `baseUrl`. Each source gets its own route group (`app/runtime/`, `app/protocol/`).

## Alternatives Considered

### Alternative 1: Single collection with MDX component differentiation

- **Description**: All docs in one collection. Protocol pages use custom MDX components (NormativeKeyword, EnvelopeViewer) to differentiate visually.
- **Pros**: Simpler config. One sidebar. Easier cross-linking.
- **Cons**: Protocol pages mixed into runtime nav. No distinct URL structure. Cannot apply different page templates per section.
- **Why rejected**: The protocol spec is a distinct product aimed at external implementers. Mixing it with runtime docs dilutes its identity as an open standard.

### Alternative 2: Completely separate Fumadocs instances

- **Description**: Two independent Fumadocs setups in separate Next.js route groups.
- **Pros**: Maximum isolation. Each can evolve independently.
- **Cons**: Duplicated Fumadocs config, shared layout code, duplicated search setup. Over-engineering for the current scale.
- **Why rejected**: Two collections in one instance give sufficient isolation without the maintenance cost.

## Consequences

### Positive

- Clean URL separation: `/runtime/*` and `/protocol/*`
- Protocol gets its own sidebar with spec-specific navigation
- Page templates can differ (e.g., protocol pages show conformance badges, normative keyword highlighting)
- Search can scope by collection
- Matches the "two-pillar" landing page narrative (Runtime + Protocol)

### Negative

- Cross-linking between collections requires full paths, not relative links
- Two sidebar configurations to maintain

### Risks

- Fumadocs multi-source is a supported but less-documented pattern. Mitigate by following Fumadocs examples closely.

## Implementation Notes

```typescript
// source.config.ts — define doc sources (no baseUrl here)
import { defineDocs, defineConfig } from "fumadocs-mdx/config";

export const runtime = defineDocs({ dir: "content/runtime" });
export const protocol = defineDocs({ dir: "content/protocol" });
export default defineConfig();
```

```typescript
// lib/source.ts — loaders with baseUrl
import { loader } from "fumadocs-core/source";
import { runtime, protocol } from "@/.source";

export const runtimeDocs = loader({
  source: runtime.toRuntime(),
  baseUrl: "/runtime",
});
export const protocolDocs = loader({
  source: protocol.toRuntime(),
  baseUrl: "/protocol",
});
```

- Each source gets its own catch-all route: `app/runtime/[[...slug]]/page.tsx` and `app/protocol/[[...slug]]/page.tsx`
- Sidebar navigation is derived from each loader's page tree — no shared sidebar config
- Protocol pages can use a custom layout with normative keyword styling (MUST/SHOULD/MAY)
- Runtime pages use domain-first sidebar: Sessions, Agents, Memory, Skills, etc.
- `meta.json` files in content directories control sidebar ordering and grouping
- OpenAPI reference and RFC pages are regular MDX within the appropriate source
