# ADR-003: ACP Internally, HTTP/SSE Externally

## Status

Accepted

## Date

2026-04-02

## Context

The previous AGH used custom PTY-based drivers to communicate with agents (Claude Code, Codex, OpenCode, Pi). Each driver scraped terminal output, used regex for readiness detection, and required per-driver hook configurations. This resulted in 4 driver implementations with significant maintenance overhead.

ACP (Agent Client Protocol) is an open standard (JSON-RPC 2.0 over stdio) supported by Claude Code, Codex, Gemini CLI, OpenCode, Copilot, Cursor, Kiro, and others. It provides structured tool calls, session management, permission delegation, and streaming updates natively.

The question is whether to use ACP on both sides (daemon↔agent AND web↔daemon) or only internally.

## Decision

Use ACP **only internally** (daemon as ACP client talking to agent subprocesses via stdio). Expose a **custom HTTP/SSE API** externally for the web UI and other consumers.

```
Web UI  ──HTTP/SSE──▶  Go Daemon (ACP Client)  ──ACP stdio──▶  claude --acp
                                                 ──ACP stdio──▶  codex --acp
CLI  ──UDS──▶  Go Daemon
```

The daemon handles all ACP bidirectional complexity (fs/readTextFile, terminal/create, request_permission) locally on the machine where agents run.

## Alternatives Considered

### Alternative 1: ACP End-to-End

- **Description**: Daemon acts as both ACP Server (Streamable HTTP for web clients) and ACP Client (stdio for agents)
- **Pros**: Single protocol everywhere. Any ACP client can connect. Protocol interoperability.
- **Cons**: ACP Streamable HTTP transport is still in **draft** status. ACP is bidirectional — agents send `fs/readTextFile`, `terminal/create`, `request_permission` back to the client. In a proxy scenario, the daemon must either resolve these locally (breaking the ACP contract with the web client) or forward them to the browser (which has no filesystem access). Permission requests become complex round-trips through HTTP to the browser.
- **Why rejected**: Draft transport, bidirectional complexity in proxy mode, browser cannot handle fs/terminal requests.

### Alternative 2: Custom Protocol Both Sides

- **Description**: Custom JSON protocol for daemon↔agent and daemon↔web
- **Pros**: Full control over both protocols
- **Cons**: Loses all ACP ecosystem benefits. Every agent needs a custom adapter instead of using native `--acp` flag.
- **Why rejected**: ACP eliminates the need for custom drivers — too valuable to skip internally.

## Consequences

### Positive

- Single ACP driver replaces 4+ custom PTY drivers
- Structured data from agents (tool calls, thoughts, plans) without regex parsing
- Native session management from ACP (session/new, session/load, session/prompt)
- Daemon resolves fs/terminal/permission locally — simple and fast
- External API is stable (we control it), not dependent on draft ACP HTTP spec
- Can add ACP Server endpoint later when Streamable HTTP stabilizes

### Negative

- Two protocols to maintain (ACP internal + HTTP/SSE external)
- Web UI cannot use off-the-shelf ACP clients directly (needs custom integration)
- Must implement AI SDK UI Message Stream format in Go for frontend compatibility

### Risks

- ACP spec may evolve — mitigated by using `acp-go-sdk` which tracks the spec
- Agent ACP support varies — some agents use adapter packages (npx wrappers) rather than native support

## Implementation Notes

- Use `github.com/coder/acp-go-sdk` for the ACP client implementation
- Provider registry stores the exact ACP command per agent (e.g., `npx -y @agentclientprotocol/claude-agent-acp@^0.24.2`)
- HTTP/SSE API compatible with Vercel AI SDK `useChat` via `x-vercel-ai-ui-message-stream: v1` format
