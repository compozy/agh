# ADR-001: Rewrite From Scratch

## Status

Accepted

## Date

2026-04-02

## Context

The current AGH codebase has ~78,800 lines of Go across 210+ files. The `internal/kernel` package alone has 51 files (~25K LOC) centered on a monolithic `Kernel` struct with 30+ fields. Every concept (workgroups, supervisors, resilience, hooks, registries, drivers, dream, memory, observability, messaging, dashboard) is tightly coupled through this struct. The `Session` struct holds a back-pointer to `Kernel`. The `SessionManager` reaches into `Kernel` for drivers, NATS, observability, config, and home paths.

The project wants to move from custom PTY-based drivers to ACP (Agent Client Protocol), remove workgroups/supervisors/master/advisor hierarchy, and adopt a simpler session-centric model. These changes affect every layer of the current system.

The codebase was generated by LLMs in a few days — there is no institutional knowledge encoded in it that would be expensive to lose.

## Decision

Rewrite the project from scratch. Use the existing codebase as reference material for concepts that proved valuable (memory system design, config merge strategy, observability patterns, TOON renderer), but do not attempt to refactor or prune the existing code.

## Alternatives Considered

### Alternative 1: Incremental Refactoring

- **Description**: Gradually remove workgroups, supervisors, and PTY drivers while keeping the existing structure
- **Pros**: Preserves working code, lower perceived risk
- **Cons**: The kernel monolith makes this nearly impossible — removing workgroups cascades through 51 kernel files, all transport scoping, prompt templates, CLI commands, and the dashboard. The coupling is too deep.
- **Why rejected**: Estimated effort exceeds a rewrite, with higher risk of leaving vestigial complexity

### Alternative 2: Fork and Strip

- **Description**: Copy the repo and delete unwanted code
- **Pros**: Starts with working infrastructure (config, SQLite, logging)
- **Cons**: Dead imports, broken tests, phantom coupling everywhere. The architecture itself (Kernel god object, NATS subject hierarchy, PTY drivers) is what needs replacing, not just features.
- **Why rejected**: Would inherit the architectural problems that caused the original failure

## Consequences

### Positive

- Clean architecture from day one, designed for ACP
- No vestigial complexity from removed features
- Opportunity to apply lessons learned (smaller packages, no god objects, CI-enforceable boundaries)
- LLMs can work with the new codebase more effectively (smaller, focused packages)

### Negative

- Temporary loss of all implemented features (memory, dream, dashboard, skills, observability)
- Features must be re-implemented incrementally
- Risk of losing subtle design insights embedded in the old code

### Risks

- Mitigated by keeping the old code as reference in `.resources/` or git history
- Phased approach ensures each feature is solid before adding the next
