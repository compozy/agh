# ADR-005: Built-In Provider Registry With ACP Commands

## Status

Accepted

## Date

2026-04-02

## Context

Different AI agents require different commands to launch in ACP mode. Research of the acpx project revealed significant variation:

| Agent | ACP Command |
|-------|-------------|
| Claude | `npx -y @agentclientprotocol/claude-agent-acp@^0.24.2` |
| Codex | `npx @zed-industries/codex-acp@^0.10.0` |
| Gemini | `gemini --acp` |
| OpenCode | `npx -y opencode-ai acp` |
| Copilot | `copilot --acp --stdio` |

Users should not need to know these details. The daemon should ship with knowledge of how to launch common agents.

## Decision

Ship a **built-in provider registry** hardcoded in Go (like acpx's `AGENT_REGISTRY`). Users can override any provider or add custom ones via `[providers.*]` in config.toml.

Built-in registry in Go:
```go
var BuiltinProviders = map[string]ProviderConfig{
    "claude":   {Command: "npx -y @agentclientprotocol/claude-agent-acp@^0.24.2"},
    "codex":    {Command: "npx @zed-industries/codex-acp@^0.10.0"},
    "gemini":   {Command: "gemini --acp"},
    "opencode": {Command: "npx -y opencode-ai acp"},
    "copilot":  {Command: "copilot --acp --stdio"},
    "cursor":   {Command: "cursor-agent acp"},
    "kiro":     {Command: "kiro-cli-chat acp"},
    "pi":       {Command: "npx pi-acp@^0.0.22"},
}
```

Config.toml override:
```toml
[providers.claude]
default_model = "claude-opus-4-20250514"
api_key_env = "ANTHROPIC_API_KEY"

[providers.my-custom]
command = "./bin/my-acp-server"
```

Resolution chain: AGENT.md override → Provider config.toml → Built-in defaults.

Provider fields: `command` (ACP launch command), `default_model`, `api_key_env`.

## Alternatives Considered

### Alternative 1: User Configures Everything

- **Description**: No built-in registry. Users must specify the full command for every provider in config.toml.
- **Pros**: No maintenance burden for built-in commands
- **Cons**: Poor onboarding experience. Users need to research ACP commands for each agent. Error-prone.
- **Why rejected**: Too much friction for common agents

### Alternative 2: Plugin-Based Discovery

- **Description**: Providers discovered at runtime via plugin mechanism
- **Pros**: Infinitely extensible
- **Cons**: Over-engineering for v1. Plugin systems add complexity (loading, versioning, trust).
- **Why rejected**: YAGNI. Config override covers custom agents sufficiently.

## Consequences

### Positive

- Zero-config for common agents: `provider: claude` just works
- Users can still override commands or add custom agents
- Easy to update built-in commands when ACP adapters release new versions

### Negative

- Built-in commands may become stale as adapter packages are updated
- Must track multiple ACP adapter versioning

### Risks

- Stale versions mitigated by config.toml override and periodic updates to built-in registry
