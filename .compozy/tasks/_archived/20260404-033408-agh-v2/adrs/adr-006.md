# ADR-006: Dual SQLite Storage (Per-Session + Global)

## Status

Accepted

## Date

2026-04-02

## Context

The system needs to persist session events (ACP messages, tool calls, agent output) for observability and replay, plus global data (session index, cross-session queries, aggregate stats). The previous AGH used per-session SQLite databases successfully.

Key query patterns:
- "What happened in session X?" → per-session query
- "What errors happened across all sessions today?" → cross-session query
- "Show the full conversation in session X" → transcript replay
- "How many tokens did I use this week?" → cross-session aggregate

## Decision

Use **two SQLite tiers**:

1. **Per-session DB** (`~/.agh/sessions/<id>/events.db`) — detailed events, ACP messages, tool calls, transcript
2. **Global DB** (`~/.agh/agh.db`) — session index, agent registry, cross-session stats, lightweight event summaries

Both use WAL mode for concurrent reads. Per-session DBs use single-writer pattern.

Additionally, a `meta.json` per session for quick reads without opening SQLite:
```
~/.agh/sessions/<id>/
├── events.db     # Detailed per-session data
└── meta.json     # Quick metadata (agent, status, created_at, updated_at)
```

## Alternatives Considered

### Alternative 1: Single Centralized DB

- **Description**: One `~/.agh/agh.db` containing everything
- **Pros**: Cross-session queries trivial, single file
- **Cons**: Grows indefinitely. Harder to clean up individual sessions (DELETE vs rm -rf). Potential write contention from multiple active sessions.
- **Why rejected**: Session cleanup should be as simple as deleting a directory

### Alternative 2: NDJSON Files (like acpx)

- **Description**: Append-only NDJSON files per session
- **Pros**: Ultra-simple, easy to stream, no SQLite dependency
- **Cons**: No indexes. Queries with filters (--type, --since, --last) require full file scan. Poor for agent-queryable observability where filtered queries are the primary use case.
- **Why rejected**: Agent-friendly observability requires indexed, filterable storage

### Alternative 3: Per-Session DB Only (no global)

- **Description**: Only per-session SQLite, no global DB
- **Pros**: Simpler — one storage tier
- **Cons**: Cross-session queries require opening every session DB and merging results. Listing sessions requires scanning filesystem.
- **Why rejected**: Cross-session queries are a core requirement for observability

## Consequences

### Positive

- Session isolation: delete session = delete directory
- Cross-session queries via global DB without opening individual session DBs
- WAL mode enables concurrent reads from CLI/API while session writes
- `meta.json` enables fast session listing without SQLite opens

### Negative

- Two storage tiers to keep synchronized
- Global DB must be updated when session events occur (dual writes)

### Risks

- Crash consistency between per-session and global DB mitigated by clear ownership split:
  - `session/` package owns per-session DB writes (detailed events)
  - `observe/` package owns global DB writes (summaries, stats) via Notifier callbacks
  - `meta.json` written atomically (temp file + rename)
  - Boot-time reconciliation: daemon scans sessions directory and reconciles with global DB on startup
