# ADR-004: Self-Contained Agent Directory With AGENT.md

## Status

Accepted

## Date

2026-04-02

## Context

Agents need a definition format that specifies their provider, model, tools, permissions, and system prompt. The previous AGH used TOML role files in `.agh/roles/` with a separate role catalog and discovery system. Reference projects use different approaches: OpenFang uses `agent.toml` manifests, OpenClaw uses JSON5 config arrays.

The system prompt is a critical part of agent identity — it defines personality, capabilities, and behavior. It should be a first-class citizen in the definition format, not a field in a config file.

## Decision

Agents are defined as self-contained directories at `~/.agh/agents/<name>/` with an `AGENT.md` file that combines YAML frontmatter (configuration) and Markdown body (system prompt):

```
~/.agh/agents/coder/
└── AGENT.md
```

```markdown
---
name: coder
provider: claude
model: claude-opus-4-20250514
tools: ["*"]
permissions: approve-reads
mcp_servers:
  - name: github
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: ""
---

You are a senior software engineer specialized in Go...
```

Frontmatter fields for v1: `name` (required), `provider` (required), `model` (optional), `command` (optional — override provider ACP command), `tools` (optional, default: `["*"]`), `permissions` (optional), `mcp_servers` (optional — merged with provider-level MCP servers).

The directory format allows future extension without changing the AGENT.md schema:
```
~/.agh/agents/coder/
├── AGENT.md
├── skills/       # future: agent-specific skills
├── memory/       # future: agent-specific memory
└── sessions/     # future: agent session history
```

## Alternatives Considered

### Alternative 1: TOML Config Files

- **Description**: `~/.agh/agents/coder.toml` with system prompt as a string field
- **Pros**: Consistent with the config.toml format
- **Cons**: System prompts are multi-line prose — awkward in TOML. No natural place for the prompt to be the "main content" of the file. Not self-contained (single file, not directory).
- **Why rejected**: Poor ergonomics for the most important part of agent definition (the system prompt)

### Alternative 2: JSON5 Config (like OpenClaw)

- **Description**: Agent configs as entries in a JSON5 file
- **Pros**: Rich typing, used by OpenClaw
- **Cons**: System prompt buried as a field. All agents in one file. Not extensible per-agent.
- **Why rejected**: Not self-contained, poor system prompt ergonomics

## Consequences

### Positive

- System prompt is first-class content (Markdown body), not a config field
- Self-contained directory enables future per-agent extensions (skills, memory)
- Frontmatter is familiar to developers (Jekyll, Hugo, Astro, etc.)
- Easy to create: just write a Markdown file with YAML header
- Can be version-controlled per-agent

### Negative

- Requires frontmatter parser (already exists in codebase: `internal/frontmatter/`)
- Directory per agent is more filesystem overhead than single config file

### Risks

- None significant — the format is proven across many tools and frameworks
