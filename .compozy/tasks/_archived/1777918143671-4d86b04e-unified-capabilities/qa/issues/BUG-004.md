# BUG-004: CLI network integration daemon omitted the network store

**Severity:** High
**Priority:** P0
**Type:** Integration
**Status:** Fixed

## Environment

- **Build:** `feat/network-slack-redesign` after Round 1 commit `971c51cf`
- **OS:** Linux
- **Browser:** N/A
- **Surface:** CLI over in-process UDS integration daemon

## Summary

`TestCLINetworkRoundTripIntegration` failed because the CLI integration harness started the UDS API server with a `NetworkService` but without the matching `NetworkStore`. Production daemon wiring already included the store, so the failure was isolated to the test daemon composition path.

## Reproduction

```bash
go test -race -tags integration ./internal/network ./internal/cli ./internal/api/udsapi
```

Observed before the fix:

- `internal/network` passed.
- `internal/api/udsapi` passed.
- `internal/cli` failed in `TestCLINetworkRoundTripIntegration`.
- The failing API response was `api: network store is required` when the CLI requested network channel data through UDS.

## Expected

The CLI integration harness should wire the UDS server the same way the production daemon does for network capability paths, including both the network service and network store.

## Root Cause

`internal/daemon/daemon.go` passed `udsapi.WithNetworkStore(deps.Registry)`, but `internal/cli/cli_integration_test.go` only passed `udsapi.WithNetworkService(networkManager)`. The integration test daemon therefore exercised a partial network API composition that production did not use.

## Fix

Added `udsapi.WithNetworkStore(registry)` to the CLI integration test daemon setup so CLI network commands run against the complete UDS API dependency set.

## Verification

```bash
go test -race -tags integration ./internal/cli -run TestCLINetworkRoundTripIntegration -count=1
go test -race -tags integration ./internal/network ./internal/cli ./internal/api/udsapi
```

Both commands passed after the fix.

## Impact

- **Users Affected:** CLI integration coverage and any future in-process daemon harnesses copied from it.
- **Frequency:** Always in `TestCLINetworkRoundTripIntegration` before the fix.
- **Workaround:** None in the test harness; production daemon wiring was already correct.
