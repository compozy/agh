# ADR-005: go:embed for Bundled Skills via fs.FS Interface

## Status

Accepted

## Date

2026-04-06

## Context

AGH ships 3-5 starter skills to provide immediate value and demonstrate the SKILL.md format. These bundled skills need to be compiled into the single binary. Two patterns observed in reference harnesses: go:embed with real SKILL.md files (GoClaw, OpenFang) and programmatic registration as Go structs (Claude Code's `registerBundledSkill()`).

The registry's `RegistryConfig.BundledFS` field must support both production use (real `embed.FS`) and testing (synthetic filesystem). This requires using the `fs.FS` interface rather than the concrete `embed.FS` type.

## Decision

Use **go:embed with real SKILL.md files** in `internal/skills/bundled/skills/`. Each bundled skill is a standard SKILL.md file in a directory, embedded into the binary via `//go:embed skills/**/SKILL.md`. At boot, the loader parses them using the same `ParseSkillFile()` path as filesystem skills.

The `RegistryConfig.BundledFS` field uses the **`fs.FS` interface** (not concrete `embed.FS`) so that:
- Production code passes the `embed.FS` (which satisfies `fs.FS`)
- Tests pass `testing/fstest.MapFS` (which also satisfies `fs.FS`)

Directory structure:
```
internal/skills/bundled/
├── embed.go              // //go:embed directive + FS accessor
└── skills/
    ├── agh-session-guide/
    │   └── SKILL.md
    ├── agh-memory-guide/
    │   └── SKILL.md
    └── agh-agent-setup/
        └── SKILL.md
```

## Alternatives Considered

### Alternative 1: Programmatic Registration

- **Description**: Define bundled skills as Go struct literals with inline Markdown content.
- **Pros**: Full Go type safety.
- **Cons**: Harder to author. Not portable. Two code paths.
- **Why rejected**: Fragments the authoring experience.

### Alternative 2: Concrete embed.FS in Config

- **Description**: Use `embed.FS` directly in `RegistryConfig`.
- **Pros**: Simpler type.
- **Cons**: `testing/fstest.MapFS` does not satisfy `embed.FS`. Tests would need real embedded files or a wrapper.
- **Why rejected**: Breaks testability. `fs.FS` is the correct abstraction.

## Consequences

### Positive

- Single code path: embedded skills parsed identically to filesystem skills
- Skills are real SKILL.md files — portable, testable, easy to contribute
- Go compiler ensures embedded files exist at build time
- Users can override bundled skills by placing a same-name skill in higher-precedence directories
- `fs.FS` interface enables clean test isolation

### Negative

- Embedded files add to binary size (trivial — typically < 50KB total)

### Risks

- Embedded SKILL.md with malformed YAML would fail at boot. Mitigation: unit tests validate all bundled skills parse correctly.
