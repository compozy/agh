# ADR-001: Three-Increment Delivery Strategy

## Status

Accepted

## Date

2026-04-06

## Context

The skills system proposal encompasses multiple subsystems: SKILL.md loading/parsing, prompt injection, memory integration, MCP lazy-load, lifecycle hooks, loading hierarchy, progressive disclosure, ClawHub marketplace, full CLI, and security scanning. A council of 5 advisors (pragmatic engineer, architect, security advocate, product mind, devil's advocate) debated whether this should ship as one deliverable or be phased.

Key constraints:
- Greenfield alpha with zero production users
- Memory system just completed as first Phase 2 feature
- Phase 3 (Agent Network Protocol) is the long-term architectural moat
- 6 competing harnesses already ship skills systems
- MCP lazy-load is novel but carries security risk (subprocess execution from skill frontmatter)
- No backward compatibility pressure

## Decision

Ship the skills system in 3 increments, each independently valuable and passing `make verify`:

**Increment 1 — Core**: SKILL.md loader + dual-scope registry (global + workspace) + prompt injection via ComposedAssembler + CLI (list, view, info, create) + security scanning (VerifyContent) + loading hierarchy (bundled → user → .agents → workspace) + hot-reload (stat-based polling) + bundled skills (go:embed).

**Increment 2 — MCP + Hooks**: MCP lazy-load with user consent gate + env scrubbing + command allowlist for non-local skills. Lifecycle hooks (on_session_created, on_session_stopped, on_prompt_assembly) with explicit execution model (ordering, timeouts).

**Increment 3 — Marketplace**: ClawHub integration + install/remove/search CLI + cryptographic provenance verification + override audit trail.

## Alternatives Considered

### Alternative 1: Ship Everything at Once

- **Description**: Full proposal as one deliverable
- **Pros**: Maximum feature surface, immediate differentiation
- **Cons**: High risk, 5+ distinct subsystems with interleaved test matrices, delays Phase 3
- **Why rejected**: Council unanimously agreed scope is too large. No validated learning until everything ships.

### Alternative 2: Minimal Core Only (Skills + Memory)

- **Description**: Strip to SKILL.md loader + prompt injection + memory-aware assembly. No hooks, no MCP, no marketplace.
- **Pros**: Fastest path to validated learning (1-2 weeks). Minimum investment.
- **Cons**: Without MCP lazy-load, AGH's skills are functionally identical to every competitor. Misses differentiation window.
- **Why rejected**: Product Mind argued memory-integrated skills is already a differentiator, but MCP lazy-load is what makes AGH worth switching to. Deferred to increment 2 rather than cut entirely.

### Alternative 3: Skill Bundles (Adjacent Opportunity)

- **Description**: Package SKILL.md + MCP + memory templates as distributable bundles (like Codex plugins)
- **Pros**: Strong product differentiation, validated by Codex's March 2026 plugin launch
- **Cons**: Higher complexity, conflates distribution with runtime, may duplicate Phase 3 concerns
- **Why rejected**: Compelling future direction but couples too many concerns for V1. Can evolve toward bundles after increment 3 lands.

## Consequences

### Positive

- Each increment is independently testable and shippable
- Security controls for MCP lazy-load ship alongside the feature (increment 2), not after
- Marketplace infrastructure only built when there's evidence of community demand
- Earlier unblocking of Phase 3 work

### Negative

- MCP lazy-load delayed to increment 2 — early adopters may not see AGH's full differentiation immediately
- Three increments means three planning/review cycles

### Risks

- **Risk**: Users only care about MCP lazy-load and skip AGH until increment 2. **Mitigation**: Increment 1's memory-integrated skills is already unique. Monitor early adopter feedback and collapse increments if demand warrants.
- **Risk**: Increment 2's security controls delay MCP shipment. **Mitigation**: Minimal viable controls (consent gate + env scrub) are small engineering cost.
