# ADR-003: Team Memory via Blackboard Enhancement

## Status

Accepted

## Date

2026-04-01

## Context

Agents within a session need to share structured knowledge (decisions, findings, patterns) that persists for the session's lifetime. AGH already has the blackboard system — a SQLite-backed, append-only store scoped to workgroups with `type`, `content`, `author`, and `scope` fields.

The question is whether to build a separate team memory system or enhance the existing blackboard.

## Decision

Reuse the existing blackboard with a new entry type convention: `type="memory"`. Memory-typed blackboard entries contain YAML frontmatter in the `content` field with `name`, `description`, and memory `type` (user/feedback/project/reference). The prompt assembler filters blackboard entries where `type="memory"` and injects them into agent system prompts.

No new SQLite tables, no schema migrations. Agents write team memories via the existing `agh state append --type memory` command.

## Alternatives Considered

### Alternative 1: Separate SQLite Table

- **Description**: New `team_memory` table with dedicated columns: `name`, `description`, `memory_type`, `author`, `scope`, `content`, `created_at`
- **Pros**: Cleaner schema, typed columns, dedicated indexes
- **Cons**: Duplicates the storage pattern of the blackboard, requires schema migration, new query types, additional API endpoints
- **Why rejected**: Over-engineering for a convention that the existing schema already supports. The blackboard's `type` field is free-form by design

### Alternative 2: File-Based Within Session Directory

- **Description**: Store team memories as markdown files in `~/.agh/sessions/<id>/memory/`
- **Pros**: Consistent with persistent memdir pattern, version-controllable
- **Cons**: Adds filesystem management to session lifecycle, must handle concurrent writes from multiple agents, cleanup on session destroy
- **Why rejected**: Filesystem concurrency is harder than SQLite (which already handles writes via single-writer goroutine). Team memories are ephemeral by nature — they don't need to outlive the session

## Consequences

### Positive

- Zero new infrastructure — reuses existing blackboard, queries, CLI commands, and API
- Agents already know how to use `agh state append`
- Team memories are automatically cleaned up with session teardown
- Existing blackboard dashboard views show team memories

### Negative

- No dedicated schema validation for memory entries (frontmatter parsing happens at read time)
- Blackboard entries are append-only — cannot update or delete a team memory, only append a new version
- Mixing memory entries with other blackboard types requires filtering in queries

### Risks

- Frontmatter in the `content` field is a convention, not enforced by the schema. Malformed entries will be silently included. Mitigated by validating frontmatter at prompt injection time and skipping malformed entries with a warning log
