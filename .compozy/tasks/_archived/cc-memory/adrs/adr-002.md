# ADR-002: Dream Consolidation via Ephemeral Agent Session

## Status

Accepted

## Date

2026-04-01

## Context

Dream consolidation requires LLM reasoning to synthesize session transcripts, blackboard entries, and existing memories into updated, well-organized memory files. AGH's kernel does not make LLM API calls directly — it orchestrates external agent processes via PTY-backed drivers.

The consolidation process needs to:
1. Read existing memory files and indexes
2. Scan recent session transcripts and blackboard snapshots
3. Synthesize patterns, merge signal, resolve contradictions
4. Write updated memory files with proper frontmatter
5. Prune the MEMORY.md index

This requires the reasoning capability of an LLM, which means we must use one of AGH's agent drivers.

## Decision

Dream consolidation spawns a lightweight, ephemeral AGH session with a single worker agent. The worker receives a restricted system prompt focused on the 4-phase consolidation cycle (orient/gather/consolidate/prune). The session auto-stops after the agent completes.

The `DreamService` uses a `SessionSpawner` callback provided by the kernel, keeping the dream package decoupled from session internals.

Triggering uses a 3-gate system (cheapest first):
1. Time gate: hours since last consolidation > 24h (one `stat` call on lock file mtime)
2. Session gate: sessions completed since last consolidation >= 3 (directory scan)
3. Lock gate: no other consolidation in progress (file lock with PID)

## Alternatives Considered

### Alternative 1: Direct Driver Invocation

- **Description**: Bypass session machinery, call `driver.Start()` directly with a consolidation prompt
- **Pros**: Lighter weight, fewer moving parts
- **Cons**: Skips session lifecycle, state store, resilience, observability, and health checking. No hook events, no dashboard visibility
- **Why rejected**: Loses all the infrastructure guarantees that sessions provide. Debugging a failed consolidation would be opaque

### Alternative 2: External Script/Binary

- **Description**: Shell out to a separate consolidation tool
- **Pros**: Maximum isolation, can be written in any language
- **Cons**: Maintenance burden, must understand memory file format, requires separate installation, no integration with AGH's driver/model selection
- **Why rejected**: Adds an external dependency to a single-binary system, duplicates knowledge of the memory format

### Alternative 3: Kernel-Internal LLM Call

- **Description**: Add an HTTP client in the kernel to call an LLM API directly for consolidation
- **Pros**: Fastest execution, no session overhead
- **Cons**: Violates AGH's core architecture — the kernel orchestrates agents, it does not make LLM calls. Would require API key management, rate limiting, model selection in the kernel
- **Why rejected**: Architectural violation. The kernel's job is orchestration, not direct LLM interaction

## Consequences

### Positive

- Reuses all existing session infrastructure (lifecycle, state, resilience, observability)
- Consolidation sessions are visible in the dashboard
- Uses the same driver/model configuration as regular sessions
- `DreamService` is decoupled from session internals via callback

### Negative

- Higher overhead than a direct API call (full session bootstrap for a single-agent task)
- Consolidation occupies a session slot (counts against `max_sessions` limit)
- Agent startup latency (driver readiness detection) adds to consolidation time

### Risks

- Consolidation session could fail silently if the agent produces no output. Mitigated by lock rollback (mtime rewound so gates pass again on next check)
- Session slot exhaustion if consolidation runs while user is at max sessions. Mitigated by checking available session slots before spawning
