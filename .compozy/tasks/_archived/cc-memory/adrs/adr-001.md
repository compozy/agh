# ADR-001: Dual Storage — Global + Workspace Memory Directories

## Status

Accepted

## Date

2026-04-01

## Context

AGH needs persistent memory that survives across sessions. The memory system must support both user-level knowledge (preferences, feedback patterns) that applies across all projects and project-level knowledge (ongoing work, external system references) that is specific to a workspace.

The existing config system already uses a two-layer resolution pattern: global `~/.agh/config.toml` overlaid with workspace `.agh/config.toml`. Memory storage should follow a consistent pattern.

Claude Code uses a single project-scoped directory (`~/.claude/projects/<slug>/memory/`), but AGH's multi-workspace architecture benefits from separating global vs project concerns.

## Decision

Use two memory directories with independent MEMORY.md indexes:

- **Global**: `~/.agh/memory/` — stores `user` and `feedback` type memories that apply across all projects.
- **Workspace**: `<workspace>/.agh/memory/` — stores `project` and `reference` type memories specific to a codebase.

Both indexes are injected into agent system prompts. Agents read full memory content on demand via `agh memory read`.

## Alternatives Considered

### Alternative 1: Workspace Only

- **Description**: Single memory directory at `<workspace>/.agh/memory/`
- **Pros**: Simpler, no global state management
- **Cons**: User preferences and feedback patterns must be re-learned per project. No cross-project knowledge sharing
- **Why rejected**: Loses institutional knowledge when switching between projects

### Alternative 2: Global Only

- **Description**: Single memory directory at `~/.agh/memory/`
- **Pros**: Simplest, all memories in one place
- **Cons**: Different projects pollute each other's context. No project isolation
- **Why rejected**: Project-specific memories (e.g., "auth rewrite uses JWT") are irrelevant noise in other projects

### Alternative 3: Claude Code Style (project-slug scoped)

- **Description**: `~/.agh/projects/<sanitized-workspace>/memory/`
- **Pros**: Matches Claude Code's pattern, keeps all state under `~/.agh/`
- **Cons**: Project memories are not colocated with the workspace, making them invisible in version control and harder to share with team
- **Why rejected**: Workspace-colocated storage enables version control and team sharing of project memories

## Consequences

### Positive

- Consistent with existing config layering pattern (global + workspace)
- User preferences follow across projects without repetition
- Project memories are isolated and colocated with their workspace
- Workspace memories can be version-controlled if desired

### Negative

- Two directories to manage, scan, and inject into prompts
- Scope resolution logic needed when agents write memories without explicit scope
- Memory consolidation (dream) must handle both directories

### Risks

- Scope ambiguity: an agent may write a project memory to global scope or vice versa. Mitigated by defaulting scope based on memory type (user/feedback → global, project/reference → workspace)
