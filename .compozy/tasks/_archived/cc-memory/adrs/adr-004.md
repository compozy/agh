# ADR-004: time.Ticker Over Cron Library for Dream Scheduling

## Status

Accepted

## Date

2026-04-01

## Context

The dream consolidation service needs periodic scheduling within the daemon process. The requirement is simple: check every ~30 minutes whether consolidation gates pass, and if so, spawn a consolidation session.

Several Go cron/scheduler libraries were evaluated: `go-co-op/gocron` v2 (~6.9k stars, actively maintained), `robfig/cron` v3 (~14.1k stars, unmaintained since 2020), and `reugn/go-quartz` (~1.9k stars, zero deps).

## Decision

Use a plain `time.Ticker` goroutine for dream scheduling. No external cron library.

The daemon lifecycle goroutine runs a ticker at a configurable interval (default 30 minutes). On each tick, it calls `DreamService.ShouldRun()` which evaluates the 3-gate system. If gates pass, it spawns a consolidation session via the `SessionSpawner` callback.

The ticker goroutine integrates with the existing `context.Context` cancellation for graceful shutdown.

## Alternatives Considered

### Alternative 1: go-co-op/gocron v2

- **Description**: Full-featured scheduler with cron expressions, singleton mode, context support
- **Pros**: Rich feature set, actively maintained, built-in overlap protection
- **Cons**: Adds external dependency with transitive deps, larger API surface than needed, features like distributed locking and electors are unused
- **Why rejected**: Over-engineered for a single fixed-interval task. The 3-gate system in `DreamService.ShouldRun()` already provides overlap protection via the lock file

### Alternative 2: robfig/cron v3

- **Description**: Popular cron library with standard expression support
- **Pros**: Mature, widely used, simple API
- **Cons**: Unmaintained since 2020, 55+ open PRs, no context propagation into jobs, known unpatched bugs
- **Why rejected**: Unmaintained library is a liability in a production daemon

### Alternative 3: reugn/go-quartz

- **Description**: Quartz-inspired scheduler with zero dependencies and excellent context support
- **Pros**: Zero deps, best-in-class context.Context design, cron expressions
- **Cons**: Pre-v1 API (v0.11.0), verbose JobDetail/JobKey/Trigger ceremony, smaller community
- **Why rejected**: API verbosity is excessive for a single periodic check. Pre-v1 stability risk

## Consequences

### Positive

- Zero new dependencies — aligns with single-binary, local-first philosophy
- Native `context.Context` integration via `select` with `ctx.Done()`
- Integrates cleanly with existing `suture` supervision and `oklog/run` lifecycle
- Simplest possible implementation — easy to understand and maintain

### Negative

- No cron expression support — only fixed intervals
- No built-in jitter or randomization (not needed for a local-only daemon)
- If future requirements need multiple scheduled tasks with cron expressions, would need to revisit

### Risks

- If dream scheduling requirements grow complex (multiple tasks, cron expressions, overlap protection beyond lock file), the ticker approach may become insufficient. Mitigated by the 3-gate system handling all complexity in `ShouldRun()`, keeping the ticker trivial
