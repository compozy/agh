# BUG-001: API server home override leaves default daemon socket on process home

**Severity:** High
**Priority:** P0
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** local `tools-registry` worktree, Task 16 QA pass
- **OS:** macOS
- **Browser:** N/A
- **URL:** UDS and HTTP API server constructors

## Summary

`udsapi.New(WithHomePaths(...))` accepted an overridden AGH home layout but kept the default config resolved from the process environment. In the isolated Task 16 QA lab, the provider `HOME` path was long enough that the stale default daemon socket exceeded the portable Unix socket limit and blocked `make verify`.

## Reproduction

```bash
env -u AGH_HOME \
  HOME="/Users/pedronauck/dev/qa-labs/agh-tools-registry-task16-20260429-073832-489478-lab/.provider-home" \
  CODEX_HOME="/Users/pedronauck/dev/qa-labs/agh-tools-registry-task16-20260429-073832-489478-lab/.provider-home/.codex" \
  go test -race ./internal/api/udsapi ./internal/config \
  -run "TestNewRequiresSessionManagerTaskServiceObserverAndWorkspaceResolver|TestLoadUsesDotEnvForAGHHome" -v
```

Observed before the fix:

- `internal/api/udsapi` failed with `udsapi: socket path too long` for the process-home-derived daemon socket.
- `internal/config` passed when `AGH_HOME` was not exported globally, confirming the UDS constructor mismatch was the remaining failure.

## Expected

When no explicit runtime config is supplied, `WithHomePaths(homePaths)` should realign the default server config to the same `homePaths` layout before transport-specific defaults are finalized.

## Root cause

Both API server constructors initialized `config` from `aghconfig.ResolveHomePaths()` before options were applied. `WithHomePaths` later replaced only `server.homePaths`; it did not rebuild the default config unless callers also supplied `WithConfig`. UDS socket validation therefore read `server.config.Daemon.Socket` from the process home, not the option-provided home.

## Fix

- Added constructor state to track whether `WithConfig` was explicitly supplied.
- Updated UDS and HTTP `WithHomePaths` to rebuild `aghconfig.DefaultWithHome(homePaths)` only when no explicit config has been provided.
- Kept explicit `WithConfig` authoritative regardless of option ordering.
- Isolated UDS dependency-constructor tests from unrelated Unix socket path length failures by using the existing `shortSocketPath` helper.

## Verification

- `go test -race -count=1 ./internal/api/udsapi ./internal/api/httpapi ./internal/config -run "TestNewWithHomePathsRealignsDefaultConfig|TestNewRequiresSessionManagerTaskServiceObserverAndWorkspaceResolver|TestLoadUsesDotEnvForAGHHome" -v`
- Evidence log: `.compozy/tasks/tools-registry/qa/logs/fix-api-homepaths-targeted-rerun.log`

## Impact

- **Users Affected:** API construction paths using `WithHomePaths` without `WithConfig`, including isolated test/runtime harnesses.
- **Frequency:** Always when process home differed from the option home and the process-home socket path was invalid or misleading.
- **Workaround:** None suitable; callers should not need to duplicate `WithHomePaths` through `WithConfig`.

## Automation Follow-up

- **Required:** Yes
- **Status:** Added
- **Spec / Command:** `TestNewWithHomePathsRealignsDefaultConfig` in `internal/api/udsapi` and `internal/api/httpapi`
- **Notes:** Regression asserts the default config daemon socket follows the overridden home layout.
