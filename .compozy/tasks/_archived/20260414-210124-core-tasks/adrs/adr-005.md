# ADR-005: Derive Actor Identity Server-Side and Allow Optional Mutable Ownership

## Status

Accepted

## Date

2026-04-13

## Context

The review identified a critical gap in the original draft: task identity fields such as creator, owner, and source context were under-specified and risked being trusted from payloads. With broad writer ingress in v1, payload-trusted identity would allow callers to forge audit trails, steal ownership, or create misleading global tasks.

The accepted redesign separates three concepts:

- immutable creator identity
- mutable operational ownership
- immutable technical ingress origin

This distinction must be explicit because AGH supports human writers, agent writers, automation, extensions, and network peers in the same domain.

## Decision

Adopt the following identity model:

- `created_by` is immutable and always derived server-side from the authenticated/request context
- `origin` is immutable technical ingress metadata and always derived server-side
- `owner` is mutable and optional

Authorization contract:

- task writes are accepted only from authenticated first-class writer surfaces
- in v1, every authenticated first-class writer surface is granted both `global` and `workspace` task creation authority by policy
- `workspace` writes still require the referenced workspace to resolve successfully in the writer context
- extensions and network peers require explicit capability checks in addition to identity derivation
- unauthenticated callers may not read or write the task domain

Ownership rules:

- a task may be created without an owner
- a task may be auto-assigned at creation
- a task may later be claimed or reassigned by an authorized writer

Mutation rules:

- `created_by` and `origin` are never editable
- structural fields such as `scope`, `workspace_id`, and `parent_task_id` are immutable after creation
- editable fields include content metadata, channel binding, and ownership

## Alternatives Considered

### Alternative 1: Trust identity fields from payloads

- **Description**: Let callers set creator, origin, and ownership directly.
- **Pros**: Simple payload design.
- **Cons**: Weak audit integrity and trivial impersonation.
- **Why rejected**: Incompatible with broad writer ingress.

### Alternative 2: Collapse creator and origin into one field

- **Description**: Use one generic “source” field for both actor and ingress context.
- **Pros**: Fewer columns.
- **Cons**: Mixes “who” with “how,” producing weak audit semantics.
- **Why rejected**: The user explicitly distinguished human/agent creation from technical ingress context.

### Alternative 3: Require ownership at creation

- **Description**: Every task must have an owner immediately.
- **Pros**: Simpler dashboards and less nullability.
- **Cons**: Weak fit for pooled, unclaimed, or later-assigned work.
- **Why rejected**: The user explicitly accepted unowned tasks.

## Consequences

### Positive

- Protects audit integrity under broad writer ingress.
- Keeps creator, owner, and ingress semantics distinct.
- Supports unowned queues, later claims, and automated assignment flows.
- Turns broad v1 authority into an explicit policy contract rather than an implicit assumption.

### Negative

- Adds more fields and validation paths.
- Requires stronger request-context derivation in all transports and integrations.

### Risks

- Incomplete context propagation in transports can still produce low-quality identity data.
- Ownership reassignment may need tighter policy later if abuse appears.

## Implementation Notes

- Derive actor identity from the actual writer surface, session, extension, or peer context.
- Reject payload attempts to override `created_by` or `origin`.
- Keep ownership nullable in storage and API contracts.
- Apply capability checks before persistence, not as a best-effort post-check.
