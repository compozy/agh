# ADR-004: Support Optional Task-to-Network-Channel Binding

## Status

Accepted

## Date

2026-04-13

## Context

The user wants tasks to be filterable and groupable by network channel so operational lanes such as finance, marketing, and development can be addressed directly in the task system.

The review also made an important framing point: channel binding should exist, but it should not dominate the task domain. Channels are an optional organizational concern layered onto tasks, not the core reason the task system exists.

AGH already uses validated channel strings across session and network flows, but it does not currently have a separate durable channels catalog.

## Decision

Keep optional task-to-network-channel binding in v1:

- `Task.network_channel` is nullable
- `TaskRun.network_channel` stores the execution-side channel snapshot
- tasks and runs may be filtered by channel
- when a new session is created for a channel-bound executable subtask, the channel is propagated into session creation unless an explicit override is allowed by policy
- network-originated writes must satisfy channel validation when a channel is already bound
- if a stored channel no longer validates against current config, tasks remain readable but new run start or session propagation is rejected until the channel is corrected or cleared

This remains an optional organizing and routing feature. It is not the central axis of task lifecycle or authorization.

## Alternatives Considered

### Alternative 1: No task-to-channel relation

- **Description**: Keep channels only on sessions and network logs.
- **Pros**: Simpler task schema.
- **Cons**: Weak fit for channel-organized workflows and poor filtering.
- **Why rejected**: The user explicitly wants channel-centric task organization.

### Alternative 2: First-class channel catalog now

- **Description**: Create a separate channel domain with foreign-key task references.
- **Pros**: Stronger referential integrity and future channel metadata.
- **Cons**: Introduces another domain without enough current need.
- **Why rejected**: YAGNI for v1.

## Consequences

### Positive

- Supports channel-based filtering and operational segmentation immediately.
- Aligns task orchestration with existing network/session vocabulary.
- Does not force channel semantics onto tasks that do not need them.

### Negative

- Adds another validation dimension to task writes.
- Uses soft validated strings instead of hard channel foreign keys.

### Risks

- Channel labels can drift without naming discipline.
- A task may retain a valid but operationally stale channel label.
- Channel mismatch handling must remain consistent across network ingress and session start flows.

## Implementation Notes

- Reuse existing channel validation rules.
- Keep the field optional and lightweight.
- Treat channel binding as mutable task metadata, not as identity or authority.
- Log and audit stale-channel rejections during run start and network ingress.
