# BUG-001: HTTP task endpoints accept unauthenticated requests

**Severity:** Critical
**Priority:** P0
**Type:** Security
**Status:** Open

## Environment

- **Build:** `95712b7-dirty`
- **OS:** macOS
- **Browser:** N/A
- **URL:** `http://127.0.0.1:50512/api/tasks`

## Summary

The HTTP task surface accepts anonymous requests as a trusted local human principal instead of rejecting them with `401/403`. During `TC-SEC-003`, unauthenticated `GET /api/tasks` returned task data with `200 OK`, and unauthenticated `POST /api/tasks` created a new task with `201 Created`.

## Reproduction

```bash
curl -i http://127.0.0.1:50512/api/tasks
curl -i -H 'Content-Type: application/json' \
  -X POST http://127.0.0.1:50512/api/tasks \
  --data '{"scope":"global","title":"anon task tc-sec-003"}'
```

Observed before the fix:

- `GET /api/tasks` returned `200 OK` with task summaries
- `POST /api/tasks` returned `201 Created` and persisted a task with `created_by.kind = "human"` and `origin.kind = "http"`

## Expected

Anonymous HTTP requests to all task endpoints should be rejected with `401 Unauthorized` or `403 Forbidden`, and no task data should be returned.

## Root cause

The HTTP task handlers derive a trusted local actor for incoming requests without an authentication gate in front of the task routes. As a result, unauthenticated HTTP calls are treated as authenticated local-user writes instead of being rejected before actor resolution.

## Fix

Not fixed in this QA round. The reproduction is stable, but addressing it requires an explicit authentication policy for the local HTTP surface and corresponding handler/middleware changes.
