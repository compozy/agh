# ADR-002: Session-as-Peer Identity Model

## Status

Accepted

## Date

2026-04-10

## Context

AGH Network requires each participant to have a unique peer identity. The daemon manages multiple agent sessions, each running a different agent role (reviewer, coder, general). The design question is: what is a "peer" — the daemon, the agent definition, or the individual session?

The daemon hosts the NATS server and is infrastructure. Agent definitions are templates reused across sessions. Sessions are the active, stateful participants in conversations.

## Decision

Each active agent session is a unique peer in the AGH Network. The peer_id is derived from the agent name and session ID suffix: `{agent_name}.{session_id}` (e.g., `reviewer.sess-a1b2c3d4`). The daemon is infrastructure only — it hosts the NATS server but does not participate as a peer.

In v1, the peer_id transitions to `nickname@fingerprint` format where the nickname is the agent name and the fingerprint is derived from an Ed25519 keypair.

## Alternatives Considered

### Alternative 1: Daemon as Single Peer

- **Description**: The daemon itself is one peer, multiplexing all agent sessions
- **Pros**: Simpler identity model, single keypair in v1
- **Cons**: Cannot address individual sessions, breaks the protocol's peer-to-peer model, conflates infrastructure with participants
- **Why rejected**: Two sessions of the same agent should be independently addressable

### Alternative 2: Agent Definition as Peer

- **Description**: The agent role (e.g., "reviewer") is the peer, shared across sessions
- **Pros**: Stable identity across sessions, simpler addressing
- **Cons**: Multiple concurrent sessions of the same agent share identity, cannot distinguish between them, breaks uniqueness
- **Why rejected**: Each session has independent state and conversations — sharing identity causes routing ambiguity

## Consequences

### Positive

- Each session is independently addressable in the network
- Peer Card carries session-specific capabilities and state
- Clean mapping: session lifecycle events (create/stop) map to peer lifecycle (join/leave)
- Natural transition to v1 nickname@fingerprint format

### Negative

- Peer identities are ephemeral — they appear and disappear with sessions
- More peers to track than if using agent-level identity
- Peer_id includes session suffix which is less human-readable

### Risks

- Many short-lived sessions could create churn in peer registries: mitigated by greet/expiration mechanism from RFC

## Implementation Notes

- Peer_id format: `{agent_name}.{session_id}` matching `[a-z0-9][a-z0-9._-]{0,127}`
- Route token: SHA-256(peer_id)[:32] as per RFC v0 Section 10.3
- Peer Card `peer_id` field matches the `from` field in envelopes
- Session Manager does NOT hold network state — Network Manager observes session lifecycle
