# ADR-001: Embedded NATS Server as Transport Layer

## Status

Accepted

## Date

2026-04-10

## Context

AGH Network v0 requires a NATS transport binding for agent-to-agent communication. The daemon is a single-binary, local-first system. Three deployment models exist: embedded NATS server in the daemon binary, external user-managed NATS server, or a hybrid approach with configurable fallback.

The daemon already embeds all its infrastructure (SQLite, HTTP server, UDS server). Adding an external dependency would break the single-binary promise and complicate the install story.

## Decision

Embed `github.com/nats-io/nats-server/v2/server` directly in the AGH daemon binary. The daemon uses `nats.InProcessServer(ns)` for its own zero-overhead connection. The loopback TCP listener remains part of the transport boundary, but AGH-supported local clients (CLI, bundled skills, extensions) do not connect to the broker directly in v0 and no broker credential is published for them.

Configuration lives in the flat `[network]` TOML section with `port` (default: random via `Port: -1`). `NoSigs: true` is hardcoded (daemon owns signal handling).

## Alternatives Considered

### Alternative 1: External NATS Server

- **Description**: Require users to install and run a separate NATS server
- **Pros**: Smaller binary, proven ops tooling, multi-machine federation trivial
- **Cons**: Breaks single-binary promise, adds install friction, not local-first
- **Why rejected**: Contradicts AGH's core design principle of zero external dependencies

### Alternative 2: Hybrid Embedded + External

- **Description**: Default to embedded but allow config to point to an external NATS server
- **Pros**: Maximum flexibility, supports both local and distributed
- **Cons**: Doubles the connection code paths, complicates testing, premature for v0
- **Why rejected**: YAGNI — federation is an explicit non-goal in v0/v1 RFCs

## Consequences

### Positive

- Single binary, zero external dependencies for networking
- In-process connection has zero TCP overhead for daemon-internal operations
- Keeps local clients on the audited CLI/UDS/Host API path instead of publishing direct broker access
- NATS server lifecycle fully controlled by daemon boot/shutdown sequence
- Testable with `DontListen: true` for unit tests

### Negative

- Adds ~15-20MB to binary size
- NATS server version is locked to the daemon's Go dependency
- No multi-machine federation without future work

### Risks

- Embedded NATS shutdown ordering bug (nats-server#5922): mitigated by drain-before-shutdown sequence
- Binary size growth: acceptable for a daemon binary, monitored in CI

## Implementation Notes

- Use `server.Options{Port: -1, NoSigs: true, Host: "127.0.0.1"}` for localhost-only binding
- Keep the broker auth token in daemon memory only; do not write it to a shared file
- Store resolved port in daemon info file for diagnostics/observability only
- Drain daemon connection before calling `ns.Shutdown()` then `ns.WaitForShutdown()`
