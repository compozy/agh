# ADR-006: Safe Spawn Requires Lineage, TTL, and Permission Narrowing

## Status

Accepted

## Date

2026-04-25

## Context

Autonomous spawn is required for delegation, but unbounded sub-agent creation can create runaway sessions, excessive cost, and broad permission exposure. AGH currently has `Manager.Create` for daemon-initiated sessions but no agent-initiated spawn contract with parent/child lineage, budgets, TTL, or permission inheritance.

## Decision

Spawned sessions must be safe by default:

- Add parent/child lineage (`ParentSessionID`) and a spawned session type or equivalent metadata.
- Default max spawn depth is `1`.
- Default max children per parent is `5`.
- Every spawned session has a TTL.
- Children auto-stop when the parent stops unless explicitly configured otherwise within hard caps.
- Child permissions, tools, and skills may only narrow parent permissions, never widen them.

The coordinator-agent may spawn workers only through this safe spawn contract.

Permission narrowing compares concrete atoms: tools, skills, MCP server IDs, workspace path grants, network channels, and environment profile grants. A child permission set must be a subset of the parent set. Unknown child atoms count as widening. Invalid spawn requests are rejected; the daemon must not silently narrow and continue.

The reaper wins over active leases. When parent-stop or TTL expiry terminates a spawned session, the daemon releases every active run owned by that session with a structured release reason such as `parent_stopped` or `ttl_expired`, emits the relevant task-run and spawn hooks, and then stops the session.

## Alternatives Considered

### Alternative 1: Fully configurable limits without hard defaults

- **Description**: Let global/workspace config define all spawn limits without conservative defaults.
- **Pros**: Maximum flexibility.
- **Cons**: Easy to misconfigure into runaway behavior.
- **Why rejected**: Alpha autonomy needs safe defaults before broad configurability.

### Alternative 2: Manual spawn only

- **Description**: Add lineage APIs but forbid automatic coordinator spawn in the MVP.
- **Pros**: Safest operationally.
- **Cons**: Delays meaningful autonomous delegation.
- **Why rejected**: The approved scope includes safe autonomous spawn.

### Alternative 3: Permissive recursive spawn

- **Description**: Let coordinator and workers spawn multiple levels with broad tools.
- **Pros**: Maximizes autonomy.
- **Cons**: High blast radius and difficult recovery.
- **Why rejected**: Permission narrowing and depth caps are required control primitives.

## Consequences

### Positive

- Enables autonomous delegation with bounded risk.
- Makes session trees observable and recoverable.
- Prevents child agents from escalating permissions.
- Gives task recovery a deterministic policy when spawned children expire or lose their parent.

### Negative

- Some complex workflows may need explicit future limit increases.
- Requires reaper and boot recovery awareness of session lineage.

### Risks

- Risk: parent stops while child has useful work in progress.
- Mitigation: reaper releases active leases with structured reasons before stopping the child, so the task service can requeue or fail according to policy and observers can audit the interruption.

- Risk: permission atom comparisons miss a new permission category.
- Mitigation: unknown child atoms count as widening and reject the spawn until the comparator supports the category.

## Implementation Notes

Extend `session.CreateOpts` or introduce `SpawnOpts` with parent, role overlay, TTL, and permission policy. Add a small permission comparator owned by the session or permission package consumed by daemon wiring. Keep daemon as the composition root for wiring spawn APIs into CLI/UDS.
