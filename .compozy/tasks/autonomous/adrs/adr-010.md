# ADR-010: Manual Operator Control Remains First-Class

## Status

Accepted

## Date

2026-04-26

## Context

The autonomy program adds coordinator-led orchestration, agent-pulled tasks, safe spawn, scheduling wakeups, and self-correction. Those capabilities must not remove existing user agency. Users still need to create tasks manually, start sessions manually, prompt sessions directly, and run manual verification or counter-check sessions. Creating a task and starting a task are distinct actions: creation captures intent; publish/start/approval allows execution by enqueueing a task run.

If autonomy creates a separate path that only coordinator-agents can use, AGH would split into two systems: manual workflows on one side and autonomous workflows on the other. That would duplicate task/session semantics, make debugging harder, and force users to choose between manual control and autonomy.

## Decision

Autonomy is additive. Manual operator control remains a first-class path.

- User-created tasks, automation-created tasks, coordinator-created tasks, and agent-created child tasks all use the same task/run model.
- User-started sessions, coordinator sessions, and spawned worker sessions all use the same session manager and lifecycle hooks.
- Manual operator commands remain explicit about workspace/session/task identity.
- Agent-facing commands infer identity from session environment and daemon context.
- Manual task assignment and autonomous claim converge on the same claim token, lease, heartbeat, completion, failure, and release rules.
- User-created tasks do not require orchestration metadata at creation time.
- Task creation alone does not enqueue claimable work.
- When a user publishes, starts, or approves a task for execution, the enqueued task run is coordinated by the coordinator by default.
- User-started sessions may claim work when their capabilities match, but users may also prompt them directly without involving the coordinator.

There is no separate manual queue, no separate autonomous queue, and no coordinator-only queue. The same task/run model supports user-created tasks, coordinator-created tasks, and agent-created child tasks.

## Alternatives Considered

### Alternative 1: Autonomy-first execution path

- **Description**: Require autonomous tasks to flow through the coordinator and scheduler while manual tasks stay on existing operator APIs.
- **Pros**: Keeps autonomous behavior isolated.
- **Cons**: Duplicates lifecycle semantics and makes claim/lease, permissions, and observability inconsistent.
- **Why rejected**: The task/run/session contracts should be shared. Separate paths create drift and hidden edge cases.

### Alternative 2: Coordinator owns all new task creation

- **Description**: Route every new autonomous-capable task through the coordinator-agent, including user-created tasks.
- **Pros**: Gives the coordinator full semantic context.
- **Cons**: Removes user control from simple directed work and makes manual verification depend on an LLM session.
- **Why rejected**: Users must be able to create tasks and start sessions directly. Coordinator involvement begins when execution is published, started, or approved and a run is enqueued, not when the task is drafted.

### Alternative 3: User-set orchestration flag at task creation

- **Description**: Require users to mark a task as orchestration-required when they create it.
- **Pros**: Explicit trigger stored on the task.
- **Cons**: Forces execution-routing decisions during task drafting and implies agents/coordinators do not create tasks.
- **Why rejected**: Task creation should remain lightweight. Publishing, starting, or approving execution is the correct point to hand the run to the coordinator.

### Alternative 4: Manual-only override mode

- **Description**: Add a separate mode where autonomy is disabled for a workspace and manual behavior uses different APIs.
- **Pros**: Clear operational boundary.
- **Cons**: Creates two systems to maintain and test.
- **Why rejected**: Explicit task metadata and shared claim/lease contracts provide enough control without a separate mode.

## Consequences

### Positive

- Preserves user agency and manual verification workflows.
- Keeps task/session lifecycle semantics unified.
- Lets users adopt autonomy gradually.
- Simplifies testing because manual and autonomous flows exercise the same safety primitives.

### Negative

- Operator APIs and agent APIs must both be covered by claim/lease and permission tests.
- Some contract DTOs need both explicit operator identity and implicit agent identity handling.

### Risks

- Risk: manual and autonomous callers diverge over time.
- Mitigation: require shared service methods for task/session ownership and add integration tests for both user-created and coordinator-created work.

- Risk: task creation accidentally triggers coordinator startup.
- Mitigation: creation never starts the coordinator; only publish/start/approve-execution actions enqueue coordinated work.

## Implementation Notes

Operator commands should stay explicit and stable. Agent commands should stay identity-implicit and session-scoped. Both surfaces call the same task/session services after identity and permission resolution.

Operator UI surfaces should keep task creation, publish/approval, run enqueue, and coordinator spawn visually distinct. Web e2e coverage should prove both manual-first bookends: a task can move from user creation to coordinated execution only after explicit execution, and a user-started session can be prompted directly without triggering coordinator orchestration.

The first MVP integration test should prove: user creates a task, user publishes, starts, or approves it, the task service enqueues a run, the coordinator wakes or spawns, a worker claims via `agh task next --wait`, heartbeats, and completes the run with the same token-fenced path used by autonomous workers. A separate counter-check test should prove a user can start a manual session and prompt it directly without triggering task orchestration.
