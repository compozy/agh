# ADR-009: Autonomy Hooks and Extension Points Are First-Class Contracts

## Status

Accepted

## Date

2026-04-25

## Context

AGH already has a typed hook system with event taxonomy, payload and patch types, dispatch methods, introspection descriptors, hook binding resources, config/agent/skill/extension declaration providers, and resource reconciliation. The autonomy TechSpec adds new runtime behavior: coordinator lifecycle, scheduler wake/recovery decisions, task claim and lease ownership, safe spawn, and memory/self-correction loops.

If those behaviors are implemented only as closed daemon internals, autonomy becomes hard to extend and future integrations will require ad-hoc callbacks, direct package coupling, or a second plugin system. That would violate AGH's existing extensibility model and create a workaround instead of a root-cause architecture.

## Decision

Autonomy must expose first-class hook and extension contracts through the existing hook and resource systems.

Add typed autonomy-related hook events, payloads, patches, dispatch methods, and introspection descriptors instead of ad-hoc callbacks. The initial hook surface should cover operator-meaningful autonomy extension points:

- Coordinator lifecycle: `coordinator.pre_spawn`, `coordinator.spawned`, `coordinator.decision`, `coordinator.stopped`, `coordinator.failed`.
- Task run ownership: `task.run.enqueued`, `task.run.pre_claim`, `task.run.post_claim`, `task.run.lease_extended`, `task.run.lease_expired`, `task.run.lease_recovered`, `task.run.released`.
- Spawn lifecycle: `spawn.pre_create`, `spawn.created`, `spawn.parent_stopped`, `spawn.ttl_expired`, `spawn.reaped`.

Scheduler wake/no-match/recovery signals remain internal metrics, logs, and task-run observability in the MVP. They are not hooks until an external policy use case requires them.

Do not add a `workflow.*` hook family in the MVP. Workflow is correlation metadata until AGH has a first-class workflow package or store.

Autonomy extension must reuse the existing resource architecture:

- Coordinator configuration is resolved from global config plus future workspace overrides.
- Coordinator agent definition may be supplied by bundled defaults and config first; workspace-scoped resource declarations are post-MVP unless a durable user-authored declaration becomes required.
- Scheduler policy must be expressed through narrow typed services, not a generic event bus.
- Situation sections, memory summarization, and evaluation/replay hooks must be extensible through existing provider/resource patterns where those domains already expose such contracts.

Safety invariants remain daemon-owned. Hooks may validate, deny, annotate, or narrow a request where the payload explicitly supports it, but hooks must not bypass atomic claim transactions, lease fencing, spawn limits, TTL, parent/child lineage, or permission narrowing. Pre hooks mutate only the request/criteria envelope; post hooks observe the committed result. `task.run.pre_claim` may deny a claim and may only add required-capability constraints or raise priority if mutation is enabled; it must never broaden criteria or change the claimant.

## Alternatives Considered

### Alternative 1: No new autonomy hooks

- **Description**: Implement coordinator, scheduler, claim/lease, and spawn as internal daemon flows only.
- **Pros**: Less initial hook boilerplate.
- **Cons**: Makes autonomy less extensible than the rest of AGH and pushes integrations toward custom callbacks.
- **Why rejected**: This is a workaround. It hides the root problem that new autonomous behavior needs explicit extension contracts.

### Alternative 2: Create a separate autonomy plugin system

- **Description**: Add autonomy-specific plugin or callback registration outside the current hooks/resources stack.
- **Pros**: Could be tailored to autonomy concepts.
- **Cons**: Duplicates existing hook binding, resources, extension declarations, permissions, and introspection.
- **Why rejected**: AGH already has the extensibility substrate; a second system increases complexity without solving a new class of problem.

### Alternative 3: Add a generic event bus

- **Description**: Publish autonomy events into a broad internal bus that arbitrary packages can subscribe to.
- **Pros**: Flexible and easy to append events.
- **Cons**: Violates the project's direct-call architecture, weakens type boundaries, and makes behavior harder to reason about.
- **Why rejected**: The existing typed hook taxonomy provides extension without introducing hidden control flow.

## Consequences

### Positive

- Autonomy remains consistent with AGH's existing extension model.
- Workspace, skill, config, and extension authors can customize autonomous behavior through documented hooks.
- Hook introspection can explain the autonomy surface to operators and agents.
- Safety-sensitive invariants stay centralized in the daemon.
- Hook names align with AGH's existing taxonomy instead of introducing an `autonomy.*` umbrella.

### Negative

- Every autonomy feature needs explicit payload, patch, dispatch, and introspection work.
- Some hook events will be observation-only to preserve claim/lease and spawn safety.

### Risks

- Risk: hook mutability could accidentally weaken safety boundaries.
- Mitigation: keep claim/lease, TTL, lineage, and permission narrowing enforced after hook patches and inside daemon-owned transactions.

- Risk: too many hook events create noise.
- Mitigation: start with lifecycle and decision points that external policy needs, and keep purely internal scheduler bookkeeping private.

## Implementation Notes

Extend `internal/hooks/events.go`, `internal/hooks/payloads.go`, `internal/hooks/dispatch.go`, and `internal/hooks/introspection.go` for autonomy events.

Existing task-domain events remain audit records written by `internal/task`. The task package consumes a narrow hook dispatcher interface injected by the daemon; the daemon adapter maps those requests to the hook runtime. Pre-commit task-run hooks dispatch at the service call site before the transactional state change. Post-commit hooks dispatch after the service writes the task audit event. Do not tail the task events table to implement pre-commit hooks.

Prefer adding resource kinds only when a durable, user-authored declaration exists. Do not create resources for transient scheduler state. The MVP should add zero new resource kinds unless implementation proves one is required; task run claim state remains in `task_runs`.

Tests must cover event taxonomy validation, introspection descriptors, resource-backed hook declarations, and safety boundaries such as hooks being unable to widen child permissions or claim a run outside `ClaimNextRun`.
