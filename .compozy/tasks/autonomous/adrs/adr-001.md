# ADR-001: Phased Autonomy Kernel Scope

## Status

Accepted

## Date

2026-04-25

## Context

The autonomy analysis identified a broad set of gaps across prompt context, agent-facing commands, task claiming, scheduling, spawn, network coordination, memory, observability, and evaluation. AGH is greenfield alpha, so the system can make clean schema and API changes, but the implementation still needs a bounded scope that can be decomposed into independently executable tasks.

The design must avoid speculative features that do not unlock local autonomous operation, including cross-daemon swarm coordination, leader election, vector memory, signed skill supply chain, generic runtime tool execution, and Prometheus/webhook integrations. The design must also preserve manual task creation and manual session startup as first-class paths.

## Decision

The TechSpec will define a broad phased autonomy program covering:

- Situation Surface
- Agent Kernel CLI
- Task Claim and Lease
- Mechanical Scheduler plus Coordinator-Agent
- Safe Spawn
- Essential Network Evolution
- Useful Memory
- Minimal Observability
- Basic Eval and Replay

The first implementation wave will prioritize local daemon autonomy through the kernel steps: contracts/config, hook taxonomy, Situation Surface, Agent Kernel CLI, task claim/lease, scheduler sweep/notify, safe spawn, and coordinator bootstrap. Network evolution, broad memory expansion, self-correction beyond minimal counters, eval/replay, and web visibility are follow-on TechSpecs unless explicitly pulled into scope later.

Autonomy is additive to operator workflows. User-created tasks and user-started sessions remain supported and use the same task/session/claim/lease contracts as autonomous work.

## Alternatives Considered

### Alternative 1: P0-only specification

- **Description**: Limit the TechSpec to Situation Surface, Agent Kernel CLI, and Task Claim and Lease.
- **Pros**: Smaller document, faster first implementation, lower coordination overhead.
- **Cons**: Omits scheduler, coordinator-agent, spawn, memory, and eval decisions that need to be coherent with the foundation.
- **Why rejected**: The user wants a large TechSpec that can later be decomposed by `cy-create-tasks`; documenting the broader target now prevents incompatible local decisions.

### Alternative 2: Full unfiltered autonomy roadmap

- **Description**: Include every proposal from the analysis, including cross-daemon swarm, contract-net, multi-home sessions, generic tools, vector memory, and advanced observability.
- **Pros**: Captures all research ideas in one design.
- **Cons**: Mixes necessary local autonomy with speculative architecture and increases the risk of overengineering.
- **Why rejected**: The immediate goal is a more integrated autonomous local system, not a distributed multi-daemon swarm or marketplace.

## Consequences

### Positive

- Keeps a coherent system target while allowing phased delivery.
- Gives `cy-create-tasks` enough structure to create dependent task files.
- Preserves explicit out-of-scope boundaries.
- Preserves manual operation while introducing autonomous execution.

### Negative

- The TechSpec will be larger than a narrow MVP-only design.
- Sequencing must be precise to avoid treating all components as one atomic implementation.

### Risks

- Risk: downstream tasks become too broad.
- Mitigation: the TechSpec must include P0/P1/P2 sequencing, dependencies, and clear component boundaries.

- Risk: local autonomy tasks accidentally include post-MVP network, memory, eval, or web work.
- Mitigation: the first task decomposition should cut at the local autonomy MVP and reference follow-on items separately.

## Implementation Notes

The TechSpec must describe the target state but split the build order into phases. Every phase after the first must name its dependencies.
