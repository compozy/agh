# ADR-002: Support Global and Workspace Task Scope with Explicit Hierarchy and Bounded Dependencies

## Status

Accepted

## Date

2026-04-13

## Context

The task domain must support durable coordination across one workspace or many workspaces. A workspace-only model would force cross-workspace orchestration to live outside the task system, which conflicts with the product goal for explicit global coordinators and workspace-scoped subtasks.

At the same time, AGH needs two different structural relationships:

- parent/child hierarchy for subtasks and decomposition
- dependency edges for blockers and orchestration across branches

The review of the original TechSpec correctly identified a risk: if dependency edges are left too open-ended, the v1 model becomes a difficult-to-operate DAG without explicit limits. The domain therefore needs both expressive structure and hard guardrails.

## Decision

Model `Task` with explicit scope, explicit hierarchy, and bounded dependency edges:

- `scope` enum: `global` or `workspace`
- `workspace_id` required only when `scope = workspace`
- `parent_task_id` for explicit task/subtask hierarchy
- separate dependency edges for `depends_on`

The default multi-workspace orchestration pattern in v1 is:

- one global coordinating task
- one or more workspace-scoped child tasks
- optional dependency edges across siblings or across trees

Guardrails are part of the decision:

- maximum hierarchy depth: `8`
- maximum dependency edges per task: `32`
- maximum direct children per task: `64`
- cycle detection occurs when a dependency edge is created
- edge validation and insertion occur in one `BEGIN IMMEDIATE` transaction so cycle checks and writes share the same lock window

Hierarchy and dependencies remain distinct concepts. Subtasks are not encoded only as edges, and dependencies are not inferred from parenthood.

## Alternatives Considered

### Alternative 1: Workspace-only tasks

- **Description**: Every task belongs to exactly one workspace.
- **Pros**: Simplest store and query model.
- **Cons**: Cannot naturally represent one coordinator over multiple workspaces.
- **Why rejected**: The user explicitly wants multi-workspace coordination in v1.

### Alternative 2: Parent/child hierarchy only

- **Description**: Represent all subtask and blocker semantics with `parent_task_id`.
- **Pros**: Simple tree queries and easier lifecycle inheritance.
- **Cons**: Cannot model lateral blockers or cross-tree sequencing cleanly.
- **Why rejected**: Insufficient for real orchestration.

### Alternative 3: Generic graph only

- **Description**: Drop explicit subtask hierarchy and represent everything as graph edges.
- **Pros**: Most flexible abstract model.
- **Cons**: Poor UX, poor explainability, and a worse fit for explicit subtasks.
- **Why rejected**: The product requirement is tasks/subtasks, not arbitrary graph nodes.

## Consequences

### Positive

- Supports both local workspace work and cross-workspace orchestration in one domain.
- Keeps subtask semantics legible through explicit parent/child structure.
- Allows blocker semantics without collapsing the model into a graph-only abstraction.
- Adds explicit operating limits before the graph becomes difficult to reason about.

### Negative

- Introduces cross-scope validation rules.
- Requires both tree validation and graph validation.
- Needs explicit failure paths for limit violations.

### Risks

- Poorly specified cross-scope rules could still permit invalid parent/child relationships.
- Dependency limits that are too strict may reject future orchestration shapes.
- Parent lifecycle rules and dependency lifecycle rules can still conflict if not reconciled centrally.

## Implementation Notes

- Enforce `scope` and `workspace_id` consistency at the store and manager layers.
- Treat graph guardrails as contract-level invariants, not optional recommendations.
- Reject self-dependency and cycles at edge-creation time.
- Expose hierarchy and dependency data separately in API responses.
