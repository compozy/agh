# ADR-001: Separate Task Coordination Records from TaskRun Execution Records

## Status

Accepted

## Date

2026-04-13

## Context

AGH is session-centric today. `internal/session/` owns live ACP-backed runtime lifecycle, `internal/store/sessiondb/` persists per-session append-only history, `internal/observe/` builds projections over session and global data, and `internal/daemon/` is the sole composition root.

That shape is a poor fit for durable work coordination. A future tasks/subtasks feature must survive session restarts, span multiple sessions, and support explicit dependency management. The analysis in `.compozy/tasks/core-tasks/analysis/analysis_agh-harness.md` shows that bolting task state onto `SessionMeta`, `sessiondb`, or `observe` would blur write/read boundaries and create a workaround architecture.

External reference systems split the problem differently:
- `paperclip` treats the durable work item as the primary issue record.
- `multica` separates issue-like coordination from queue/execution rows.
- `goclaw` keeps shared board tasks apart from comments, audit activity, and execution-side announce/recovery flows.

AGH needs a model that preserves session simplicity while introducing a durable orchestration domain.

## Decision

Introduce a dedicated daemon-owned task domain in a new `internal/task/` package with two first-class persisted entities:

- `Task` as the durable coordination record
- `TaskRun` as the durable execution record linked to a task and, once started, optionally to a session

`Task` carries coordination metadata such as scope, ownership, hierarchy, dependencies, status, and user/agent intent. `TaskRun` carries execution lifecycle, scheduling/claiming state, session linkage, attempt history, and runtime outcome details.

Canonical `Task.status` is not derived ad hoc by callers. It is owned by a central `TaskManager`, which reconciles task mutations and run lifecycle transitions.

The persistence for this domain belongs in the global database, not in `sessiondb`, because tasks are cross-session coordination artifacts.

## Alternatives Considered

### Alternative 1: Single issue/task record with embedded execution state

- **Description**: Store coordination and execution lifecycle in one table, closer to `paperclip`.
- **Pros**: Simpler query model, fewer tables, straightforward task detail APIs.
- **Cons**: Mixes queueing, retries, runtime linkage, and coordination concerns into one aggregate; harder to preserve clean session boundaries.
- **Why rejected**: AGH already has a strong distinction between durable coordination state and live runtime/session state. A single aggregate would increase coupling and make queue-first execution harder.

### Alternative 2: Execution-first queue rows as the primary task model

- **Description**: Make execution rows the main entity, closer to `multica`'s `agent_task_queue`.
- **Pros**: Strong for dispatch, claim, retry, and resumable execution.
- **Cons**: Weaker for explicit subtasks, dependency graphs, durable ownership, and non-running coordination state.
- **Why rejected**: The user wants core tasks/subtasks, not only an execution queue. AGH needs a durable coordination layer above execution.

## Consequences

### Positive

- Preserves the existing `session` package as a live runtime boundary.
- Supports retries, multiple attempts, and session linkage without corrupting the durable task record.
- Makes global/API/CLI task queries cleaner because coordination and execution can be filtered independently.
- Creates a natural place for future projections, audit trails, and richer orchestration features.

### Negative

- Adds a new domain package and multiple persistence tables.
- Requires explicit reconciliation rules between `Task` and `TaskRun`.
- Introduces more joins and more lifecycle rules than a single-table approach.

### Risks

- Reconciliation complexity could grow if status rules are underspecified.
- Callers may try to bypass the manager and treat `TaskRun` as authoritative status.
- Over-modeling execution details could recreate automation-like duplication if boundaries are not kept sharp.

## Implementation Notes

- The task domain should be wired only from `internal/daemon/`, following existing composition-root rules.
- The initial write/read API surface should parallel the patterns already used for `session` and `automation`: manager interface in `internal/api/core/`, transport handlers in HTTP/UDS, CLI commands that talk to the daemon rather than the filesystem directly.
- `sessiondb` remains the source of session event history; task/run projections may reference sessions but should not duplicate the full transcript model.
