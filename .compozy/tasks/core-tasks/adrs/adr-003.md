# ADR-003: Use Queue-First TaskRun Lifecycle with Central TaskManager Authority

## Status

Accepted

## Date

2026-04-13

## Context

AGH needs broad task writes in v1. Humans, agents, automation, extensions, and network peers must all be able to create tasks or enqueue work. That implies work may exist before any live session exists.

At the same time, the redesign clarified two critical points:

- `Task` is an explicit coordination resource, not AGH's universal execution envelope
- canonical `Task.status` must remain manager-owned even with broad writer ingress

The original draft correctly chose queue-first execution, but it left too much room to read `TaskRun` as a second universal execution system parallel to `session`. The accepted redesign keeps queue-first execution while making session-backed execution the concrete runtime boundary.

## Decision

Adopt a queue-first `TaskRun` lifecycle:

`queued -> claimed -> starting -> running -> completed | failed | cancelled`

Key rules:

- `TaskRun` may exist before any session exists
- `session_id` is nullable until execution starts
- canonical `TaskRun.status` is manager-owned, just like canonical `Task.status`
- external writers may create tasks, enqueue runs, and request lifecycle actions through the task service
- external writers may not mutate canonical `Task.status` or `TaskRun.status` directly
- `TaskManager` reconciles canonical task state
- executable subtasks start in dedicated sessions by default
- attaching an existing session is explicit and reserved for resume, continue, or handoff flows
- non-human run lifecycle actions require idempotency keys
- on daemon boot, orphaned `claimed` runs are re-queued and orphaned `starting` or `running` runs are failed during startup recovery

Automation boundary:

- automation may continue running non-task operational jobs outside the task domain
- when automation chooses a task-backed path, `TaskRun` is the sole execution pipeline for that work item
- automation must not create a parallel automation run/session for the same task-backed unit of work

This keeps queueing durable and transport-friendly without turning the task domain into a replacement for `internal/session`.

## Alternatives Considered

### Alternative 1: Session-first runs

- **Description**: Create `TaskRun` only when a session already exists.
- **Pros**: Simpler execution model and fewer states.
- **Cons**: Poor fit for automation, network ingress, retries, and deferred work.
- **Why rejected**: The accepted writer set requires queued work before session startup.

### Alternative 2: Direct multi-writer task status mutation

- **Description**: Allow any authorized writer to set canonical task state directly.
- **Pros**: Simple for callers.
- **Cons**: Conflicting transitions, weak invariants, and inconsistent lifecycle policy.
- **Why rejected**: Conflicts with the central-authority requirement.

### Alternative 3: Universal task-based execution engine

- **Description**: Treat `TaskRun` as the default wrapper for all execution in AGH.
- **Pros**: One apparent abstraction for all work.
- **Cons**: Overloads the task domain, duplicates session responsibility, and violates the explicit-resource model the user approved.
- **Why rejected**: `Task` is explicit coordination, not a mandatory wrapper around every runtime action.

## Consequences

### Positive

- Supports asynchronous ingress from many surfaces.
- Preserves one source of truth for canonical task lifecycle.
- Keeps queueing durable without requiring immediate session allocation.
- Aligns execution with the accepted “subtasks get dedicated sessions by default” model.

### Negative

- Requires explicit reconciliation logic.
- Requires idempotency and replay protection for non-human callers.
- Adds manager complexity around start, attach, cancel, and reconciliation flows.

### Risks

- Duplicate queued runs can appear without strong idempotency handling.
- Claims and starts can race if execution and persistence are not coordinated carefully.
- Cancellation semantics can become inconsistent if run lifecycle and session stop behavior drift apart.

## Implementation Notes

- Persist all run states in the global store.
- Keep canonical task status transitions inside `TaskManager`.
- Keep canonical run status transitions inside `TaskManager`.
- Treat `StartRun` and `AttachRunSession` as different operations with different policy.
- Document dedicated-session default explicitly in transport and API contracts.
- Run startup recovery before accepting new lifecycle mutations after daemon boot.
