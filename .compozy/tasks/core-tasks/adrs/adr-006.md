# ADR-006: Execute Subtasks Through an Injected Session Bridge with Dedicated Sessions by Default

## Status

Accepted

## Date

2026-04-13

## Context

The redesign clarified that tasks are not the universal execution engine of AGH. `session` remains the general execution primitive. At the same time, executable work inside the task domain needs a clean and observable way to start, attach, and stop sessions.

The review also called out an architectural boundary issue: `internal/task` should not import `internal/session` directly. That would blur the domain boundary and make `TaskManager` too tightly coupled to runtime implementation details.

## Decision

Use an injected execution bridge defined by `internal/task` and wired by `internal/daemon`.

Execution rules:

- `Task` itself does not create sessions
- executable subtasks are the default unit of task-backed execution
- starting an executable subtask creates a dedicated session by default
- attaching an existing session is allowed only in explicit resume, continue, or handoff flows
- attachment is valid only before a run reaches `running`, only when the run has no bound session, and only when the target session is not already attached to another non-terminal run
- cancellation first requests cooperative stop and escalates to forced stop after the daemon-configured grace timeout
- daemon startup performs orphaned-run recovery before new task-backed execution begins

`internal/task` depends only on a small interface for start, attach, request-stop, and force-stop behavior.

## Alternatives Considered

### Alternative 1: Direct import of `internal/session`

- **Description**: Let `TaskManager` call session package APIs directly.
- **Pros**: Fewer abstractions.
- **Cons**: Strong coupling, harder testing, and broken package boundaries.
- **Why rejected**: Violates AGH's composition-root discipline.

### Alternative 2: Reuse current session for subtask execution by default

- **Description**: Attach most subtasks to the caller's existing session automatically.
- **Pros**: Lower startup overhead and easier conversational continuity.
- **Cons**: Blurs execution boundaries, weakens cancellation isolation, and hurts audit clarity.
- **Why rejected**: The user explicitly chose dedicated sessions by default.

### Alternative 3: Force every execution to allocate a session with no attach path

- **Description**: Never allow attach or reuse.
- **Pros**: Very simple default model.
- **Cons**: Prevents explicit resume and handoff flows.
- **Why rejected**: Too rigid for real recovery and continuation scenarios.

## Consequences

### Positive

- Preserves the session package as the runtime boundary.
- Gives executable subtasks clean session isolation by default.
- Keeps resume and handoff possible without making them the default.
- Makes cancellation and audit behavior easier to reason about.

### Negative

- Requires a bridge interface and adapter wiring.
- Introduces explicit start-versus-attach policy in the task service.

### Risks

- Poor adapter design could still leak session concerns into task logic.
- Dedicated-session defaults can increase resource use if task volumes spike.

## Implementation Notes

- Define the bridge interface in `internal/task`.
- Implement it in the daemon wiring layer over `internal/session`.
- Treat explicit attach as a different policy path than default start.
- Make `session_id` single-assignment once bound.
- Record forced-stop escalation details in task audit data.
