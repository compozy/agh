# ADR-002: AI SDK + Harnss Hybrid Streaming Architecture

## Status

Accepted

## Date

2026-04-03

## Context

The web client needs to consume the daemon's SSE streaming API (`POST /api/sessions/:id/prompt`) which outputs events in the Vercel AI SDK `x-vercel-ai-ui-message-stream: v1` format. The stream includes text deltas, reasoning deltas, tool call events, tool results, permission requests, and finish signals. We need to decide how to parse, buffer, and render these events with high performance (60fps during streaming) while supporting complex UI features like collapsible tool cards, thinking block rendering, and permission prompt handling.

The reference implementation (`.resources/harnss`) uses a `SimpleStreamingBuffer` class with `requestAnimationFrame` flush pattern that coalesces rapid SDK events into batched React state updates.

## Decision

Adopt a hybrid approach: use Vercel AI SDK (`@ai-sdk/react` `useChat`) as the SSE transport layer for connection management, reconnection, and basic message framing, but intercept the event stream to feed a custom `SimpleStreamingBuffer` (adapted from harnss) for fine-grained rendering control.

Specifically:
- `useChat` establishes the SSE connection to `/api/sessions/:id/prompt`
- Custom `onToolCall`, `onFinish`, and stream part callbacks extract events before AI SDK's default processing
- A `SimpleStreamingBuffer` (ref-based, no React dependency) accumulates text and thinking deltas
- A `requestAnimationFrame` scheduler flushes buffer contents to Zustand store at ~60fps
- Zustand `sessionStore` holds the active session's rendering state (messages array, streaming flags, pending permissions)
- TanStack Query handles server-side cache for agents list, sessions list, and session event history

## Alternatives Considered

### Alternative 1: Pure Custom SSE Parser

- **Description**: Build a fully custom SSE parser using `fetch` + `ReadableStream`. No AI SDK dependency. Hand-written event routing, reconnection, and error handling.
- **Pros**: Maximum control over every aspect of SSE parsing. Zero dependency on external SDK. Full flexibility for custom event types.
- **Cons**: Significant code to write and maintain (SSE parsing, reconnection with backoff, message format mapping). The daemon already outputs Vercel-compatible format, making this integration free.
- **Why rejected**: Reinvents transport-layer plumbing that AI SDK provides battle-tested. The daemon's output format was designed for this SDK.

### Alternative 2: AI SDK Full Delegation

- **Description**: Fully delegate to AI SDK `useChat` — let it manage messages, tool calls, and all state internally. Rendering components consume AI SDK's message model directly.
- **Pros**: Minimal custom code. Proven patterns from AI SDK ecosystem.
- **Cons**: AI SDK's internal message model doesn't natively support: permission prompts with allow/reject options, thinking blocks with overlap detection, tool card expand/collapse persistence, harnss-style `UIMessage` fields (subagent steps, checkpoint IDs, queued state). Would need extensive workarounds.
- **Why rejected**: The rendering requirements (from harnss patterns) exceed what AI SDK's built-in UI layer supports without heavy customization that defeats the purpose.

## Consequences

### Positive

- Best of both worlds: proven SSE transport + full rendering control
- Directly reuses harnss patterns (SimpleStreamingBuffer, rAF flush, UIMessage model)
- Streaming performance guaranteed at 60fps via rAF coalescing
- AI SDK handles reconnection and error recovery transparently
- Clean separation: AI SDK owns transport, Zustand owns UI state

### Negative

- Coupling to AI SDK's stream format (mitigated: daemon already speaks it)
- Need to intercept AI SDK's internal processing to extract custom event types (tool cards, permissions)
- Two state management layers (AI SDK's internal messages + our Zustand store) require careful synchronization

### Risks

- AI SDK may change its internal callback API in future versions — mitigated by pinning version and wrapping in an adapter
- Custom event types (permission, data-agh-event) not natively supported by AI SDK — mitigated by using stream part callbacks

## Implementation Notes

- Install `@ai-sdk/react` and `ai` packages
- SimpleStreamingBuffer adapted from `.resources/harnss/src/lib/streaming-buffer.ts`
- rAF flush pattern: accumulate in ref → schedule single rAF → flush to Zustand store
- Zustand store shape: `{ messages: UIMessage[], isStreaming: boolean, pendingPermission: PermissionRequest | null }`
- TanStack Query keys: `['agents']`, `['sessions']`, `['session', id, 'events']`
