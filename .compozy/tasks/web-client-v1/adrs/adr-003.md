# ADR-003: State Management — Zustand + TanStack Query

## Status

Accepted

## Date

2026-04-03

## Context

The web client has two distinct categories of state: server state (agents list, sessions list, session event history — data fetched from daemon API) and client/UI state (active session's streaming buffer, pending permissions, sidebar collapse, tool card expand states). We need to decide how to manage each category.

## Decision

Use Zustand for client/UI state and TanStack Query for server state.

**Zustand stores:**
- `sessionStore` — active session rendering state: messages array, streaming buffer ref, isStreaming flag, pending permission request, active session ID
- Scoped to the active chat view; resets on session switch

**TanStack Query:**
- `['agents']` — `GET /api/agents` with staleTime: 60s (agents rarely change)
- `['sessions']` — `GET /api/sessions` with refetchInterval: 5s (sessions can change state)
- `['session', id, 'history']` — `GET /api/sessions/:id/history` fetched on session navigation
- Mutations: `createSession`, `stopSession`, `resumeSession`

## Alternatives Considered

### Alternative 1: TanStack Query Only

- **Description**: Use TanStack Query for all state, including streaming/UI state via mutable query keys.
- **Pros**: Single state management system. Well-documented patterns.
- **Cons**: Streaming state updates at 60fps don't fit TanStack Query's cache-invalidation model. Would require constant `setQueryData` calls that defeat the purpose of the cache.
- **Why rejected**: TanStack Query is designed for server state with request/response patterns, not for high-frequency real-time UI state.

### Alternative 2: React Context + TanStack Query

- **Description**: React Context for session/streaming state, TanStack Query for server state.
- **Pros**: No additional dependency (Zustand). Works with React's built-in primitives.
- **Cons**: Context changes trigger re-renders for all consumers. Streaming at 60fps would cause excessive re-renders in components that don't need the update (sidebar, header). Requires extensive memoization.
- **Why rejected**: Performance characteristics are worse than Zustand's selector-based subscription model for high-frequency updates.

## Consequences

### Positive

- Clear separation: server state in Query cache, UI state in Zustand
- Zustand's selector subscriptions prevent unnecessary re-renders during streaming
- TanStack Query handles caching, refetching, and stale-while-revalidate for API data
- Both libraries are already in package.json (Zustand 5.0.11, TanStack Query 5.90.0)

### Negative

- Two state management paradigms to understand
- Need discipline to not leak server state into Zustand or UI state into Query cache

### Risks

- State synchronization between Query and Zustand (e.g., session state changes) — mitigated by using Query's `onSuccess` callbacks to sync relevant fields to Zustand
