# ADR-001: Harness Web Lite — SPA focada no loop de conversa

## Status

Accepted

## Date

2026-04-03

## Context

AGH v2 backend está completo com HTTP/SSE API, ACP subprocess management, e session lifecycle. O projeto `.resources/harnss` é um app Electron desktop rico com sidebar, chat virtualizado, tool cards, e permission management. Precisamos definir a estratégia de produto para a primeira versão do web client que consome a API do daemon.

O web client precisa atender tanto devs individuais quanto tech leads de equipes pequenas, com foco em experiência visual rica para interação com agentes de coding (Claude, Codex, Gemini, etc.).

## Decision

Adotar a abordagem "Harness Web Lite" — construir uma SPA web que replica o core de conversa do harnss, adaptado para browser (HTTP/SSE em vez de Electron IPC).

Escopo V1:
- Sidebar com hierarquia Agentes → Sessões
- Chat view com streaming (text + reasoning), markdown renderizado
- Tool cards colapsáveis copiados do harnss
- Permission prompt completo (allow once/always, reject once/always)
- Indicadores de estado (processing, streaming, pending permission)

Fora de escopo V1:
- Terminal embutido, browser panel, file explorer
- Spaces/projects (conceito do harnss Electron)
- Dashboard de observabilidade, custos, métricas
- Temas customizáveis além de light/dark

## Alternatives Considered

### Alternative 1: Dashboard-First

- **Description**: Começar pelo painel de observabilidade (saúde do daemon, sessões ativas, eventos, custos) e adicionar conversa como feature secundária.
- **Pros**: Atende tech leads que querem monitorar agentes; boa visão de saúde do sistema.
- **Cons**: Demora mais para entregar o valor principal (conversar com agentes); vai contra a necessidade primária de "experiência visual rica".
- **Why rejected**: O problema principal é a limitação do terminal para interação rica, não monitoramento.

### Alternative 2: Full Parity com Harnss

- **Description**: Replicar o harnss completo incluindo terminal embutido, browser panel, file explorer, spaces/projects, e todas as ferramentas laterais.
- **Pros**: Feature-complete desde o início.
- **Cons**: Escopo enorme para V1; muitas features dependem de Electron (terminal real, filesystem direto) e não funcionam em browser puro; alto risco de atraso.
- **Why rejected**: Muitas features são específicas de desktop e não fazem sentido em uma SPA web. YAGNI — começar pelo core e expandir.

## Consequences

### Positive

- Entrega rápida do loop de conversa completo
- Reutiliza patterns provados e battle-tested do harnss
- Foco claro permite alta qualidade na experiência core
- Base sólida para adicionar observabilidade e features avançadas em fases futuras

### Negative

- Sem observabilidade ou dashboard na V1
- Sem terminal embutido (usuários precisam do CLI para operações avançadas)
- Sem conceito de projects/workspaces na primeira versão

### Risks

- O streaming SSE do daemon pode ter diferenças sutis com o protocolo Electron IPC do harnss — adaptar os event handlers requer atenção
- Permission flow via HTTP pode ter latência maior que IPC local — mitigado pelo fato do daemon rodar localmente

## Implementation Notes

- Stack: React 19 + Vite 8 + TanStack Router/Query + shadcn/ui (base-nova) + Tailwind v4 (já configurado em web/)
- Usar `systems/` directory para feature modules (app-renderer-systems pattern)
- API proxy já configurado: `/api` → `localhost:2123`
- Copiar/adaptar patterns do harnss: UIMessage model, StreamingBuffer, tool card renderers, permission UI
