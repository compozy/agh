# ADR-006: Keep ACP Mock Implemented in Go

## Status

Accepted

## Date

2026-04-17

## Context

The original E2E planning documents described a Node/TypeScript ACP mock driver plus `@copilotkit/aimock` for deterministic assistant streaming. The shipped implementation took a different path:

- `internal/testutil/acpmock/cmd/acpmock-driver/` is a Go ACP subprocess
- runtime and browser E2E both register it through temporary AGH agent definitions
- the driver and daemon share the same ACP SDK and test toolchain
- no `driver/dist/index.js` or `@copilotkit/aimock` dependency exists in the repository

That shipped path is already the operational truth. The gap is documentary, not architectural. Without an explicit ADR, the task pack continues to describe a non-existent Node driver and makes review notes look like missing-file bugs instead of stale documents.

## Decision

Standardize the ACP mock implementation on the shipped Go driver under `internal/testutil/acpmock/cmd/acpmock-driver/`.

The E2E harness contract is:

- mock agents are registered only through temporary AGH agent definitions
- fixture routing uses structured prompt metadata plus exact user-text matching
- pathological ACP behaviors are modeled through constrained fixture primitives, not ad hoc test-only seams
- runtime and browser lanes may prebuild the daemon and mock-driver binaries once per lane and reuse them through `AGH_TEST_DAEMON_BIN` and `AGH_TEST_ACPMOCK_DRIVER_BIN`

ADR-001 remains as historical context only and is superseded by this ADR.

## Alternatives Considered

### Alternative 1: Reintroduce the planned Node driver

- **Description**: revive `driver/dist/index.js` and `@copilotkit/aimock` so the implementation matches the original docs.
- **Pros**: restores document parity with the old task pack.
- **Cons**: adds a second ACP implementation language, extra CI/build surface, and ongoing SDK parity tax for no product gain.
- **Why rejected**: the shipped Go driver already satisfies the runtime/browser needs with less complexity and tighter coupling to production ACP behavior.

### Alternative 2: Keep the Go driver but leave the docs ambiguous

- **Description**: accept the shipped code and avoid an explicit architectural statement.
- **Pros**: less documentation work now.
- **Cons**: reviewers and future tasks keep reasoning from stale assumptions.
- **Why rejected**: the suite’s confidence claims depend on an honest contract, not accidental tribal knowledge.

## Consequences

### Positive

- One ACP protocol implementation language across runtime code and deterministic mocks.
- No cross-language parity maintenance for ACP request/response evolution.
- Browser and runtime E2E reuse the same fixture and diagnostics model.
- Lane-level prebuild caching is straightforward because both binaries are Go commands.

### Negative

- All mock-driver fault injection lives in Go fixture/driver code rather than reusable JS libraries.
- The docs must be updated anywhere they still mention the phantom Node driver.

### Risks

- Teams may over-read the deterministic Go driver as equivalent to real-provider coverage. Mitigate through ADR-007 and explicit lane documentation.
