# BUG-005: UDS/HTTP observe parity test expected a stale harness augmenter sequence

## Status

Fixed and re-verified.

## Severity

Medium.

## Scenario

`make test-e2e-runtime` failed in `TestUDSTransportObserveHarnessLifecycleParityMatchesHTTP` while waiting for observe harness events.

## Expected

The parity test should wait for the current runtime event sequence and then compare HTTP and UDS projections.

## Actual

The test expected only one `harness.augmenter_applied` event after turn context resolution. The runtime emitted both `durable_memory` and `situation` augmenter events, so the helper kept polling until its timeout despite the correct events already being present.

## Root Cause

The test's expected lifecycle sequence lagged the current harness behavior.

## Fix

`internal/api/udsapi/transport_parity_integration_test.go` now expects both turn augmenter events before asserting HTTP/UDS parity.

## Evidence

- Reproduction failure: `.compozy/tasks/tools-refac/qa/logs/final-test-e2e-runtime.log`
- Re-verified focused test: `go test -race -count=1 -tags integration ./internal/api/udsapi -run TestUDSTransportObserveHarnessLifecycleParityMatchesHTTP`
