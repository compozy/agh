# BUG-002: MCP auth-required discovery failure made builtin tool surfaces unavailable

## Status

Fixed and re-verified.

## Severity

High.

## Scenario

Task 13 real-scenario QA configured an isolated remote MCP server with OAuth PKCE auth and no token, then executed:

```bash
agh tool list -o json
agh tool info agh__mcp_auth_status -o json
agh tool invoke agh__mcp_auth_status --input '{"server_name":"linear-qa"}' -o json
```

## Expected

External MCP tools that require login should not break builtin AGH tool discovery. `agh__mcp_auth_status` must remain callable so agents can diagnose the MCP auth state while login/logout stay on management surfaces.

## Actual

The registry failed the entire operator tool surface:

```text
error: tool_unavailable: mcp authentication is unavailable
```

The same failure prevented `agh__mcp_auth_status` from being inspected or invoked.

## Root Cause

`MCPProvider.List` propagated auth-blocking errors from one external MCP source. Because registry listing/indexing treats provider errors as fatal, one unauthenticated remote MCP server prevented unrelated builtin tools from being listed or invoked.

## Fix

`internal/tools/mcp.go` now treats auth-blocking MCP discovery errors as source-local: the unavailable external MCP source is skipped during dynamic discovery, while the builtin registry remains available. Non-auth discovery errors still fail closed.

Regression coverage was added in `internal/tools/mcp_test.go`.
