## Shared Decisions

## Shared Learnings

- Direct rooms must keep being described as restricted two-party runtime visibility, not cryptographic privacy. RFC 004 signs `surface`, `thread_id`, `direct_id`, and `work_id` when present, but signing is integrity/authenticity only.
- Active RFC/glossary docs now have a Vitest docs scan in `packages/site/lib/protocol-rfc-hard-cut.test.ts`; later tasks can expand or mirror that guard when they update runtime, contracts, site docs, CLI, and tools.
- Old vocabulary still intentionally appears in TechSpec/ADR/task instructions and old/historical RFC artifacts as delete-target or provenance text. Later implementation tasks still need to remove/reject it from their active surfaces.
- `kind:"capability"` is work-linked in runtime validation and now requires `work_id`, matching the TechSpec table even though task_02 only called out `receipt` and `trace` explicitly.
- Public API/CLI/store schema hard-cut work remains for later tasks: task_02 maps existing public `InteractionID` DTO values into internal `WorkID` only where needed to compile, and the current SQLite column name remains `interaction_id` until the store migration task.
- Direct-room known vector for later store/API tests: `DirectRoomIdentity("builders", "coder.sess-abc", "reviewer.sess-xyz")` => `direct_99401d24bee62651d189e5a561785466`.
- Agent prompt metadata now includes `trust:"untrusted"` to match the rendered `<network-message trust="untrusted">` wrapper; acpmock fixtures can match this field.
- Public-thread `capability` messages that carry `work_id` still open directed lifecycle work and therefore need a `to` peer even though their conversation container is `surface:"thread"`.
- Durable conversation writes may represent the sent side in the conversation store, while E2E file-audit prompt assertions should wait on delivered rows for receiver-facing wrapper artifacts.
- `internal/network.ValidateDirectRoomBinding` returns `ErrDirectRoomCollision` when a persisted direct ID does not match the deterministic channel/peer pair; store tasks should use this instead of duplicating hash logic.
- Runtime work lifecycle exposes `ErrWorkContainerMismatch`/`ReasonCodeWorkContainerMismatch`; router rejects cross-container continuations and does not emit rejection receipts for trace/receipt lifecycle messages.
- Store/globaldb does not import `internal/network`; task 04 mirrored the durable conversation ID/direct-room/work validation grammar in `internal/store` DTOs.
- Migration 17 owns final conversation table/index creation so stale legacy `network_audit_log` / `network_timeline_log` tables are rebuilt before final indexes reference `surface`, `thread_id`, `direct_id`, or `work_id`.
- `network_direct_rooms` enforces `UNIQUE(channel, peer_a, peer_b)` and `CHECK(peer_a < peer_b)`; `network_work` is bound to exactly one thread or direct room through SQLite checks and foreign keys with delete restriction.
