# ADR-001: Separate Public Threads from Direct Rooms

## Status

Accepted

## Date

2026-05-04

## Context

AGH Network v0 currently mixes public channel discussion and bilateral handoff semantics across a flat channel timeline plus a peer-room projection. The operator wants Slack-like public thread navigation for N-to-N channel discussion, while preserving restricted one-to-one agent conversations as a distinct product surface. Treating both as one universal container forces visibility, addressing, and workflow semantics into the same primitive and makes the UI ambiguous.

## Decision

AGH Network will model two channel-scoped conversation containers:

- `public_thread` for public N-to-N conversation inside a channel
- `direct_room` for restricted one-to-one conversation inside a channel

`channel` remains the audience/discovery scope above both containers. `peer` remains an identity participant, not a conversation container. Peer activity views remain auxiliary UX filters and are not the same thing as `direct_room`.

`direct_room` is intentionally two-party only in this cut. The runtime derives a stable `direct_id` from `(channel, sorted(peer_a, peer_b))` using a domain-separated SHA-256 hash and persists `(channel, peer_a, peer_b)` under a unique constraint. The resolver must be order-independent and concurrency-safe.

## Alternatives Considered

### Alternative 1: Universal thread container for all conversation

- **Description**: Every message, including bilateral handoffs, lives in one thread model. Peer rooms are derived views only.
- **Pros**: Fewer protocol primitives; one conversation model.
- **Cons**: Conflates public discussion and restricted conversation; makes visibility rules hard to explain; forces bilateral handoff noise into public-thread semantics.
- **Why rejected**: The product goal explicitly wants public thread UX and restricted bilateral direct UX to stay distinct.

### Alternative 2: Keep flat channel timelines plus peer rooms only

- **Description**: Preserve current channel + peer split and add more grouping metadata without first-class public threads.
- **Pros**: Smaller change set; closer to current runtime.
- **Cons**: Does not solve parallel public-topic mixing in channels; keeps the main UX problem intact.
- **Why rejected**: It patches symptoms instead of creating a clean public conversation model.

## Consequences

### Positive

- Public discussion and bilateral handoff become separate, legible product concepts.
- Visibility rules become explicit instead of inferred from message kind.
- Web navigation can become `channel -> public threads` and `channel -> direct rooms`.

### Negative

- The protocol and store gain one more conversation primitive.
- Existing `/network` channel and peer timeline assumptions become obsolete.

### Risks

- Direct-room identity could become underspecified if pair resolution is vague.
- Overlapping thread/direct affordances could still confuse users if naming is sloppy.
- Expanding `direct_room` beyond two peers would invalidate the current `peer_a` / `peer_b` schema and must be treated as a future migration with a new ADR.

## Implementation Notes

- `public_thread` and `direct_room` are both channel-scoped.
- No private-thread feature lands in this cut.
- No group direct-room feature lands in this cut.
- `network_direct_rooms` stores `peer_a` and `peer_b` in lexicographic order with `UNIQUE(channel, peer_a, peer_b)` and `CHECK(peer_a < peer_b)`.
- Resolve-only operations must not update `last_activity_at`; only message writes move activity.
- The implementation must delete the current mental model that “peer room is the primary direct conversation primitive” without replacing it with another ambiguous projection.
