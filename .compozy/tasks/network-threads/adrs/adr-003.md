# ADR-003: Make direct a conversation surface, not a message kind

## Status

Accepted

## Date

2026-05-04

## Context

In AGH Network v0, `kind:"direct"` mixes three dimensions:

- end-user visibility expectations
- addressing semantics
- event type

Once `public_thread` and `direct_room` are both explicit conversation containers, `direct` as a message kind becomes redundant and confusing. Message kinds should describe what happened; conversation surface should describe where it happened.

## Decision

Introduce an explicit conversation surface on conversation-bearing envelopes:

- `surface:"thread"`
- `surface:"direct"`

Delete `kind:"direct"` from the protocol. Keep event kinds focused on event semantics:

- `greet`
- `whois`
- `say`
- `capability`
- `receipt`
- `trace`

Conversation-bearing messages require either:

- `surface:"thread"` with `thread_id`
- `surface:"direct"` with `direct_id`

RFC 004 signed content must include `surface`, `thread_id`, `direct_id`, and `work_id` when those fields are present. Trust validation happens before routing, and routing uses `channel + surface + container id + to`.

## Alternatives Considered

### Alternative 1: Keep kind=direct and add surface temporarily

- **Description**: Add surface but allow `kind:"direct"` for compatibility or reduced churn.
- **Pros**: Smaller implementation delta.
- **Cons**: Preserves the core ambiguity and teaches two overlapping ways to describe the same thing.
- **Why rejected**: The user explicitly chose the full hard cut while the project is still pre-launch.

### Alternative 2: Keep visibility encoded only in `to`

- **Description**: Let `to` imply bilateral restricted conversation without a surface field.
- **Pros**: Minimal wire change.
- **Cons**: `to` is addressing, not visibility or container identity; does not model a direct room cleanly.
- **Why rejected**: Endereçamento and visibilidade should not share one overloaded primitive.

## Consequences

### Positive

- Protocol vocabulary becomes orthogonal: `kind` = event, `surface` = conversation container.
- Public threads can carry targeted work visibly without pretending to be private.
- Direct rooms become explicit protocol/runtime surfaces instead of UI-only projections.

### Negative

- All message validation, examples, CLI help, and docs must change together.
- Existing `direct`-specific lifecycle language has to be rewritten.

### Risks

- If `surface` rules are not enforced consistently, visibility bugs could appear in API/Web layers.
- If direct-room IDs are unstable, bilateral history can fragment.

## Implementation Notes

- `greet` and `whois` remain non-conversation kinds and MUST NOT carry `surface`, `thread_id`, `direct_id`, or `work_id`.
- `say`, `capability`, `receipt`, and `trace` become conversation-bearing kinds.
- Normal direct-room chat is `kind:"say"` with `surface:"direct"`; `DirectBody` is deleted.
- `receipt` and `trace` inherit visibility from the conversation container where their `work_id` is bound.
- In AGH Runtime, control-plane ergonomics may auto-resolve `thread_id` or `direct_id`, but the wire contract stays explicit.
- `kind:"direct"` must be rejected by envelope validation, HTTP, UDS, CLI, native tools, extension Host API, generated contracts, and docs examples.
