# ADR-002: Rename interaction_id to work_id and narrow it to lifecycle-bearing work

## Status

Accepted

## Date

2026-05-04

## Context

`interaction_id` in AGH Network v0 currently carries both “work or conversation progression” language and lifecycle semantics. With first-class public threads and direct rooms, the name becomes actively misleading: conversation grouping belongs to the conversation container, while lifecycle state belongs to directed work. Greenfield alpha allows a hard cut now, before external adoption makes ambiguity expensive.

## Decision

Perform a hard cut rename from `interaction_id` to `work_id` across RFCs, runtime, store, API contracts, CLI, web, docs, prompts, and tests. `work_id` names a lifecycle-bearing unit of directed work inside exactly one conversation container.

`work_id` is not a conversation identifier. A conversation may contain zero, one, or many work units. A work unit never spans multiple conversation containers.

`work_id` is a network-level lifecycle marker in this cut, not a task queue. `task_runs` remains the only durable queue and the only owner of claim-token, lease, heartbeat, complete, fail, release, and cancel transitions.

## Alternatives Considered

### Alternative 1: Keep interaction_id and redefine the docs only

- **Description**: Preserve the field name but explain it as work-only from now on.
- **Pros**: Less implementation churn in the short term.
- **Cons**: Keeps a misleading name in the protocol forever; leaks ambiguity into prompts, docs, and code review.
- **Why rejected**: Greenfield hard-cut timing makes this the cheapest moment to remove the ambiguity.

### Alternative 2: Collapse work and conversation into thread_id

- **Description**: Remove `interaction_id` and let thread identity carry lifecycle.
- **Pros**: Fewer identifiers.
- **Cons**: Threads should not complete/fail/cancel; one thread can contain multiple concurrent work units.
- **Why rejected**: Conflates conversation topology with work lifecycle semantics.

## Consequences

### Positive

- `thread_id` / `direct_id` mean conversation; `work_id` means work.
- Agent prompts and operator docs become materially easier to understand.
- Lifecycle logic stays intact without owning conversation grouping.

### Negative

- This is a breaking rename across every exported surface.
- Existing tests, OpenAPI, and web generated types all need to move in the same change.

### Risks

- Partial rename would leave the repo in an inconsistent state.
- Archived docs and examples may keep stale terminology unless explicitly updated or marked historical.

## Implementation Notes

- Delete targets include all `interaction_id` fields, docs text, validation paths, query params, and examples on supported current surfaces.
- Archived artifacts can remain historical, but active docs and runtime surfaces must hard cut.
- Store `work_id` in `network_work`, bound to exactly one `(channel, surface, thread_id|direct_id)` container.
- Network code must correlate task ingress through `trace_id`, `causation_id`, and task metadata; it must not reuse `work_id` as `task_runs.id`.
- Required hard-cut symbol renames:
  - `Envelope.InteractionID` -> `Envelope.WorkID`
  - `NetworkSendRequest.InteractionID` -> `NetworkSendRequest.WorkID`
  - `NetworkSendPayload.InteractionID` -> `NetworkSendPayload.WorkID`
  - `NetworkMessageEntry.InteractionID` -> `NetworkConversationMessage.WorkID`
  - `InteractionState` -> `WorkState`
  - `Interaction` -> `Work`
  - `OpenInteraction` -> `OpenWork`
  - `ApplyInteractionEnvelope` -> `ApplyWorkEnvelope`
  - `ReasonCodeInteractionClosed` -> `ReasonCodeWorkClosed`
  - CLI `--interaction-id` -> `--work`
- No alias or compatibility reader may accept `interaction_id` on active surfaces.
