## Shared Decisions
- Canonical extension manifest tool resource IDs are currently derived as `ext__<extension_name>__<tool_key>` through `tools.CanonicalToolID`. Later executable-runtime tasks should preserve that canonical identity unless they deliberately introduce a new spec change.
- `ToolID` and `ToolsetID` validate during text/JSON marshal. Encode-before-validate code paths can fail before resource validation when given syntactically invalid IDs.
- Agent `tools` and `deny_tools` now accept exact canonical ToolIDs or namespace-prefix wildcard patterns ending in `*`; agent `toolsets` accepts only canonical ToolsetIDs. The legacy `["*"]` default is removed.
- Tool policy `trusted_sources` entries use `mcp:<server_name>` or `extension:<extension_name>`. Config validation resolves MCP entries against configured MCP servers and syntax-validates extension owners because extension loading is outside `internal/config`.
- Task 03 policy projections enforce descriptor visibility: operator/internal descriptors stay operator-diagnostic only and are absent from session-callable projections with `visibility_denied`.

## Shared Learnings
- The repo had a Mage `Boundaries` task but no Makefile wrapper; Task 01 added `make boundaries` as the documented boundary evidence command.
- `internal/tools` package boundary is additionally covered by a focused `go list` test that rejects daemon/API/CLI/extension/MCP/session/task/skills/network imports.
- `internal/config` now imports `internal/tools` validators directly without creating a cycle; policy and dispatch tasks should consume parsed config rather than re-parsing agent grammar.
- `internal/store` should not import `internal/tools` directly for lineage validation because the existing resources/store relationship creates an import-cycle risk; share grammar only by extracting a neutral package if a future task needs a single implementation.
- Task 03 implements registry indexing, toolset expansion, effective policy evaluation, operator/session projections, and concrete lineage subset validation; actual backend dispatch remains for Task 04.
- Task 04 makes `RuntimeRegistry.Call` the safe executable path; native, extension, and MCP adapters from later tasks should enter through registry handles rather than bypassing schema validation, hooks, budgets, redaction, and event emission.
- Registry-owned tool hook payloads and tool-family matchers are now canonical `tool_id` only. ACP permission-family matchers still retain `tool_name` because that is the permission request vocabulary.
- Task 05 keeps `internal/tools` domain-neutral: static built-in descriptors and generic native provider support live there, while concrete skill/network/task adapters belong in daemon composition.
- Task 06 keeps extension tools manifest-authoritative and cold until a later runtime provider reconciles matching handler/schema/risk proof metadata. Extension manifests now require backend metadata for `resources.tools`; `extension_host` tools require a handler and automatically require `tool.provider`.
- Task 07 makes executable extension-host tools available only through `RuntimeRegistry.Call` and the existing subprocess manager. The daemon must not execute third-party TypeScript handlers in-process; runtime `provide_tools` descriptors are proof for manifest-authoritative descriptors, not the authority.
- Create-extension tool-provider scaffolds use `extension.json` for tool schemas because TOML inline/object schemas do not decode into the current manifest `json.RawMessage` fields.
- Task 09 keeps remote MCP credentials AGH-owned: token material is read only inside `internal/mcp` for outbound headers, registry/API-facing diagnostics are redacted, and mcp-go OAuth helper constructors/stores must not become AGH credential authority.
- Task 10 keeps the hosted MCP bind nonce as a TTL/single-use correlation value only; authorization depends on UDS peer credentials plus expected AGH binary validation and every hosted call still goes through `RuntimeRegistry.Call`.
- Task 10 intentionally injects only the AGH-hosted stdio MCP server into ACP session payloads; configured remote HTTP/SSE/stdio MCP servers remain daemon-owned registry backends, not ACP `mcpServers`.
