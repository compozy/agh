# BUG-003: Smoke P0 deny-all command did not execute any test

**Severity:** High
**Priority:** P0
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** local `tools-registry` worktree, Task 16 QA pass
- **OS:** macOS
- **Browser:** N/A
- **URL:** `internal/tools` smoke regression coverage

## Summary

The smoke regression suite listed `go test ./internal/tools -run TestPolicyDenyAll` for TC-SEC-001, but no `TestPolicyDenyAll` existed. The command exited 0 with `[no tests to run]`, so the P0 smoke lane could silently miss deny-all dispatch coverage across executable backends.

## Reproduction

```bash
go test -race -count=1 ./internal/tools -run TestPolicyDenyAll -v
```

Observed before the fix:

- Output contained `testing: warning: no tests to run`.
- Exit code was 0, which would allow a shell-only smoke runner to treat missing P0 coverage as a pass.

## Expected

TC-SEC-001 must execute a real test proving `deny-all` blocks native Go, extension-host, and MCP backend calls before provider invocation.

## Root cause

Task 15’s smoke command referenced the intended test name before the matching coverage existed. Existing policy and dispatch tests covered adjacent branches but did not provide the exact cross-backend P0 smoke entrypoint.

## Fix

- Added `TestPolicyDenyAll` in `internal/tools/dispatch_test.go`.
- The test covers native Go, extension-host, and MCP descriptors.
- Each subtest asserts `RuntimeRegistry.Call` returns approval-required under `deny-all`, emits a denied event with approval-required/unreachable reasons, and never invokes the provider handle.
- The smoke runner now treats `[no tests to run]` / missing `=== RUN` as a hard failure for Go smoke cases.

## Verification

- `go test -race -count=1 ./internal/tools -run TestPolicyDenyAll -v`
- Evidence log: `.compozy/tasks/tools-registry/qa/logs/fix-policy-deny-all-targeted-rerun.log`

## Impact

- **Users Affected:** Release QA operators relying on the smoke suite to catch deny-all regressions.
- **Frequency:** Always before the fix for TC-SEC-001.
- **Workaround:** None acceptable; missing P0 coverage must fail the lane.
