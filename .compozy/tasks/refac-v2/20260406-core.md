## 1. Package: `internal/session`

**Files**: 17 (9 production, 8 test)  
**Production LOC**: ~2,864  
**Imports**: acp, store, config, workspace  
**Imported by**: apicore, daemon, observe, cli, apisupport, httpapi, udsapi (43 consumers)

### 1.1 Code Smells

#### S-1: `transcript.go` is a standalone domain module squeezed into session (643 LOC)

**Severity**: P1 | **Action**: (B) Package-level split  
**File**: `internal/session/transcript.go`  
**Lines**: 1-643

The transcript rendering subsystem (types `TranscriptMessage`, `TranscriptRole`, `TranscriptToolResult`, `canonicalEventPayload`, and all parsing/assembly logic) is self-contained and has **zero coupling to session state machine or manager logic**. It only needs `store.SessionEvent` as input and `acp.EventType*` constants for matching. This file alone is 22% of the package.

The module handles three distinct event parsing formats (canonical, legacy, loose), each with its own complex branch tree. This is a separate bounded context (event normalization/replay) that happens to be accessed via the Manager.

**Recommendation**: Extract to `internal/transcript/` package. The manager would call `transcript.Assemble(events)`.

#### S-2: `interfaces.go` is a mixed bag of types, adapter, and AgentProcess struct (225 LOC)

**Severity**: P2 | **Action**: (A) File-level split  
**File**: `internal/session/interfaces.go`  
**Lines**: 1-225

This file contains:
- `AgentProcess` struct + constructor + methods (lines 16-136) -- a wrapper DTO
- `AgentDriver`, `EventRecorder`, `Notifier`, `PromptAssembler` interfaces (lines 138-165)
- `ACPDriverAdapter` concrete adapter (lines 167-225)

The `AgentProcess` wrapper re-declares all public fields of `acp.AgentProcess` plus lifecycle function pointers. The `ACPDriverAdapter` is the only consumer of `acp.Driver` and uses `proc.native.(*acp.AgentProcess)` to unwrap -- this is a classic **Middle Man** where session wraps acp.AgentProcess only to re-wrap it.

**Recommendation**: Split into `interfaces.go` (pure interfaces), `agent_process.go` (AgentProcess), and `driver_adapter.go` (ACPDriverAdapter). Consider whether the wrapping is needed at all (see S-7 below).

#### S-3: Duplicate sort logic for `[]*SessionInfo`

**Severity**: P3 | **Action**: (D) Inline fix  
**File**: `internal/session/manager.go:257-263` and `internal/session/query.go:226-243`

Both `List()` and `sortSessionInfos()` implement identical sort comparators:
```go
sort.Slice(out, func(i, j int) bool {
    if out[i].CreatedAt.Equal(out[j].CreatedAt) {
        return out[i].ID < out[j].ID
    }
    return out[i].CreatedAt.Before(out[j].CreatedAt)
})
```
`List()` should call `sortSessionInfos()`.

#### S-4: `marshalAgentEvent` contains legacy reverse-engineering logic (50 LOC)

**Severity**: P2 | **Action**: (D) Inline fix  
**File**: `internal/session/manager_prompt.go:181-233`

`marshalAgentEvent` parses `event.Raw` back into `map[string]any`, then calls `legacyToolName`, `buildToolResult`, etc. from transcript.go. This is **forward-path code re-using backward-path (replay) logic**, coupling the recording path to transcript parsing. The canonical envelope should be built directly from the `AgentEvent` fields without reverse-engineering raw JSON.

#### S-5: `Create` and `Resume` share ~70% structure (duplication)

**Severity**: P1 | **Action**: (D) Inline fix  
**File**: `internal/session/manager_lifecycle.go:17-296`

`Create` (lines 17-131) and `Resume` (lines 177-296) follow the same flow: resolve workspace, resolve agent, assemble prompt, generate/reuse ID, reserve slot, create session dir, open store, start driver, activate, watch, notify. They differ only in metadata source and whether `ResumeSessionID` is set. This is textbook **Duplicated Code** across two 115-line functions.

**Recommendation**: Extract a shared `startSession(ctx, sessionSetup)` where `sessionSetup` carries the variant inputs. This would eliminate ~80 lines.

### 1.2 Architectural Boundaries

#### S-6: `session/` owns both runtime orchestration AND query/read-model concerns

**Severity**: P2 | **Action**: (B) Package-level split

The manager handles:
1. Active session lifecycle (create, stop, resume, prompt, approve permission)
2. Query operations against on-disk metadata (ListAll, Status, Events, History, Transcript)

These are two distinct responsibilities. The query path opens SQLite databases directly via `openQueryRecorder`, reads meta.json files, and merges with in-memory state. The lifecycle path owns the process, state machine, and event recording.

**Recommendation**: The query code could move to a `sessionquery/` package or stay in session but behind a separate interface (e.g., `SessionReader`). Not critical at current LOC, but would improve testability.

#### S-7: `AgentProcess` wrapper creates unnecessary indirection

**Severity**: P2 | **Action**: (C) Extraction / Inline

`session.AgentProcess` wraps `acp.AgentProcess` 1:1, adding function-pointer-based lifecycle hooks (`waitFn`, `stderrFn`, `approvePermissionFn`). The only reason for this wrapper is to avoid importing `acp.AgentProcess` directly. But `session/interfaces.go` already imports `acp` for `acp.ACPCaps`, `acp.ApproveRequest`, and `acp.AgentEvent`. The wrapper creates a **Middle Man**: 6 methods that simply delegate.

The `native any` field + type assertion pattern (`proc.native.(*acp.AgentProcess)`) is fragile and defeats Go's type safety.

**Recommendation**: Use `acp.AgentProcess` directly. The `AgentDriver` interface already isolates the boundary.

### 1.3 Coupling Analysis

| Metric | Value |
|--------|-------|
| Afferent coupling (imports by others) | **43 files** (very high -- most imported package) |
| Efferent coupling (imports of others) | acp, store, config, workspace (4 packages) |

**Assessment**: `session` is the most-imported package in the system, so changes here have high blast radius. The efferent coupling to 4 packages is manageable but means this package cannot be tested without mocking all four.

### 1.4 SOLID Violations

- **SRP**: The Manager has 3 reasons to change: lifecycle logic, query logic, transcript rendering. (See S-1, S-6)
- **ISP**: `EventRecorder` (5 methods) is consumed but not overly wide. `Notifier` (3 methods) is fine. `AgentDriver` (4 methods) is appropriately scoped.
- **DIP**: `NewManager()` defaults to `acp.New()` and `store.OpenSessionDB()` -- concrete dependencies in the constructor default. These are injected via functional options, so the violation is mild. However, `config.ResolveHomePaths()` and `config.EnsureHomeLayout()` are called directly in the constructor, coupling the manager to filesystem side-effects at construction time.

## 2. Package: `internal/acp`

**Files**: 11 (6 production, 5 test)  
**Production LOC**: ~2,960  
**Imports**: config, testutil, acp-go-sdk  
**Imported by**: session, daemon, apicore, httpapi, udsapi, observe, cli (37 consumers)

### 2.1 Code Smells

#### A-1: `handlers.go` is the largest file in all four packages (774 LOC)

**Severity**: P0 | **Action**: (A) File-level split  
**File**: `internal/acp/handlers.go`

This file contains:
- Wire types (5 structs, ~60 LOC)
- Inbound JSON-RPC dispatch (`handleInbound`, ~90 LOC)
- File system handlers (read/write, ~30 LOC)
- Permission handler (`handleRequestPermission`, ~100 LOC)
- Session update handler + event translation (~90 LOC)
- Terminal management subsystem (terminalManager + managedTerminal + 7 methods, ~200 LOC)
- Token usage conversion (~50 LOC)
- Utility functions (~80 LOC)

The terminal management subsystem alone is a distinct domain that manages subprocess lifecycle, output buffering, and process signals. It shares nothing with permission handling or event translation.

**Recommendation**: Split into:
- `handlers.go` -- dispatch + fs handlers (~120 LOC)
- `terminal.go` -- terminalManager + managedTerminal (~250 LOC)
- `event_translate.go` -- translateSessionUpdate + token usage conversion (~100 LOC)
- Move wire types to `types.go` or a dedicated `wire_types.go`

#### A-2: `permission.go` mixes policy engine + pending permission state machine (546 LOC)

**Severity**: P1 | **Action**: (A) File-level split  
**File**: `internal/acp/permission.go`

This file contains two unrelated subsystems:
1. **Static permission policy** (`permissionPolicy`, `authorize`, `isAllowed`, `resolvePath`, etc.) -- path resolution and mode-based authorization
2. **Interactive pending permission state** (`registerPendingPermission`, `ResolvePermission`, `lookupPendingPermissionLocked`, etc.) -- runtime request/response lifecycle on `AgentProcess`

The pending permission methods are on `AgentProcess` and belong with the process lifecycle, not with the policy evaluation engine.

**Recommendation**: Split into `permission_policy.go` (static engine) and keep pending-permission methods in `types.go` or a new `process_permissions.go`.

#### A-3: `handleRequestPermission` is 100+ LOC with deep nesting

**Severity**: P2 | **Action**: (D) Inline fix  
**File**: `internal/acp/handlers.go:228-324`

This function handles 3 branches (non-interactive, interactive-resolved, interactive-timeout/cancelled) with nearly identical event emission code copied 3 times:
```go
p.emitPromptEvent(AgentEvent{
    Type:       EventTypePermission,
    SessionID:  string(request.SessionId),
    TurnID:     turnID,
    RequestID:  requestID,
    Timestamp:  timeNowUTC(),
    Title:      title,
    ToolCallID: strings.TrimSpace(string(request.ToolCall.ToolCallId)),
    Action:     string(permissionRequestToolGrant),
    Resource:   resource,
    Decision:   string(appliedDecision),
    Raw:        cloneRawJSON(raw),
})
```
This struct literal is duplicated **4 times** with minor field variations.

**Recommendation**: Extract `emitPermissionEvent(turnID, requestID, title, resource, decision, raw)` helper.

#### A-4: `handleInbound` is a 90-line switch statement

**Severity**: P2 | **Action**: (D) Inline fix  
**File**: `internal/acp/handlers.go:104-194`

The inbound dispatch has 9 cases, each with identical unmarshal + call + error-conversion boilerplate. The repetitive pattern is:
```go
var request SomeType
if err := json.Unmarshal(params, &request); err != nil {
    return nil, acpsdk.NewInvalidParams(...)
}
response, err := p.handleSomething(ctx, request)
if err != nil { return nil, requestError(err) }
return response, nil
```
This is a classic **Repeated Switch/Dispatch** smell.

**Recommendation**: Extract a generic dispatch helper or use a handler map. However, given Go's type system, the switch is idiomatic. The boilerplate could be reduced with a helper like `dispatchHandler[Req, Resp](params, handler)`.

#### A-5: `types.go` has grown to 429 LOC, mixing process state with DTOs

**Severity**: P2 | **Action**: (A) File-level split  
**File**: `internal/acp/types.go`

Contains:
- Event type constants (12 constants)
- `StartOpts`, `PromptRequest`, `ACPCaps` -- public API DTOs
- `TokenUsage` with Merge/IsZero -- domain type
- `AgentEvent` -- stream item DTO
- `AgentProcess` (34 fields, 200+ LOC of methods) -- core runtime struct
- `activePromptState`, `pendingPermission`, `lockedBuffer` -- internal state types
- Utility functions (`chooseInt64`, `chooseFloat64`, etc.)

**Recommendation**: Split into:
- `types.go` -- public DTOs (StartOpts, PromptRequest, ACPCaps, AgentEvent, TokenUsage)
- `process.go` -- AgentProcess struct and all its methods
- Keep utility functions with their primary consumer

### 2.2 Architectural Boundaries

#### A-6: Terminal management is an independent subsystem

**Severity**: P2 | **Action**: (B) Package-level split (future)

The `terminalManager` + `managedTerminal` + `terminalOutputWriter` types (200+ LOC) handle subprocess execution, output capture with bounded buffer, and exit status tracking. This subsystem is only connected to the rest of `acp` through `AgentProcess.terminals` and permission checks. It could be extracted to `internal/acp/terminal/` or kept as a file-level split for now.

### 2.3 Coupling Analysis

| Metric | Value |
|--------|-------|
| Afferent coupling | **37 files** (second most imported) |
| Efferent coupling | config (1 internal package), acp-go-sdk (1 external) |

**Assessment**: Low efferent coupling (good). High afferent coupling means the public API surface area matters. The `AgentEvent` and `TokenUsage` types are consumed across the entire system.

### 2.4 DRY Violations

- `cloneRawJSON` duplicated as `cloneRawMessage` in session/transcript.go (see cross-package S-CP-2)
- `timeNowUTC()` is a module-level function that could use `time.Now().UTC()` directly
- Permission event emission duplicated 4x (see A-3)

## 3. Package: `internal/store`

**Files**: 18 (10 production, 8 test)  
**Production LOC**: ~3,640  
**Imports**: fileutil, workspace, testutil  
**Imported by**: session, observe, daemon, apicore, cli, workspace (46 consumers)

### 3.1 Code Smells

#### ST-1: `migrate_workspace.go` is a 586-LOC migration file

**Severity**: P1 | **Action**: Review for deletion  
**File**: `internal/store/migrate_workspace.go`

Given the project's **zero legacy tolerance** policy ("Never write migration, compat, or defensive code for old state -- delete the old thing instead of working around it"), this entire file is a candidate for deletion. It handles migrating from `workspace` column to `workspace_id` foreign key, reconciling legacy session metadata, and creating intermediate tables.

Per CLAUDE.md: "No production users exist. Never sacrifice code quality for backward compatibility."

**Recommendation**: Delete this file entirely. If the old schema is encountered, `openGlobalSQLite` should either recreate the database or fail fast.

#### ST-2: `types.go` is a mega-types file with 328 LOC of flat structs

**Severity**: P2 | **Action**: (A) File-level split  
**File**: `internal/store/types.go`

Contains 14 types covering 4 distinct domains:
1. Session events: `SessionEvent`, `EventQuery`, `TurnHistory`, `TokenUsage`
2. Session index: `SessionInfo`, `SessionListQuery`, `SessionStateUpdate`, `SessionMeta`
3. Observability: `EventSummary`, `EventSummaryQuery`, `TokenStats`, `TokenStatsUpdate`, `TokenStatsQuery`
4. Permissions: `PermissionLogEntry`, `PermissionLogQuery`
5. Reconciliation: `ReconcileResult`

Each has a `Validate()` method following the same pattern.

**Recommendation**: Split by domain: `types_session.go`, `types_observe.go`, `types_permission.go`.

#### ST-3: `GlobalDB` implements 2 interfaces from 2 different packages

**Severity**: P1 | **Action**: Analysis finding  
**File**: `internal/store/global_db.go:21`

```go
var _ SessionRegistry = (*GlobalDB)(nil)
var _ aghworkspace.WorkspaceStore = (*GlobalDB)(nil)
```

`GlobalDB` implements both `store.SessionRegistry` (11 methods!) and `workspace.WorkspaceStore`. This violates ISP -- consumers that only need workspace CRUD must accept a type that also carries session/observe/permission methods.

The `SessionRegistry` interface itself has **11 methods** spanning 4 domains:
- Session CRUD: RegisterSession, UpdateSessionState, ListSessions, ReconcileSessions
- Observability: WriteEventSummary, ListEventSummaries, UpdateTokenStats, ListTokenStats
- Permissions: WritePermissionLog, ListPermissionLog
- Lifecycle: Close

**Recommendation**: Split `SessionRegistry` into smaller interfaces:
- `SessionIndexer` (Register, Update, List, Reconcile)
- `EventSummaryStore` (Write, List)
- `TokenStatsStore` (Update, List)
- `PermissionLogStore` (Write, List)

Consumers can accept only the interface they need.

#### ST-4: `sql_helpers.go` has boilerplate nullable helpers (183 LOC)

**Severity**: P3 | **Action**: (D) Inline fix  
**File**: `internal/store/sql_helpers.go`

Contains 6 `nullable*` functions, 3 `null*` functions, and `clause` builders. These are fine as internal helpers but contribute to the overall LOC. The `newID` function is duplicated (see cross-package finding).

#### ST-5: `global_db_workspace.go` has overly defensive workspace normalization

**Severity**: P3 | **Action**: (D) Inline fix  
**File**: `internal/store/global_db_workspace.go`

`normalizeWorkspaceForInsert` and `normalizeWorkspaceForUpdate` share 90% of their logic with `normalizeWorkspaceRecord`. The two wrapper functions differ only in ID and timestamp defaulting. This is acceptable but worth noting.

### 3.2 Architectural Boundaries

#### ST-6: `store` handles 5 distinct domains in one package

**Severity**: P1 | **Action**: (B) Package-level split

The package manages:
1. **Per-session event storage** (SessionDB, writer loop, events table)
2. **Global session index** (sessions table, CRUD, reconciliation)
3. **Global observability** (event_summaries, token_stats tables)
4. **Global permission audit** (permission_log table)
5. **Workspace persistence** (workspaces table, implements workspace.WorkspaceStore)
6. **Session metadata** (meta.json read/write)
7. **Migration** (legacy schema migration)

This is the most SRP-violating package in the group. Each domain could change independently.

**Recommendation**: Consider splitting into:
- `store/sessiondb/` -- per-session SQLite (SessionDB, schema, writer loop)
- `store/globaldb/` -- global database (GlobalDB, session index, observe, permissions)
- Keep `store/` as the shared types/helpers package

Or, more conservatively, split the types and keep the implementations co-located since they share `sql_helpers.go`.

#### ST-7: Circular-ish dependency: store -> workspace -> store

**Severity**: P2 | **Action**: Analysis finding

`store/global_db_workspace.go` imports `workspace` for `workspace.Workspace` type and error sentinels.
`workspace/resolver.go` likely depends on `store` for the `WorkspaceStore` interface.

This is not technically circular because workspace defines the interface and store implements it (correct DI direction). But the types in `workspace.Workspace` are used as DTOs in both directions, creating tight coupling.

### 3.3 SOLID Violations

- **SRP**: Package has 5+ reasons to change (see ST-6)
- **ISP**: `SessionRegistry` has 11 methods (see ST-3). Severely violates ISP.
- **OCP**: Adding a new global table (e.g., for skills state) requires modifying `GlobalDB`, `schema.go`, and `types.go`. Not extensible.
- **DIP**: `store` depends on `workspace` for workspace types. The direction is correct (store implements workspace's interface), but the type coupling is deep.

## 4. Package: `internal/config`

**Files**: 12 (6 production, 6 test)

### 4.1 Code Smells

#### C-1: `config.go` is 511 LOC with 12 Validate methods

**Severity**: P2 | **Action**: (A) File-level split  
**File**: `internal/config/config.go`

The file mixes:
1. 12 config struct definitions (DaemonConfig, HTTPConfig, etc.)

Each config section has a parallel overlay type with `*T` pointer fields and an `Apply` method. This is 10 overlay types with 10 Apply methods. The pattern is correct for TOML partial-merge semantics, but it's verbose. No practical alternative in Go without reflection.

#### C-4: `agent.go` mixes agent parsing, workspace discovery, and skill directory logic

**Severity**: P2 | **Action**: (A) File-level split  
**File**: `internal/config/agent.go` (293 LOC)

Contains:
1. `AgentDef` type + YAML frontmatter parser (true agent concerns)
2. `WorkspaceDiscoveryRoot` + `WorkspaceDiscoveryRoots()` -- workspace discovery concern
3. `LoadWorkspaceAgentDefs` -- workspace-scoped agent loading
4. `AgentsDir()`, `SkillsDir()` -- directory resolution for discovery roots

The `WorkspaceDiscoveryRoot` type and `SkillsDir()` method are consumed by the skills subsystem, not the agent parsing subsystem. The discovery root concept should arguably live in workspace or a shared discovery package.

**Recommendation**: Move `WorkspaceDiscoveryRoot` and `WorkspaceDiscoveryRoots` to a shared location or keep in config but split to `discovery.go`.

### 4.2 Architectural Boundaries

#### C-5: `config` is a "God package" for the entire system

**Severity**: P1 | **Action**: Analysis finding

With **72 importing files**, `config` is the most imported package. It's essentially the system's type registry, providing:
- Configuration types + loading
- `HomePaths` (filesystem layout)
- `AgentDef` (agent definition parsing)
- `ProviderConfig` + `ResolvedAgent` (provider resolution)
- `MCPServer` (MCP server definition)
- `PermissionMode` (permission enum)
- `WorkspaceDiscoveryRoot` (workspace discovery)
- Bootstrap functions (install flow)

This package has **7 distinct responsibilities**. Any change to `config` invalidates the entire build cache.

**Recommendation**: Consider extracting:
- `internal/agentdef/` -- AgentDef parsing, validation, frontmatter
- `internal/discovery/` -- WorkspaceDiscoveryRoot, discovery roots
- Keep `config/` for TOML config loading, HomePaths, and provider resolution

#### C-6: `config.ResolveAgentName` is a 6-line function that doesn't belong

**Severity**: P3 | **Action**: (C) Extraction  
**File**: `internal/config/bootstrap.go:32-40`

This function is called only from `session/manager_workspace.go`. It's a simple "use name, or fall back to config default" function. It's in `bootstrap.go` but has nothing to do with bootstrap.

**Assessment**: Config has zero efferent internal coupling (correct -- leaf package). But the extreme afferent coupling means this package is the hottest change target in the system. Any type change here forces recompilation of everything.

## 5. Cross-Package Findings

### CP-1: Duplicated `newID` function (P1, Action: C)

**Files**: `internal/session/manager_helpers.go:115-129` and `internal/store/sql_helpers.go:169-183`

These are **character-for-character identical** implementations:
```go
func newID(prefix string) string {
    var random [8]byte
    if _, err := rand.Read(random[:]); err != nil {
        now := time.Now().UTC().UnixNano()
        ...
    }
    ...
}
```

**Recommendation**: Extract to a shared utility package (e.g., `internal/idgen/` or add to existing `internal/testutil/` as a non-test function, or create `internal/xid/`).

### CP-2: Duplicated `cloneRawJSON` / `cloneRawMessage` (P2, Action: C)

**Files**: `internal/acp/handlers.go:753-759` (`cloneRawJSON`) and `internal/session/transcript.go:595-602` (`cloneRawMessage`)

Identical implementation with different names:
```go
func cloneRawJSON(value json.RawMessage) json.RawMessage {
    if len(value) == 0 { return nil }
    cloned := make([]byte, len(value))
    copy(cloned, value)
    return cloned
}
```

**Recommendation**: Consolidate into one shared location or have session import from acp (though that increases coupling). Best to extract to a `internal/jsonutil/` package.

### CP-3: Duplicated `TokenUsage` type (P0, Action: C)

**Files**: `internal/acp/types.go:120-133` and `internal/store/types.go:64-77`

Both packages define a `TokenUsage` struct with **identical fields**:
```go
type TokenUsage struct {
    TurnID           string
    InputTokens      *int64
    OutputTokens     *int64
    TotalTokens      *int64
    ThoughtTokens    *int64
    CacheReadTokens  *int64
    CacheWriteTokens *int64
    ContextUsed      *int64
    ContextSize      *int64
    CostAmount       *float64
    CostCurrency     *string
    Timestamp        time.Time
}
```

The `acp.TokenUsage` has `Merge` and `IsZero` methods; `store.TokenUsage` has `Validate`. The session manager manually copies field-by-field between them in `manager_prompt.go:160-173`.

**Recommendation**: Define `TokenUsage` in one package. Since `store` is the persistence layer, it should own the type. `acp` would import `store.TokenUsage` or a shared `internal/types/` package. Alternatively, since acp has the richer type (with Merge/IsZero), move the canonical type to acp and have store use it.

### CP-4: Duplicated `SessionInfo` type (P1, Action: C)

**Files**: `internal/session/session.go:44-57` and `internal/store/types.go:88-98`

Both define `SessionInfo` with overlapping fields. The session version has `acp.ACPCaps` and `Workspace string`; the store version has `ACPSessionID *string`. These are different read models for different layers, which is intentional. However, the manual mapping between them creates coupling.

**Recommendation**: This is likely intentional (session has runtime fields, store has persistence fields). Document the distinction. The field-by-field copy in `sessionInfoFromMeta` is acceptable.

### CP-5: `normalizeSessionType` duplicated in session and store (P2, Action: C)

**Files**: `internal/session/session.go:358-367` and `internal/store/sql_helpers.go:58-64`

The session version works with `SessionType` enum; the store version works with `string`. They normalize the same concept but in different type systems.

**Recommendation**: Extract session type constants to a shared location, or have store call through to session's normalizer.

### CP-6: session depends on store for meta.json I/O but also re-reads it independently (P2, Action: D)

The session manager calls `store.ReadSessionMeta`, `store.WriteSessionMeta`, and `store.SessionDBFile` / `store.SessionMetaFile` directly. This means session has deep knowledge of the on-disk layout. This is acceptable given the flat package structure, but creates coupling that would complicate extraction.

## 6. Subpackage Grouping Recommendation

### Should `session`, `acp`, and `store` live under an `agent/` parent?

**Analysis**:

The cohesion between these packages is real:
- `session` depends on `acp` and `store` (primary consumer of both)
- `acp` and `store` are rarely used independently of `session`
- All three form the "agent runtime core" of the system

However, **grouping them under `agent/` would not improve the architecture** for several reasons:

1. **`config` has zero coupling to the other three** -- it's a standalone leaf package. Including it in an `agent/` group would be incorrect.

2. **`store` serves multiple consumers beyond session** -- it's imported by 46 files including observe, daemon, apicore, and workspace. It's not purely an "agent" concern.

3. **`acp` is the protocol layer** -- it represents the external ACP protocol, not a subdomain of "agent management."

4. **Go subpackages don't share access** -- `agent/session` importing `agent/store` has no advantage over `internal/session` importing `internal/store`. There's no package-level visibility benefit.

### Recommended grouping instead:

Rather than creating a parent package, the better refactoring is **extracting shared types**:

```
internal/
  config/         -- Config loading, HomePaths, PermissionMode (standalone leaf)
  agentdef/       -- AgentDef parsing, frontmatter, validation (extracted from config)
  acp/            -- ACP protocol client, subprocess management
    terminal/     -- Terminal subsystem (extracted from handlers.go) [optional]
  store/          -- SQLite persistence
    sessiondb/    -- Per-session event storage [optional future split]
  session/        -- Session lifecycle manager
  transcript/     -- Event replay/normalization (extracted from session)
  idgen/          -- Shared ID generation (extracted from session + store)
```

This preserves the flat-with-discipline architecture while addressing the real problems (duplication, SRP violations, oversized files).
