# API Layer Refactoring Analysis

**Date**: 2026-04-06
**Scope**: `internal/apicore`, `internal/httpapi`, `internal/udsapi`, `internal/apisupport`, `internal/apitest`
**Total LOC**: ~12,565 (production + test across 59 Go files)

> **Corrections applied (2026-04-06)**: Package naming updated to `api/httpapi`, `api/udsapi`, `api/contract` (see subpackage grouping section). File counts in some sections are stale -- httpapi has 11 prod/10 test files, udsapi has 12 prod/8 test files in the current tree. The summary document is the authoritative source for corrected data.

---

## Executive Summary

The API layer is already well-factored relative to most codebases -- `apicore` successfully extracts shared handler logic so that `httpapi` and `udsapi` are thin transport wrappers. However, analysis reveals several significant structural issues:

1. **13 empty placeholder files** (1-line `package` declarations) across httpapi/udsapi indicate a prior file-per-domain split that was superseded by the apicore extraction but never cleaned up
2. **Massive test duplication** -- httpapi and udsapi have near-identical handler tests (~860 and ~930 lines respectively) testing the same BaseHandlers logic
3. **`BaseHandlers` + `BaseHandlerConfig` god struct** (14 fields each, 604 lines in handlers.go) violates SRP -- it handles sessions, agents, observe, daemon status, memory, and workspaces all in one struct
4. **`apisupport` is a vestigial package** -- a single 147-line file that should be inlined into `apicore`
5. **Five flat packages should become subpackages** under a common `api/` parent

**Findings by severity**: P0: 1 | P1: 5 | P2: 7 | P3: 5

## Package Analysis

### 1. `internal/apicore` (9 source files, 2,047 LOC production)

#### Finding 1.1: `BaseHandlers` God Struct
- **Severity**: P1 (high)
- **Action**: (B) Package-level split
- **Smell**: Large Class, Divergent Change, SRP violation
- **Location**: `handlers.go:44-63` (BaseHandlers struct), `handlers.go:126-578` (handler methods), `memory.go`, `workspaces.go`

The `BaseHandlers` struct has 14 fields and serves as the handler for 6 distinct API domains: sessions, agents, observe, daemon, memory, and workspaces. Adding a new domain requires modifying this struct and its config. The struct is changed for at least 6 unrelated reasons.

`BaseHandlerConfig` is an exact field-for-field mirror of `BaseHandlers` (lines 28-42 vs 44-63), adding 15 lines of pure boilerplate duplication. The `NewBaseHandlers` constructor (lines 66-107) copies every field one-by-one.

**Recommendation**: Either eliminate `BaseHandlerConfig` in favor of functional options (matching the pattern httpapi/udsapi already use), or keep the config but generate the handler struct fields from it. Consider splitting BaseHandlers into domain-specific handler groups (SessionHandlers, MemoryHandlers, etc.) that share a common `HandlerContext` holding the cross-cutting concerns (logger, transport name, etc.).

#### Finding 1.2: `parsers.go` Depends on `gin.Context`
- **Severity**: P2 (medium)
- **Action**: (D) Inline fix
- **Smell**: Coupling to transport framework
- **Location**: `parsers.go:9` (imports gin), `parsers.go:16-37` (ParseSessionEventQuery), `parsers.go:40-57` (ParseObserveEventQuery)

`ParseSessionEventQuery` and `ParseObserveEventQuery` accept `*gin.Context` to extract query parameters. This couples the "core" package to Gin. The lower-level parsers (`ParseOptionalTime`, `ParseOptionalInt`, `ParseOptionalInt64`) correctly accept `string` values instead.

**Recommendation**: Refactor the two query parsers to accept a `url.Values` or individual string parameters, making apicore truly framework-agnostic. The Gin-specific extraction (`c.Query(...)`) should happen at the call site.

#### Finding 1.3: `errors.go` Passes Through to `apisupport`
- **Severity**: P2 (medium)
- **Action**: (C) Extraction / inline
- **Smell**: Middle Man, unnecessary delegation
- **Location**: `errors.go:33-40`

```go
func StatusForSessionError(err error) int {
    return apisupport.StatusForSessionError(err)
}
func StatusForWorkspaceError(err error) int {
    return apisupport.StatusForWorkspaceError(err)
}
```

These are pure pass-through functions. `apicore` already imports `apisupport`, but these functions add no value -- they just delegate. Similarly, `workspaces.go:184-198` has 6 private pass-through functions to `apisupport`:

```go
func filterSessionInfosByWorkspaceID(...) { return apisupport.FilterSessionInfosByWorkspaceID(...) }
func validateAbsolutePath(...)            { return apisupport.ValidateAbsolutePath(...) }
func validateAbsolutePaths(...)           { return apisupport.ValidateAbsolutePaths(...) }
func trimStringSlice(...)                 { return apisupport.TrimStringSlice(...) }
```

**Recommendation**: Inline `apisupport` into `apicore` (see Finding 4.1). All these pass-throughs become direct calls.

#### Finding 1.4: `memory.go` Is the Largest File (413 LOC)
- **Severity**: P2 (medium)
- **Action**: (A) File-level split
- **Smell**: Large module
- **Location**: `memory.go:1-413`

Contains 7 handler methods, 9 private helper functions, and 2 exported utility functions. The memory domain is complex enough (scope resolution, workspace inference, health aggregation, consolidation) to warrant its own file structure within apicore.

**Recommendation**: This is well-organized within the file, but if BaseHandlers were split into domain groups (Finding 1.1), memory would naturally become its own handler group.

#### Finding 1.5: Duplicated Exported/Unexported Function Pairs
- **Severity**: P3 (low)
- **Action**: (D) Inline fix
- **Smell**: Duplicate code, unnecessary indirection
- **Location**: `memory.go:217-220`, `memory.go:339-341`, `memory.go:344-347`, `memory.go:381-383`, `memory.go:398-399`

Several functions exist in both exported and unexported forms:
- `ResolveMemoryLocation` -> `resolveMemoryLocation` (line 218-219)
- `MemoryHealthWorkspaces` -> `memoryHealthWorkspaces` (line 339-341)
- `ResolveMemoryWriteScope` -> `resolveMemoryWriteScope` (line 344-347)
- `ParseOptionalMemoryScope` -> `parseOptionalMemoryScope` (line 381-383)
- `ResolveMemoryWorkspace` -> `resolveMemoryWorkspace` (line 398-399)

The exported versions exist solely to enable testing from httpapi/udsapi test packages. This is a test smell -- it exposes internal API surface just for test access.

**Recommendation**: These exported wrappers exist because httpapi/udsapi tests need to call the underlying logic. If tests were consolidated (Finding 3.1), these could be removed.

### 2. `internal/httpapi` (10 source files, 1,377 LOC production; 11 test files, 3,515 LOC test)

#### Finding 2.1: 6 Empty Placeholder Files (Dead Code)
- **Severity**: P1 (high)
- **Action**: (D) Inline fix (delete)
- **Smell**: Dead code, vestigial structure
- **Location**: `agents.go`, `daemon.go`, `memory.go`, `observe.go`, `stream.go`, `workspaces.go` (all 1 line: `package httpapi`)

These files contain only the package declaration. They were likely created during the original file-per-domain split but became empty when handler logic was extracted to `apicore.BaseHandlers`. They add noise to the package listing and confuse the file count (21 files reported, only 10 have content).

**Recommendation**: Delete all 6 files immediately. Zero risk.

#### Finding 2.2: `prompt.go` Contains Duplicated Payload Types
- **Severity**: P1 (high)
- **Action**: (C) Extraction
- **Smell**: DRY violation, Divergent Change
- **Location**: `prompt.go:33-63`

`agentEventPayload` and `tokenUsagePayload` in prompt.go are near-clones of `apicore.AgentEventPayload` and `apicore.TokenUsagePayload`, differing only in that timestamps are serialized as strings instead of `time.Time`. This leads to the adapter function `agentEventPayloadFromEvent` (lines 375-397) which field-by-field copies from the apicore payload and then formats the timestamp.

Similarly `tokenUsagePayloadFromUsage` (lines 399-422) copies all fields from `apicore.TokenUsagePayloadFromUsage` and reformats the timestamp.

**Recommendation**: Add a `TimestampString` field or use a custom JSON marshaler on the apicore types. Alternatively, accept this as intentional transport-specific formatting, but consolidate the adapter into apicore as a variant conversion function.

#### Finding 2.3: `shared.go` Is a Thin Alias File
- **Severity**: P3 (low)
- **Action**: (D) Inline fix
- **Smell**: Middle Man
- **Location**: `shared.go:1-31`

Contains 4 type aliases and 5 wrapper functions that delegate to apicore. While this provides package-local naming convenience, it adds an indirection layer with no logic. The wrappers:
```go
func respondError(c, status, err) { apicore.RespondError(c, status, err, true) }
func statusForSessionError(err)    { return apicore.StatusForSessionError(err) }
func prepareSSE(c)                 { return apicore.PrepareSSE(c) }
func writeSSE(writer, msg)         { return apicore.WriteSSE(writer, msg) }
func writeSSERaw(writer, id, raw)  { return apicore.WriteSSERaw(writer, id, raw) }
```

The only one with transport-specific logic is `respondError` (hardcodes `maskInternalErrors=true`). The rest are pure pass-throughs.

**Recommendation**: Keep `respondError` (it carries transport policy). Use `apicore.*` directly for the rest, or keep as convenience if the team prefers shorter names.

#### Finding 2.4: `shared_test.go` Contains Heavy Alias Boilerplate
- **Severity**: P2 (medium)
- **Action**: (C) Extraction
- **Smell**: DRY violation, test infrastructure coupling
- **Location**: `shared_test.go:1-68` (12 type aliases + 8 wrapper functions)

The test file creates 12 type aliases (`sessionPayload`, `agentPayload`, `workspacePayload`, etc.) and 8 wrapper functions (`statusForMemoryError`, `payloadJSON`, `observeEventAfterCursor`, etc.) just to bring apicore types into the httpapi test namespace. The nearly identical file exists in udsapi (`shared_test.go:1-87`, 14 type aliases + 14 wrapper functions).

**Recommendation**: These would be unnecessary if tests were consolidated (see Finding 3.1), or if the transport packages imported apicore types directly in tests.

#### Finding 2.5: `server.go` Duplicates Default-Setting Logic
- **Severity**: P2 (medium)
- **Action**: (D) Inline fix
- **Smell**: Duplicated conditional chains
- **Location**: `server.go:207-297`

The `New()` constructor has a long sequence of nil/zero checks with defaults (lines 229-277) that closely mirrors the same pattern in `udsapi/server.go:194-272`. Both packages: set default logger, default `now()`, default `pollInterval`, default `startedAt`, default `agentLoader`, validate required fields (sessions, observer, workspaces), then build handlers.

**Recommendation**: Extract a shared `ServerConfig` builder or validation function into apicore that handles the common defaults and required-field validation. Transport-specific differences (socket path vs TCP host/port) stay local.

#### Finding 2.6: `server.go` `handlerConfig` Mirrors `apicore.BaseHandlerConfig`
- **Severity**: P3 (low)
- **Action**: (D) Inline fix
- **Smell**: Data Clump
- **Location**: `server.go:78-93`

`handlerConfig` has 12 fields that are a subset of `apicore.BaseHandlerConfig` (plus `staticFS`). The `newHandlers` function (lines 486-515) simply copies these fields into `apicore.BaseHandlerConfig`. This is an intermediate struct that adds no value.

**Recommendation**: Use `apicore.BaseHandlerConfig` directly (add `StaticFS` to it, or pass separately).

### 3. `internal/udsapi` (8 source files, 644 LOC production; 12 test files, 3,062 LOC test)

- **Severity**: P0 (critical)
- **Action**: (C) Extraction
- **Smell**: Duplicated Code (cross-package), Shotgun Surgery
- **Location**: `handlers_test.go` (929 LOC) vs `httpapi/handlers_test.go` (859 LOC)

| `TestListSessionsHandlerFiltersByWorkspace` | 38 | 37 |

| `TestListWorkspacesHandlerReturnsRows` | 33 | 28 |
| `TestGetWorkspaceHandlerReturnsDetail` | 58 | 58 |
| `TestUpdateWorkspaceHandlerUpdatesWorkspace` | 48 | 45 |
| `TestDeleteWorkspaceHandlerReturnsNoContent` | 18 | 18 |
| `TestResolveWorkspaceHandlerReturnsWorkspace` | 38 | 40 |
| `TestStopSessionHandlerReturnsStopped` | 18 | 18 |
| `TestSessionEventsHandlerReturnsFilteredEvents` | ~50 | ~62 |
| `TestSessionHistoryHandlerReturnsTurns` | ~40 | ~42 |
| `TestSessionTranscriptHandlerReturnsMessages` | ~30 | ~33 |
| `TestListAgentsHandlerReturnsAgent(s)` | ~40 | ~44 |

The differences are trivial: httpapi uses `json.Marshal` while udsapi uses `mustJSONBody`; httpapi has one extra CORS test; udsapi has an extra `TestApproveSessionRejectsWithNotImplemented` test.

These tests are testing `apicore.BaseHandlers` behavior through the transport layer. Every time handler behavior changes, both test suites must be updated in lockstep (Shotgun Surgery).

**Recommendation**: The apicore test suite (`handlers_test.go`, 273 LOC) already tests `BaseHandlers` directly. The transport tests should only test transport-specific behavior:
- httpapi: CORS, static file serving, prompt SSE formatting, error masking, AI SDK headers
- udsapi: socket lifecycle, approve-not-implemented, raw SSE prompt streaming

Delete or reduce ~600+ lines from each transport test suite. The existing apicore tests cover the handler logic.

#### Finding 3.3: `server.go` Structural Duplication with httpapi
- **Severity**: P1 (high)
- **Action**: (B) Package-level split or (C) Extraction
- **Smell**: DRY violation, Shotgun Surgery
- **Location**: `server.go` (482 LOC) vs `httpapi/server.go` (651 LOC)

Both server files share:
1. **Identical constants**: `defaultPollInterval`, `defaultReadHeaderTimeout`, `defaultIdleTimeout` (both packages define all 3)
2. **Identical type aliases**: `AgentLoader`, `SessionManager`, `Observer`, `DreamTrigger`, `WorkspaceService` (5 aliases each)
3. **Identical functional options**: `WithHomePaths`, `WithConfig`, `WithLogger`, `WithStartedAt`, `WithNow`, `WithPollInterval`, `WithSessionManager`, `WithObserver`, `WithWorkspaceResolver`, `WithMemoryStore`, `WithDreamTrigger`, `WithAgentLoader`, `WithEngine` (13 identical `Option` functions each)
4. **Nearly identical `New()` constructors**: Same default-setting, same validation checks
5. **Nearly identical `Start()`/`Shutdown()` lifecycle**: Same mutex pattern, same `serveDone` goroutine, same `streamCancel`, same `waitForServeDone`
6. **Identical `waitForServeDone()`**: Copied verbatim
7. **Identical `Handlers` wrapping pattern**: Both wrap `*apicore.BaseHandlers`
8. **Identical `handlerConfig` struct**: Same fields (minus httpapi's `staticFS` and `httpPort`)

The `Server` struct fields are also near-identical (udsapi has `socketPath` instead of `host`/`port`/`actualPort`).

**Recommendation**: Extract a `ServerBase` struct into apicore (or a new `api/transport` package) that handles:
- Common fields (homePaths, config, logger, startedAt, now, pollInterval, sessions, observer, workspaces, memoryStore, dreamTrigger, agentLoader)
- Common option functions
- Common defaults/validation in a `Configure()` method
- Common Start/Shutdown lifecycle (parameterized by listener type)

Transport packages then embed `ServerBase` and add only their specific fields and logic.

#### Finding 3.4: `shared.go` / `shared_test.go` Near-Identical to httpapi
- **Severity**: P2 (medium)
- **Action**: (C) Extraction
- **Smell**: DRY violation
- **Location**: `shared.go` (31 LOC), `shared_test.go` (87 LOC)

Both transport packages have `shared.go` files with type aliases and wrapper functions for apicore. The udsapi version has 3 type aliases + 5 wrappers; httpapi has 4 type aliases + 5 wrappers. They differ only in the `maskInternalErrors` parameter to `RespondError`.

`shared_test.go` files are even more duplicated: udsapi has 14 type aliases + 14 wrapper functions; httpapi has 12 type aliases + 8 wrapper functions.

**Recommendation**: Consolidate type aliases into apicore or delete them in favor of direct `apicore.` references.

#### Finding 3.5: `helpers_test.go` Duplication with httpapi
- **Severity**: P2 (medium)
- **Action**: (C) Extraction
- **Smell**: DRY violation
- **Location**: `helpers_test.go` (142 LOC) vs `httpapi/helpers_test.go` (138 LOC)

Both files define identical helper functions: `newTestHandlers`, `newTestHandlersWithWorkspace`, `newTestRouter`, `newTestHomePaths`, `writeAgentDef`, `newSessionInfo`, `newSession`, `performRequest`, `decodeJSONResponse`, `parseSSE`, `discardLogger`.

**Recommendation**: Move shared test helpers to `apitest` package (which already exists for this purpose). The transport-specific helpers (e.g., `shortSocketPath`, `freeTCPPort`, `newUnixClient`) stay local.

## Group-Level Summary

### Subpackage Grouping Recommendation

**Strongly recommended**: Reorganize the 5 packages under a common `internal/api/` parent:

```
internal/api/
    core/         <-- current apicore + apisupport merged
    contract/     <-- NEW: shared API contract DTOs (extracted from CLI)
    httpapi/      <-- current httpapi (transport-specific only; explicit name avoids net/http ambiguity)
    udsapi/       <-- current udsapi (transport-specific only)
    testutil/     <-- current apitest
```

> **UPDATED (2026-04-06)**: Names changed from `api/http` to `api/httpapi` and `api/uds` to `api/udsapi` per Codex review (avoid ambiguity with Go's `net/http`). Contract types co-located inside `api/` subtree instead of standalone `apitypes/` package.

**Rationale**:
1. These packages form a cohesive group with tight coupling (apicore is imported by every other package in the group)
2. Go's flat namespace means `apicore`, `apisupport`, `apitest` live alongside unrelated packages like `acp`, `config`, `session` -- grouping clarifies the architecture
3. The `api` prefix is already baked into every package name (`apicore`, `apisupport`, `apitest`) -- the naming convention signals they belong together
4. Subpackages enable shorter import aliases (`api/core` instead of `apicore`, `api/http` instead of `httpapi`)
5. This reduces the flat package count from 20 to 17 (5 become 1 parent + 4 children)

**Alternative considered**: Keep flat but rename. This preserves the current structure but misses the organizational benefit. Not recommended given the user's explicit desire to reduce flat package count.

### Coupling Analysis

**Afferent coupling** (packages that depend on these):
- `apicore`: imported by httpapi, udsapi, apitest, daemon (high -- expected for core)
- `apisupport`: imported only by apicore (should be merged)
- `apitest`: imported only by test files in apicore, httpapi, udsapi (correct)
- `httpapi`: imported only by daemon (correct)
- `udsapi`: imported only by daemon (correct)

**Efferent coupling** (packages these depend on):
- `apicore` -> acp, apisupport, memory, observe, session, store, config, workspace (8 deps -- high but acceptable for a coordination layer)
- `httpapi` -> acp, apicore, apitest (test), memory, observe, session, store, config, workspace (9 deps)
- `udsapi` -> acp, apicore, apitest (test), memory, observe, session, store, config, workspace (9 deps)

httpapi and udsapi have identical dependency sets despite being different transports. This confirms they should share more infrastructure.

**Circular dependencies**: None detected. Dependency flow is strictly: daemon -> httpapi/udsapi -> apicore -> apisupport -> domain packages.

### SOLID Assessment

| Principle | Status | Notes |
|---|---|---|
| **SRP** | Violated | BaseHandlers handles 6 domains (Finding 1.1) |
| **OCP** | Partially violated | Adding a new API domain requires modifying BaseHandlers, BaseHandlerConfig, both RegisterRoutes functions |
| **LSP** | OK | No inheritance hierarchy to violate |
| **ISP** | OK | SessionManager interface is broad (11 methods) but all callers use most methods |
| **DIP** | Good | apicore defines interfaces consumed by transports; domain packages supply implementations |

### Recommended Refactoring Order

1. **Quick wins first** (trivial effort, immediate cleanup):
   - Delete 13 empty placeholder files
   - Inline apisupport into apicore
   
2. **Test consolidation** (moderate effort, highest maintenance impact):
   - Move duplicated handler tests to apicore test suite
   - Reduce transport tests to transport-specific behavior only
   - Move shared test helpers from helpers_test.go to apitest

3. **Structural improvements** (significant effort, long-term benefit):
   - Extract common server lifecycle into apicore
   - Reorganize into `internal/api/` subpackage tree
   - Consider splitting BaseHandlers into domain handler groups
