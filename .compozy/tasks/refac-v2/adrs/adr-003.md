# ADR-003: Split Persistence into store/sessiondb and store/globaldb

## Status

Accepted

## Date

2026-04-06

## Context

`internal/store` currently contains multiple persistence concerns:

- per-session event storage and writer lifecycle
- global index storage
- workspace persistence
- observability queries
- migration and schema helpers

The code already exposes concrete structural seams through `SessionDB` and `GlobalDB`, but they still live inside one broad package. The `refac-v2` architecture needs a clearer persistence boundary that reflects the actual storage model: one SQLite database per session and one global SQLite database for registry and observability concerns.

## Decision

Adopt this persistence target structure:

- `internal/store` keeps shared persistence types, narrow interfaces, validation helpers, and small common utilities.
- `internal/store/sessiondb` owns per-session event storage and its dedicated writer loop.
- `internal/store/globaldb` owns the global registry, workspace store integration, and observability-facing storage.

Migration/schema helpers remain close to the package they serve unless a later phase proves that a separate schema boundary is necessary.

## Alternatives Considered

### Alternative 1: Keep one store package with file splits only

- **Description**: Preserve `internal/store` as a single package and rely on better file organization plus interface splitting.
- **Pros**: Lower import churn and simpler migration.
- **Cons**: Keeps persistence responsibilities coupled behind one package boundary.
- **Why rejected**: `refac-v2` is explicitly aiming for clearer package-level boundaries, not just improved file layout.

### Alternative 2: Split into three packages immediately, including schema/migration

- **Description**: Create `store/sessiondb`, `store/globaldb`, and a separate schema or migration package.
- **Pros**: Highest boundary clarity.
- **Cons**: Adds another level of abstraction before there is evidence that schema logic has an independent lifecycle.
- **Why rejected**: Current evidence supports the session/global split strongly, but a third persistence subpackage is still YAGNI.

## Consequences

### Positive

- Makes the two database lifecycles explicit in the package graph.
- Enables narrower interfaces for consumers in `session`, `workspace`, `observe`, and `daemon`.
- Improves test isolation by aligning package boundaries with database ownership.

### Negative

- Requires import and constructor migration across several core packages.
- Forces decisions about where shared validation and SQLite helpers live.

### Risks

- Shared helpers can become a dumping ground if `internal/store` is not kept intentionally small.
- Migration logic might end up duplicated if package responsibilities are not enforced.
- Mitigation: define a narrow shared surface in `internal/store` and keep schema helpers package-local unless reuse is proven.

## Implementation Notes

- `SessionDB` and `GlobalDB` should remain concrete types; the refactor is about ownership and imports, not introducing an abstract repository layer.
- Existing interface consumers should be narrowed as part of the split, especially the current broad `SessionRegistry` surface.
