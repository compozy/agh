# ADR-004: Use Phased Cutovers with Same-Phase Bridge Removal and Layered Verification

## Status

Accepted

## Date

2026-04-06

## Context

The chosen `refac-v2` architecture introduces package moves, contract extraction, and runtime-boundary changes across API, daemon, session, workspace, and store. Direct cutovers are desirable because the project has a zero-legacy tolerance policy, but some transitions may still need short-lived forwarding code to keep the tree compiling while imports move in a controlled sequence.

The repository already defines two important verification surfaces:

- `make verify` for fmt, lint, unit tests, and build
- `make test-integration` for wider runtime coverage across daemon, API, CLI, persistence, and workspace flows

## Decision

Use phased cutovers as the default rollout strategy, with these rules:

- Prefer direct package cutovers inside each phase.
- Allow temporary bridges, re-exports, or forwarders only when they reduce mechanical migration risk.
- Any bridge introduced during a phase must be removed before that phase is considered complete.
- Run `make verify` for every step that moves packages or shared contracts.
- Run `make test-integration` for every phase that touches `api/*`, `cli`, `daemon`, `session`, `store`, or `workspace`.

## Alternatives Considered

### Alternative 1: Direct cutovers only

- **Description**: Forbid all temporary bridges and require each phase to land as a clean cutover.
- **Pros**: Cleanest history and least chance of transitional leftovers.
- **Cons**: Larger diffs and higher mechanical risk during package migrations.
- **Why rejected**: Some structural moves are easier and safer when performed with a very short-lived bridge inside one phase.

### Alternative 2: Allow bridges across multiple phases

- **Description**: Permit compatibility layers to persist until later cleanup phases.
- **Pros**: Reduces immediate migration pressure.
- **Cons**: Violates the project’s zero-legacy tolerance and invites lingering transitional code.
- **Why rejected**: Transitional code must not become a second architecture.

## Consequences

### Positive

- Keeps the migration pragmatic without normalizing compatibility debt.
- Makes verification expectations explicit for every structural phase.
- Aligns the rollout with the repository’s existing build and test workflow.

### Negative

- Increases the operational cost of each phase.
- Requires discipline to avoid leaving bridges behind.

### Risks

- Teams may under-scope phases and trigger expensive repeated integration runs.
- Temporary bridges could accidentally persist if phase completion criteria are vague.
- Mitigation: define phase exit criteria explicitly in the TechSpec and tie them to verification commands.

## Implementation Notes

- The TechSpec should annotate which phases require integration validation and why.
- Temporary bridges should be limited to mechanical package moves, not behavioral compatibility code.
