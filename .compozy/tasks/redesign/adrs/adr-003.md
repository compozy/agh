# ADR-003: Adopt `motion` (framer-motion successor) for UI animations

## Status

Accepted

## Date

2026-04-18

## Context

The redesign mock (`docs/design/web-inspiration/`) defines minimal motion — CSS transitions on hover/focus (150ms), `active:translate-y-px` on buttons, a keyframe `dot-pulse` for live indicators, and a `shimmer` utility for skeletons. `DESIGN.md §9` codifies this as "minimal, purposeful motion, no bounce/spring, no entrance animations".

However, the real operator app ships complex stateful transitions that CSS-only cannot handle cleanly:

- **Split-pane detail panel** — select a row, detail enters from the right; close, it exits. Requires exit animations (element unmounts).
- **Sidebar collapse / expand** — width transition plus staggered label fades.
- **Route transitions** — Session page → Tasks list should have a soft crossfade, not a hard swap.
- **Modal / drawer enter-exit** — shadcn's Radix primitives need framer-style exit handling to avoid layout thrashing on close.
- **Toast / notification stacking**.
- **Tool-call card state transitions** — RUNNING → DONE badge with a brief fade.

CSS-only handles enter-animations but cannot animate elements that unmount; Tailwind's `data-[state=open]:animate-…` pattern helps but is brittle for orchestrated sequences. Existing shadcn wrappers in `web/src/components/ui/` already use Radix `data-state` attributes which integrate poorly with CSS-only timing when combined with layout changes.

## Decision

**Adopt the `motion` package (the current name for what was framer-motion) as a peer dep of `@agh/ui` and a dep of `web/`.**

Scope:

- `@agh/ui` exports primitives that internally use `motion` for unmount animations — Dialog overlay, Popover, Sheet, Drawer, Toast, and a generic `<AnimatePresence>` re-export if needed.
- `web/` uses `motion` for layout-level transitions: sidebar collapse, split-pane enter/exit, route transitions (via TanStack Router's built-in + `AnimatePresence`).
- CSS still owns simple transitions (hover/focus colors, `active:translate-y-px`, `dot-pulse`, `shimmer`). `motion` is only used when an element mounts/unmounts or when timing has to synchronize across siblings.
- `prefers-reduced-motion` is honored globally — `@agh/ui` sets a motion config respecting the media query; no component opts out.
- No layout-id magic on the hot path (session event stream). Layout animations are opt-in per surface.

Bundle budget: `motion` gzipped is ~30KB. Acceptable for the operator UI (no SSR, desktop-first). Budget enforced via Vite bundle analyzer in CI — any PR raising the first-page bundle >10% requires explicit approval.

## Alternatives Considered

### Alternative 1: motion-one

- **Description:** Same author's lightweight lib (~4KB); Web Animations API wrapper.
- **Pros:** Tiny bundle.
- **Cons:** No first-class `AnimatePresence` equivalent for unmount orchestration; React integration is a thinner wrapper that still needs manual hook wiring for exit animations. Less community / docs coverage.
- **Why rejected:** Unmount animations are the main reason we need a lib; motion-one's ergonomics for that case are worse than bare CSS.

### Alternative 2: CSS + tw-animate-css only

- **Description:** Tailwind's animate utilities plus hand-rolled keyframes; no JS dep.
- **Pros:** Zero added dep; matches DESIGN.md's "minimal motion" principle.
- **Cons:** Cannot animate elements that unmount (the exact case we have). `data-state` gymnastics for Radix primitives become brittle once multiple primitives orchestrate together. Route transitions are not possible without reimplementing route-outlet buffering.
- **Why rejected:** Forces workarounds that cost more than the lib itself.

## Consequences

### Positive

- **Clean unmount animations** — Dialog, Sheet, Popover exit properly without layout jank.
- **Route-level transitions** feasible (though intentionally restrained).
- Aligns with current-generation React animation ecosystem (Radix + motion is a well-documented pairing).

### Negative

- **+30KB gzip** on the first-page bundle.
- One more API surface the team must learn. Contributors writing stories / components must understand when CSS suffices vs. when `motion` is appropriate.

### Risks

- **Risk:** Overuse — team starts wrapping every component in `motion.div` for fun.
- **Mitigation:** Style guide in `packages/ui/README.md` (to be added in this spec) plus code review discipline. Motion usage outside `@agh/ui` primitives or explicit layout transitions is a review flag.
- **Risk:** Motion library bypasses `prefers-reduced-motion`.
- **Mitigation:** Centralize motion config in `@agh/ui`'s root provider; test with `prefers-reduced-motion` simulated in Playwright and in one story per animated primitive.

## Implementation Notes

- Install with `pnpm add motion` in both `packages/ui/package.json` (as peer + dev) and `web/package.json` (as runtime).
- Root provider placement: `@agh/ui` exports `<UIProvider>` which wraps a `MotionConfig reducedMotion="user"`; `web/src/main.tsx` wraps the app in it once.
