# ADR-004: Phased rollout — tokens+primitives → shell → Tasks → Session → remaining domains → Settings

## Status

Accepted

## Date

2026-04-18

## Context

The migration touches ~200 files across `packages/ui/`, `web/src/components/`, `web/src/systems/`, and route files. Shipping all at once ("big-bang") is unreviewable; shipping one page at a time forces primitive churn inside every PR. A phased rollout with a clear dependency chain minimizes both review burden and coupling risk.

ADR-002 forbids coexistence of old and new. That means the order matters: dependencies must land first or we regress. The primitives layer (`@agh/ui`) is the root dependency of everything else.

The eight redesigned mock pages (Tasks, Network, Automation, Bridges, Knowledge, Skills, Session, Settings) have very different complexity profiles:

- **Tasks**: 27 domain components, multiple views (List/Kanban/Dashboard/Inbox), richest visual surface — best proving ground for the new vocabulary.
- **Session**: 19 domain components, the most stateful flow (live event stream, composer, inspector), critical path for daily operator use.
- **Settings**: 12 components across 11 sub-routes, structurally simplest but volumetrically large.
- **Network, Automation, Bridges, Knowledge, Skills**: split-pane list + detail, similar shape, can be batched.
- **Workspace, Daemon, Dashboard home**: not in the mock; derived from the same system (ADR-??).

## Decision

**Six-phase rollout. Each phase is one or more PRs; a phase must finish before the next starts. `make verify` + Playwright visual snapshots (ADR-005) gate each PR.**

### Phase 1 — Tokens + primitive library (packages/ui)

- Extend `packages/ui/src/tokens.css` with missing tokens: `--radius-chip`, `--radius-mono-badge`, `--font-display` (Playfair Display, marketing-scoped), `--font-wordmark` (NuixyberNext), `--duration-fast/--duration-base/--duration-slow`, `--ease-out`, `--ease-in-out`.
- Consolidate `web/src/components/ui/**` into `@agh/ui` (Dialog, Popover, Combobox, Command, Sheet, Select, Tabs, Tooltip, ScrollArea, Switch, Toggle, …). Stories move with them.
- Consolidate `web/src/components/design-system/**` into `@agh/ui` as rebuilt primitives: `Pills`, `PageHeader`, `SearchInput`, `Empty`, `Section`, `Metric`, `StatusDot`, `ConnectionIndicator`.
- Add new primitives from the mock: `Sidebar` (workspace rail + panel shell), `SplitPane`, `MonoBadge`, `KindChip`, `CodeBlock`, `ToolCallCard` (shell only, no state), `ChatMessageBubble` (shell only).
- Remove dead `TextureCanvas`, `MetricStrip`, `PillButton` (superseded by `Pills` + `Button`).
- Install `motion`, add `<UIProvider>` wrapping MotionConfig.
- Every new primitive has a `.stories.tsx` with variants, empty/filled states, and a `reducedMotion` variant where animation exists.
- End gate: `@agh/ui` Storybook passes a11y + visual snapshot baseline; `web/` still compiles (via shims from old import paths re-exporting new ones, removed in Phase 2).

### Phase 2 — App shell

- Rewrite `web/src/components/app-sidebar.tsx` on top of `@agh/ui` Sidebar. Adopt workspace rail + agent tree + nav rows + system footer from the mock.
- Rewrite `__root.tsx` + `_app.tsx` layout: header (sticky, blurred), split-pane wrapper, content column.
- Wire route-level motion (soft crossfade between top-level routes, no layout-id).
- Delete the shims from Phase 1 — all primitive imports now resolve directly to `@agh/ui`.
- End gate: every existing page renders inside the new shell without visual changes to page interior (screens stay on their old primitives where not yet migrated — this is the one exception to ADR-002, scoped to the inside of list/detail panels until their own phase). Actually: *every page's interior stays visually identical until its domain phase; the shell + sidebar change only*.

### Phase 3 — Tasks (richest visual surface)

- Rewrite `web/src/systems/tasks/components/**` to use `@agh/ui` primitives only.
- Implement the 4 views: List (split-pane), Kanban (4 columns), Dashboard (metric grid + charts), Inbox (approval flow).
- Update `/tasks`, `/tasks/new`, `/tasks/$id`, `/tasks/$id/edit`, `/tasks/$id/runs/$runId` routes.
- End gate: Playwright visual snapshots for all 5 route states + empty + loading + error.

### Phase 4 — Session (critical path)

- Rewrite `web/src/systems/session/**`: message thread (system/user/agent/tool/diff roles), composer, right inspector panel (trace timeline, usage, memory, files).
- Keep live SSE event wiring intact; only visual layer changes.
- End gate: run a real session end-to-end in the UI and compare against the mock for the five message roles.

### Phase 5 — Remaining domains (batched)

- Network (channels + peers), Automation (jobs + triggers), Bridges, Knowledge, Skills (installed + marketplace).
- Each domain is one PR following the split-pane list+detail pattern established in Phase 3.
- Workspace, Daemon, home dashboard derived from the system pattern (decision per ADR-001 consequences + "non-mocked screens" decision).

### Phase 6 — Settings

- 11 sub-routes, all using `@agh/ui` primitives. Table-heavy, uses the new `Section`, `Pills`, `Field`, `Switch`, `Metric`.
- Last because volumetrically large but structurally isolated — migrating it last does not block other domains.
- End gate: Playwright snapshots for all 11 sub-routes + save-bar + dirty state.

## Alternatives Considered

### Alternative 1: Big-bang single PR

- **Description:** One PR migrates everything.
- **Pros:** No coordination overhead.
- **Cons:** Unreviewable (~200 files). Blocks parallel work. Rollback is a full revert.
- **Why rejected:** Review quality and iteration speed matter more than PR count.

### Alternative 2: Per-page sequential (no separate primitives phase)

- **Description:** Migrate Tasks first; build primitives as Tasks needs them; subsequent pages extend the primitive set as needed.
- **Pros:** Each PR is a vertical slice.
- **Cons:** Primitive API churns every PR; later pages force breaking changes to earlier pages' components; every PR bundles primitive work + page work, making review fuzzy.
- **Why rejected:** Primitive stability is a foundation; deferring it trades predictability for perceived progress.

### Alternative 3: Domain-parallel rollout after Phase 1–2

- **Description:** After primitives + shell, run Tasks, Network, Session, Settings in parallel PRs by different contributors.
- **Pros:** Ship faster.
- **Cons:** Increases risk of conflicting primitive tweaks landing in parallel; visual parity drift. Only works if Phase 1 primitives are truly stable.
- **Why rejected for now:** Can be revisited after Phase 3 proves Phase 1 primitives are stable. Listed here so contributors understand why the sequence is strict.

## Consequences

### Positive

- Each phase has a clear end gate + measurable output.
- Review is focused per phase (primitive people review Phase 1; domain people review Phase 3+).
- Rollback unit is a single phase PR, not the whole redesign.

### Negative

- Phase 2 introduces one small violation of ADR-002 (page interiors use old primitives briefly, until their own phase). Managed by requiring the shim removal at the end of Phase 2 and Phases 3–6 each cleaning their own interior.
- Timeline longer than big-bang.

### Risks

- **Risk:** Phase 1 takes longer than expected and blocks everything.
- **Mitigation:** Phase 1 is the tightest-scoped — every primitive has an existing implementation to migrate; no green-field design work. Stories-first discipline (build the story, then the component) keeps scope honest.
- **Risk:** A domain phase reveals a missing primitive.
- **Mitigation:** Add it to `@agh/ui` in the same PR as the domain that needs it, with a story; no scope creep allowed (one primitive, exact contract, reviewed on the same PR).

## Implementation Notes

- Phase boundaries are enforced via task files (see `cy-create-tasks`) — each phase becomes a task cluster with explicit blocks/blockedBy.
- A Phase 1 companion doc lives at `packages/ui/README.md` listing primitive inventory, "when to use CSS vs motion", and Storybook contribution rules.
