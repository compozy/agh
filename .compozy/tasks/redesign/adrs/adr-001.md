# ADR-001: Consolidate UI primitives into @agh/ui

## Status

Accepted

## Date

2026-04-18

## Context

The operator web app (`web/`) currently owns three overlapping primitive layers:

- `packages/ui/src/components/` ā€” `@agh/ui`: 12 base-ui / shadcn-derived primitives (Button, Badge, Card, Alert, Input, Label, Kbd, Progress, Separator, Skeleton, Spinner, Table) with Storybook coverage and tokens alignment.
- `web/src/components/ui/` ā€” 27 local shadcn primitives (Dialog, Popover, Combobox, Command, Sidebar, Sheet, Select, Tabs, Tooltip, ScrollArea, Switch, Toggle, ā€¦) used only inside `web/`.
- `web/src/components/design-system/` ā€” 13 branded components (Pill, PillButton, PageContent, Panel, Metric, SectionHeading, StatusDot, ConnectionIndicator, MetricStrip, Toolbar, TextureCanvas, DesignSystemShowcase).

The redesign mock at `docs/design/web-inspiration/` introduces a new shared vocabulary (Sidebar shell, PageHeader, Pills, SearchInput, Empty, Section, Metric, MonoBadge, KindChip, CodeBlock, ChatMessage shell, ToolCallCard shell) that overlaps heavily with `design-system/` and will be consumed across every screen. Keeping three separate sources of primitives makes token + visual consistency impossible to enforce and forces every new primitive to choose a home arbitrarily.

Greenfield alpha ā€” per `CLAUDE.md` we never preserve backwards-compat. No production users exist.

## Decision

**`@agh/ui` (packages/ui) becomes the single source of generic primitives. `web/src/components/ui/` and `web/src/components/design-system/` are deleted. Domain compositions stay in `web/src/systems/`.**

Concretely:

- Every generic primitive ā€” whether existing (Button, Dialog, Popover, Combobox, Sidebar shell, Sheet, ā€¦) or new from the mock (PageHeader, Pills, SearchInput, Empty, Section, Metric, MonoBadge, KindChip, CodeBlock, SplitPane, ChatMessageBubble, ToolCallCard shell) ā€” is exported from `@agh/ui` and has a colocated `.stories.tsx`.
- `web/src/components/` is reduced to the app shell (`app-sidebar.tsx` becomes thin composition over `@agh/ui` Sidebar) plus cross-system compositions that do not belong to a single domain.
- Domain components (e.g. `tasks-detail-panel`, `session-inspector`, `agent-sidebar-group`) stay inside `web/src/systems/<domain>/components/`. They compose `@agh/ui` primitives; they never redefine primitives.
- The `design-system-showcase` route (`/design-system`) becomes a thin consumer of `@agh/ui` exports, not a standalone primitive library.

Import rules, enforced by lint / codeowners:

- `web/src/**` imports primitives only from `@agh/ui`.
- `@agh/ui` does not import from `web/src/**`.
- `web/src/systems/<domain>/**` imports from `@agh/ui` and from the same domain; never from another domain.

## Alternatives Considered

### Alternative 1: Two-layer split (@agh/ui + web/components/ui)

- **Description:** Keep `@agh/ui` as "atomic" primitives (Button, Badge, Input) and `web/src/components/ui/` as "complex" primitives (Dialog, Popover, Sidebar).
- **Pros:** Smaller `@agh/ui` surface; complex primitives stay close to the app.
- **Cons:** The atomic/complex boundary is arbitrary; every new primitive triggers a placement debate. Two token-consuming locations means drift is likely. Storybook split adds coordination cost.
- **Why rejected:** The boundary is not defensible and does not solve the consolidation problem ā€” it just splits it.

### Alternative 2: Keep three layers, just repaint

- **Description:** Leave structure as-is; only update styles/tokens in each layer.
- **Pros:** Lowest immediate refactor cost.
- **Cons:** Preserves the overlap (Pill exists in `design-system/` and will be re-introduced by the mock as Pills). Token drift is guaranteed. Review burden compounds.
- **Why rejected:** Violates the greenfield-alpha mandate; locks in tech debt the redesign was meant to pay down.

### Alternative 3: Move everything into @agh/ui including domain compositions

- **Description:** Put ChatMessage, ToolCallCard, session inspector, agent tree inside `@agh/ui` as "headless" primitives.
- **Pros:** One package to version.
- **Cons:** Couples UI package to API contracts, query hooks, zustand stores. Makes `@agh/ui` unpublishable (it depends on OpenAPI types) and impossible to test in isolation.
- **Why rejected:** Domain coupling belongs in `web/src/systems/` ā€” app-renderer-systems pattern already documented.

## Consequences

### Positive

- **One source of truth** for tokens + primitives; DESIGN.md maps 1:1 to `@agh/ui` exports.
- **Storybook split becomes coherent** ā€” `@agh/ui` owns primitive stories, `web/` owns screen-level stories.
- **Domain systems stay thin** ā€” they compose known primitives instead of re-implementing them.
- Enables independent versioning / publishing of `@agh/ui` later if needed.

### Negative

- Large diff: ~40 primitive files move or delete; every screen component that imports from `@/components/ui` or `@/components/design-system` has its import rewritten. Estimated ~200 files touched.
- `@agh/ui` grows from 12 to ~35+ exports ā€” larger surface to maintain.
- Primitives that are genuinely app-specific (e.g. currently-local Combobox preset) need either generalization or deliberate staying-in-web justification.

### Risks

- **Risk:** Primitive generalization hides app-specific defaults, causing regressions.
- **Mitigation:** Migrate primitive-by-primitive with storybook parity before swapping call sites. Playwright visual snapshots (ADR-005) catch regressions.
- **Risk:** Import cycle between `@agh/ui` and `web/` if a primitive accidentally pulls from the app.
- **Mitigation:** CI grep rule forbidding `@agh/ui ā†’ web/` imports; tsconfig path check in `make verify`.

## Implementation Notes

- Before deletion, confirm every `web/src/components/ui/*` import site resolves to a `@agh/ui` export of the same name (or an intentional rename).
- `web/src/components/design-system/design-system-showcase.tsx` is the primary consumer of the branded set ā€” rewriting it is the tracking unit for "design-system folder is gone".
- The `@agh/ui` tokens file (`packages/ui/src/tokens.css`) is already aligned with DESIGN.md; gaps (missing `--radius-chip`, `--font-display`, `--duration-*`) are addressed inside this migration (ADR-004 consequence), not a separate task.
