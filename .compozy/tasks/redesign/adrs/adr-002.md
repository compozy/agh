# ADR-002: Greenfield migration — delete without backwards-compat

## Status

Accepted

## Date

2026-04-18

## Context

AGH is explicitly in greenfield alpha (`CLAUDE.md`: "No production users exist. Never sacrifice code quality for backward compatibility. Never write migration, compat, or defensive code for old state — delete the old thing instead of working around it.").

The redesign touches every screen, every primitive, and most routing/layout scaffolding. Any "compat shim", "feature flag per route", or "old-new coexistence" strategy would double the maintenance surface for months, hide dead paths, and violate the project's own discipline.

## Decision

**Migration is greenfield in-place. Old primitives and styles are deleted, not deprecated. Old classes, variants, imports, and CSS utilities are removed from the tree the same PR the new one lands.**

Concretely:

- When a domain is migrated (ADR-004 rollout), its screens are rewritten using new `@agh/ui` primitives, old imports are removed, and dead code in `web/src/components/ui/` or `web/src/components/design-system/` is deleted if no other domain depends on it.
- No `<LegacyX>` / `<NewX>` pair. No `experimental_X` export. No feature flag gating visual output.
- Stories that describe the old look are deleted alongside their components.
- `make verify` must pass on every PR — there is no "tolerate temporarily" grace period.

## Alternatives Considered

### Alternative 1: Feature-flag by route

- **Description:** Introduce a GrowthBook/env flag per route; old and new render side-by-side until all routes opt in.
- **Pros:** Allows partial rollout; reviewers can toggle.
- **Cons:** Doubles component tree; requires wrapper layer; stories must cover both states; cross-route navigation mixes styles. Flag cleanup becomes its own PR stack.
- **Why rejected:** No production users, no stakeholder needs the toggle, and the flag infrastructure itself becomes debt.

### Alternative 2: Temporary compat shims

- **Description:** Keep old component exports as thin wrappers that forward to the new API; remove shims in a final cleanup PR.
- **Pros:** Each screen migration is smaller.
- **Cons:** Wrappers drift from both sides; the "final cleanup PR" never arrives on time; external consumers of `@agh/ui` (if any emerge) lock into the shim.
- **Why rejected:** Violates the "delete the old thing" rule in `CLAUDE.md` and historically never ships the cleanup step.

## Consequences

### Positive

- **Honest PRs** — every diff tells the whole story; no hidden "still using the old thing" state.
- **Smaller cognitive load** — contributors learn one set of primitives, not a new and legacy set.
- **Verification is single-path** — `make verify` either passes or fails; there is no "yellow" in-between.

### Negative

- **Large PRs per domain.** Each domain-migration PR replaces ~10–30 files simultaneously, making review denser.
- **No incremental safety net.** If a regression ships, rollback means reverting the whole migration PR.

### Risks

- **Risk:** A domain PR lands with a regression because the old fallback was removed.
- **Mitigation:** Playwright visual snapshots (ADR-005) are the regression gate. Screens without snapshots get them before migration, not after. Each PR sits at least one overnight CI cycle before merge.
- **Risk:** A primitive shared across domains breaks in the middle of a migration.
- **Mitigation:** Primitives migrate first (rollout phase 1, ADR-004); domains migrate on top of already-stable primitives.

## Implementation Notes

- Every PR commit message prefix is `feat:` or `refactor:` (matching `CLAUDE.md` commit style). Never `chore: cleanup` — cleanup is part of the feature PR, not a follow-up.
- Delete `.old_project/` pattern is already established — this ADR extends that discipline to the web tree.
