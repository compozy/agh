# ADR-001: Capability Is the Single Network Capability Artifact

## Status

Accepted

## Date

2026-04-20

## Context

The current branch introduced explicit, structured capability catalogs for agent directories and projected them into network discovery through brief `greet` metadata and rich `whois` responses. At the same time, the network protocol still keeps `recipe` as a separate first-class wire artifact with overlapping semantics around reusable procedures and capability requirements.

This split creates a persistent modeling problem:

- `capability` already carries outcome, context, expected artifacts, execution outline, constraints, and examples
- `recipe` separately carries a reusable transferable procedure with versioning and requirement metadata
- both concepts answer nearly the same question for authors and peer integrators: "what can this peer do, and how is that shared?"

Because AGH is still alpha/research with zero legacy tolerance, the architecture should optimize for the clearest long-term DX rather than preserving the current split.

## Decision

`Capability` becomes the single surviving concept for authored delegation offers and transferable network artifacts.

`Recipe` is removed as a separate protocol/runtime concept.

The unified model keeps the name `capability` and extends it to cover the useful transport semantics that previously lived in `recipe`.

## Alternatives Considered

### Alternative 1: Keep capabilities and recipes as separate concepts

- **Description**: Preserve the current model and refine the boundary through docs or smaller schema changes.
- **Pros**: Minimal conceptual churn from the current branch state.
- **Cons**: Leaves authors with two overlapping concepts, two mental models, and a cross-type dependency path.
- **Why rejected**: The overlap is architectural, not just documentary. The split keeps the core DX confusing.

### Alternative 2: Introduce a new unified name

- **Description**: Replace both `capability` and `recipe` with a third name.
- **Pros**: Avoids historical baggage from either term.
- **Cons**: Forces a larger rename surface and adds a second round of conceptual migration while the current capability work is already good and adopted in-branch.
- **Why rejected**: `Capability` already matches the preferred authoring and discovery language. Renaming adds churn without improving clarity enough.

## Consequences

### Positive

- Authors get one concept instead of two overlapping ones.
- Runtime, protocol, and docs converge on the same artifact.
- Dependency/composition can live inside one table using `capability.id`.

### Negative

- The current RFC and runtime wire model need a real redesign, not a localized rename.
- Existing code paths for `recipe` validation, routing, and lifecycle must be reworked.

### Risks

- The unified capability schema could become bloated if it tries to preserve every past field without discipline.
- Mitigation: keep the model field-based and outcome-oriented, and only absorb transport semantics that are still necessary.

## Implementation Notes

- The current capability catalog remains the source of truth.
- The network RFC should be rewritten so discovery, transfer, and interaction all speak about `capability`.
- `recipe` should disappear from runtime types, protocol docs, and user-facing docs.
