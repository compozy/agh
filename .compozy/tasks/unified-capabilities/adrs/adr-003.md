# ADR-003: Replace `recipe` Wire Semantics with `capability` While Preserving Interaction Behavior

## Status

Accepted

## Date

2026-04-20

## Context

Today `recipe` is not just an RFC example. It is a real wire kind with concrete runtime behavior:

- `KindRecipe` is part of the envelope registry
- recipe payloads have their own validation rules
- recipe messages can be broadcast or directed
- recipe messages can open interactions and participate in lifecycle handling beside `direct`

If capability becomes the single surviving concept, the protocol still needs one wire-level way to transmit a reusable capability artifact without forcing everything through `direct`.

## Decision

Replace `kind: "recipe"` with `kind: "capability"` and preserve the current operational role that `recipe` had.

`kind: "capability"`:

- may be broadcast to a channel
- may be sent directly to a peer
- may open an interaction
- participates in the same lifecycle rules that currently apply to `direct` and `recipe`

Discovery remains split from transfer:

- brief capability discovery stays in `greet`
- rich capability discovery stays in explicit `whois`
- transferable capability artifacts use `kind: "capability"`

## Alternatives Considered

### Alternative 1: Capability transfer only through `direct`

- **Description**: Remove the dedicated artifact kind and tunnel capability payloads through `direct`.
- **Pros**: Fewer wire kinds.
- **Cons**: Overloads `direct`, weakens protocol clarity, and loses the clean distinction between conversation and artifact transfer.
- **Why rejected**: A unified model still benefits from a dedicated artifact-carrying kind.

### Alternative 2: Keep the wire name `recipe`

- **Description**: Internally unify the model but preserve `kind: "recipe"` on the wire.
- **Pros**: Smaller protocol rename surface.
- **Cons**: Leaves the main conceptual confusion alive in the protocol.
- **Why rejected**: If the concept is unified, the wire should say so explicitly.

## Consequences

### Positive

- Preserves the useful runtime behavior already implemented for shared procedural artifacts.
- Removes cross-type reasoning from the protocol and lifecycle model.
- Keeps discovery and transfer distinct without keeping two artifact concepts.

### Negative

- Requires touching envelope kinds, validation, routing, lifecycle, delivery help text, and RFC examples.
- Existing tests that assert `recipe` need to be rewritten around `capability`.

### Risks

- The new `capability` kind could be confused with passive discovery metadata if docs are not explicit.
- Mitigation: define a clear distinction between capability discovery projections and transferable capability envelopes.

## Implementation Notes

- Replace `KindRecipe`, `RecipeBody`, and `Recipe` runtime types with capability equivalents.
- Update lifecycle helpers so `capability` inherits current recipe behavior.
- Rewrite RFC 003 worked examples to show `capability` broadcast and follow-up directed interaction.
