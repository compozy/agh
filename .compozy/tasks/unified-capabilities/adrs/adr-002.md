# ADR-002: Keep Current Capability Authoring Layouts and Use a Canonical Structured Schema

## Status

Accepted

## Date

2026-04-20

## Context

The current capability implementation already gives AGH a local-first authoring model with clear filesystem ergonomics:

- `capabilities.toml`
- `capabilities.json`
- `capabilities/*.toml`
- `capabilities/*.json`

That surface is already aligned with AGH agent-directory conventions and solves the original authoring problem well. The unification effort is about collapsing the conceptual split between `capability` and `recipe`, not about inventing a new storage UX.

The open design question is how much of the old recipe artifact model should survive in the unified capability schema.

## Decision

Keep the existing local capability catalog layouts and extend the capability schema rather than replacing the authoring surface.

The unified capability remains a structured object. Its canonical content is the field-based capability document itself, not a primary `inline|uri|content_type` body.

Additional schema decisions:

- `digest` is computed by the runtime from a canonical capability representation
- `version` is optional and authored locally
- `requirements` remains in the model and references `capability.id`
- `execution_outline`, `examples`, `artifacts_expected`, `context_needed`, and `constraints` remain first-class fields

## Alternatives Considered

### Alternative 1: Replace authoring with artifact-centric files

- **Description**: Move to a new format where each capability is primarily a transferable body artifact with metadata wrapped around it.
- **Pros**: Closer to the old recipe artifact framing.
- **Cons**: Discards a good local authoring UX and recent branch work.
- **Why rejected**: The current capability catalogs are already the better authoring surface.

### Alternative 2: Use two shapes of the same concept

- **Description**: Keep one local authored shape and define a separate canonical wire shape with explicit transformation.
- **Pros**: Allows aggressive wire optimization.
- **Cons**: Reintroduces conceptual duplication inside the supposedly unified model.
- **Why rejected**: The new model should collapse concepts, not recreate them as layered twins.

## Consequences

### Positive

- Preserves the branch's strongest UX improvement: straightforward local authoring.
- Makes digesting and validation deterministic because the runtime owns canonicalization.
- Keeps the unified capability easy to inspect and reason about without hiding meaning inside a body blob.

### Negative

- The wire payload may be larger than a body-centric artifact model in some cases.
- Canonicalization rules must be specified carefully for stable digest computation.

### Risks

- Over time, too many optional fields could make capability documents noisy.
- Mitigation: define required versus optional fields clearly and apply YAGNI to transfer-only additions.

## Implementation Notes

- Reuse `internal/config/capabilities.go` as the basis for the new schema and validation rules.
- Add canonicalization rules before digest computation.
- Treat `requirements` as a self-referential dependency list inside the unified capability space.
