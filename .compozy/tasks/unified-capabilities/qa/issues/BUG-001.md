# BUG-001: Filtered rich discovery returned a mismatched brief capability list

**Severity:** High
**Priority:** P0
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** `a572694e`
- **OS:** Darwin 25.3.0
- **Browser:** N/A
- **URL:** `/api/network/peers/:id` and rich `whois` discovery payloads

## Summary

When rich `whois` discovery was requested with `agh.include=["capability_catalog"]` and a capability ID filter, the response returned a filtered rich `capability_catalog` but preserved the original unfiltered `peer_card.capabilities` brief list. That made the brief and rich discovery views disagree on the same peer and could blank or misalign operator-visible capability summaries after refresh.

## Reproduction

```bash
go test -v -tags integration ./internal/network -count=1 -run 'Test(RoutersExchangeBroadcastCapabilityTransfers|RoutersPreserveCapabilityLifecycleAcrossPeers|DirectedWhoisRichDiscoveryDeliversPeerCardAndCapabilityCatalog|DirectedWhoisRichDiscoveryFilteringRefreshesRemotePresence)$'
```

Observed before the fix:

- A filtered rich discovery response returned a reduced `capability_catalog`, but the emitted `peer_card.capabilities` still reflected the unfiltered source peer card.
- Remote presence refresh paths could lose coherent capability summaries because brief and rich payloads no longer described the same subset.

## Expected

If a rich `capability_catalog` response is filtered, the brief `peer_card.capabilities` list surfaced with that same response must be derived from the same filtered subset and stay API-coherent with the rich catalog.

## Root cause

`buildWhoisResponseEnvelope` cloned and filtered the rich capability catalog but reused the original `PeerCard` brief discovery list unchanged. The response therefore mixed filtered rich data with stale unfiltered brief data.

## Fix

Projected the response `peer_card.capabilities` brief list from the same filtered rich catalog selection and added focused router coverage so brief and rich discovery stay aligned through `whois` filtering.

## Verification

- `go test -v ./internal/network -count=1 -run 'Test(RouterWhoisRichCapabilityDiscoveryReturnsCapabilityCatalog|RouterWhoisRichCapabilityDiscoveryFiltersRequestedIDsInCatalogOrder)$'`

- `make verify`

## Impact

- **Frequency:** Always when rich discovery filtering was used
