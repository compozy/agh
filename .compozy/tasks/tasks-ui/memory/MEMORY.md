## Shared Decisions
- Durable actor-scoped inbox triage state lives in GlobalDB `task_triage_state` and is exposed through the task store `GetTaskTriageState` / `UpsertTaskTriageState` surface.
- Inbox lane precedence is fixed to `archived -> approvals -> failed_runs -> blocked -> my_work`, and dismissals are activity-sensitive: a dismissed item resurfaces as unread when newer task/run/event activity advances beyond `last_seen_activity_at`.
- Stable task live replay semantics are backed by persisted `task_events` sequence order; reconnect-aware task streaming should reuse `after_sequence` rather than inventing a second cursor model.
- OpenAPI response documentation now supports explicit per-response content types in `internal/api/spec/spec.go`; task-native live streams should be documented as `text/event-stream` in the shared spec/codegen surface instead of falling back to generic session-stream assumptions.
- Tasks frontend entrypoint: `/_app/tasks` is the canonical route and `TasksPageShell` is the canonical shell. Sub-surfaces (list/kanban/dashboard/inbox/detail/run-detail/multi-agent live) must mount inside this shell instead of creating parallel `/_app` areas.
- Tasks frontend data access: the tasks system barrel at `@/systems/tasks` is the only allowed path to task APIs from routes/components. Mutation invalidation flows through `tasksKeys.lists()/runsRoot()/timelineRoot()/treeRoot()/runDetails()/triageRoot()` plus `[...tasksKeys.all, 'dashboard'|'inbox']`, so any new mutation hook must mirror this fan-out rather than invalidating ad-hoc keys.
- Task templates (`one_shot/recurring/epic/remote_peer/human_in_loop/blank`) live in `web/src/systems/tasks/lib/task-templates.ts` as frontend presets that layer defaults onto the create payload. They are not backend capabilities; new modal/empty-state flows must consume the template helpers rather than reinventing template logic.
- Kanban grouping rules are canonical in `web/src/systems/tasks/lib/task-grouping.ts`: `draft+pending` => "Pending" column, `failed+canceled` => "Failed" column, all other statuses are 1:1. Future board surfaces must reuse `groupTasksForKanban`/`getKanbanColumns` instead of inventing new mappings.

## Shared Learnings
- `internal/task/manager.go` now owns draft publication (`PublishTask`) plus approval-aware and attempt-aware canonical status reconciliation, which downstream task/API/read-model work should treat as authoritative.
- The daemon boot/runtime path only enables task services when the backing registry satisfies the full `task.Store` contract, so store/test doubles must keep pace when new live-read methods are added.
- Shared contract changes can fail extension integration through `TestReferenceExtensionsEndToEnd` because the reference SDK build runs `make codegen-check`; when task/host contract surfaces change, regenerate `sdk/typescript/src/generated/contracts.ts` (and companion codegen outputs) before expecting integration or repo verification to pass.
