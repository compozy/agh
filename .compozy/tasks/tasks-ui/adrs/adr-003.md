# ADR-003: Add Dedicated Task Live Surfaces Instead of Client-Side Stitching

## Status

Accepted

## Date

2026-04-17

## Context

The current backend gives the frontend enough data to build static or partially live task screens by combining:

- `GET /api/tasks/{id}`
- `GET /api/tasks/{id}/runs`
- session transcript/history endpoints
- session SSE

That composition is workable for prototypes, but it is not a clean architecture for full-parity task detail, run detail, or multi-agent live views. It pushes domain joins, active-run switching, and event ordering into the client.

## Decision

Ship dedicated task live surfaces in v1:

- a task timeline/events API
- a task-specific SSE stream
- a task tree live-view API for parent/child execution state
- a task-run detail API

The frontend may still reuse session surfaces where appropriate, but task pages must depend primarily on task-native live contracts rather than stitching session data together as the main design.

## Alternatives Considered

### Alternative 1: Temporary client-side composition

- **Description**: Build task live screens by merging task detail responses with session transcript/history/SSE in the browser.
- **Pros**: Faster initial implementation; reuses existing transports.
- **Cons**: Client complexity, duplicated join logic, fragile active-run switching, and weaker API semantics.
- **Why rejected**: The user explicitly chose first-class task live APIs for v1.

### Alternative 2: Reuse generic observe event streaming

- **Description**: Feed task screens from the generic observe stream.
- **Pros**: Reuses an existing stream transport.
- **Cons**: Observe events are session-centric and do not express task-native timeline, run detail, or tree semantics cleanly.
- **Why rejected**: It would overload a generic stream instead of exposing a task-native contract.

## Consequences

### Positive

- Makes task pages simpler and more stable.
- Keeps event ordering and aggregation logic in the backend, where the domain context already exists.
- Creates explicit API contracts for timeline, run detail, and tree live views.

### Negative

- Requires new endpoints, new payload types, and new SSE documentation in the OpenAPI surface.
- Increases backend implementation scope in the first wave.

### Risks

- Task live payloads can diverge from existing task detail views if designed separately. Mitigation: derive live payloads from shared task/read-model types and avoid parallel semantic definitions.
- SSE reconnect and cursor handling can become complex. Mitigation: define explicit event IDs/cursors for task streams and keep stream scope narrow per task or task tree.

## Implementation Notes

- Treat `GET /api/task-runs/{id}` as a required new read endpoint, not an optional enhancement.
- Keep session SSE focused on session UX; task SSE should model task-domain events directly.
- The task tree live view should solve the multi-agent screen without N+1 detail fetches.
