# ADR-001: First-Class Tasks Area in the Main App Shell

## Status

Accepted

## Date

2026-04-17

## Context

The Paper task screens represent a major operator surface, not a one-off modal or hidden experiment. The existing `web/` application already organizes top-level operator features under the `_app` shell with a matching route, a route-level page hook under `web/src/hooks/routes`, and a dedicated domain system under `web/src/systems/<domain>/`.

The task UI must fit that architecture instead of introducing a parallel feature layout. It must also behave as a first-class workspace-aware area alongside automation, bridges, network, knowledge, and skills.

## Decision

Implement tasks as a first-class area in the main app shell:

- add a primary route at `/_app/tasks`
- add a sidebar navigation entry for Tasks
- add a dedicated route hook at `web/src/hooks/routes/use-tasks-page.ts`
- add a dedicated domain system at `web/src/systems/tasks/`
- keep route files thin and move API/query/domain orchestration into the tasks system and route hook

## Alternatives Considered

### Alternative 1: Isolated route without primary navigation

- **Description**: Ship a tasks route but keep it outside the main sidebar.
- **Pros**: Lower rollout risk; less visible while the feature matures.
- **Cons**: Treats tasks as a secondary feature even though the design and scope are first-class. Reduces discoverability and breaks consistency with other operator surfaces.
- **Why rejected**: The initiative is full parity now, not an experimental soft launch.

### Alternative 2: Hidden or experimental route

- **Description**: Ship tasks behind an internal or experimental gate.
- **Pros**: Limits blast radius while backend work lands.
- **Cons**: Conflicts with the agreed rollout scope and weakens the value of implementing the full Paper surface.
- **Why rejected**: The user explicitly chose a first-wave, full-parity delivery.

## Consequences

### Positive

- Matches the current app-shell and systems architecture.
- Makes tasks discoverable and coherent with the rest of the operator UI.
- Keeps feature boundaries clean: routes for composition, systems for domain logic.

### Negative

- Requires sidebar, route-tree, and workspace-shell integration from the start.
- Raises the quality bar because tasks become part of the main operator flow immediately.

### Risks

- If tasks are added to the sidebar before backend parity exists, the route can feel incomplete. Mitigation: the techspec includes the backend work needed for full parity in the same first wave.

## Implementation Notes

- Reuse the established route/system pattern used by `automation`, `network`, and `knowledge`.
- Keep reusable task UI inside `web/src/systems/tasks/components/`.
- Keep route-local selection, tabs, URL state, and dialog orchestration inside the route hook.
