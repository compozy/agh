# ADR-004: Use Observer-Backed Read Models for Dashboard, Inbox, and Aggregate Task Views

## Status

Accepted

## Date

2026-04-17

## Context

The current task manager already provides CRUD, detail, dependencies, and run lifecycle surfaces. Separately, the observer layer already computes aggregate task read-side data such as:

- task summary buckets
- task metrics
- task health

The Paper dashboard and inbox screens are aggregate operator views. They are not merely expanded versions of task detail. Adding them directly to task detail/list payloads would overload the task manager with read-side concerns that fit the observer layer better.

## Decision

Expose dashboard, inbox, and aggregate task operator views through dedicated observer-backed read models and endpoints.

The task manager remains responsible for:

- task CRUD
- task detail
- dependencies
- run lifecycle commands
- task-native point reads

The observer/read side becomes responsible for:

- dashboard summary and metrics
- inbox-style triage views
- health-oriented aggregate views
- other task aggregates optimized for operator inspection

## Alternatives Considered

### Alternative 1: Expand the task manager with aggregate views

- **Description**: Add dashboard and inbox-style aggregates directly to the task manager API surface.
- **Pros**: Keeps all task endpoints under one logical owner.
- **Cons**: Mixes command/detail responsibilities with read-side aggregation concerns. Increases coupling between task write paths and UI-specific projections.
- **Why rejected**: The observer already owns the right computation model for these aggregates.

### Alternative 2: Build aggregates client-side from list/detail endpoints

- **Description**: Load tasks and derive dashboard/inbox in the frontend.
- **Pros**: Avoids new backend aggregate endpoints.
- **Cons**: Incomplete, expensive, and semantically weak for charts, queue health, triage, and personalized feeds.
- **Why rejected**: The existing analysis already shows client derivation is insufficient for design parity.

## Consequences

### Positive

- Matches the existing separation between task write flows and observability/read-side aggregation.
- Reuses existing observer computations instead of duplicating them.
- Produces cleaner APIs for dashboard and inbox screens.

### Negative

- Requires new contract types and transport wiring for observer-backed task endpoints.
- Adds another task-related API family that frontend code must understand.

### Risks

- Read-side freshness can lag behind write-side mutations if event propagation is inconsistent. Mitigation: keep observer projections derived from durable state and define expected freshness/reconnect behavior in the API contracts.
- Inbox semantics can grow beyond the current observer model. Mitigation: define the minimal inbox read model needed for the Paper screens and keep triage mutations explicit.

## Implementation Notes

- Extend observer-facing APIs rather than bloating `GET /api/tasks`.
- Keep inbox payloads task-native even if they are computed from observer state.
- Continue to expose health in `/api/observe/health`, but add task-specific aggregate endpoints for the richer Paper screens.
