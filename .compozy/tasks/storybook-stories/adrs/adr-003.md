# ADR-003: stories/ Subfolder Placement, Opt-in Autodocs

## Status

Accepted

## Date

2026-04-17

## Context

Two placement conventions exist in the codebase:
- `web/src/components/design-system/` groups stories under a `stories/` subfolder with a shared `story-frame.tsx` helper.
- Unit tests are co-located (`foo.tsx` + `foo.test.tsx`).

We need one convention for the ~79 stories created by this spec, applied uniformly across `packages/ui/src/components`, `web/src/components/ui`, and `web/src/systems/<name>/components`.

Separately, the storybook-stories skill suggests `tags: ["autodocs"]` universally; the user requested not to apply autodocs to every story.

## Decision

**Placement**: every component category folder owns a `stories/` subfolder. Example:
- `packages/ui/src/components/stories/button.stories.tsx`
- `web/src/components/ui/stories/dialog.stories.tsx`
- `web/src/systems/session/components/stories/chat-header.stories.tsx`
- Nested `tool-renderers/` gets its own `tool-renderers/stories/`.

**Autodocs**: off by default. Only add `tags: ["autodocs"]` to a story meta when the component is a public design-system primitive (`packages/ui/src/components/*`) or when the story explicitly documents configuration via `argTypes`. System and web/ui stories render visually only.

## Alternatives Considered

### Alternative 1: Co-located stories

- **Description**: `button.tsx` next to `button.stories.tsx` in the same folder.
- **Pros**: Matches the existing test file convention; zero nesting.
- **Cons**: Diverges from the already-established `design-system/stories/` pattern; mixing prod + story files in large folders (e.g. `session/components/`) bloats the listing.
- **Why rejected**: Loses the visual grouping that `stories/` subfolder gives in large folders.

### Alternative 2: Mixed by layer

- **Description**: `stories/` subfolder for base UI, co-located for systems.
- **Pros**: Fits each layer's folder size.
- **Cons**: Two conventions to remember and enforce.
- **Why rejected**: Consistency > micro-optimization.

## Consequences

### Positive

- Listing a component folder shows only production code; stories are a well-known sibling.
- Consistent with existing `design-system/stories/` layout — no migration needed for those files.
- Autodocs remain where they add value (design system) without generating noise for every domain component.

### Negative

- One extra directory per component folder.
- Relative imports from story to component are `../component` — longer paths but explicit.

### Risks

- Mis-placed stories would be missed by the Storybook glob. Mitigation: both globs already catch anything matching `**/*.stories.@(ts|tsx)` recursively.

## Implementation Notes

- Create `stories/` folders as needed during implementation; do not pre-create empty ones.
- Reuse `web/src/components/design-system/stories/story-frame.tsx` where it adds value; do not generalize it further.
- Tag policy is enforced in code review, not lint — write it in `.claude/skills/storybook-stories/SKILL.md` during the rollout.
