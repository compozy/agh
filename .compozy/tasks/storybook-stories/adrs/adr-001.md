# ADR-001: Dual Storybook Topology

## Status

Accepted

## Date

2026-04-17

## Context

The web monorepo hosts UI code in three locations: the reusable primitives package `packages/ui/src/components` (`@agh/ui`), the shadcn-style composed primitives in `web/src/components/ui`, and domain components under `web/src/systems/<name>/components`. Today a single Storybook instance lives in `web/.storybook` and indexes only `web/src/**/*.stories.@(ts|tsx)`. Stories for `packages/ui` therefore cannot be authored in-place without reaching across workspace boundaries with a relative glob, which couples the library's documentation to the web app and bypasses the library's own dependency graph.

## Decision

Run two independent Storybook 10 instances:
- `packages/ui/.storybook` — indexes `packages/ui/src/**/*.stories.@(ts|tsx)`; dev server on port 6007; renders only base primitives; no MSW, no router, no query client.
- `web/.storybook` (existing) — continues to index `web/src/**/*.stories.@(ts|tsx)` on port 6006; carries MSW + TanStack Query + router decorators.

Each workspace owns its Storybook config, scripts (`storybook`, `build-storybook`), and dev dependencies. The two instances share the same Tailwind v4 token layer via `@agh/ui/tokens.css`.

## Alternatives Considered

### Alternative 1: Single Storybook with Cross-Workspace Glob

- **Description**: Extend `web/.storybook/main.ts` to `stories: ["../src/**/*.stories.@(ts|tsx)", "../../packages/ui/src/**/*.stories.@(ts|tsx)"]`.
- **Pros**: One command to run, one URL to share, zero new config.
- **Cons**: Couples the library's stories to the web app's Vite config, node_modules, and provider decorators. If anyone publishes `@agh/ui` or extracts it, stories need to move anyway.
- **Why rejected**: Violates the workspace's dependency direction (packages/ui must not depend on web) and forces library stories to inherit web decorators (QueryClientProvider, Router) that base primitives do not need.

### Alternative 2: Stories Under web/src/stories/ui

- **Description**: Keep one Storybook, author stories under `web/src/stories/ui/*.stories.tsx` that import from `@agh/ui`.
- **Pros**: No config change.
- **Cons**: Breaks co-location — changing a primitive requires editing files in a different workspace. Drift risk is high.
- **Why rejected**: Co-location is a prerequisite per the storybook-stories skill.

## Consequences

### Positive

- Each Storybook composes only what its layer needs; no leaked decorators.
- `packages/ui` becomes independently consumable/publishable.
- Faster startup for pure design-system browsing (no MSW, no query client bootstrap).

### Negative

- Two dev servers and two build pipelines to maintain.
- Contributors need to know which port serves which layer.

### Risks

- Divergence between theme/token loading in the two instances. Mitigation: both import `@agh/ui/tokens.css` and `tw-animate-css` in their `preview.ts`, and the web instance additionally imports `web/src/styles.css`.
- Accidental cross-imports from `packages/ui` stories into `web/src`. Mitigation: oxlint import rules already restrict this path; document in READMEs.

## Implementation Notes

- Add `packages/ui/.storybook/{main.ts,preview.ts}` mirroring the web config minus MSW and provider decorators.
- Add `storybook` and `build-storybook` scripts to `packages/ui/package.json` and install Storybook 10 devDependencies in that workspace.
- Add root turbo pipeline entries if turbo tasks gate on storybook builds.
