# ADR-001: Provider-Scoped Bridge SDK and Runtime Model

## Status

Accepted

## Date

2026-04-15

## Context

AGH needs production bridge providers for Telegram, Slack, Discord, WhatsApp, Microsoft Teams, Google Chat, GitHub, and Linear. The previous bridge-adapter direction assumed one bridge instance per launched extension process, but that conflicts with provider-level multi-tenant modes such as GitHub App installations and Linear OAuth organizations.

The existing `sdk/examples/telegram-reference/` proves the negotiated bridge runtime concept, but its current shape is still instance-scoped. A production bridge SDK must support provider-scoped runtimes that multiplex multiple `BridgeInstance` records internally.

## Decision

Create a shared `internal/bridgesdk` package for provider-scoped bridge runtimes. Each provider lives in `extensions/bridges/<platform>/` as one standalone Go binary sharing the root `go.mod` and importing `internal/bridgesdk` for daemon protocol concerns.

The SDK provides:

- JSON-RPC peer (bidirectional request/response over stdio)
- provider-scoped initialization handshake
- provider instance cache and secret-binding materialization
- health check handler with customizable health probes
- delivery acknowledgment with sequence tracking
- Host API client for provider-owned bridge instances
- HTTP webhook server scaffold with guard rails and signature verification hooks
- adapter-local dedup, batching, and error classification helpers
- graceful shutdown coordination

Each provider implements only:

- platform-specific webhook parsing and signature verification
- platform API client behavior
- message and typed interaction mapping between platform format and AGH bridge events
- provider-specific config interpretation
- platform-specific secret requirements

## Alternatives Considered

### Alternative 1: Inline per provider, refactor later

- **Description**: Copy `telegram-reference` into each provider and refactor shared code later
- **Pros**: Lower initial abstraction cost
- **Cons**: immediate duplication, divergent hardening behavior, poor multi-tenant evolution
- **Why rejected**: provider-scoped runtime behavior is core architecture, not optional cleanup

### Alternative 2: One process per bridge instance

- **Description**: Launch one subprocess for every configured bridge instance
- **Pros**: stronger tenant isolation, simpler per-process config story
- **Cons**: conflicts with provider-level multi-tenant models, duplicates connections and caches, makes provider installs operationally expensive
- **Why rejected**: the approved bridge model keeps `BridgeInstance` as the daemon-owned routing unit while provider runtimes multiplex them internally

## Consequences

### Positive

- one provider runtime can manage many bridge instances cleanly
- protocol and operational hardening update in one place
- consistent error handling, health reporting, dedup, batching, and shutdown across providers
- future providers become straightforward additions

### Negative

- runtime contract changes require updating all provider binaries
- adds one more internal package to maintain

### Risks

- provider-scoped abstractions may not fit every tenant model immediately
- mitigate by validating Telegram first, then Slack or GitHub to exercise provider multiplexing pressure
