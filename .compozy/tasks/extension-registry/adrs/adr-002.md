# ADR-002: Separate CLI Namespaces for Skills and Extensions

## Status

Accepted

## Date

2026-04-13

## Context

AGH has two distinct artifact types: **skills** (prompt-only SKILL.md documents) and **extensions** (subprocess-based with extension.toml manifests, capabilities, Host API). Both need remote registry operations (search, install, remove, update). The question is whether to unify or separate the CLI surface.

Skills and extensions have fundamentally different lifecycles: skills are loaded into memory and injected into prompts; extensions are launched as subprocesses with health monitoring and restart logic.

## Decision

Maintain separate CLI namespaces:
- `agh skill search/install/remove/update` for skills
- `agh extension search/install/remove/update` for extensions

Both use the shared `RegistrySource` interface internally, but the user-facing commands remain distinct. This preserves the existing CLI structure (`agh skill list/view/info/create/search/install/remove/update` already exists) and avoids confusion between lightweight skills and heavyweight extensions.

## Alternatives Considered

### Alternative 1: Unified `agh extension` namespace

- **Description**: Merge everything under `agh extension`. Skills become a "type" of extension. `agh extension install --type skill <slug>`.
- **Pros**: Single command surface, conceptual simplification.
- **Cons**: Breaks existing `agh skill` commands, conflates fundamentally different artifact types, users must always specify `--type`.
- **Why rejected**: Skills and extensions have different install paths, different runtimes, and different security models. Forcing them into one namespace creates ambiguity.

### Alternative 2: New `agh registry` namespace

- **Description**: A third command group `agh registry search/install` as the unified entry point for remote operations.
- **Pros**: Clean separation between local management and remote operations.
- **Cons**: Adds a third concept to learn, fragments the user experience across three namespaces, confusing overlap with existing install commands.
- **Why rejected**: Three namespaces is worse than two. Users shouldn't need to learn that "search" lives in a different command tree than "list".

## Consequences

### Positive

- No breaking changes to existing CLI commands
- Clear mental model: `agh skill` = prompt documents, `agh extension` = runtime plugins
- Each namespace can evolve independently
- Tab-completion and help text remain unambiguous

### Negative

- Some duplication in CLI flag definitions between `agh skill install` and `agh extension install`
- Users working with both types need to know which namespace to use

### Risks

- None significant. The separation mirrors the internal architecture accurately.

## Implementation Notes

- New commands for extensions: `search`, `install <slug>`, `remove`, `update [name|--all]`
- Skills already have all these commands — only need to wire them to the new `RegistrySource` interface
- Both command sets share the same `--from` flag to specify registry source
