# ADR-003: Config from Root Only, Agents/Skills from All Dirs

## Status

Accepted

## Date

2026-04-06

## Context

Workspaces support multi-root: a primary `root_dir` plus optional `additional_dirs`. When scanning `.agh/` folders across multiple directories, we need clear rules for which resources are discovered from where and how conflicts are resolved.

The key question: should `config.toml` be loaded from all directories (creating an N-layer merge chain) or only from the root directory?

## Decision

**Asymmetric discovery rules:**
- `config.toml`: loaded ONLY from `root_dir/.agh/config.toml` — additional dirs do NOT contribute config
- `agents/`: discovered from ALL dirs (root + additional), merged by name, root wins
- `skills/`: discovered from ALL dirs (root + additional), merged by name, root wins

**Merge precedence** (left wins):
```
root_dir/.agh/ > add_dirs[0]/.agh/ > add_dirs[1]/.agh/ > ... > ~/.agh/
```

## Alternatives Considered

### Alternative 1: Config from all dirs (full N-layer merge)

- **Description**: Each additional dir's `.agh/config.toml` contributes a layer to the config cascade
- **Pros**: Maximum flexibility, every dir can customize behavior
- **Cons**: N config files merging creates impossible-to-debug precedence issues. Devil's Advocate called this "a recipe for bugs impossible to debug." Current 3-layer merge (defaults → global → workspace) is already at the complexity limit.
- **Why rejected**: Config defines project-level behavior (providers, permissions, limits). Multiple conflicting sources create ambiguity. One project = one config.

### Alternative 2: Agents/skills from root only

- **Description**: Only `root_dir/.agh/` contributes all resources — additional dirs provide no `.agh/` scanning
- **Pros**: Simplest model, no merge questions
- **Cons**: Defeats the purpose of multi-root — users add additional dirs precisely to bring their resources into scope
- **Why rejected**: The whole point of additional dirs is resource sharing across project boundaries

## Consequences

### Positive

- Config cascade stays at 3 layers (defaults → global → root workspace) — predictable and debuggable
- Agents/skills benefit from multi-root discovery — a shared-libs dir can contribute its agents
- Clear mental model: "config comes from your project, resources come from everywhere"

### Negative

- Asymmetry between config and resources may confuse users who add a shared-config dir as additional_dir expecting its config.toml to apply
- Documented limitation: needs prominent documentation

### Risks

- Users who want shared config across workspaces must use `~/.agh/config.toml` (global) instead of per-dir config — may not be obvious. Mitigated by documentation and `agh workspace info` showing which config is active.

## Implementation Notes

- Config loading: `config.Load(WithWorkspaceRoot(root_dir))` — unchanged signature, uses only root_dir
- Agent scanning: new function iterates `[root_dir, add_dirs...]`, collects `<dir>/.agh/agents/*/AGENT.md`, merges by name
- Skills scanning: Resolver provides paths to skills registry, which handles loading
