# ADR-002: Random Hex ID with Human-Friendly Name Field

## Status

Accepted

## Date

2026-04-06

## Context

The workspace entity needs an identifier strategy. The ID must be stable (survive directory renames), usable in CLI commands, and unique across all registered workspaces.

## Decision

Use `ws_<random-hex>` as the primary key (16-char hex, same pattern as `sess_` and `turn_`), with a separate `name TEXT UNIQUE` column for human-friendly CLI usage. The `name` defaults to `filepath.Base(root_dir)` and auto-deduplicates with `-2` suffix on collision. Users can reference workspaces by either ID or name in CLI/API.

## Alternatives Considered

### Alternative 1: Slug derived from dirname as PK

- **Description**: Use `filepath.Base(root_dir)` as the primary key (e.g., `myapp`)
- **Pros**: Human-readable, no lookup needed, matches CLI ergonomics
- **Cons**: Guaranteed collisions — two dirs named `app` at different paths cannot coexist. Data model contradiction between PK uniqueness and real-world path diversity.
- **Why rejected**: Codex review identified this as the most critical flaw — slug collisions are guaranteed in common scenarios (monorepos, multiple projects with same basename)

### Alternative 2: Content-addressed hash of root_dir

- **Description**: `sha256(canonical_root_dir)[:12]` as PK
- **Pros**: Deterministic, no collision for different paths, no external state needed
- **Cons**: Not human-readable, changes if directory moves (hash of new path differs), no stable identity across renames
- **Why rejected**: Doesn't survive directory renames, which is a stated requirement

## Consequences

### Positive

- IDs are stable across directory renames (random, not path-derived)
- Consistent with existing `sess_`, `turn_` ID patterns in the codebase
- Human-friendly `name` field provides CLI ergonomics without compromising identity
- `root_dir UNIQUE` constraint prevents duplicate registrations for the same path

### Negative

- Two lookups needed: by ID (PK) and by name (UNIQUE index) — requires two query methods
- Name deduplication logic adds minor complexity

### Risks

- Name collisions resolved with `-2` suffix may confuse users with many similarly-named projects — mitigated by allowing `agh workspace edit --name` to rename

## Implementation Notes

- ID generation: reuse existing `store.newID("ws")` function
- Name dedup: on insert conflict, append `-2`, `-3`, etc. until unique
- Lookup: `Get(idOrName)` tries ID first, then name, similar to how git resolves branch names vs commit hashes
