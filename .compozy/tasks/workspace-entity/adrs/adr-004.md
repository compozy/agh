# ADR-004: New internal/workspace/ Package

## Status

Accepted

## Date

2026-04-06

## Context

The workspace concept involves: persistent storage (SQLite table), resolution logic (filesystem scanning, config cascade, resource merging), caching (mtime-based invalidation), and CLI/API surface. This logic needs a home in the package layout.

## Decision

Create a new `internal/workspace/` package containing the Resolver, types, WorkspaceStore interface, and functional options. This package sits between `config/` (types it consumes) and `session/` (which consumes the Resolver via interface injection).

Package structure:
```
internal/workspace/
    workspace.go   -- Workspace, ResolvedWorkspace, error types
    resolver.go    -- Resolver struct, Resolve, ResolveOrRegister, Register/Unregister/Update/List/Get
    store.go       -- WorkspaceStore interface
    options.go     -- functional options for NewResolver
```

## Alternatives Considered

### Alternative 1: Extend store/ with workspace CRUD

- **Description**: Add `WorkspaceInfo` struct and CRUD methods directly to `store/GlobalDB`
- **Pros**: No new package, ships fastest, follows existing pattern of store/ owning all persistence
- **Cons**: Resolution logic (scanning, merging, caching) doesn't belong in a persistence package. store/ would grow from pure data access to include filesystem walking and resource merging.
- **Why rejected**: store/ should remain pure persistence. Resolution logic is complex enough to warrant separation.

### Alternative 2: Extend session/ with workspace logic

- **Description**: Add workspace resolution to session.Manager, since it's the primary consumer
- **Pros**: No new package, workspace logic close to its primary user
- **Cons**: Creates coupling — every workspace feature requires modifying session/. Other consumers (daemon/ for dreams, observe/ for reconciliation) would need to import session/ for workspace capabilities.
- **Why rejected**: Workspace has consumers beyond session/ — embedding it there creates wrong dependency direction

### Alternative 3: Extend config/ with workspace resolution

- **Description**: config.Load() already does workspace-aware cascade — enrich it with multi-root and resource discovery
- **Pros**: Natural extension of existing config cascade
- **Cons**: config/ is currently stateless (loads files, returns struct). Adding SQLite persistence, caching, and CRUD mutates its purpose entirely.
- **Why rejected**: config/ should remain a stateless loader. Adding persistent state changes its architectural role.

## Consequences

### Positive

- Clean package boundary with single responsibility
- Follows AGH pattern: new capability = new package wired into daemon/
- workspace/ can evolve independently (Phase 2: security scopes, Phase 3: network workspaces)
- Testable in isolation with mock WorkspaceStore

### Negative

- One more package in the layout (12 instead of 11)
- Cross-cutting change touches session/, store/, daemon/, httpapi/, udsapi/, cli/, config/, acp/, observe/, memory/, skills/

### Risks

- Package boundary may feel over-engineered for alpha — mitigated by keeping the package small (4 files) and the Resolver API minimal

## Implementation Notes

- Dependency flow: `workspace/` imports `config.HomePaths` and `config.Config` (types only)
- `session/` consumes `workspace.Resolver` via interface injection
- `store/` implements `workspace.WorkspaceStore`
- `daemon/` wires everything as composition root
