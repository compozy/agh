# ADR-001: Resolver with Persistent Backing over Manager Entity

## Status

Accepted

## Date

2026-04-06

## Context

AGH needs a formal Workspace concept to replace bare path strings threaded through session, config, store, and daemon packages. The workspace must support multi-root directories, resource discovery (agents/skills), config cascade, session ownership, and CLI CRUD operations.

A council debate between four architectural archetypes (architect-advisor, pragmatic-engineer, devils-advocate, the-thinker) surfaced a core tension: should workspace be a CRUD entity with full lifecycle management (Manager pattern), or a resolver that computes workspace state from the filesystem with a persistent hint table?

## Decision

Use a **Resolver with persistent backing** — a hybrid that stores registration hints in SQLite but treats the filesystem as the source of truth. The Resolver computes `ResolvedWorkspace` snapshots on demand by scanning `.agh/` directories and merging resources.

Key semantics:
- `agh workspace add` does not "create" a workspace — it registers a resolution hint
- `agh workspace remove` does not delete files — it removes the alias
- `Resolve()` always reads current filesystem state, using cached snapshots when mtimes match
- The backing table provides stable IDs for session foreign keys and CLI queries

## Alternatives Considered

### Alternative 1: Full Manager Entity (Architect position)

- **Description**: New `internal/workspace/` package with a `Manager` struct owning Create/Update/Delete lifecycle, similar to `session.Manager`
- **Pros**: Clean package boundary, familiar CRUD pattern, explicit lifecycle ownership
- **Cons**: Creates a registry that drifts from filesystem truth, requires reconciliation logic, over-models a concept whose lifecycle is owned by the filesystem
- **Why rejected**: The filesystem owns workspace existence — the system should remember how to find it, not own its lifecycle

### Alternative 2: Table in store/ only (Pragmatic position)

- **Description**: Add `workspaces` table and `WorkspaceInfo` struct to existing `store/` package, no new package
- **Pros**: Ships fastest, no new patterns to learn, minimal structural change
- **Cons**: Mixes resolution logic into store/ which should be pure persistence, workspace scanning and merging logic doesn't belong in a storage package
- **Why rejected**: Workspace resolution (scanning, merging, caching) is complex enough to warrant its own package boundary

### Alternative 3: Pure Resolver without persistence (Thinker position)

- **Description**: Enrich `config.Load()` to walk `.agh/` overlays and produce deterministic environment snapshots — no SQLite table, no stored entity
- **Pros**: No drift between registry and filesystem, no stale data, pure functional resolution
- **Cons**: Cannot support `agh workspace list`, session-to-workspace FK queries, dream consolidation grouping, or workspace naming/aliasing
- **Why rejected**: CLI CRUD operations and session queries require a stable backing store

## Consequences

### Positive

- Filesystem remains source of truth — no reconciliation logic needed
- Stable IDs enable session FK, dream grouping, and CLI queries
- Resolver pattern naturally supports caching with mtime invalidation
- Clean package boundary keeps resolution logic isolated

### Negative

- Backing table can become stale if directories are moved/deleted (mitigated by staleness checks on List/Resolve)
- Two concepts to reason about: the stored hint vs the resolved snapshot

### Risks

- Symlink targets changing after registration — mitigated by re-evaluating `filepath.EvalSymlinks()` on every `Resolve()` call
- Path mutations (rename/move) orphaning workspace rows — mitigated by `os.Stat()` check on `List()` results

## Implementation Notes

- Package: `internal/workspace/` with Resolver as primary type
- Table: `workspaces` in existing global DB (`~/.agh/agh.db`)
- Interface: `WorkspaceStore` defined in `workspace/`, implemented in `store/`
- Wiring: `daemon/` injects Resolver into `session.Manager` via new `WithWorkspaceResolver` option
