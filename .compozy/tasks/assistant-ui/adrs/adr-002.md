# ADR-002: Keep the Go Daemon as the Sole AI SDK Wire Encoder

## Status

Accepted

## Date

2026-04-21

## Context

`internal/api/httpapi/prompt.go` already translates ACP `session/update` notifications into AI SDK v6 UI Message Stream frames. It is ~340 LOC of focused Go and emits the full set of frames the assistant-ui runtime consumes. There is also a published npm package, `@mcpc-tech/acp-ai-provider`, that implements `LanguageModelV3` over ACP. The temptation is to add that package to a Node sidecar and let AI SDK handle the wire format in JavaScript.

AGH has a hard architectural constraint: the daemon is a single Go binary. Sidecars require a written techspec and are reserved for cases where there is no reasonable Go alternative.

## Decision

The Go daemon remains the sole emitter of the AI SDK UI Message Stream. No Node sidecar. No `LanguageModelV3` shim. ACP → AI SDK frame translation continues to live in `internal/api/httpapi/prompt.go`.

## Alternatives Considered

### Alternative 1: Node sidecar using `@mcpc-tech/acp-ai-provider`

- Description: run a TS process that wraps ACP as a `LanguageModelV3`. The Go daemon would proxy SSE through it.
- Pros: reuses upstream provider; frees Go from tracking AI SDK frame schema.
- Cons: breaks single-binary; adds deploy and supervision surface; the provider flattens permission/plan/mode semantics into string-prefixed prompt text (`"System: ..."`, `"[Tool Call: name(args)]"`); adds an extra hop and an extra process failure mode.
- Why rejected: architectural violation and semantic loss with no real upside for AGH.

### Alternative 2: Rewrite `prompt.go` in terms of a Go-native LanguageModel interface

- Description: write a Go equivalent of `LanguageModelV3` inside AGH and express frame emission through it.
- Pros: theoretically cleaner abstraction.
- Cons: duplicates a TypeScript-only spec in Go; the AI SDK wire format is the real contract, not the provider interface.
- Why rejected: the frame protocol is the interface we need to be correct about; an extra abstraction does not help.

### Alternative 3: Move frame emission to the browser

- Description: emit raw ACP JSON-RPC on the wire and translate in the client.
- Pros: zero Go-side knowledge of AI SDK.
- Cons: re-introduces the client-side mapping layer we want to delete; ACP over the wire would conflict with the AI SDK Message Stream contract assistant-ui expects.
- Why rejected: contradicts ADR-001 and the whole migration premise.

## Consequences

### Positive

- Preserves single-binary delivery and supervision model.
- Keeps ACP-specific semantics (permission, plan, mode, terminal) available as first-class custom data parts.
- Keeps the full SSE protocol observable in AGH logs and dashboards.

### Negative

- AGH Go code must track AI SDK v6 frame schema changes.
- Any AI SDK protocol upgrade (for example, AI SDK v7 shape changes) is a Go-side task rather than a dependency bump on a sidecar.

### Risks

- Risk: AI SDK v6 frame schema evolves in minor releases.
  - Mitigation: keep contract-level integration tests that assert the exact SSE payload; regenerate the OpenAPI fixtures on upgrade.
- Risk: ACP events gain new kinds (new tool lifecycle states, new plan semantics) faster than we port them.
  - Mitigation: `data-agh-event` remains a generic fallback, so unmapped events stay visible without blocking a user's turn.

## Implementation Notes

- `internal/api/httpapi/prompt.go` stays the canonical encoder.
- New ACP event types map to new AI SDK frames (or new `data-agh-*` parts) in Go, not in JS.
- No frontend translation layer, no client-side adapter for the live stream.
