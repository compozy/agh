# ADR-002: Re-Resolve Provider-Owned Runtime Fields On Session Override

## Status

Accepted

## Date

2026-04-20

## Context

Today `AgentDef` may contribute `provider`, `command`, and `model`, while `Config.ResolveAgent()` also merges provider defaults. If a session switches from one provider to another, preserving an agent's provider-specific `command` or `model` can create invalid mixed-runtime configurations, such as a Claude ACP command paired with a Codex provider selection.

The feature needs one deterministic rule for which runtime fields belong to the session-selected provider and which remain owned by the agent definition.

## Decision

When a session provider override is present, AGH will re-resolve the **runtime layer** from the chosen provider:

- `provider` comes from the session selection.
- `command` comes from the chosen provider configuration.
- `model` defaults come from the chosen provider configuration.

The agent definition continues to own behavior-local fields:

- `prompt`
- `tools`
- `permissions`
- agent-local MCP servers and other agent metadata

Per-session model override is explicitly out of scope for this feature.

## Alternatives Considered

### Alternative 1: Preserve `agent.command` and `agent.model` when defined

- **Description**: Only change the provider name and keep explicit agent runtime fields.
- **Pros**: Smaller short-term diff against current resolution behavior.
- **Cons**: Creates invalid runtime mixes and unclear precedence rules.
- **Why rejected**: The result is not a coherent engine switch.

### Alternative 2: Allow provider override only when the agent has no explicit `command` or `model`

- **Description**: Reject overrides for agents with provider-specific runtime fields.
- **Pros**: Avoids mixed-runtime states.
- **Cons**: Makes the feature surprisingly unavailable for many useful agents.
- **Why rejected**: Too restrictive for the intended flexibility.

## Consequences

### Positive

- Provider override semantics stay coherent and easy to reason about.
- Runtime launch state remains internally consistent for the chosen engine.
- The change can be encapsulated in config-resolution helpers instead of scattered across session start logic.

### Negative

- Agents that currently rely on provider-specific explicit `command` or `model` lose those fields during an overridden session.
- A future per-session model picker will need a separate design instead of piggybacking on this feature.

### Risks

- Runtime ownership boundaries may be unclear in the code if provider override is implemented by ad-hoc field mutation.
- Mitigation: add an explicit config-resolution helper for "agent + optional session provider" semantics.
- Provider override also swaps the provider-owned MCP layer, which can surprise operators if not tested explicitly.
- Mitigation: assert in tests that provider MCP servers are replaced while global and agent MCP layers remain intact.

## Implementation Notes

- Prefer a dedicated config helper over mutating `AgentDef` in many call sites.
- The helper should take an `AgentDef` plus optional `provider` override, clone the agent definition, overwrite `Provider`, clear explicit `Command` and `Model` when an override is present, then reuse the shared resolution path.
- `ResolvedAgent.Provider` is the source-of-truth field that flows into session persistence and transport payloads.
- Provider-level MCP server layers should continue to come from the selected provider, then merge with agent/global MCP layers using the existing resolution rules.
