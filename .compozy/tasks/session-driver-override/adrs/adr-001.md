# ADR-001: Model Session Driver Selection As A Provider Override

## Status

Accepted

## Date

2026-04-20

## Context

AGH sessions currently choose an `agent_name`, then resolve the effective runtime through `internal/config.ResolveAgent()`. The session manager itself still owns one injected `AgentDriver`, which is the daemon's ACP transport adapter, not a registry of Claude/Codex/OpenCode-specific runtime implementations.

The requested feature is to let one agent keep a default ACP runtime while allowing each session to choose a different engine for that conversation. The design must fit the current architecture without introducing a second agent-selection system or a parallel runtime catalog unless it is strictly necessary.

## Decision

AGH will model per-session ACP driver selection as an optional **session-level provider override** layered on top of the existing agent-resolution flow.

- `agent_name` remains the identity and behavior source for the session.
- A new optional session `provider` field selects the ACP runtime/provider for that session.
- The session manager keeps the current global ACP `AgentDriver`; it does not grow a per-session driver registry.

## Alternatives Considered

### Alternative 1: Treat the feature as an agent swap

- **Description**: Selecting another runtime would require choosing another `agent_name`.
- **Pros**: Reuses existing create-session semantics with minimal transport changes.
- **Cons**: Conflates prompt/behavior identity with runtime choice and forces duplicate agents for simple engine changes.
- **Why rejected**: The user explicitly wants one agent definition with per-session runtime flexibility.

### Alternative 2: Introduce a separate session-driver catalog

- **Description**: Add a new top-level abstraction distinct from both agents and providers.
- **Pros**: Could model runtime engines independently from agent config.
- **Cons**: Adds a new concept, new persistence rules, and new validation paths without a clear need in the current codebase.
- **Why rejected**: Existing provider resolution already expresses the needed runtime selection.

## Consequences

### Positive

- Preserves current architecture and keeps the change centered on config/session resolution.
- Lets one agent definition run on multiple ACP providers without duplicating prompts or tools.
- Avoids introducing a new runtime abstraction that would need separate UI, API, and persistence semantics.

### Negative

- The user-facing wording must stay precise so "provider" and "agent" are not confused.
- Provider override semantics must be carefully defined for runtime-owned fields such as command and model.

### Risks

- Existing code and UI often use "agent" as shorthand for the whole session identity.
- Mitigation: expose `agent_name` and `provider` as distinct session fields and keep validation explicit.

## Implementation Notes

- The implementation should extend session creation and resume flows with an optional provider value.
- The current daemon-owned ACP transport adapter remains unchanged.
