# ADR-003: Persist Effective Session Provider And Fail Explicitly On Mismatch

## Status

Accepted

## Date

2026-04-20

## Context

Sessions already persist runtime metadata such as `agent_name` and `acp_session_id`, but not provider/runtime selection. The requested behavior requires a session that was started with a non-default engine to resume with that exact engine later. The user also chose deterministic behavior over automatic fallback when the requested engine is unavailable.

Without persistence, a resumed session may silently change engines if the agent default changes. Without strict failure rules, AGH may resume on a different runtime than the operator expects.

## Decision

AGH will persist the **effective session provider** in session metadata and reuse it on resume. If the requested or persisted provider cannot be resolved at create or resume time, AGH will fail explicitly and will not fall back to the current agent default.

Legacy sessions that predate this field are handled as a one-time repair path: if inactive session metadata has no stored provider, AGH resolves the effective provider from the stored agent plus workspace config, persists it immediately, and only then applies the strict persisted-provider rule.

## Alternatives Considered

### Alternative 1: Persist overrides only when the user explicitly selected one

- **Description**: Sessions created with the default provider keep resolving against current agent config.
- **Pros**: Smaller persistence surface.
- **Cons**: Session identity remains partly time-dependent and harder to reason about.
- **Why rejected**: The approved design favors deterministic session runtime identity.

### Alternative 2: Fall back silently to the current agent default

- **Description**: If a selected provider is missing, resume on whatever the agent resolves to now.
- **Pros**: More tolerant in degraded environments.
- **Cons**: Changes the conversation engine invisibly and undermines operator trust.
- **Why rejected**: The user explicitly rejected silent fallback.

## Consequences

### Positive

- Session runtime identity becomes stable across resume.
- Error handling is predictable and operator-visible.
- Session APIs can report the effective provider directly instead of reconstructing it indirectly.

### Negative

- Create and resume can fail more often after config edits that remove providers.
- Session metadata, session info, and global indexes must grow a provider field.

### Risks

- Legacy sessions are coupled once to whatever provider their agent resolves to at first post-upgrade repair time.
- If that repair cannot resolve the agent or provider anymore, resume and reconcile fail explicitly until the session is repaired or removed.

## Implementation Notes

- The persisted field should represent the effective provider used by the session runtime.
- The one-time repair path should run before resume and reconcile persist session rows into the global index.
- Error messages should name both the session id and the missing provider; structured logs may additionally include `agent_name`.
