# ADR-005: Migrate Session Provider State In Place And Repair Legacy Metadata Once

## Status

Accepted

## Date

2026-04-20

## Context

The feature adds an effective provider field to the session read model, but AGH already has developer-local session indexes and on-disk session metadata in use. `CREATE TABLE IF NOT EXISTS` will not update an existing `sessions` table, and older session metadata files do not contain a provider field at all.

The project is greenfield alpha, but deleting every local session index is unnecessary when SQLite can add the column safely in place. The design still needs a crisp rule for legacy sessions whose metadata has no persisted provider.

## Decision

AGH will migrate provider persistence in two explicit steps:

- The global `sessions` table adds `provider TEXT NOT NULL DEFAULT ''` through the existing session-column migration path.
- Legacy inactive session metadata with a blank provider is repaired once by resolving the effective provider from the stored agent plus resolved workspace config, then persisting that provider back to disk before reconcile or resume continues.

After that one-time repair, normal strict semantics apply: create and resume use the persisted provider and fail explicitly when it is unavailable.

## Alternatives Considered

### Alternative 1: Drop and recreate the global sessions index

- **Description**: Delete or rebuild the `sessions` table and repopulate it from current on-disk state only.
- **Pros**: Simple schema shape after migration.
- **Cons**: Unnecessarily wipes local session catalog state for every developer.
- **Why rejected**: In-place migration is cheaper and avoids gratuitous local disruption.

### Alternative 2: Fail every legacy session whose metadata has no provider

- **Description**: Treat blank provider as irrecoverable and require manual cleanup.
- **Pros**: No transitional repair logic.
- **Cons**: Breaks all pre-feature sessions immediately after upgrade.
- **Why rejected**: Too harsh for a local developer workflow when the provider can be derived once.

### Alternative 3: Re-resolve provider on every resume when the field is blank

- **Description**: Keep using current agent resolution until the session happens to be recreated.
- **Pros**: Minimal migration work.
- **Cons**: Reintroduces silent runtime drift and undermines deterministic persistence.
- **Why rejected**: The approved design requires persisted runtime identity, not repeated fallback.

## Consequences

### Positive

- Existing global DBs migrate safely without requiring table deletion.
- Pre-feature sessions become deterministic after one repair write.
- Reconcile and resume can rely on a persisted provider field after migration.

### Negative

- The first post-upgrade read of a legacy session depends on current workspace and agent resolution.
- Migration and repair need dedicated coverage in both global DB and session tests.

### Risks

- Legacy repair can still fail if the workspace or agent definition no longer resolves.
- Mitigation: fail explicitly with session id and provider/agent context, and keep repair limited to inactive sessions.

## Implementation Notes

- Add the new column in the existing `migrateSessionColumns` path before any session index reads.
- Ensure legacy copy-style migrations that rebuild the `sessions` table also include the new column.
- Run the metadata repair before `ReconcileSessions` upserts session rows, so the global index does not keep blank providers.
