# BUG-002: ACP stop-path could fail full verification with a stale process-group EPERM

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** `84e62142` + local task_08 changes
- **OS:** macOS
- **Browser:** n/a
- **URL:** n/a

## Summary

The final `make verify` pass intermittently failed in `internal/acp` even after the subprocess tree had already exited. The failure came from forced process-group cleanup during stop handling and blocked the task closeout despite provider-override scenarios already being green.

## Reproduction

```bash
make verify
```

Observed before the fix:

- `internal/acp TestStopTerminatesWrappedProcessTree` failed intermittently
- the failure surfaced as `wait for process tree exit: signal process group ... operation not permitted`

## Expected

If the follow-up wait proves the process group is already gone, stop should succeed and verification should not fail on a stale signal error.

## Root cause

`internal/procutil.KillCommandProcessGroupAndWait` joined the result of the `SIGKILL` attempt with the result of `WaitForCommandProcessGroupExit`. Under full-suite load, the process group could disappear between those two steps, leaving a transient `EPERM` from the signal attempt even though the wait confirmed the tree was gone. That stale `EPERM` was still returned and failed ACP shutdown tests.

## Fix

- Added `joinProcessGroupKillResult` in `internal/procutil/process_group_unix.go`
- Suppressed `EPERM` only when `WaitForCommandProcessGroupExit` succeeds, preserving real wait failures and non-`EPERM` signal errors
- Added focused regression tests in `internal/procutil/process_group_unix_test.go`

## Verification

- `go test ./internal/procutil ./internal/acp -run 'TestJoinProcessGroupKillResult|TestStopTerminatesWrappedProcessTree'`
- `go test -race ./internal/acp -run TestStopTerminatesWrappedProcessTree -count=20`
- `make codegen-check`
- `make verify`

## Impact

- **Users Affected:** maintainers and CI/full-gate runs exercising ACP shutdown tests
- **Frequency:** sometimes under the full race-enabled suite
- **Workaround:** rerun the full gate; there was no deterministic fix without code changes
