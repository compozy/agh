# BUG-001: Removed-provider resume returned a masked 500 instead of an explicit provider failure

**Severity:** High
**Priority:** P0
**Type:** Functional | UI | Data
**Status:** Fixed

## Environment

- **Build:** `84e62142` + local task_08 changes
- **OS:** macOS
- **Browser:** Chromium via Playwright
- **URL:** `/api/sessions/{id}/resume` and `/session/{id}`

## Summary

When a persisted session was resumed after its provider disappeared from workspace config, the backend returned a generic 500 response body instead of an explicit provider/session failure. That broke the task-08 requirement for visible, operator-facing mismatch semantics and prevented the web resume-failure UX from proving the missing provider end to end.

## Reproduction

```bash
AGH_E2E_QA_OUTPUT_DIR=/Users/pedronauck/dev/compozy/agh/.compozy/tasks/session-driver-override \
  bunx playwright test e2e/session-provider-override.spec.ts
```

Observed before the fix:

- resume returned `500 Internal Server Error`
- the response body was only `{"error":"Internal Server Error"}`
- the inline resume-failure panel could not prove the missing provider/session details from the backend contract

## Expected

Resume must fail explicitly with a client-visible error that includes the session id and missing provider, and the web UI must render the dedicated inline failure state from that response.

## Root cause

Provider-resolution failures coming out of `ResolveProvider`/`ResolveSessionAgent` were plain config errors, so `internal/api/core/statusForSessionError` did not recognize them as provider-availability failures. The transport mapped them to HTTP 500, and `RespondError` correctly masked the 500 body, which erased the session/provider detail that the feature depends on.

## Fix

- Added the config-level sentinel `aghconfig.ErrProviderUnavailable`
- Wrapped unknown and invalid provider-resolution failures with that sentinel in `internal/config/provider.go`
- Mapped that sentinel to `400 Bad Request` in `internal/api/core/session_workspace.go`
- Added narrow config/core regressions plus HTTP and UDS integration coverage for removed-provider resume failures
- Added browser E2E coverage to prove the inline resume-failure state end to end

## Impact

- **Users Affected:** any operator resuming a session whose persisted provider was later removed or renamed
- **Frequency:** always when the provider is unavailable
- **Workaround:** manually restore the missing provider in workspace config; no safe UI workaround existed
