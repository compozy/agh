# ADR-004: Network Manager as Boot-Phase Observer

## Status

Accepted

## Date

2026-04-10

## Context

The AGH Network runtime needs to integrate with the daemon lifecycle. The design question is whether the Network Manager should be a boot-phase component (initialized at daemon startup, like the Observer) or created on-demand (like session-scoped resources).

The Network Manager owns the embedded NATS server, manages peer registrations, handles message routing, and delivers inbound messages to sessions. It needs access to session lifecycle events and config.

The Architect advisor recommended the Network Manager be an observer of sessions, not a property of sessions — avoiding coupling that would require session schema migrations for every network protocol change.

## Decision

The Network Manager is a boot-phase component initialized in a new `bootNetwork` phase between `bootRuntime` and `bootHooks`. It follows the Observer pattern: receives session lifecycle events via the existing `Notifier` interface, manages peer state in-memory, and encapsulates all NATS server details. Because the session manager is already constructed during `bootRuntime`, network integration with `session.Manager` is done via post-construction setters rather than constructor injection.

Shutdown order: drain daemon NATS connection → stop NATS server → after sessions but before registry close.

## Alternatives Considered

### Alternative 1: On-Demand Creation

- **Description**: Create Network Manager only when the first session requests network participation
- **Pros**: No startup cost when network unused
- **Cons**: First session incurs NATS server startup latency, complicates lifecycle management, harder to test
- **Why rejected**: NATS server startup is fast (~100ms), and boot-phase initialization provides predictable lifecycle

### Alternative 2: Network as Session Property

- **Description**: Each session owns its own NATS connection and peer state
- **Pros**: Full isolation between sessions
- **Cons**: Duplicated NATS connections, no shared peer registry, can't route between sessions efficiently, couples network state to session model
- **Why rejected**: Creates N connections instead of 1, and the Architect specifically warned against coupling network into the session model

## Consequences

### Positive

- Predictable startup/shutdown lifecycle aligned with existing daemon patterns
- Single NATS server instance shared by all sessions
- Network state isolated from session state — protocol changes don't require session migrations
- Testable via factory injection (same pattern as httpFactory, udsFactory)
- Clean shutdown: drain → stop NATS → cleanup

### Negative

- NATS server starts even if no session uses the network (when enabled)
- Boot sequence grows by one phase

### Risks

- NATS server startup failure could block daemon boot: mitigated by making network optional (enabled=false default) and non-fatal startup with degraded mode

## Implementation Notes

- New `bootNetwork` phase in `internal/daemon/boot.go`
- `networkManagerFactory` in daemon struct (same pattern as `httpFactory`)
- Network Manager added to `RuntimeDeps` for server wiring
- Config: `[network] enabled = false` by default — NATS server only starts when enabled
- Interface consumed by session manager: `NetworkPeerLifecycle` with `JoinSpace(sessionID, peerID, space)` and `LeaveSpace(sessionID)`
- Wiring model: `bootNetwork` calls `sessions.SetNetworkPeerLifecycle(networkManager)` and `sessions.SetTurnEndNotifier(networkManager.OnTurnEnd)` after the network manager is created
- Shutdown registered in `bootCleanup` LIFO stack
