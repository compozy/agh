# ADR-008: Direct Interfaces and Notifier Pattern (No Event Bus)

## Status

Accepted

## Date

2026-04-02

## Context

The previous AGH used an embedded NATS message bus for all inter-component communication (agent messaging, hook events, topology updates, status broadcasts). The Vertical Slices proposal suggested an in-process event bus as a simpler alternative.

Research found that only 1/10 major Go projects (Grafana) uses an event bus, and Grafana has been simplifying theirs over time. The Codex review warned: "Do not let the event bus become the new kernel." Rich workflows (start session → open DB → spawn agent → stream events → expose via API) are not naturally event-only flows.

## Decision

Use **direct function calls through small interfaces** for synchronous request/response workflows. Use a **Notifier interface** for asynchronous fan-out (observability recording, SSE streaming to web clients).

No event bus. No NATS. No reflection-based message routing.

Synchronous path (interfaces defined where consumed):
```go
// session/ defines what it needs from storage
type EventRecorder interface {
    Record(ctx context.Context, ev Event) error
}

// session/ defines what it needs from ACP
type AgentDriver interface {
    Start(ctx context.Context, opts StartOpts) (*Process, error)
    Prompt(ctx context.Context, proc *Process, msg PromptRequest) error
    Stop(ctx context.Context, proc *Process) error
}
```

Asynchronous fan-out (Notifier pattern):
```go
// session/ defines notifications it emits
type Notifier interface {
    OnSessionCreated(ctx context.Context, s *Session)
    OnSessionStopped(ctx context.Context, s *Session)
    OnAgentEvent(ctx context.Context, sessionID string, ev Event)
}
```

The `daemon/` package wires a concrete Notifier that fans out to `observe/` (event recording) and `httpapi/` (SSE streaming). Explicit, typed, debuggable.

## Alternatives Considered

### Alternative 1: In-Process Event Bus

- **Description**: `map[string][]func(Event)` with Publish/Subscribe methods
- **Pros**: Slices fully decoupled. Adding observers is zero-change to producers.
- **Cons**: Hidden dependency graph. Poor debuggability (breakpoints don't show call chain). Race-prone for command flows. Becomes the new god object.
- **Why rejected**: Codex review + Go ecosystem evidence strongly advise against it

### Alternative 2: Embedded NATS

- **Description**: In-process NATS for pub/sub (as in previous AGH)
- **Pros**: Battle-tested, subject-based routing, request/reply
- **Cons**: Heavy dependency for intra-process communication. Subject hierarchy was tightly coupled to workgroup concept. Unnecessary for a single-binary daemon.
- **Why rejected**: Over-engineering for single-process communication

### Alternative 3: Channel-Based Fan-Out

- **Description**: Go channels for event distribution
- **Pros**: Native Go concurrency primitive
- **Cons**: Channel management complexity (buffering, backpressure, close semantics). Subscriber management boilerplate.
- **Why rejected**: Notifier interface is simpler and more explicit

## Consequences

### Positive

- Debuggable: step through function calls with a debugger
- Type-safe: compiler catches interface mismatches
- No hidden message routing or reflection
- Explicit dependency graph visible in import statements
- Easy for LLMs to understand and modify

### Negative

- Adding a new observer requires modifying the Notifier implementation in `daemon/`
- Less decoupled than a bus (producers know about the Notifier interface)

### Risks

- If fan-out requirements grow significantly, can introduce a typed event registry later
- The Notifier interface can evolve (add methods) without changing the pattern
