# ADR-009: Agent-First Observability

## Status

Accepted

## Date

2026-04-02

## Context

The previous AGH had a sophisticated observability system with a real-time web dashboard (topology canvas, terminal emulation, event timeline, connection tracking). The dashboard was designed for human operators to visually monitor agent sessions.

The new AGH is designed as an **agent-first system** — the primary consumers of observability data are other LLMs/agents, not humans. An agent like Claude Code needs to query "what happened in this session?" or "why did this agent fail?" via CLI commands and structured output, not by looking at a dashboard.

## Decision

Observability is designed for **programmatic consumption by agents/LLMs**:

1. **All ACP events persisted** in SQLite (tool calls, agent messages, thoughts, plans, errors)
2. **CLI-queryable** with rich filters:
   ```bash
   agh session events <id> --type tool_call --last 20
   agh session events <id> --since 1h --follow
   agh session history <id>
   agh session status <id>
   agh observe events --session <s> --agent <a> --type <t> --follow
   agh observe health
   ```
3. **Cross-session queries** via global DB (`agh observe events --since 24h`)
4. **Structured output formats**: `--output human` (default for terminals), `--output json` (for LLMs), `--output toon` (Token-Oriented Object Notation, LLM-optimized)
5. **HTTP API** exposes the same queries for the web UI

No dedicated observability dashboard in v1. The web UI (separate spec) may include an observability view, but it is not the primary consumption path.

## Alternatives Considered

### Alternative 1: Dashboard-First Observability

- **Description**: Real-time web dashboard with topology visualization, terminal views, event timeline (like the previous AGH)
- **Pros**: Rich visual experience for human operators
- **Cons**: Significant implementation effort. The primary users (other agents) can't use a dashboard. Dashboard was ~5K lines of Svelte + complex WebSocket infrastructure.
- **Why rejected**: Wrong audience for v1. Agents need CLI/API access, not visual dashboards.

### Alternative 2: Log-Only Observability

- **Description**: Structured logging (slog) to files, grep for queries
- **Pros**: Zero additional infrastructure
- **Cons**: Unfiltered logs are overwhelming. No indexed queries. No cross-session aggregation. Poor agent experience.
- **Why rejected**: Agents need filtered, structured queries — not raw log files

## Consequences

### Positive

- Agents can self-diagnose issues via CLI commands
- Same data accessible via CLI, HTTP API, and future web UI
- Structured output formats (JSON, TOON) are LLM-native
- Cross-session queries enable trend analysis and debugging

### Negative

- No visual dashboard in v1 (humans use CLI or web UI chat view)
- Requires SQLite schema design for efficient filtered queries

### Risks

- If human operators need visual observability, it can be added as a web UI feature later without changing the underlying storage or query layer
