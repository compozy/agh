# ADR-007: Background Sessions With CLI Prompt

## Status

Accepted

## Date

2026-04-02

## Context

The previous AGH model was task-oriented: `agh session start "Build a REST API"` created a session with a goal and immediately spawned supervisor/worker agents. The new model is conversational: users create sessions and interact through messages over time.

The question is how users interact with sessions — TUI, web UI, CLI, or a combination.

## Decision

Sessions are **background processes** managed by the daemon. No TUI for conversation. Two interaction paths:

1. **CLI** (for agents and automation): `agh session prompt <id> "message"` sends a message and returns the response. This is the agent-friendly path — other agents/LLMs can interact with AGH sessions programmatically.

2. **Web UI** (for humans, separate spec): A web application using Vercel AI SDK components (`useChat`) connects to the daemon's HTTP/SSE API. This provides a rich conversational UI with streaming, tool call visualization, etc. The web UI is out of scope for this TechSpec.

Session lifecycle:
```bash
agh session new --agent coder [--cwd /repo] [--name my-session]  # Returns session ID
agh session prompt <id> "explain the auth module"                  # Send message
agh session list                                                   # List sessions
agh session status <id>                                            # Check state
agh session stop <id>                                              # Stop session
agh session resume <id>                                            # Resume stopped session
agh session wait <id>                                              # Block until done
```

The `cwd` belongs to the session (not the agent). The same agent definition can be used across different workspaces.

## Alternatives Considered

### Alternative 1: Built-in TUI Chat

- **Description**: `agh chat coder` opens an interactive terminal chat
- **Pros**: No web UI needed for basic interaction
- **Cons**: Significant implementation effort (TUI framework, streaming, tool call display). Duplicates what the web UI provides. Not useful for agent-to-agent interaction.
- **Why rejected**: Web UI with AI SDK components provides better UX with less effort. CLI prompt covers the agent-friendly path.

### Alternative 2: Implicit Daemon (auto-start)

- **Description**: Daemon starts automatically on first `agh session new`
- **Pros**: One less command for users
- **Cons**: Hides daemon lifecycle. Harder to debug when daemon has issues. Users can't control when the daemon runs.
- **Why rejected**: Explicit daemon control is simpler to reason about and debug

## Consequences

### Positive

- Sessions survive terminal disconnects (background daemon manages them)
- Agent-friendly: any LLM can send prompts via CLI
- Clean separation: daemon manages lifecycle, web UI handles human interaction
- `cwd` per session enables same agent across multiple projects

### Negative

- No built-in conversational UI in the terminal
- Requires web UI setup for human-friendly interaction (separate spec)

### Risks

- None significant — CLI prompt covers the automation case, web UI covers the human case
