# ADR-002: Pragmatic Flat Architecture With Discipline

## Status

Accepted

## Date

2026-04-02

## Context

The previous AGH architecture collapsed into a monolithic `kernel` package because there were no enforced boundaries between concerns. Three architectural approaches were evaluated for the rewrite:

1. **Pragmatic Flat** (12 packages) — simple, but no formal boundary protection
2. **Ports & Adapters** (16 packages) — strong boundaries via `domain/` + `infra/` split
3. **Vertical Slices** (16 packages) — feature-isolated slices communicating via event bus

Research of 10 major Go daemons (Prometheus, Docker, Consul, Vault, Containerd, Kubernetes, Grafana, Ollama, PocketBase, soft-serve) revealed that 0/10 use formal hexagonal architecture and 0/10 use formal vertical slices. The dominant pattern is pragmatic flat with feature-organized packages. Only Grafana uses an event bus, and has been simplifying it over time.

Codex review recommended Approach 3 but warned that the event bus must not become "the new kernel" — use it only for fan-out, not for commands.

## Decision

Use **Pragmatic Flat with Discipline**: 11 Go-idiomatic packages under `internal/`, no domain/infra layer split, no event bus. Communication between packages via direct function calls through small interfaces. A Notifier pattern for fan-out (observability, SSE streaming). CI-enforceable import rules to prevent re-coupling.

```
cmd/agh/main.go
internal/
  config/     # TOML, home paths, agent def parsing, validation
  acp/        # ACP client: subprocess spawn, JSON-RPC over stdio
  session/    # Session lifecycle, Manager, state machine
  store/      # SQLite (per-session + global)
  observe/    # Event recording, health, queries
  httpapi/    # HTTP/SSE server
  udsapi/     # UDS server
  cli/        # Cobra commands
  daemon/     # Composition root, lock, boot, shutdown
  logger/     # slog setup
  version/    # Build metadata
```

That is **11 packages** under `internal/` plus the `cmd/agh/` entry point.

Key rules:
- **Designed for incremental extension** — every new capability (memory, skills, agent networking) must arrive as a new package wired into `daemon/`, without modifying existing packages. Keep public APIs stable: add, don't change.
- `daemon/` is the only package that imports all others (composition root)
- No package imports `daemon/`, `httpapi/`, `udsapi/`, or `cli/`
- Interfaces defined where consumed (Go-style) — use small interfaces and dependency injection so future features plug in rather than fork existing code
- No back-pointers between packages
- CI grep checks enforce boundaries

## Alternatives Considered

### Alternative 1: Pragmatic Flat (no discipline)

- **Description**: 12 packages with no enforced import rules
- **Pros**: Simplest possible structure
- **Cons**: LLMs are opportunistic — without enforced rules, cross-package coupling will grow. This is what happened with the original kernel.
- **Why rejected**: Too weak for a codebase maintained by LLMs. No protection against re-coupling.

### Alternative 2: Ports & Adapters (Hexagonal)

- **Description**: `domain/` (pure types + interfaces) never imports `infra/` (adapters)
- **Pros**: Strongest boundary protection, CI-enforceable, excellent testability
- **Cons**: More ceremony than needed for runtime-heavy software. Premature packages like `natsbus/`, `ptyhost/`. Interface inflation and mock-heavy tests. 0/10 major Go daemons use this pattern.
- **Why rejected**: Too ceremonial for v1. The system is orchestration around subprocesses and SQLite, not a rich pure-domain system.

### Alternative 3: Vertical Slices with Event Bus

- **Description**: Feature-isolated slices communicate only via in-process event bus
- **Pros**: Adding features without touching existing code. Good locality of change.
- **Cons**: Event bus becomes hidden dependency graph. Rich workflows (start session → open DB → spawn agent → attach streams) are not naturally event-only. Race-prone orchestration. Only 1/10 Go projects uses a bus (Grafana), and they're simplifying it.
- **Why rejected**: Bus-as-communication-backbone is anti-idiomatic Go and creates an "in-process distributed monolith."

## Consequences

### Positive

- Aligned with how successful Go daemons are actually structured
- LLM-friendly: small packages, explicit wiring, no hidden message routing
- Testable: each package testable through its public API with real dependencies (SQLite temp DBs, ACP protocol harnesses)
- Extensible: new features become new packages wired in `daemon/`

### Negative

- No formal domain/infra boundary — requires discipline via CI rules rather than compiler enforcement
- If CI rules are not maintained, coupling can grow

### Risks

- Re-coupling risk mitigated by CI grep checks and the Notifier interface pattern
- If a package exceeds ~800 lines, extract sub-packages (like Docker does with `daemon/cluster/`)
