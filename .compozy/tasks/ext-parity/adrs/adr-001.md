# ADR-001: Adopt a Shared Resource Runtime as the Authoritative Extensibility Control Plane

## Status

Accepted

## Date

2026-04-15

## Context

AGH already has a serious extension host, but its extensibility runtime is fragmented across manifest parsing, capability maps, handshake flags, manager-owned catalogs, and domain-specific reconciliation code. This fragmentation is the root cause behind repeated gaps such as unwired `tool.*` and `permission.*` hooks, incomplete `provide_tools` negotiation, and bundle/automation/bridge logic being reconciled through separate ownership models.

The architectural comparison against `compozy` showed that the missing piece is not another isolated API. The missing piece is a shared runtime graph where extensible resources can be published, listed, reconciled, and projected into the subsystems that actually execute behavior.

During technical clarification, the chosen direction was the definitive path rather than another incremental patch. The user also chose a persisted generic resource store and explicitly selected that this store should be authoritative for all resources in the covered domains, not only extension-owned resources.

## Decision

Introduce a persisted shared resource runtime as the authoritative control plane for the first implementation wave of extensibility.

The first-wave covered families are:

- `hooks`
- `tools`
- `agents`
- `mcp_servers`
- `skills`
- `automations`
- `bridges`
- `bundles`

This runtime will:

- store canonical resource records in persistent storage;
- own reconcile semantics for covered domains;
- become the source of truth for all resources in those domains, regardless of whether they originate from extensions, daemon-owned configuration, or bundle activation;
- turn existing domain managers into projectors, reconcilers, or execution adapters instead of primary authorities.

## Alternatives Considered

### Alternative 1: Close only the currently visible gaps

- **Description**: Wire missing hooks, enable `provide_tools`, add manifest `tools`, and leave each domain with its own catalog and reconciliation model.
- **Pros**: Smaller short-term change set; faster first delivery.
- **Cons**: Preserves the structural reason the same extensibility work keeps reopening; every new surface remains a shotgun-surgery change.
- **Why rejected**: The user explicitly chose the definitive solution over another targeted pass.

### Alternative 2: Make the shared runtime authoritative only for extension-owned resources

- **Description**: Keep daemon-native automation, bridge, and bundle state in their current domain stores and use the new runtime only for extension contributions.
- **Pros**: Smaller initial migration; lower short-term disruption to current subsystems.
- **Cons**: Creates split-brain ownership; duplicates domain invariants; keeps bundle and runtime reconciliation fragmented.
- **Why rejected**: The user explicitly chose a single authoritative store for all resources in the covered domains.

### Alternative 3: Use an in-memory runtime and rebuild it at boot

- **Description**: Keep a generic resource graph, but do not persist canonical records.
- **Pros**: Less schema work; simpler first implementation.
- **Cons**: Weak recovery semantics; harder boot determinism; poor fit for authoritative ownership and debugging.
- **Why rejected**: The chosen direction requires a persisted source of truth.

## Consequences

### Positive

- Eliminates the repeated pattern of adding a new surface in multiple disconnected places.
- Gives AGH a unified extensibility model broad enough to cover workflow-adjacent domains, not only hook execution.
- Makes tools, bridges, automations, and bundles first-class extensible resources under one control plane.

### Negative

- Broadens the blast radius of the refactor because multiple subsystems must stop being their own source of truth.
- Requires new persistence schema, runtime APIs, and reconciliation contracts before domain cutovers can happen.

### Risks

- A too-generic runtime can become a god object if domain validation is not kept explicit.
- Domain-specific invariants could be weakened during migration if projection boundaries are vague.
- Boot and reconcile performance may regress if store and projector responsibilities are not bounded.

## Implementation Notes

- The shared runtime should live as a new internal subsystem rather than be embedded into `internal/extension/manager.go`.
- Each covered domain should define typed validators and projectors over canonical resource records.
- Existing domain-specific catalogs should be removed as each family cuts over to the shared runtime.
