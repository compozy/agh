# ADR-004: Use Snapshot-First Reconcile for Resource Consistency

## Status

Accepted

## Date

2026-04-15

## Context

The first version of the shared resource runtime originally proposed a `Watch` store contract, a public resource event stream, and optional watchpoint persistence. That design assumed change notification could be part of the platform correctness path.

After review, that assumption proved weak for AGH's actual environment:

- the canonical store is SQLite, which does not provide a durable notification primitive comparable to `LISTEN/NOTIFY`;
- the project architecture explicitly avoids event-bus style control flow;
- the system is a single binary, so deterministic post-write and boot-time reconcile are simpler than inventing a generic watcher substrate.

The platform still needs a way to drive projection into hooks, automation, bridges, and bundles, but it does not need a first-version watch contract to do that correctly.

## Decision

Make full-snapshot reconcile the correctness path for the shared resource runtime.

For each migrated resource kind:

- the daemon runs reconcile on boot;
- the daemon runs reconcile after successful writes affecting that kind;
- the projector receives the full persisted snapshot for that kind;
- no public `Watch` store method, `resources/watch`, or `/api/resources/events/stream` endpoint exists in v1.

Change notification may be introduced later as an internal invalidation hint, but it must remain an optimization. Correctness must always be recoverable from persisted snapshots alone.

## Alternatives Considered

### Alternative 1: Public watch contract on the resource store

- **Description**: Expose `Watch` from the store and let projectors or clients react to streaming changes.
- **Pros**: Lower-latency propagation; attractive API surface.
- **Cons**: Pushes event-delivery semantics onto SQLite-backed infrastructure that does not support them natively; risks becoming an event bus in disguise.
- **Why rejected**: Too much complexity for v1, and a poor fit for AGH's architecture constraints.

### Alternative 2: Public event stream plus watchpoint persistence

- **Description**: Emit change events, persist cursors or watchpoints, and let consumers resume from checkpoints.
- **Pros**: Better recovery story than transient in-memory watch callbacks.
- **Cons**: Substantially more machinery, more schema, and more failure modes; solves a problem AGH does not yet need.
- **Why rejected**: Over-engineered for the current alpha stage and not required for correctness.

### Alternative 3: Imperative direct updates per subsystem with no shared reconcile path

- **Description**: Skip snapshot reconcile and update each subsystem directly on every write.
- **Pros**: Lower apparent runtime overhead.
- **Cons**: Recreates the same fragmented update model the refactor is trying to remove; weaker boot determinism.
- **Why rejected**: Preserves structural drift and makes recovery harder.

## Consequences

### Positive

- Keeps the correctness model simple and deterministic.
- Aligns with the existing single-binary, direct-call architecture.
- Removes a large class of watcher, fanout, and checkpoint failure modes from v1.

### Negative

- Post-write reconcile reads full snapshots more often.
- Change propagation may be slightly less immediate than a streaming design.

### Risks

- Some families may reconcile larger snapshots than strictly necessary.
- Developers may be tempted to reintroduce hidden watcher dependencies later.
- Reconcile cost could grow if snapshot size is not monitored.

## Implementation Notes

- The daemon composition root should explicitly trigger reconcile after successful writes and at boot.
- Projectors must treat the full snapshot as authoritative input.
- Bundles still depend on automation and bridge projections; those families should migrate before bundle composition.
