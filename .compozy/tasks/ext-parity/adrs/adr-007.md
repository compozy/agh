# ADR-007: Use Optimistic Concurrency and Serialized Source Snapshots

## Status

Accepted

## Date

2026-04-15

## Context

The revised TechSpec narrowed the resource runtime, but it temporarily removed explicit versioning from canonical records. That created two correctness gaps:

1. direct mutations had no compare-and-set protection against lost updates;
2. `resources/snapshot` could race with itself when retries or overlapping submissions from the same source arrived out of order.

The later review pass exposed two more correctness gaps:

3. snapshots could bypass ownership boundaries by overwriting daemon-owned or foreign-source records if `(kind, id)` matched;
4. bootstrap and crash-respawn behavior for `resource_source_state` and session nonces was still under-specified.

Timestamp-only approaches are not sufficient for canonical desired-state correctness. The runtime needs explicit concurrency semantics both for individual record mutations and for source-scoped snapshot replacement.

## Decision

Use optimistic concurrency for direct resource mutations and serialized snapshot replacement for extension sources.

Specifically:

- every persisted `ResourceRecord` carries a monotonic `Version`;
- direct `Put` and `Delete` operations require an expected version, with `0` reserved for create-only semantics;
- every extension `resources/snapshot` call carries a monotonic `source_version` for that active source session;
- snapshot application is serialized per `(source_kind, source_id)`;
- the daemon pairs snapshot sequencing with a server-issued session nonce for the active source session;
- session nonces are daemon-generated with at least 128 bits of entropy and only the active nonce for that live session is accepted;
- stale `source_version` values are rejected before diff and apply;
- new live sessions start at `source_version=1`, and source-state bootstrap or reset happens atomically with `INSERT ... ON CONFLICT DO UPDATE` under the per-source lock;
- snapshots may create new records for their own source, but they can update or delete only records already stamped with that same source;
- if `(kind, id)` already belongs to the daemon or another source, snapshot apply fails with conflict instead of overwriting the record;
- snapshot diff and record updates execute in one transaction and bump `Version` on every changed record.

## Alternatives Considered

### Alternative 1: Last-writer-wins

- **Description**: Accept whichever write commits latest and overwrite prior state silently.
- **Pros**: Simple implementation.
- **Cons**: Lost updates become invisible and debugging becomes harder.
- **Why rejected**: Canonical desired state should not accept silent corruption from concurrent writes.

### Alternative 2: Timestamp-based conflict detection

- **Description**: Use `updated_at` comparisons instead of explicit versions.
- **Pros**: Avoids adding version columns.
- **Cons**: Weaker semantics, harder to reason about, and vulnerable to clock or ordering ambiguity.
- **Why rejected**: Timestamps are observational metadata, not concurrency control.

### Alternative 3: One global lock for all resource mutations

- **Description**: Serialize every write through one global mutex.
- **Pros**: Very simple correctness model.
- **Cons**: Unnecessarily broad contention and poor scaling across unrelated kinds and sources.
- **Why rejected**: The runtime only needs per-record CAS and per-source snapshot serialization.

## Consequences

### Positive

- Makes write conflicts explicit instead of silent.
- Prevents retried or stale snapshots from overwriting newer source state.
- Gives tests concrete correctness properties to verify.
- Protects daemon-owned and foreign-source records from snapshot stomping.

### Negative

- Adds version fields and source-state metadata to persistence.
- Increases write-path complexity.

### Risks

- Callers may misuse create-only versus update semantics if contracts are unclear.
- Stale source-version handling must stay aligned with extension session lifecycle.
- Poor error reporting around conflicts can make clients retry incorrectly.
- Source reset semantics must delete both source state and owned records together or the next session bootstrap becomes inconsistent.

## Implementation Notes

- Persist source snapshot state in a dedicated `resource_source_state` table keyed by `(source_kind, source_id)`.
- Store the active source-session nonce alongside the last accepted snapshot version.
- Reject snapshots whose nonce does not match the currently active live session before the per-source diff starts.
- Operator APIs should not be able to mutate extension-owned records outside the authorized write path.
- Operator-driven source reset should remove the source row and source-owned records in the same transaction.
- Integration tests should cover stale `expected_version`, stale `source_version`, replayed or stale-nonce snapshots, and foreign-source overwrite attempts.
