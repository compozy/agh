# ADR-002: Migrate Covered Domains Through Phased Clean Cutovers

## Status

Accepted

## Date

2026-04-15

## Context

The chosen architecture is broad enough that a single big-bang rewrite would create unnecessary execution risk and make task decomposition poor. At the same time, the user explicitly rejected long-lived compatibility layers, deprecated contracts, and repeated interim fixes that preserve the wrong architecture.

AGH is still in alpha with no production users, so there is no requirement to preserve legacy runtime structure for external compatibility. The migration strategy should therefore optimize for end-state architecture quality while still allowing progressive delivery through independently scoped tasks.

## Decision

Implement the shared resource runtime through phased clean cutovers by domain or resource family.

For each phase:

1. add the shared runtime support needed for the target family;
2. migrate the target family to read and reconcile from the shared resource runtime;
3. remove the replaced legacy authority and runtime path in the same phase;
4. ship the phase only when verification gates pass.

No long-lived dual-write, deprecation layer, feature-flagged compatibility shim, or legacy mirror path will be kept after a family has cut over.

## Alternatives Considered

### Alternative 1: Big-bang rewrite of all covered domains

- **Description**: Replace hooks, tools, agents, MCP servers, skills, automations, bridges, and bundles in one implementation wave.
- **Pros**: Fastest path to a single architectural end state; no transitional boundaries.
- **Cons**: High integration risk; hard to review; poor fit for later task generation and incremental validation.
- **Why rejected**: The user wants progressive task creation and staged execution.

### Alternative 2: Long-lived dual-write or compatibility layer

- **Description**: Keep old domain authorities active while the shared runtime writes mirrored state during a transition period.
- **Pros**: Lower immediate migration risk; easier rollback.
- **Cons**: Preserves the wrong architecture; creates split authority; adds cleanup work that history shows tends to linger.
- **Why rejected**: The user explicitly rejected deprecated or stopgap architecture.

### Alternative 3: Feature-flagged compatibility shims

- **Description**: Keep old APIs and wire them through adapters into the new runtime for multiple releases.
- **Pros**: Reduces short-term call-site churn.
- **Cons**: Leaves compatibility code in core paths and obscures the real ownership model.
- **Why rejected**: Alpha status removes the need to trade architecture quality for compatibility.

## Consequences

### Positive

- Preserves architectural integrity while still producing task-sized implementation slices.
- Forces each phase to end in a coherent state rather than an indeterminate half-migrated system.
- Keeps future task decomposition straightforward for `cy-create-tasks`.

### Negative

- Each phase requires deliberate cutover planning and may touch public or internal contracts.
- Rollback is harder than with dual-write because replaced paths are removed rather than kept warm.

### Risks

- Family boundaries may be chosen poorly and create cross-phase churn.
- A phase can stall if prerequisites are not identified early.
- Removing old paths in-phase increases the importance of verification quality.

## Implementation Notes

- Build sequencing in the TechSpec should define dependency order so tasks can be created progressively.
- After foundational runtime and protocol steps, family migrations should be organized as parallel lanes where dependencies permit.
- Family cutovers should be organized around ownership boundaries, not arbitrary file counts.
- The extension manager should be reduced as families leave its current control-plane responsibilities.
