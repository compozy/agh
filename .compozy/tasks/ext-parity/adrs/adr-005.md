# ADR-005: Make Resource Access Server-Authoritative

## Status

Accepted

## Date

2026-04-15

## Context

The shared resource runtime becomes the authoritative desired-state store for multiple extensibility families. That makes mutation authority a control-plane concern, not an implementation detail.

If callers can self-assert `owner`, `source`, or `scope`, the runtime becomes vulnerable to privilege escalation and state forgery. Examples include:

- an extension publishing records that masquerade as daemon-owned resources;
- a workspace-scoped caller writing global resources;
- one source deleting records owned by another source;
- oversized or malformed specs entering the canonical store and breaking reconcile downstream.

These risks must be addressed in the store write path, not deferred to projectors or transport handlers.

The revised TechSpec also adds extension read surfaces (`resources/list` and `resources/get`). Those reads can leak credentials or other sensitive config if they are not governed by the same server-side authority model as writes.

## Decision

All resource mutations are server-authoritative.

Specifically:

- the daemon stamps `owner` and `source` from the authenticated actor and execution path;
- scope is required on every mutation, narrowed from granted authority, and never trusted from payload alone when omitted or malformed;
- the store resolves a kind-specific validator before persistence and rejects invalid or oversized specs;
- read paths are server-authoritative too: extension reads are filtered by same-source identity, granted kinds, and scope;
- `resources/snapshot` is source-scoped and can modify only the calling extension's own records within granted kinds and scopes;
- extension sessions do not receive direct resource `Put` or `Delete` in v1; they publish through `resources/snapshot`;
- delete operations are allowed only for daemon-authorized callers or records owned by the calling source;
- mutating resource endpoints and source snapshots are rate-limited as part of the control-plane defense boundary;
- HTTP resource mutation routes remain disabled until explicit operator auth middleware exists for those routes.

## Alternatives Considered

### Alternative 1: Trust payload metadata and validate later

- **Description**: Accept caller-supplied `owner`, `source`, and `scope`, then rely on projectors or runtime paths to catch bad inputs.
- **Pros**: Simpler write path; fewer cross-cutting checks.
- **Cons**: Invalid or forged state enters the canonical store; privilege boundaries become inconsistent.
- **Why rejected**: The canonical store cannot safely persist untrusted authority metadata.

### Alternative 2: Transport-level checks only

- **Description**: Enforce authorization in HTTP, UDS, or subprocess handlers but keep the store blind to mutation authority.
- **Pros**: Keeps the store API smaller.
- **Cons**: Duplicates logic across transports; non-transport call sites can bypass rules; store invariants remain weak.
- **Why rejected**: Authorization must be part of the canonical mutation path.

### Alternative 3: Add ACLs later after the runtime lands

- **Description**: Ship the resource runtime first and retrofit source ownership and scope enforcement later.
- **Pros**: Faster initial implementation.
- **Cons**: Bakes unsafe semantics into the first version and makes later tightening more disruptive.
- **Why rejected**: Security and authority are foundational, not optional polish.

## Consequences

### Positive

- Prevents forged desired state from entering the canonical store.
- Gives every transport and subsystem one mutation-authority model.
- Makes bundle ownership and extension snapshots safer by default.
- Prevents cross-source resource reads from leaking sensitive specs.

### Negative

- Adds complexity to the store write path and validator registration.
- Requires capability-checker, handshake, and API contracts to stay aligned with store semantics.

### Risks

- Overly strict scope rules can block legitimate extension use cases if grants are underspecified.
- Payload limits that are too low can reject valid definitions.
- Delete authorization must be implemented carefully to avoid stranded records.
- Source-scoped reads are intentionally conservative and may delay future observability use cases until an explicit public-read design exists.

## Implementation Notes

- `CapabilityChecker` should gate `resources/*` access by kind and scope.
- `CapabilityChecker` should gate `resources/list`, `resources/get`, and `resources/snapshot`; direct extension CRUD stays unavailable in v1.
- `internal/extension/surfaces` should define which kinds and operations can be granted to extension sessions.
- Read paths should filter by scope and caller authority where exposure differs by workspace or source.
- Operator policy for resource grants and rate limits should live in `[extensions.resources]` under `internal/config.ExtensionsConfig`.
- Bundle-owned deletes should use owner filters rather than arbitrary identifier lists.
- HTTP resource writes should only be enabled behind the daemon's operator authority boundary; until then, UDS is the mutating control plane.
