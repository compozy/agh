# BUG-001: API bundlepkgStableID uses colon separator instead of SHA256 hash

**Severity:** Medium
**Priority:** P2
**Component:** `internal/api/core/bundles.go:333-335`
**Related:** TC-FUNC-014

## Environment

- Code review of `internal/api/core/bundles.go`
- Function `bundlepkgStableID` at line 333

## Description

The API layer's `bundlepkgStableID` function uses a simple string concatenation with `:` separator, while the service layer's `stableID` uses SHA256 hashing. The API function is used to generate IDs for the activation payload's jobs, triggers, and bridges — but these IDs do NOT match the actual resource IDs generated by the service layer.

### Expected Behavior

The job/trigger/bridge IDs in the API response payload should match the actual resource IDs used internally by the service and stored in the database.

### Actual Behavior

- Service layer: `stableID("job", activationID, jobName)` → `job_<sha256_hex>`
- API layer: `bundlepkgStableID("job", activationID, jobName)` → `job:activationID:jobName`

The API response contains IDs like `job:act_abc123:daily-report` while the actual stored resources have IDs like `job_def456`.

## Reproduction

1. Activate a bundle with jobs via POST `/api/bundles/activations`
2. Compare `activation.jobs[0].id` in the response with the actual job ID in the automation system
3. IDs don't match

## Impact

- API consumers cannot use the job/trigger/bridge IDs from activation responses to query other subsystems (automation, bridges)
- Inventory items show the real IDs, but inline jobs/triggers/bridges show different IDs
- Confusion when correlating resources across subsystems

## Suggested Fix

Import and use the service layer's `stableID` function, or export it as a public function. Alternatively, populate job/trigger/bridge IDs from the resolved inventory items which use the correct IDs.

## Workaround

Use the `inventory` array's `resource_id` field instead of the inline job/trigger/bridge `id` fields.
