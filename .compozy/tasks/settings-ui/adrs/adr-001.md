# ADR-001: Use a consolidated settings namespace with a dedicated settings shell

## Status

Accepted

## Date

2026-04-17

## Context

The current AGH HTTP API is organized by operational domains such as `/api/automation`, `/api/network`, `/api/skills`, and `/api/memory`. The `settings-ui` analysis showed that the Paper settings screens cut across those operational domains and need a cohesive configuration surface that does not exist today. The web app also already follows a route-per-area pattern, and the user explicitly chose nested routes for settings rather than a single local-state page.

## Decision

Introduce a dedicated consolidated settings API namespace under `/api/settings/*` and a dedicated web settings shell under `/_app/settings/*`. Each settings section gets its own route and its own settings resource, while operational pages such as Automation, Network, and Skills remain separate destinations linked from settings.

The settings shell may also depend on existing operational routes where the product already has a separate runtime surface. In particular, the Hooks & Extensions section is allowed to compose existing extension operations from `/api/extensions`, but those routes must be available on the HTTP transport used by the web app.

## Alternatives Considered

### Alternative 1: Extend each existing operational namespace with its own settings endpoints

- **Description**: Add endpoints like `/api/network/settings`, `/api/automation/settings`, and `/api/skills/settings`.
- **Pros**: Keeps API ownership aligned with existing runtime domains.
- **Cons**: Spreads one product area across many unrelated namespaces, makes cross-section UI composition harder, and creates inconsistent client integration patterns.
- **Why rejected**: The settings screens are a unified operator surface and need one discoverable contract rather than a set of dispersed operational add-ons.

### Alternative 2: Use one monolithic `/api/settings` document and one single web page

- **Description**: Expose all settings through one large payload and render the UI as a single page that switches sections with local state.
- **Pros**: Minimal route count and fewer endpoint definitions.
- **Cons**: Produces a coarse contract, weakens cache invalidation and test isolation, and does not fit the existing route-per-surface pattern in `web/`.
- **Why rejected**: The implementation needs section-level loading, validation, mutations, and tests without turning settings into one oversized resource.

## Consequences

### Positive

- Settings become a first-class product surface in both the daemon and the web app.
- Each section can evolve independently while still living under one coherent namespace.
- The web app can follow the existing `routes` + `hooks/routes` + `systems/*` pattern cleanly.

### Negative

- Adds a new API namespace instead of piggybacking on the existing route tree.
- Requires deliberate duplication boundaries so settings resources do not drift from operational domain behavior.
- Some settings pages now depend on both `/api/settings/*` and a small set of supporting operational APIs such as `/api/extensions`.

### Risks

- Settings endpoints may become a thin wrapper layer with duplicated mapping code if they are not anchored to shared domain services.
- Operators may expect operational actions to live inside settings once a consolidated namespace exists.
- HTTP transport gaps can still break the web shell if operational APIs needed by settings are only registered on UDS.

## Implementation Notes

- Add `registerSettingsRoutes(api, handlers)` under `internal/api/httpapi/routes.go` and equivalent UDS registration for parity.
- Add web routes under `web/src/routes/_app/settings.tsx` plus child routes for each section.
- Keep `automation`, `network`, and `skills` pages as existing operational routes and link to them from settings.
- Register `/api/extensions` on HTTP as an explicit dependency of the Hooks & Extensions screen.
- Apply the loopback-only HTTP mutation policy from ADR-004 to settings and extension mutations.
