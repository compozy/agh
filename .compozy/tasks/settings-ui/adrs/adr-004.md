# ADR-004: Restrict HTTP settings mutations to loopback-bound servers in v1

## Status

Accepted

## Date

2026-04-17

## Context

The AGH web app uses the HTTP transport, but the current HTTP middleware is focused on origin checking and local-first usage, not on remote administrative authentication. The `settings-ui` effort introduces powerful mutation endpoints that can rewrite canonical config, install or enable extensions, and restart the daemon. Exposing those capabilities over non-loopback HTTP without an explicit auth design would create a privileged remote-management surface by accident.

At the same time, the local web app still needs a complete settings experience, including the Hooks & Extensions screen, which depends on operational extension routes being available on HTTP.

## Decision

In v1, read-only settings routes are available on both HTTP and UDS, but mutating settings routes are available over HTTP only when the daemon HTTP server is bound to a loopback host. The same rule applies to HTTP extension mutations that the settings UI depends on.

When the HTTP server is bound to a non-loopback host, settings mutations and HTTP extension mutations return `403`, and UDS remains the authoritative privileged transport for local CLI and trusted automation. This effort does not introduce authenticated remote HTTP administration.

## Alternatives Considered

### Alternative 1: Allow full settings mutation over any HTTP bind and rely on existing origin checks

- **Description**: Expose the complete settings mutation surface over HTTP regardless of bind host, relying on existing Origin validation and local deployment assumptions.
- **Pros**: Simplest transport story and no per-route restriction logic.
- **Cons**: Turns unauthenticated HTTP into a remote config-write and daemon-restart surface when AGH is bound beyond loopback.
- **Why rejected**: Origin checks are not a substitute for an explicit administrative auth design.

### Alternative 2: Introduce bearer-token or session authentication as part of this effort

- **Description**: Add remote-capable HTTP authentication together with the settings surface.
- **Pros**: Enables authenticated remote administration in one step.
- **Cons**: Significantly expands the scope into security, credential lifecycle, storage, rotation, and compatibility decisions unrelated to the Paper settings UI delivery.
- **Why rejected**: Remote administration is a separate design problem and should not be smuggled into this feature.

## Consequences

### Positive

- The local web app keeps a complete settings surface without opening remote admin accidentally.
- HTTP extension parity needed by the Hooks & Extensions screen becomes safe to ship in the same release.
- The privileged transport boundary stays explicit: HTTP for local UI, UDS for trusted local automation and CLI.

### Negative

- Settings edits from a remotely bound HTTP daemon are intentionally unavailable in v1.
- The product surface must explain why editing is disabled when AGH is exposed beyond loopback.

### Risks

- Future contributors may accidentally add new mutating HTTP routes without applying the same restriction.
- Users running AGH behind a reverse proxy may expect remote settings editing and treat the restriction as a regression if it is not documented clearly.

## Implementation Notes

- Add a shared HTTP mutation guard for `/api/settings/*` mutating routes and `/api/extensions` mutating routes.
- Keep read-only settings routes available on HTTP even when bound non-loopback.
- Return a structured `403` error that explains remote HTTP settings mutation is disabled in v1.
- Document authenticated remote management as future work rather than partially implementing it here.
