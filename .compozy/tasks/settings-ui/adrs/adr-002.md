# ADR-002: Persist settings by writing canonical config overlays instead of creating a new settings store

## Status

Accepted

## Date

2026-04-17

## Context

AGH already has a canonical configuration model based on `aghconfig.Config`, with built-in defaults, a global config file at `~/.agh/config.toml`, optional workspace overlays at `<workspace>/.agh/config.toml`, and a global and workspace MCP sidecar JSON file. The config package already knows how to load, validate, and merge these sources. It also has a bootstrap-only TOML writer today, but that writer re-encodes the whole overlay document and drops user comments and formatting. The user explicitly chose direct persistence to the daemon's canonical config rather than a new database-backed or runtime-only settings store.

## Decision

Persist settings mutations by editing the canonical config overlays that AGH already loads: the global home config for global settings, the workspace overlay config for workspace-scoped settings, and the MCP sidecar JSON file where MCP server scope requires it. Do not introduce a new settings-specific persistence store.

TOML-backed settings writes must use comment-preserving document edits rather than blind whole-file re-encoding. If a requested mutation cannot be expressed without rewriting unrelated TOML structure, that mutation must be rejected as unsupported instead of silently canonicalizing the file. MCP JSON writes are implemented as a new writer with explicit target selection and source-precedence rules.

## Alternatives Considered

### Alternative 1: Create a new daemon-owned settings database or overlay store

- **Description**: Persist settings in SQLite or another daemon-managed store, then merge that store on top of file config at runtime.
- **Pros**: Avoids rewriting TOML files and could simplify fine-grained updates.
- **Cons**: Creates a second source of truth, complicates precedence rules, and diverges from the existing config loading model.
- **Why rejected**: The codebase already treats file-backed config as canonical, and a second persistence layer would add avoidable configuration ambiguity.

### Alternative 2: Keep settings runtime-only and require explicit export/apply

- **Description**: Hold settings edits in memory or a draft layer until the user explicitly exports them back to file config.
- **Pros**: Safer experimentation and no immediate file writes.
- **Cons**: Adds another state machine, weakens parity with daemon boot behavior, and makes the web app responsible for a draft model the rest of AGH does not use.
- **Why rejected**: The chosen product behavior is direct edit of canonical settings, with restart signaling where needed.

## Consequences

### Positive

- The daemon boots from the same data the web app edits.
- Existing config validation and precedence rules remain authoritative.
- Workspace-scoped settings can reuse the existing overlay model instead of inventing a parallel one.
- Users keep file-backed config as the single source of truth without losing comments and unrelated structure on every save.

### Negative

- MCP servers remain a special case because they already have TOML and sidecar JSON precedence behavior.
- Comment-preserving TOML edits are more complex to implement than a raw encoder-based rewrite.

### Risks

- Incorrect write targeting could clobber global config when the user intended a workspace overlay, or vice versa.
- MCP dual-source writes can become confusing if the API does not expose which source is currently effective.

## Implementation Notes

- Replace the bootstrap-only `writeConfigOverlayFile` path with a reusable settings editor that preserves comments and untouched TOML structure.
- Model write targets explicitly: `global-config`, `workspace-config`, `global-mcp-sidecar`, `workspace-mcp-sidecar`.
- For `mcp-servers`, accept `target=auto|config|sidecar`; `auto` edits the highest-precedence definition in the selected scope and writes new entries to the sidecar by default.
- Validate the merged effective config before committing a write.
- Return semantic write-target metadata and restart requirements in mutation responses instead of absolute filesystem paths.
