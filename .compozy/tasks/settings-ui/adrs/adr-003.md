# ADR-003: Keep settings mutations restart-aware and separate from operational workflows

## Status

Accepted

## Date

2026-04-17

## Context

The settings screens mix static config, live health metrics, manual actions, and links to operational areas. Most of the underlying daemon subsystems are configured at boot and do not expose a safe general-purpose hot-reload surface today. At the same time, the user wants direct persistence to canonical config and runtime application where that is already supported, with the UI clearly signaling when a restart is required.

## Decision

Treat settings mutations as restart-aware operations. Every settings read model and mutation response will declare whether the change applied immediately, requires a daemon restart, or triggered an existing operational action. Settings pages remain configuration-oriented and link out to operational pages for entity management and deep runtime workflows.

For the AGH default detached runtime, implement restart through a dedicated relaunch helper process invoked as an internal subcommand, `agh daemon relaunch`. The helper is spawned first, detached into its own process group, and given an operation id plus restart context. The current daemon then persists restart state, performs graceful shutdown, and releases lock plus `daemon.json`. The helper waits for those release conditions, launches the replacement daemon, and updates a persisted restart-operation record until the replacement daemon marks the operation `ready`.

Do not use in-process `syscall.Exec`, and do not spawn the replacement daemon before the current process has released singleton resources.

## Alternatives Considered

### Alternative 1: Attempt hot-reload for every settings section

- **Description**: Add broad in-memory reconfiguration across daemon subsystems so every settings mutation applies immediately.
- **Pros**: Best possible UX with fewer restarts.
- **Cons**: High implementation risk, broad runtime churn, and significant architectural work across network, automation, memory, observe, and logger subsystems.
- **Why rejected**: The current codebase does not expose safe generic reconfiguration hooks for most of these services, and forcing that into this effort would bloat the scope.

### Alternative 2: Require restart for every settings mutation, even where runtime apply already exists

- **Description**: Persist all changes and always block on manual restart.
- **Pros**: Simple semantics and low daemon complexity.
- **Cons**: Needlessly degrades UX and ignores existing mutable surfaces such as skill enable/disable or already-available operational triggers.
- **Why rejected**: The product direction explicitly allows immediate apply where the daemon already supports it.

### Alternative 3: Restart the daemon in-process with `syscall.Exec`

- **Description**: Replace the running daemon process image directly after persisting settings.
- **Pros**: Small apparent implementation footprint and no separate helper process.
- **Cons**: Does not provide a safe handoff around singleton lock/socket release, complicates rollback and readiness staging, and is brittle inside a live Go runtime with active servers and open file descriptors.
- **Why rejected**: AGH currently owns lock, socket, HTTP server, and persisted daemon info throughout its lifecycle, so a relaunch coordinator is safer than direct process replacement.

## Consequences

### Positive

- The UI can explain exactly what happened after each save.
- The implementation can stay incremental: safe runtime applies first, restart-required for the rest.
- Operational pages keep their own entity workflows instead of bloating settings sections.
- Restart gets an explicit, testable handoff protocol instead of ad hoc process replacement.
- Restart outcome survives the initiating connection drop because the operation state is persisted outside the daemon process.

### Negative

- The product surface must carry restart metadata and user guidance consistently.
- Some settings sections will feel asymmetric because only parts of the screen apply live.
- Restart now requires one more internal process mode and additional restart-status diagnostics.
- The restart path now includes one more persisted artifact family under the AGH home directory.

### Risks

- If restart metadata is wrong, operators may assume settings are active when they are not.
- Pressure to add more live-apply behavior later could erode subsystem safety boundaries.
- Helper spawn or replacement-daemon boot failures must be surfaced clearly after the initiating API request loses its connection.

## Implementation Notes

- Add a shared settings mutation result payload with `applied`, `restart_required`, `restart_scope`, and `warnings`.
- Add a daemon action endpoint for restart plus a status endpoint so the settings UI can complete the explicit restart flow without leaving the app.
- Persist restart-operation state under `~/.agh/restarts/<operation-id>.json`, updating it through `pending`, `stopping`, `waiting_release`, `starting`, `ready`, or `failed`.
- Factor the detached daemon launch logic from `internal/cli/daemon.go` into reusable relaunch code shared by CLI and daemon-owned restart actions.
- Implement the helper as `agh daemon relaunch`, reusing the current executable path, inherited environment, and detached process-group semantics.
- The helper should wait for daemon lock release and `daemon.json` removal before launching the replacement process, then wait for readiness before reporting success.
- The restart action response should include `active_session_count` so the UI can warn before terminating live sessions.
- If AGH is later installed under a supervisor such as `launchd` or `systemd`, restart should be delegated to the service manager instead of using the helper path.
- Reuse existing operational endpoints where the Paper screen already maps to a domain action, such as memory consolidation.
