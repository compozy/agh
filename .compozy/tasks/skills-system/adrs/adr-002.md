# ADR-002: Dual-Scope Registry (Global + Workspace)

## Status

Accepted

## Date

2026-04-06

## Context

The AGH daemon is a single global process, but sessions are workspace-scoped — different sessions can have different workspaces. The skills system discovers skills from both global directories (`~/.agh/skills/`, `~/.agents/skills/`) and workspace directories (`<workspace>/.agents/skills/`, `<workspace>/.agh/skills/`). A single daemon-global registry cannot correctly serve sessions in different workspaces because workspace-level skills differ.

The memory system already solved this problem: `memory.Store` is global, but `memoryStore.ForWorkspace(workspace)` returns a workspace-scoped view. Skills should follow the same pattern.

## Decision

Implement a **dual-scope registry**:

1. **Global layer**: Bundled + user-level skills loaded at boot, stored in the registry, polled by the Watcher. Shared across all sessions.
2. **Workspace layer**: Workspace skills loaded lazily per workspace path via `registry.ForWorkspace(ctx, workspace)`. Cached with mtime+size staleness check. Not polled — checked on-demand at prompt assembly time.
3. `ForWorkspace()` merges global skills with workspace skills, with workspace taking precedence on name collisions.

This follows the `memory.Store.ForWorkspace()` pattern already established in the codebase.

## Alternatives Considered

### Alternative 1: Single Global Registry With All Directories

- **Description**: Scan all directories (including workspace) at boot into one flat registry.
- **Pros**: Simple. One map.
- **Cons**: Daemon boot has no workspace context. Different sessions in different workspaces would get the same skill set. Wrong by construction.
- **Why rejected**: Fundamentally incompatible with AGH's per-session workspace model.

### Alternative 2: Per-Session Registry Instance

- **Description**: Create a new Registry for every session, loading all 4 levels fresh.
- **Pros**: Perfect isolation. No caching complexity.
- **Cons**: Redundant work — global skills reloaded for every session. Memory overhead for many concurrent sessions.
- **Why rejected**: Over-isolates. Global skills are identical across sessions and should be loaded once.

## Consequences

### Positive

- Sessions in different workspaces get correctly scoped skill sets
- Global skills loaded once at boot — no per-session overhead
- Workspace skills discovered without daemon restart
- Cache invalidation is automatic via mtime check
- Follows established `memory.Store.ForWorkspace()` pattern

### Negative

- More complex than a flat registry
- Workspace cache can grow with many workspaces (mitigated by TTL eviction)

### Risks

- Workspace cache entries accumulate if many workspaces used. Mitigation: evict entries older than 10 minutes.
- Concurrent ForWorkspace() calls for the same workspace could race. Mitigation: cache keyed by workspace path under RWMutex.
