# ADR-003: Composed PromptAssembler with PromptProvider Interface

## Status

Accepted

## Date

2026-04-06

## Context

AGH's current `memory.Assembler` implements the `session.PromptAssembler` interface directly, coupling memory document injection with prompt assembly orchestration. The skills system needs to inject a skill catalog into the same prompt. Additionally, the current daemon only constructs a `PromptAssembler` inside the `cfg.Memory.Enabled` branch — disabling memory silently drops prompt assembly entirely, which would prevent skills from working independently.

## Decision

Refactor prompt assembly into a **composed pipeline**:

1. Define a `PromptProvider` interface in `session/` with a single method: `PromptSection(ctx, workspace) (string, error)`.
2. `memory.Assembler` becomes a `PromptProvider` (memory section only).
3. `skills.CatalogProvider` becomes a `PromptProvider` (skill catalog section only).
4. A new `ComposedAssembler` in `daemon/` implements `session.PromptAssembler` by chaining multiple `PromptProvider` instances in deterministic order: agent system prompt → memory context → skill catalog.
5. `ComposedAssembler` is constructed **unconditionally** in `daemon.boot()`, outside any feature-flag branch. When memory is disabled, the memory provider is omitted. When skills are disabled, the skills provider is omitted. Zero providers means only the base agent prompt is returned.
6. `daemon/` wires the providers during boot.

## Alternatives Considered

### Alternative 1: Extend memory.Assembler Directly

- **Description**: Add `SkillRegistry` field to `memory.Assembler`, inject skill catalog inside existing `Assemble()`.
- **Pros**: Minimal code change. No new interfaces.
- **Cons**: Increases coupling. Memory package would import skills. Violates single-responsibility. Skills disabled when memory disabled.
- **Why rejected**: Creates coupling that compounds with each new prompt layer, and inherits the memory-enabled gate bug.

### Alternative 2: New internal/prompt/ Package

- **Description**: Dedicated prompt package owning the full pipeline with formal `PromptLayer` interface.
- **Pros**: Clean separation. Formal pipeline.
- **Cons**: New package for composition logic. Over-engineers for 2-3 providers.
- **Why rejected**: The composition belongs in `daemon/` (the composition root). YAGNI.

## Consequences

### Positive

- Each prompt provider is independently testable
- New prompt layers added by implementing one interface and wiring in `daemon/`
- `memory/` and `skills/` packages remain fully decoupled
- Existing `session.PromptAssembler` interface unchanged
- Skills work independently of memory enabled/disabled state
- Memory works independently of skills enabled/disabled state

### Negative

- Slightly more indirection in prompt assembly
- `daemon/` gains a `ComposedAssembler` type

### Risks

- Provider ordering matters (memory before skills). Mitigation: ordering is explicit in `daemon/` constructor.
