# ADR-004: Stat-Based Polling for Hot-Reload

## Status

Accepted

## Date

2026-04-06

## Context

The skills registry needs to detect when SKILL.md files are created, modified, or deleted without requiring a daemon restart. Three mechanisms were evaluated: fsnotify (filesystem watcher), version-bump polling on CLI mutations only, and stat-based periodic polling. Codex (GPT-5.4) was consulted for a production reliability analysis.

Key constraints:
- Daemon runs on macOS (primary) and Linux
- Skills live under `<root>/<skill-name>/SKILL.md` (nested structure)
- 50-300 skills typical across watched directories
- Skill changes are infrequent (1-10/hour during active dev, near-zero in production)
- Correctness matters more than sub-second responsiveness
- CLI commands are local-only (no daemon RPC) — cannot trigger immediate registry refresh

## Decision

Use **stat-based periodic polling for global directories + lazy mtime check for workspace directories**.

**Global directories** (`~/.agh/skills/`, `~/.agents/skills/`):
- Watcher goroutine scans every 3 seconds (configurable via `skills.poll_interval` in config)
- Collect `mtime` + `size` for each SKILL.md via `os.Stat()`
- If changed: re-parse affected files, atomic swap global skills map, bump `globalVersion`
- Bundled skills (go:embed) are immutable — no polling

**Workspace directories** (`<workspace>/.agents/skills/`, `<workspace>/.agh/skills/`):
- NOT polled by the Watcher
- Checked lazily in `ForWorkspace()` at prompt assembly time via mtime comparison against cached snapshots
- Re-scanned only when stale

**Version-aware caching** lives inside `CatalogProvider`, not in `session/`. Sessions call `PromptAssembler.Assemble()` as before — no version awareness needed at the session layer.

No fsnotify in Increment 1. Can be layered later as a wake-up optimization.

## Alternatives Considered

### Alternative 1: fsnotify File Watcher

- **Description**: Use `github.com/fsnotify/fsnotify` to watch all skill directories.
- **Pros**: Sub-100ms reaction to changes. Event-driven model.
- **Cons**: On macOS, fsnotify uses `kqueue` (not FSEvents), consuming one FD per watched entry. No recursive watching. Known issues with atomic-save editors. New dependency.
- **Why rejected**: Reliability concerns on macOS outweigh speed benefits for infrequent changes.

### Alternative 2: Version-Bump on CLI Mutations Only

- **Description**: No filesystem watching. Bump version only when CLI commands mutate skills.
- **Pros**: Simplest implementation.
- **Cons**: Does not detect external edits. CLI is local-only — no RPC to daemon to trigger refresh.
- **Why rejected**: Missing external edits is poor DX. Local-only CLI makes immediate refresh impossible.

## Consequences

### Positive

- Zero new dependencies
- Correct by construction — answers "what exists on disk right now"
- Works identically on macOS and Linux
- Handles atomic-save editors, git operations, all mutation patterns
- Negligible CPU cost for small file sets
- Configurable interval via `skills.poll_interval`

### Negative

- Up to 3-second latency for global dir changes (configurable)
- Workspace changes detected on next prompt assembly (lazy)
- Polling loop runs even when no changes occur

### Risks

- Large skill directories: cap at 300 candidates per root, skip `.git/` and `node_modules/`
- fsnotify can be layered later as optimization (wake-up hint → immediate rescan, polling as safety net)
