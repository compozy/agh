# ADR-001: Three-Tier Extension Model (Go-Native / WASM / Subprocess)

## Status

Accepted

## Date

2026-04-10

## Context

AGH needs an extension architecture that allows third-party developers to extend the daemon's capabilities. The industry research across 6 agent harnesses (Claude Code, OpenClaw, OpenFang, GoClaw, Hermes, Pi-Mono) shows convergence on multi-tier models that separate fast in-process extensions from rich out-of-process extensions. AGH already has the skeleton of this model: Go interfaces (L1), a WASM executor stub (L2), and the ACP subprocess pattern (L3).

The key tension is between simplicity (just subprocess) and the performance/security benefits of in-process WASM sandboxing.

## Decision

Adopt a three-tier extension model:

- **L1 (Go-native)**: Compiled-in extensions using Go interfaces. For first-party, high-performance functionality. Already exists via `AgentDriver`, `Executor`, `PromptAssembler` interfaces.
- **L2 (WASM via Extism/wazero)**: In-process sandboxed extensions for fast, stateless operations (hooks, validators, transformers). ~1-10μs per call. Deny-by-default security.
- **L3 (Subprocess via JSON-RPC)**: Out-of-process extensions for stateful, long-running operations (agent drivers, memory backends, observe exporters). ~100-500μs per call. Process-isolated.

Each tier maps to different use cases with different trade-offs:

| Tier | Latency | Security | Language Support | Use Cases |
|---|---|---|---|---|
| L1 | Fastest | Trusted | Go only | Core providers, built-in agents |
| L2 | ~1-10μs | Sandboxed (Wasm) | Rust, Go, TS, C, 16+ langs | Hooks, validators, transformers |
| L3 | ~100-500μs | Process-isolated | Any language | Agent drivers, memory backends, exporters |

## Alternatives Considered

### Alternative 1: Subprocess Only

- **Description**: Only support L3 subprocess extensions via JSON-RPC stdio. No WASM.
- **Pros**: Simplest to implement. Any language. No new runtime dependency.
- **Cons**: ~100-500μs per call is too slow for synchronous hook chains (27 events, multiple hooks per event). No in-process sandboxing.
- **Why rejected**: Hook dispatch is on the critical path of every session operation. Subprocess latency for synchronous hooks degrades the entire pipeline. WASM provides the sub-millisecond execution needed for hooks without sacrificing security.

### Alternative 2: WASM Only (No Subprocess)

- **Description**: All extensions run as WASM modules in-process.
- **Pros**: Maximum security (sandbox), fastest cross-language execution, portable .wasm binaries.
- **Cons**: WASM cannot maintain persistent connections (no network), cannot manage subprocess lifecycles, limited filesystem access. Cannot serve as agent drivers or memory backends.
- **Why rejected**: Rich extensions like agent drivers and memory backends require full system access that WASM cannot provide.

### Alternative 3: Go Native Plugins

- **Description**: Use Go's `plugin` package for dynamic loading.
- **Pros**: In-process, shared memory, fastest possible.
- **Cons**: No Windows support, requires CGO, no unloading, no security isolation, exact build-environment matching. Every major Go project has rejected this.
- **Why rejected**: Contradicts AGH's single-binary/zero-CGO constraint. No security isolation. Platform-limited.

## Consequences

### Positive

- Extensions can target the tier that matches their use case
- WASM hooks can run on the critical path without performance degradation
- Subprocess extensions get full system access when needed
- Go-native extensions remain the fastest path for core functionality
- Each tier has appropriate security isolation

### Negative

- Three tiers mean three execution paths to maintain and test
- Developers must choose which tier to target (complexity for extension authors)
- WASM adds ~5-8MB to the binary (Extism + wazero runtime)

### Risks

- WASM ecosystem maturity: Extism has 38 Go importers. Mitigation: wrap behind AGH's `Executor` interface so the adapter can be swapped.
- Testing matrix: 3 tiers × N extension points. Mitigation: share test infrastructure via the `Executor` interface contract.

## Implementation Notes

- L1 already exists (Go interfaces in `internal/session/interfaces.go`, `internal/hooks/executor.go`)
- L2 requires filling the WASM stub in `internal/hooks/executor_wasm_stub.go` with Extism
- L3 requires generalizing the ACP subprocess lifecycle in `internal/acp/` into a shared pattern
