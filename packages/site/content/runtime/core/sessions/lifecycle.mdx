# Session Lifecycle

A session is AGH's core runtime unit. It wraps an ACP-compatible agent subprocess with lifecycle management, event recording, and permission enforcement. Every session follows a strict state machine that governs what operations are valid at each point in its life.

## State machine

Sessions move through four states in a fixed order. No state can be skipped, and there are no backward transitions.

```
                 ┌────────────┐
                 │  starting  │
                 └─────┬──────┘
                       │ ACP connection established,
                       │ agent subprocess ready
                       ▼
                 ┌────────────┐
          ┌──────│   active   │──────┐
          │      └─────┬──────┘      │
          │            │             │
     user prompts      │        events stream
     arrive here       │        from here
                       │
                       │ stop requested or
                       │ process exited
                       ▼
                 ┌────────────┐
                 │  stopping  │
                 └─────┬──────┘
                       │ subprocess terminated,
                       │ state persisted
                       ▼
                 ┌────────────┐
                 │  stopped   │
                 └────────────┘
                   (terminal)
```

| State | Description |
|-------|-------------|
| `starting` | The session has been created and AGH is spawning the agent subprocess, establishing the JSON-RPC connection, and negotiating capabilities. No prompts are accepted yet. |
| `active` | The agent is running and ready. Prompts can be sent, events stream back, and permissions are enforced. This is the normal operating state. |
| `stopping` | A stop has been requested. AGH waits for any in-flight prompts to complete, sends a cooperative cancel to the agent, and then terminates the subprocess. |
| `stopped` | The session is finished. All events have been persisted. The session can be [resumed](/runtime/sessions/resume) if the agent supports it. |

The state machine enforces these transitions strictly. Attempting an invalid transition — for example, going directly from `starting` to `stopped` — returns an error.

## Creating a session

To create a session, you need a running daemon and an agent definition. The `agh session new` command handles the full flow:

```bash
agh session new --agent general
```

```
session created
  id:        a1b2c3d4
  agent:     general
  workspace: ~/projects/myapp
  state:     active
```

You can also specify a name and workspace:

```bash
agh session new --agent general --name "code-review" --workspace ~/projects/myapp
```

Behind the scenes, session creation follows this sequence:

1. **Pre-create hook** fires, allowing hooks to inspect or reject the request
2. The session manager resolves the agent definition from your configuration
3. A per-session directory and SQLite database are created at `~/.agh/sessions/\<id\>/`
4. The ACP driver spawns the agent CLI as a subprocess and establishes a JSON-RPC connection over stdio
5. AGH negotiates capabilities — available modes, models, and whether the agent supports session loading
6. The permission mode is set based on your configuration (default: `approve-reads`)
7. The session transitions from `starting` to `active`
8. **Post-create hook** fires, and observers are notified

### Via the HTTP API

You can also create sessions through the HTTP API:

```bash
curl -X POST http://localhost:2123/api/sessions \
  -H "Content-Type: application/json" \
  -d '{"agent_name": "general", "name": "my-session"}'
```

The response includes the full session payload with its ID, state, and capabilities.

## Sending prompts

Once a session is `active`, send prompts through the CLI or API:

```bash
agh session prompt a1b2c3d4 "Explain this codebase"
```

Each prompt creates a **turn** — a complete request-response cycle. During a turn, AGH:

1. Sends the prompt to the agent via JSON-RPC
2. Streams events back on a Go channel (messages, tool calls, thoughts, usage)
3. Records each event in the per-session SQLite database
4. Forwards events to SSE subscribers in real time
5. Tracks token usage and updates the global catalog

A session can only process one prompt at a time. If a prompt is already in flight, AGH tracks the in-flight state and blocks the stop sequence until the prompt setup completes.

## Stopping a session

Stop a session cooperatively:

```bash
agh session stop a1b2c3d4
```

The stop sequence is designed to give the agent time to finish gracefully:

1. The session transitions from `active` to `stopping`
2. AGH waits for any in-flight prompt setup to complete (up to 5 seconds)
3. A cooperative cancel signal is sent to the agent subprocess
4. If the agent doesn't exit within the stop timeout, AGH sends `SIGTERM`
5. If the process still doesn't exit, AGH escalates to `SIGKILL`
6. The **pre-stop hook** fires, final state is persisted
7. The session transitions to `stopped` and the **session-stopped hook** fires
8. The session is removed from the active session map

### Stop reasons

Every stopped session records **why** it stopped. This classification is persisted in the database and visible in session listings:

| Reason | Description |
|--------|-------------|
| `completed` | The agent finished its work and exited cleanly |
| `user_canceled` | The user explicitly stopped the session |
| `max_iterations` | The session hit its configured iteration limit |
| `loop_detected` | AGH detected a repeating pattern and intervened |
| `timeout` | The session exceeded its time limit |
| `budget_exceeded` | The token budget was exhausted |
| `error` | An unclassified error occurred |
| `agent_crashed` | The agent subprocess exited unexpectedly |
| `hook_stopped` | A lifecycle hook rejected continuation |
| `shutdown` | The daemon is shutting down |

You can inspect the stop reason for any session:

```bash
agh session status a1b2c3d4
```

```
id:          a1b2c3d4
agent:       general
state:       stopped
stop_reason: completed
stop_detail: agent exited cleanly
created:     2025-01-15T10:30:00Z
stopped:     2025-01-15T10:45:00Z
```

## Session types

Sessions are classified by how they were created:

| Type | Description |
|------|-------------|
| `user` | Created by a user via CLI or web UI — the default |
| `dream` | Created by the memory consolidation system for background processing |
| `system` | Created internally by AGH for system operations |

The session type affects how the session appears in listings and which hooks are triggered.

## Daemon shutdown behavior

When the daemon shuts down (via `agh daemon stop` or a system signal), all active sessions are stopped gracefully:

1. Each active session receives a `shutdown` stop cause
2. The cooperative stop sequence runs for each session in parallel
3. Sessions that don't exit within the timeout are force-terminated
4. Final state is persisted with stop reason `shutdown`
5. The global database is closed and the daemon lock is released

Sessions stopped by a daemon shutdown can be [resumed](/runtime/sessions/resume) when the daemon restarts.

## Environment variables

AGH sets several environment variables in the agent subprocess, making session context available to the agent and its tools:

| Variable | Description |
|----------|-------------|
| `AGH_SESSION_ID` | The session's unique identifier |
| `AGH_SESSION_CHANNEL` | The network channel, if the session is connected to one |
| `AGH_PEER_ID` | A network peer identifier derived from agent name and session ID |
| `AGH_BIN` | Path to the AGH binary |

## Configuration

Session behavior is controlled through your AGH configuration file (`~/.agh/config.toml`):

```toml
[limits]
max_sessions = 10         # Maximum number of active sessions
max_concurrent_agents = 5  # Maximum concurrent agent subprocesses

[session.limits]
timeout = "30m"           # Session idle/execution timeout

[permissions]
mode = "approve-reads"    # Default permission mode for new sessions
```

See the [Permissions](/runtime/sessions/permissions) page for details on permission modes.
