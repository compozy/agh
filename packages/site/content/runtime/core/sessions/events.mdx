# Event Streaming

Every interaction in an AGH session produces events. When an agent thinks, calls a tool, generates a response, or requests a permission, AGH records a typed event with a timestamp and payload. These events are persisted in a per-session SQLite database and can be streamed in real time via Server-Sent Events (SSE).

Events are the foundation of AGH's observability. They power the web UI's live session view, the CLI's `--follow` mode, session resume, and the global audit trail.

## Event types

AGH defines the following event types, each representing a distinct moment in a session turn:

| Type | Description |
|------|-------------|
| `user_message` | A user-submitted prompt or message chunk |
| `agent_message` | An assistant response chunk (streamed incrementally) |
| `thought` | Internal reasoning from the agent (when the agent exposes chain-of-thought) |
| `tool_call` | A tool invocation initiated by the agent Рђћ includes the tool name, input, and call ID |
| `tool_result` | The result returned after a tool executes Рђћ includes stdout, stderr, content, or error |
| `plan` | A plan update from the agent describing its intended approach |
| `permission` | A permission decision event Рђћ records what was requested and the decision applied |
| `usage` | Token usage metadata for the turn Рђћ input, output, cache, and cost information |
| `system` | A system-level ACP update (connection state changes, capability negotiation) |
| `done` | Signals the end of a prompt turn Рђћ includes the stop reason for the turn |
| `error` | An error that occurred during prompt processing |
| `session_stopped` | Emitted when the session transitions to the `stopped` state |

## Event payload structure

Every event shares a common envelope with type-specific fields:

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | One of the event type constants above |
| `session_id` | string | The session this event belongs to |
| `turn_id` | string | The prompt turn that produced this event |
| `timestamp` | datetime | UTC timestamp of when the event occurred |
| `text` | string | Message content, thought text, or error description |
| `title` | string | Tool name or operation title (for `tool_call` events) |
| `tool_call_id` | string | Unique identifier linking a `tool_call` to its `tool_result` |
| `stop_reason` | string | Why the turn ended (for `done` events) Рђћ e.g., `end_turn` |
| `action` | string | Action classification (for permission events) |
| `resource` | string | Resource path (for permission events) |
| `decision` | string | Permission decision applied |
| `error` | string | Error message (for `error` events) |
| `request_id` | string | Permission request identifier |
| `usage` | object | Token usage breakdown (for `usage` events) |

Not all fields are populated on every event. A `user_message` event typically only has `type`, `session_id`, `turn_id`, `timestamp`, and `text`. A `tool_call` event includes `title` (the tool name) and `tool_call_id`.

### Token usage payload

The `usage` event includes detailed token accounting:

| Field | Type | Description |
|-------|------|-------------|
| `input_tokens` | integer | Tokens in the prompt sent to the model |
| `output_tokens` | integer | Tokens in the model's response |
| `total_tokens` | integer | Combined input and output |
| `thought_tokens` | integer | Tokens used for chain-of-thought reasoning |
| `cache_read_tokens` | integer | Tokens served from prompt cache |
| `cache_write_tokens` | integer | Tokens written to prompt cache |
| `context_used` | integer | Context window tokens consumed |
| `context_size` | integer | Total context window size |
| `cost_amount` | float | Estimated cost for this turn |
| `cost_currency` | string | Currency code (e.g., `USD`) |

## Subscribing to events

### CLI Рђћ live follow

Stream events from a running session in your terminal:

```bash
agh session events a1b2c3d4 --follow
```

Events appear as they happen:

```
[user_message]   Explain the session manager
[thought]        Let me look at the session package...
[tool_call]      Read (internal/session/manager.go)
[tool_result]    (file content returned)
[agent_message]  The session manager handles lifecycle...
[usage]          tokens: 1420 input, 380 output
[done]           end_turn
```

Press `Ctrl+C` to stop following.

### CLI Рђћ historical query

Query past events without following:

```bash
# Last 20 events
agh session events a1b2c3d4 --last 20

# Events from the last 5 minutes
agh session events a1b2c3d4 --since 5m

# Filter by event type
agh session events a1b2c3d4 --type tool_call
```

### HTTP Рђћ Server-Sent Events

The web UI and custom clients subscribe to real-time events via SSE:

```bash
curl -N http://localhost:2123/api/sessions/a1b2c3d4/events/stream
```

Each SSE message contains a JSON-encoded event payload:

```
data: {"type":"agent_message","session_id":"a1b2c3d4","turn_id":"t1","timestamp":"2025-01-15T10:30:00Z","text":"The session manager handles..."}

data: {"type":"usage","session_id":"a1b2c3d4","turn_id":"t1","timestamp":"2025-01-15T10:30:01Z","usage":{"input_tokens":1420,"output_tokens":380}}

data: {"type":"done","session_id":"a1b2c3d4","turn_id":"t1","timestamp":"2025-01-15T10:30:01Z","stop_reason":"end_turn"}
```

The SSE stream stays open as long as the session is active. When the session stops, a `session_stopped` event is emitted and the stream closes.

## Event persistence

### Per-session database

Each session stores its events in an isolated SQLite database at:

```
~/.agh/sessions/<session-id>/events.db
```

The events table records every event with full fidelity:

| Column | Type | Description |
|--------|------|-------------|
| `id` | text | Unique event identifier |
| `sequence` | integer | Monotonically increasing sequence number |
| `turn_id` | text | The prompt turn this event belongs to |
| `type` | text | Event type constant |
| `agent_name` | text | Name of the agent that produced this event |
| `content` | text | JSON-encoded canonical event payload |
| `timestamp` | text | ISO 8601 UTC timestamp |

Events are indexed by type, timestamp, sequence, and turn for efficient querying.

### Token usage table

Token usage is tracked per turn in a separate table:

| Column | Type | Description |
|--------|------|-------------|
| `turn_id` | text | Primary key Рђћ one row per prompt turn |
| `input_tokens` | integer | Prompt tokens |
| `output_tokens` | integer | Response tokens |
| `total_tokens` | integer | Combined total |
| `thought_tokens` | integer | Chain-of-thought tokens |
| `cache_read_tokens` | integer | Cached prompt tokens |
| `cache_write_tokens` | integer | Tokens written to cache |
| `context_used` | integer | Context window usage |
| `context_size` | integer | Context window capacity |
| `cost_amount` | real | Estimated cost |
| `cost_currency` | text | Currency code |

### Write buffering

Events are written asynchronously through a dedicated writer goroutine with a buffered queue (default capacity: 256 events). This keeps event persistence off the critical path Рђћ the agent continues working while events are flushed to disk. A drain timeout of 5 seconds ensures all events are persisted before a session is finalized.

### Global database

In addition to per-session storage, AGH maintains a global database (`~/.agh/agh.db`) that stores session metadata, aggregated token statistics, and a permission audit log. The global database provides a fast, single-file view of all sessions without needing to open each session's individual database.

## Event recording flow

The path from agent to disk follows this sequence:

```
Agent subprocess
      Рћѓ
      Рћѓ JSON-RPC over stdio
      Рќ╝
ACP Driver (event stream)
      Рћѓ
      РћюРћђРћђРќХ Session event handler
      Рћѓ         Рћѓ
      Рћѓ         РћюРћђРћђРќХ Per-session SQLite (full event)
      Рћѓ         Рћѓ
      Рћѓ         РћћРћђРћђРќХ Global observers (summary)
      Рћѓ
      РћћРћђРћђРќХ SSE subscribers (real-time)
```

1. The agent subprocess emits events over its JSON-RPC connection
2. The ACP driver parses events and delivers them on a Go channel
3. The session's event handler receives each event and:
   - Marshals it to canonical schema via the `transcript` package
   - Sends it to the per-session SQLite writer for persistence
   - Dispatches it to global observers for aggregation and metrics
4. SSE subscribers receive events in real time for live UI updates

## Hook run recording

The per-session database also records lifecycle hook executions:

| Column | Type | Description |
|--------|------|-------------|
| `id` | text | Unique hook run identifier |
| `hook_name` | text | Name of the hook that ran |
| `event` | text | The lifecycle event that triggered the hook |
| `source` | text | Where the hook is defined |
| `mode` | text | Hook execution mode |
| `duration_ns` | integer | Execution time in nanoseconds |
| `outcome` | text | Whether the hook succeeded or failed |
| `patch_applied` | text | Any patch the hook applied to the session |
| `error` | text | Error message if the hook failed |
| `required` | integer | Whether the hook was mandatory |

## Configuration

Event recording behavior is controlled through your configuration:

```toml
[observability]
enabled = true            # Enable global event recording
retention_days = 30       # How long to keep events
max_global_bytes = 0      # Global DB size limit (0 = unlimited)

[observability.transcripts]
enabled = true            # Enable full transcript capture
segment_bytes = 0         # Segment size (0 = no segmentation)
max_bytes_per_session = 0 # Per-session size limit (0 = unlimited)
```
