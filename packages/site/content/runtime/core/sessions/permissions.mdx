# Permissions

AGH enforces a permission model that controls what an agent can do during a session. Every tool call the agent makes — reading a file, writing code, running a terminal command — is checked against the session's permission mode. This gives you control over agent autonomy without having to review every action manually.

## Permission modes

AGH supports three permission modes, ordered from most restrictive to most permissive:

| Mode | Behavior |
|------|----------|
| `deny-all` | Every operation requires explicit user approval. The agent cannot read files, write files, or run commands without you granting permission first. |
| `approve-reads` | Read operations (viewing files, listing directories) are auto-approved. Write operations (editing files, creating terminals, running commands) require your approval. This is the **default mode**. |
| `approve-all` | All operations are auto-approved. The agent runs with full access to the workspace. Use this when you trust the agent completely and want maximum speed. |

### Choosing a mode

The default `approve-reads` mode strikes a balance: agents can freely explore and understand your codebase, but they need your permission before making changes. This is appropriate for most interactive development sessions.

Use `deny-all` when:
- Running an unfamiliar agent for the first time
- Working in a sensitive codebase or production environment
- You want to review every action the agent takes

Use `approve-all` when:
- The agent is well-tested and the workspace is non-critical
- You're running automated or batch operations
- Speed matters more than per-operation review

### Setting the default mode

Configure the default permission mode in your AGH configuration:

```toml
[permissions]
mode = "approve-reads"
```

This applies to all new sessions. You can override it per session at creation time.

## Permission operations

When AGH enforces permissions, it classifies each agent action into an operation type:

| Operation | Description | Auto-approved in |
|-----------|-------------|-----------------|
| `fs/read_text_file` | Reading a file's contents | `approve-reads`, `approve-all` |
| `fs/write_text_file` | Writing or modifying a file | `approve-all` only |
| `terminal/create` | Creating a terminal or running a command | `approve-all` only |
| `session/request_permission` | Agent requesting a blanket permission grant | `approve-all` only |

When an operation is not auto-approved, AGH pauses the agent and sends a permission request to the user.

## Approval flow

When an agent requests an operation that requires approval, the following happens:

```
Agent requests tool call
        │
        ▼
┌───────────────┐
│  Check mode   │
│  and operation│
└───────┬───────┘
        │
  ┌─────┴─────┐
  │            │
auto-       requires
approved    approval
  │            │
  ▼            ▼
execute   ┌──────────────┐
tool      │  Emit         │
          │  permission   │
          │  event        │
          └──────┬───────┘
                 │
                 ▼
          ┌──────────────┐
          │  Wait for     │
          │  user decision│
          └──────┬───────┘
                 │
          ┌──────┴──────┐
          │             │
       allowed       rejected
          │             │
          ▼             ▼
       execute       agent
       tool          notified
```

### Permission decisions

When prompted, you can respond with one of four decisions:

| Decision | Effect |
|----------|--------|
| `allow-once` | Approve this specific request only. The agent will ask again next time. |
| `allow-always` | Approve this request and all future requests of the same type for this session. |
| `reject-once` | Deny this specific request. The agent can try a different approach. |
| `reject-always` | Deny this request and automatically reject all future requests of the same type. |

### Approving via CLI

When a permission request is pending, the CLI displays the request details and waits for your input:

```
Permission requested:
  tool:     Write
  resource: internal/session/manager.go
  action:   write_text_file

  [a]llow once  [A]llow always  [r]eject once  [R]eject always
```

### Approving via HTTP API

The web UI and custom clients approve permissions through the API:

```bash
curl -X POST http://localhost:2123/api/sessions/a1b2c3d4/approve \
  -H "Content-Type: application/json" \
  -d '{"request_id": "req-xyz", "turn_id": "t1", "decision": "allow-once"}'
```

The `request_id` identifies the specific permission request and is included in the permission event emitted to SSE subscribers.

## Workspace boundaries

Permissions are enforced within the session's workspace boundary. The agent can only access files within the workspace directory that was set when the session was created. Attempts to read or write files outside the workspace are rejected regardless of the permission mode.

This means `approve-all` is "approve all operations **within the workspace**" — not unrestricted access to the entire filesystem.

## Permission events

Every permission decision is recorded as an event in the session's event stream:

```json
{
  "type": "permission",
  "session_id": "a1b2c3d4",
  "turn_id": "t1",
  "request_id": "req-xyz",
  "timestamp": "2025-01-15T10:30:00Z",
  "action": "write_text_file",
  "resource": "internal/session/manager.go",
  "decision": "allow-once"
}
```

These events are:
- Persisted in the per-session SQLite database
- Streamed to SSE subscribers in real time
- Aggregated in the global permission audit log

This gives you a complete audit trail of what the agent was allowed and denied during a session.

## Mode mapping to agent capabilities

Different agents expose different permission modes under different names. AGH maps its modes to agent-specific identifiers automatically:

| AGH mode | Agent mode candidates |
|----------|-----------------------|
| `approve-all` | `full-access`, `full_access`, `bypassPermissions`, `auto` |
| `approve-reads` | `read-only`, `read_only`, `plan`, `ask` |
| `deny-all` | `read-only`, `read_only`, `plan`, `ask` |

AGH selects the first matching mode from the agent's list of supported modes. If no match is found, the session falls back to the agent's default mode.

## Configuration reference

```toml
[permissions]
# Default permission mode for new sessions.
# Options: "deny-all", "approve-reads", "approve-all"
# Default: "approve-reads"
mode = "approve-reads"
```
