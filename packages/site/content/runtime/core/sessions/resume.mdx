# Resume and Replay

AGH persists every event from every session in a dedicated SQLite database. This means sessions can be **resumed** after being stopped — whether you stopped them intentionally, the daemon shut down, or the agent crashed. The resume system reconstructs the conversation transcript from stored events and restores the agent to its prior context.

## When to resume

Resume is useful in several scenarios:

- **Picking up where you left off** — You stopped a session yesterday and want to continue the same conversation today.
- **Recovering from a crash** — The agent process exited unexpectedly and you want to restore context without starting over.
- **Daemon restart** — The daemon was stopped (or the machine rebooted) and you want to continue active sessions.
- **Iterating on a task** — You reviewed the agent's work, want to give follow-up instructions, and don't want to lose the prior context.

## Resuming a session

Resume a stopped session with the CLI:

```bash
agh session resume a1b2c3d4
```

```
session resumed
  id:        a1b2c3d4
  agent:     general
  workspace: ~/projects/myapp
  state:     active
```

The session retains the same ID, workspace, and agent configuration. It returns to the `active` state and is ready to accept prompts.

### Via the HTTP API

```bash
curl -X POST http://localhost:2123/api/sessions/a1b2c3d4/resume
```

## What happens during resume

The resume flow reconstructs the session's context through these steps:

1. **Metadata validation** — AGH reads the session's persisted metadata and verifies that the workspace still exists and the agent is still available in your configuration.

2. **Pre-resume hook** — Lifecycle hooks can inspect or modify the resume request before it proceeds.

3. **ACP session loading** — AGH attempts to resume using the ACP `LoadSession` method, which tells the agent to restore its internal state from a previous session ID. This is the preferred path because the agent can restore its own context natively.

4. **Transcript reconstruction** — If the agent doesn't support `LoadSession` (or if the previous session resources are unavailable), AGH falls back to reconstructing the conversation. The `transcript` package assembles canonical messages from the persisted event stream, giving the agent the conversation history as context.

5. **Permission mode restore** — The session's permission mode is reapplied.

6. **Post-resume hook** — Observers are notified that the session has been successfully resumed.

## Transcript reconstruction

When an agent cannot natively restore its session, AGH reconstructs the conversation from persisted events. The `transcript` package reads all events from the session's SQLite database and assembles them into canonical messages.

Each message has a role:

| Role | Source |
|------|--------|
| `user` | User-submitted prompts |
| `assistant` | Agent-generated responses and reasoning |
| `tool_call` | Tool invocations initiated by the agent |
| `tool_result` | Results returned from tool execution |

The assembly process:

1. Events are sorted by sequence number, timestamp, and ID for deterministic ordering
2. User messages are emitted immediately as they appear
3. Agent messages and thoughts are accumulated and flushed at turn boundaries
4. Tool calls are tracked with their lifecycle state — the call and its result are merged into a coherent pair
5. Tool results include structured fields: stdout, stderr, file paths, content, and error information

The reconstructed transcript gives the agent enough context to continue the conversation naturally, even though it's running in a new subprocess.

## Fallback behavior

If the ACP `LoadSession` call fails because the agent reports that session resources are missing (for example, the agent's own persistence was cleared), AGH automatically retries the resume as a fresh session with the reconstructed transcript injected as context. This means resume works even in degraded conditions — you may lose some agent-internal state, but the conversation history is preserved.

## Identifying resumable sessions

List stopped sessions that can be resumed:

```bash
agh session list --all
```

```
ID        AGENT    STATE    STOP REASON     WORKSPACE          AGE
a1b2c3d4  general  stopped  user_canceled   ~/projects/myapp   2h
b2c3d4e5  general  stopped  completed       ~/projects/other   1d
c3d4e5f6  coder    stopped  shutdown        ~/projects/lib     3d
```

Any stopped session with persisted events can be resumed. The stop reason doesn't affect whether resume is available — sessions stopped by crashes, timeouts, or user cancellation are all resumable.

### Checking agent capabilities

Not all agents support native session loading. You can check what an agent supports by examining the session's ACP capabilities:

```bash
agh session status a1b2c3d4
```

The `supports_load_session` field in the capabilities indicates whether the agent can natively restore session state. If it's `false`, AGH will always use transcript reconstruction.

## Resume vs. new session

| Aspect | Resume | New session |
|--------|--------|-------------|
| Session ID | Same as original | New ID generated |
| Conversation history | Restored from events | Empty |
| Workspace | Same as original | Can be different |
| Agent | Same as original | Can be different |
| Permission mode | Restored | From config defaults |
| Per-session database | Continued (events appended) | New database created |

Choose resume when you want to continue a conversation with its full context. Choose a new session when you want a clean slate or need to change the agent or workspace.

## Common scenarios

### Resuming after a daemon restart

When the daemon stops, all sessions are persisted with stop reason `shutdown`. After restarting the daemon, resume any session:

```bash
agh daemon start
agh session resume a1b2c3d4
```

### Resuming after an agent crash

If an agent subprocess exits unexpectedly, the session is recorded with stop reason `agent_crashed`. Resume works the same way — AGH will spawn a new subprocess and restore context:

```bash
agh session resume a1b2c3d4
```

> **Note:** After a crash, the agent's internal state may be incomplete. AGH restores the conversation transcript, but any in-memory state the agent held (caches, computed results) may be lost. The agent starts fresh with the conversation history as its context.

### Reviewing events before resuming

Before resuming, you may want to review what happened in the session:

```bash
# Show the last 50 events
agh session events a1b2c3d4 --last 50

# Show only agent messages
agh session events a1b2c3d4 --type agent_message
```

This helps you understand where the session left off and craft a good follow-up prompt.
