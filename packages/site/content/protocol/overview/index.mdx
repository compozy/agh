AGH Network is the open coordination layer behind AGH. It exists because once teams rely on more
than one specialized agent, they need a clean way to discover peers, delegate work, exchange
updates, and keep control over who is responsible for what.

The point is not to force every runtime into the same implementation. The point is to give runtimes
a shared boundary for agent-to-agent coordination.

<OperatorNote>
  If you already operate a runtime, the adoption path is to map your current agent model to the
  protocol boundary. AGH Runtime is the reference implementation, not a mandatory prerequisite.
</OperatorNote>

## Protocol vs Runtime

- **AGH Runtime** is the local control plane. It runs sessions, keeps history durable, manages
  memory and skills, exposes operator surfaces, and makes the system practical to run.
- **AGH Network** is the protocol boundary. It defines how agents discover peers, delegate work,
  exchange updates, and preserve interoperability across runtimes.

You can implement AGH Network without running AGH Runtime, and you can run AGH Runtime without
enabling the network surface.

## What AGH Network unlocks

- **Discovery instead of hard-wired glue** — agents can announce capabilities and find the right specialist.
- **Delegation with visible state** — handoffs, updates, and receipts move through a shared contract instead of disappearing into prompts.
- **Cross-runtime coordination** — teams can adopt the boundary without replacing their entire operator surface.
- **Incremental adoption** — implement the core contract first, then layer in deeper transport or trust behavior later.

## What AGH Network is not

- **Not a workflow engine.** It defines coordination semantics, not a full orchestration platform.
- **Not a runtime replacement.** AGH Runtime is the reference implementation, not a mandatory control plane.
- **Not a lock-in strategy.** The value comes from a shared protocol boundary, not a captive ecosystem.
