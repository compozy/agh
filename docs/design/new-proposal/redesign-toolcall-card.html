Redesign · Toolcall Card · v2

Move INPUT / OUTPUT into the header so the collapsed card is one row, not three.

Today the card reserves 3 rows even when closed — the disclosure labels dominate the transcript. Three approaches below: chips, icons, segmented tabs. All resolve the closed-state vertical bloat; each chooses a different tradeoff between clarity, density, and interaction model.

Strengths

  • Labels remain readable for new users
  • Byte meter signals "is there content?" without clicking
  • Open chips repaint orange — clear visual feedback

Costs

  • ~140px reserved in header for the chip group
  • On narrow widths file-path truncates earlier
  • Two open panels stack vertically — long outputs scroll within

Best for

  • Default transcript view
  • First-run users who don't know Input vs Output
  • Cards where Input is short (CLI args)

1 row closed · ~50px header reserved

Strengths

  • ~50px header footprint — half of variant A
  • Pip pattern reads as "notification" — proven UX language
  • Scales to N actions if you ever add Diff, Logs, etc.

Costs

  • First-time users won't know what {} means

Best for

  • Power users in long transcripts
  • Sidebars or compact views
  • Pair with command palette / keyboard shortcuts (i / o)

CSegmented tabs — exclusive switcher

1 row closed · one panel at a time

A two-segment switcher acts like radio tabs: only one panel is visible at a time. When neither is selected the card is fully closed (one row). A subtle close button appears once something is open. Best when outputs are long (diffs, stdout) and Input would compete for screen real estate.

  • Debug views where focus matters more than context
